# 如何开始审查代码

> 原文：<https://dev.to/shubheksha/how-to-start-reviewing-code-3jk6>

我的工作目标之一是在编写代码的同时审查代码。我一直在尝试审查越来越多的代码，虽然这很有趣，但也带来了一些挑战。我想谈谈我是如何做到这一点的，以及在我努力做得更好的过程中，是什么帮助了我。

1.  这不是火箭科学。说真的，不是的。很长一段时间以来，我一直认为审查代码的人拥有审查他们正在查看的代码片段的所有上下文，所以，当然，他们会发现这样做很容易，不费力。这很少是真的，即使是对那些系统有很多活动部件的小公司。没有人知道系统每个部分的一切。记住这一点很好。审查代码是一种技能，可以像写代码一样学习。你做得越多，你就做得越好。审查代码也有助于你写出更好的代码，因为你知道你需要注意什么。不要让对不了解你所评审的代码的恐惧阻碍你。

2.  会犯错误。这样或那样的东西总是会从缝隙中溜走，因为人类编写代码，人类审查代码，我们都天生容易犯各种各样的错误。没关系。你做不到完美，并不意味着你根本就不应该做。如果你对某件事不确定，请别人再看一眼也是完全没问题的。有第二双眼睛永远不会有坏处。

    与这一点不太相关的是有一种无可指责的文化，如果你在生产中搞砸了什么，人们不会找你麻烦。主要的焦点是诊断和修复问题，而不是归咎于谁(谁编写/审查了错误的代码)。犯错是有空间的，重要的是从中吸取教训。我很幸运能在有这种文化的地方工作，所以感觉不那么令人生畏。😊

3.  代码评审对于双方的误解和缺乏同理心来说已经非常成熟了。代码评审的核心是协作。作为评审者，提醒自己你是在评审某人的代码，而不是作为一个人来评判他们，这一点很重要；同样重要的是要记住，无论你的评审者告诉你什么，都不是人身攻击。我觉得这很大程度上是由你的团队和整个公司的工程文化决定的。在发表评论前注意这一点可以大大减少任何摩擦的可能性。

4.  我想提到的另一点是在评审代码时，初级-高级的动态。作为一个职业生涯早期的人，为一个比你有更多经验和/或背景的人审查代码是非常令人畏惧的。然而，这不应该阻止你审查他们的代码。有几种方法可以做到这一点:

    *   对现有的拉请求进行跟踪审查:这是一个很好的方法，可以学习发现人们在审查代码时注意什么，以及他们如何交流，作者如何接收(你如何交流对变更的反馈同样重要，如果不比你交流的更重要的话)
    *   当审查更复杂的东西时，与另一个工程师或作者合作:这让你可以实时提问，并获得关于正在发生的事情的更多背景信息
5.  询问更多的上下文是可以的。当你对一个团队和/或领域是全新的，而你周围的每个人似乎都比你知道得多，这一开始可能会很可怕。如果在查看了相关信息并做了调查后，有些事情还是没有意义，可以要求更多的澄清。

    让我给你举个例子:
    上周，我被要求为一项我从未接触过或使用过的服务审核一份小型公关，我的第一反应是让团队中有更多背景知识的人来做这件事。然而，我把它作为一个学习的机会，并试图弄清楚在提议的改变中发生了什么。看完之后，尽管总的拉请求是有意义的，但有些事情却不是，所以我鼓起勇气声明我是这个服务的新手，并要求作者做更多的澄清。你猜怎么着——他们非常乐意提供这些！一旦我做到了这一点，我就能够批准他们的改变，同时带走一些东西:

    *   在这次积极的经历之后，我会更有信心要求澄清
    *   我对该服务的功能有了更好的理解，并且可以更容易地查看相关的变更
    *   我们确定了如何改进我们正在查看的函数的文档，以便其他人可以更容易地理解它

    这是一个双赢的局面！🎉

一旦我开始把代码评审看作是另一种我可以学习并随着时间的推移变得更好的技能，我对做这些事情就更有信心了。现在没那么可怕了，我期待着做更多的事情。💪🏼
