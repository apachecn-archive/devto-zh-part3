随着您作为软件开发人员的进步，您应该开始建立一种天生的意识，即什么时候应该以某种方式完成某件事情，也许是为了将来以某种方式证明它，或者使它更有弹性。这是一种奇怪的感觉，但你只是感觉到了一些东西。

我发现随着我的进步和学习，这种情况发生得越来越频繁。但我从未停下来考虑*为什么*我会想出我所做的解决方案，或者*为什么*我会做出某个决定而不是另一个决定的原因。

我是第一原则思维的忠实粉丝，这个概念真的让我明白，要想在职业生涯中取得真正的进步，我需要从为什么开始。理解我的决策和解决方案的原因，并能够以简洁的方式进行交流。

让我们举一个例子，我们的 SMS 系统存在一些弹性问题。一开始，想出这个问题的解决方案似乎是黑白分明的。但是，我决定更深入一点，为什么我们要解决这些弹性问题？难道仅仅是为了让开发人员“相信”我们的系统足够健壮来处理故障吗？部分，但不是全部。我们希望提高系统弹性的原因是，使用我们产品的人可以更快地获得关于他们消息状态的反馈。如果消息在管道的某个地方失败了，我们需要确保我们通过管道向客户大声喊回来，这样他们就知道我们失败了。这是一个比信息逃逸到以太网更好的用户体验。此外，我们希望允许用户重试发送消息，所以这也是我们解决方案中的一个考虑因素。

也许你可能是一个以产品为中心的天才，但认为这并不适用，但我敢打赌，你已经想出了一些解决方案，而没有花时间来权衡其他选择，并能够有意义地证明其背后的推理。

在选择技术时会出现一个常见的例子。开发者就像寒鸦，他们喜欢新的闪亮的东西。有时，新的闪亮的东西是惊人的，例如，无服务器技术就像一个金戒指。但是其他时候，你可能会遇到一些像是象征性的箔片的东西。看起来很棒，有它的位置，但你已经有一些漂亮闪亮的箔片，这样更好吗？也许不是。这并不是说技术没有价值，但它们必须与当前的方法和其他选择进行认真的权衡。不要说“X 将彻底改变我们的工作方式，它具有 Y 和 Z 的特性”，走出去，调查它与其他类似技术相比有什么优势，学习曲线是什么样的等等。如果您有信心，甚至可以创建一个概念证明。所有这些都会帮助你找到原因。为什么要用这种技术？为什么你首先需要改变技术？诸如此类…

作为这个过程的一个例子，我考虑将一个用 Node 编写的基于无服务器的 API 移植到 Golang。我想做 Golang，因为它看起来很酷，Docker 是用它编写的，而且因为我读过它，它非常快——就是这样。然而，当我调查时，我发现 Python、Go 和 Node 有相似的冷启动时间，在移植上没有什么优势。就我们正在进行的处理而言，Node 同样能够胜任这一特定任务，而且我们已经拥有了用 Node 编写的所有库的优势。我决定不这么做，而是投入时间优化现有的节点 API。这比移植到我几乎不知道的新语言上更好地利用了时间。

当你看到新的东西，或者有远大的梦想时，从为什么开始。你会因此成长得更多。
