# 了解你的用户——行为数据简介

> 原文:[https://dev.to/tdwright/understanding-your-users-安-行为数据介绍-2f1g](https://dev.to/tdwright/understanding-your-users--an-introduction-to-behavioural-data-2f1g)

作为一名开发者，了解人们如何使用你的应用对你的使命至关重要。如果你不知道你的用户是如何与你的代码交互的，你怎么能想出如何改进它呢？作为我关于行为数据收集和分析的新博客系列的第一部分，这篇文章涵盖了一些基础知识。因此，如果你是行为数据的新手，想了解更多，请继续阅读…

这个系列源于我过去几个月写的一些笔记。我在研究行为数据时积累的一些实用技巧。当我写这篇文章时，我意识到它已经膨胀到几千字，所以我决定把它做成一个系列。这第一部分是介绍，是关于行为数据收集和分析的内容和原因的基础(非技术性)入门。

“理解用户行为”显然是一个非常宽泛的术语。有许多方法可以解决这个问题。即使你从来没有想过，当你为你的项目做决定的时候，你也可能试图进入你的用户的头脑。我们都有一个常识(“启发式”)理解人们在给定的情况下会如何表现。嘿，无论如何，我们在某种程度上可能正在[吃](https://en.wikipedia.org/wiki/Eating_your_own_dog_food),对吗？

即使我们开始更严格地去理解用户，也有很多选择。首先，我们可以查看用户提供给我们的数据。也许你的数据库中已经有了一些信息，你可以利用这些信息进行一些推断。

例如，如果用户成功发布了一篇博客文章，我们可以推断:

1.  他们当时有动力使用你的博客应用
2.  他们(至少在一定程度上)成功地完成了他们计划要做的事情

但这留下了许多未解之谜！他们的意思是只有这么少的格式吗？或者他们尝试了，但是遇到了一些问题？他们完成一个简单的动作，比如添加图片，花费的时间是否比我们预期的要长？

回答这样的问题(或者至少尝试这样做)可能是一个很好的解决问题和提供功能的方法，也是一个很强的信号，可以让你优先考虑自己的时间。那么，我们如何着手回答这些问题呢？

可用性研究人员有各种锦囊妙计来帮助回答这样的问题。这些包括可能突然想到的访谈和调查，但也包括观察方法、任务分析和 A/B 测试。虽然从适当的可用性研究中获得的洞见是非常宝贵的，但是进行研究和综合研究结果的过程会花费大量的时间和金钱。

我今天要谈论的技术非常快速和便宜，但是将极大地增加你理解用户如何与你的代码交互的能力。这反过来会让你对下一个 bug、下一次迭代，甚至下一个产品做出明智的决定。

行为数据跟踪有点像日志记录:通过向我们的应用程序添加几行额外的代码，我们可以开始收集大量关于用户行为的数据，而无需他们做任何事情。每当用户做了我们感兴趣的事情，我们就会触发一个事件，在云中进行聚合和分析。

## [](#what-to-track)跟踪什么？

如果你已经读到这里，我将假设你对获得一些关于你自己的用户行为的数据很感兴趣。你可能已经跃跃欲试，想要直接进入科技领域。在本系列接下来的几篇文章中，我将讨论连接所有东西的机制，但是现在，让我们后退一步，想想我们可能想要跟踪什么。

你会看到很多指南都提倡正确的跟踪策略。可能会有你应该填写的电子表格。这些“策略”中的一些似乎主要是为了消除你对行为数据的热情。我的建议？别理他们。

对于需要跟踪的内容，你可能已经心中有数了。从那个开始。当你开始使用这些数据来尝试和回答真正的问题时，让你的追踪进化以满足你的需求。

[![](../Images/9be1eb5ca07e7f726b3c1b66c2e9c387.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--52PIDIBt--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://i1.wp.com/blog.tdwright.co.uk/wp-content/uploads/2019/01/EventsToTrack-1.png%3Fresize%3D620%252C349)

有些事件会比其他事件更有用。虽然跟踪比你需要的更多的事件没有太大的害处，但是它会浪费时间，扰乱你的分析尝试。一旦你列出了一长串可能要追踪的事情，就用红笔从头到尾看一遍。

当我评估要跟踪的事件时，我发现考虑它是否属于以下四个类别之一会有所帮助:

1.  **“脉动”事件:**这些是用户活动的良好指标。例如，“*我们通过查看独特用户的数量来统计每周活跃用户……*”这本身就是一个有用的指标，但也是其他指标的分母。考虑一下你是否希望这适用于所有用户，或者某个子集(比如注册用户)，并适当地设计你的事件。
2.  **KPI 事件:**你的目标是一定数量的行动吗？也许你一周需要购买 100 件商品？或者你可能需要一天完成 50 个注册。我发现最好是直接跟踪这些事情，否则你最终会依赖糟糕的代理，这将不可避免地造成混乱。
3.  **“砂砾”事件:**当出现问题时，并不总是错误或 bug。有时你的用户界面会无意中鼓励“不正确”的行为。例如，跟踪向用户显示错误消息的情况，可以帮助发现趋势，并让您消除摩擦的根源。
4.  **触发事件:**我提到过 Mixpanel 允许用户触发目标消息来响应事件。考虑一下您可能想要触发的消息是否是对您已经在跟踪的事件的响应。

有了好的事件选择，你将开始对你的用户(总的来说)如何与你的应用程序交互有一个好的感觉。考虑使用您的“脉冲”事件作为基线，以帮助您区分有用的信号和噪音。像“每成功注册的电子邮件验证错误”或“每活跃用户的购买量”这样的指标可能是有用的风向标(或者也许是矿中的金丝雀)。

记住另一件有帮助的事情是，我们只处理*事件*数据(即“发生的事情”)。你想要回答的一些问题需要你混合*状态*数据(即“事物的本来面目”)。这在这些托管平台中可能很难做到，因为最好的方法通常涉及到来自数据库的数据。在 Mixpanel 中，你可以添加属性到你的用户记录中，但是仍然非常有限。如果你发现自己处于这样的情况下，试着想想你是否可以激发一个事件作为某个状态的代理，或者状态之间的转换。
