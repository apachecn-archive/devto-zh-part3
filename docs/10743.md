# 技术哲学

> 原文:[https://dev.to/pbeekums/philosophy-of-technology-2n4e](https://dev.to/pbeekums/philosophy-of-technology-2n4e)

在制造还是购买的决策中，要考虑很多因素。现有系统的技术能力如何？功能是我自己产品的核心产品吗？建造起来会有多复杂？购买要花多少钱？转换成本呢？

我发现自己越来越多地考虑的一件事是，开发我正在评估的技术的人的哲学。这不是我通常明确列出的东西，也很少在与其他开发人员的讨论中提到。

所有的软件系统都有技术上的限制，通常表现为缺陷。但是局限性是可以克服的，错误也是可以修复的。经过几年的开发，软件看起来很少是一样的。

最大的问题是什么可能会改变，什么永远不会改变。反应 vs Vue/Angular 是一个很好的例子。它们都是流行的前端框架。他们都关心性能。他们都很好地测试了他们的产品。它们有各种各样的优点和缺点，我们可以吹毛求疵，但这些都可能随着版本的不同而变化。

几乎永远不会改变的是他们软件开发的基本方法。React 的理念是一切都由代码驱动。加载应用程序的 HTML 模板只有最低限度。从那里代码决定显示什么。HTML 只是代码的输出。React 对于像我这样认为 HTML 模板使软件开发变得繁琐的人来说是完美的。将代码从可视化定义中分离出来很大程度上是肤浅的，会导致比它解决的问题更多的问题。

Vue 和 Angular 采取了不同的方法。他们认为 HTML 模板在前端开发中有重要的地位。因此，代码有时由 HTML 中定义的“指令”触发。对于那些认为 HTML 与代码分离可以在开发软件时提供更好的清晰度的人来说，这是完美的。

虽然我有明显的偏见，但这两种方法都完全有效。许多高质量的软件都是用这两种方法构建的。但是如果你的软件开发风格倾向于其中之一，那么你应该选择适合你风格的框架。不要选择 React，认为它有一天会为 HTML 模板提供更大的作用。不要认为 Vue/Angular 有一天会让你从代码中定义一切。

因此，如果您自己的偏好倾向于这些哲学之一，不要让这里或那里的一个错误诱使您选择违背您基本偏好的技术。bug 可以修复。框架的核心是什么？

再深入一点，有时技术方法提供了一个指示器，指示开发人员如何做出其他决策。这方面的一个例子是 [Sugar.js](https://sugarjs.com/dates/) 。当我在[研究日期/时间解析器](https://blog.professorbeekums.com/need-date-time-picker/)时，我发现糖在技术上更胜一筹。它是最准确的，而且是用 Javascript 编写的，所以我可以在前端运行它，而不用调用 API。

但是，它们最初覆盖了本机对象，如 Date。他们创造了一种不使用这种方法的方法，因为许多开发人员讨厌这种方法，这阻止了他们使用这个库。现在，虽然技术本身已经发生了变化，[对这种策略有效性的信念](https://sugarjs.com/extending/)成为开发者对风险与回报的容忍度的指标。

就我个人而言，我已经被覆盖本机对象的库烧够了，我相信这种回报永远不值得我去冒险。这是我根据自己的经历得出的偏见。其他开发商有其他经验，因此对此事有不同的看法。没关系。对我来说，这意味着我知道我和糖的开发者有着根本的不同意见。无论我多么喜欢他们的库的功能，我都必须记住，将来的某个版本会做出决定，让我不想升级。这意味着这个库是我的软件中可能替换的对象，我必须考虑到这一点。

还有其他类似的例子。有一种流行的后端技术，我不会给它命名。我的朋友喜欢它。开发人员社区似乎普遍喜欢它。当我在以前的工作中使用它的时候，我甚至喜欢上了它(别人在维护它)。在许多方面，它使生活变得容易多了。

不幸的是，开发人员做出了许多令人困惑的选择。我认为这些选择是糟糕的 UX，尽管我们不经常谈论后端技术的 UX。然而，当我在论坛上看到其他开发人员对这些决策提出问题时，他们的回答却不尽如人意。这项技术的开发人员没有解决这些问题，而是求助于久经考验的指责用户的策略。

这些开发人员非常聪明，他们构建了一个优秀的软件工程。但是软件开发已经够紧张的了。我不需要一个库/框架来增加不必要的压力。我只是不相信这个软件的未来版本会最小化我的焦虑，所以我选择不使用它，尽管我的同龄人都很惊愕。

说到信任，最简单和最不主观的选择是当技术开发人员有相似的态度，但是没有交付高质量的工作。我曾经在一个利基行业寻找一个框架。其中一个有很多问题，最大的问题是其中一个页面花了 30 分钟才加载完毕。我发现这是因为框架不能在虚拟机上很好地运行，因为它们经常碰到文件系统。框架开发人员没有解释为什么他们需要攻击文件系统，而是指责虚拟机在同步文件时速度太慢。

还有很多其他的例子，这些开发人员将人们在他们的框架中遇到的问题归咎于其他技术。但是最终，在某个时候，框架开发人员需要对他们的技术应该解决的用例负责。如果他们不这样做，那么人们就不能信任他们的技术，因为他们不能信任那些开发者。

当选择一个库、框架或服务时，我们不仅仅是在软件的当前状态上下赌注。我们正在对这项技术几年后的发展进行长期押注。虽然该技术的个别开发人员可能会改变，但他们的替代者也相信围绕该技术的哲学。注意到这一理念是在你自己的软件中选择使用技术的一个重要因素。
