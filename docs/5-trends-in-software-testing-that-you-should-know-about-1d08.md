# 你应该知道的软件测试的 5 个趋势

> 原文：<https://dev.to/blizzerand/5-trends-in-software-testing-that-you-should-know-about-1d08>

现代 web 和移动组件是用户和企业更新最快、最频繁的软件包之一。因此，在部署之前，需要对其进行持续和稳定的测试。

虽然自动化的好处是允许团队减少手动测试工作的数量并增加测试以确保有效的操作，但它也显著减少了需要手动执行的重复过程的数量。

在过去的十年里，在移动计算、web 开发和人工智能领域有很多活动。API 架构的采用越来越多。服务器端逻辑是在 API 的帮助下设计的，这使得它们可以被公众访问。前端代码已经转向面向移动和网络的 JavaScript 平台。

鉴于一些细节往往因不同的堆栈而异，本文试图列出 2018 年软件测试的一些更普遍的趋势。

## API 和微服务测试自动化

微服务是开发软件的一种必要方式，这种软件是专门为测试预定义的条件而设计的。这些类型的服务可以被模拟，当在它们之间建立连接时，整个测试任务可以被分成不同的部分。

在这种情况下，每个服务都被设计为能够执行一组特定的任务。这允许用户和开发者自由地修改应用程序中需要修改的较小区域，而不是修改整个系统。您可以使用[单元测试](https://softwareengineering.stackexchange.com/questions/252748/is-it-actually-worth-unit-testing-an-api-client)或微服务集成测试的变体，因为每个服务或 API 端点都可以被视为一个单独的单元。

## 采用开源测试套件和工具

开源软件工具对组织特别有帮助，而且[趋势表明它们会一直存在](https://opensource.com/article/17/11/10-open-source-technology-trends-2018)。除了成本方面，使用开源工具还有许多不同的优势，因为它通常是免费供公众使用的。

开源软件工具易于定制，更加灵活，并对公众开放。同时，由于设计者在它的设计中有发言权，用户可以以他们需要的方式来设计它，同时还允许与强大的测试自动化特性进行多种集成。

一些常用的 AI/web 和移动开发框架都是开源组件。很有可能，您正在使用的大多数测试框架都是开源的。

一个经常讨论的问题是安全漏洞。根据定义，开源工具是对公众开放的，但这并没有使它们比专有的替代工具更安全。然而，开源工具有多双眼睛审查代码版本也是事实，因此，定位和修复潜在 bug 的机会大大增加了。

越来越多的公司接受开源软件服务来执行他们的管理、自动化、开发和敏捷策略的测试，以及缺陷管理的责任。随着开放源码工具越来越活跃，这有可能导致开放支持社区的发展和增长。

## 左移和右移测试

当讨论 DevOps 流程的质量软件时，有两个基本参数很重要

1.  左移测试——当测试是持续集成或 CI 的一部分时。
2.  右移测试——根据用户反馈扩展测试。

### 左移测试

左移测试是在测试满足产品验收标准所需的特定产品功能以及等待验证以满足业务验收标准的假设时制定的场景。

简单来说，[左移测试](https://resources.whitesourcesoftware.com/blog-whitesource/shift-left-the-software-development-seismic-shift)指的是在构建产品特性之前定义测试。左移测试对于快速高效地开发高质量的软件是必不可少的。

大多数组织发现为应用程序部署新的补丁是一项挑战。需要通过功能测试或回归测试进行严格的测试。这可以确保修补程序更新不会破坏现有系统的稳定。

如果测试在生产周期的早期开始，团队通常会更加关注质量。这节省了大量的工作时间，并减少了软件开发过程中所需的迭代次数。

### 右移测试

尽管在应用程序生命周期的早期阶段进行测试是必不可少的，但这远远不够。来自用户的持续反馈也同样重要，这就是右移测试的用武之地。

有些方面并不总是在开发人员或工程师的权限范围内，可能会被遗忘。一个准备好部署的应用程序不能承受这样的失败，右移测试有助于在公开部署应用程序之前消除这些问题。

右移测试允许持续监控应用程序的可用性和性能。测试人员应该知道用户是如何理解应用程序的，即使是在信息或需求收集阶段。这里的重点是在测试过程中尽可能多地覆盖范围。

## 质量工程超过质量保证

随着软件开发领域的不断变化，市场上关于新技术的讨论永无止境。质量保证或 QA 是一个在软件测试中遵循瀑布方法的过程。这是一个循序渐进的过程，旨在使 QA 彻底但耗时。

这也是为什么 QA 为了跟上软件测试中不断变化的动态而打一场败仗的部分原因。的确，当涉及到完成一个应用程序的开发生命周期时，QA 过程有时也会成为一个瓶颈。

由于其自然的逐步过程，QA 过程不能从一个级别前进到另一个级别，直到前面的阶段完成和验证。

这是大量 QA 流程和数据堆积的主要原因之一。另一方面，当谈到质量工程时，测试和自动化可以在过程的早期引入，而不是在最终的验收阶段。

如果没有软件开发的最新发展趋势，2018 年 QA 流程的改进思路就无法实现。作为一个简单的规则，UX 越好，它被用作系统/软件性能工程或 SPE 而不是过程测试的机会就越高。

## 针对移动设备的集成测试

移动应用是个人生活中不可或缺的一部分。测试应用不仅仅是测试应用逻辑，还有其他同样重要的细节。例如，您需要验证 UI 和 UX 操作是否如预期的那样呈现，以及它是否有响应。纯粹的设备类型、应用程序和平台软件的更新数量在增加其复杂性方面发挥了作用。从企业角度测试移动应用的选项包括:

1.  单元测试
2.  用户界面测试
3.  模糊测试
4.  性能试验
5.  端到端测试
6.  生产前测试
7.  金丝雀(后期制作)测试

你可以在媒体上阅读更多关于 [7 个测试场景的内容。](https://medium.com/@FizzyInTheHall/testing-mobile-apps-a-primer-889f62a85e40)

随着成本考虑和市场准备度在未来扮演更重要的角色，移动应用程序测试自动化应该会有自己的一席之地。持续不断的创新对持续增长至关重要，如果不结合使用上述技术进行持续测试，这一点很难实现。这些平台将帮助开发人员和测试人员在测试过程之前检查问题和错误。

## 结论

移动设备的渗透率越来越高，这让应用程序开发人员和 QA 测试人员非常头疼。这种增长伴随着软件测试的更新技术和移动行业的整体发展。

测试人员需要不断提高他们的技能，企业家需要被他们的产品和解决方案的最佳选择所引导，组织需要优先考虑健壮的测试过程。这是为了确保向客户和最终用户提供近乎完美的应用和解决方案。

不断发展的测试趋势将继续向移动测试人员提出问题和挑战，并将继续使开发沿着成本效益和有效性的道路前进。当与该领域中足够数量的测试和技术专业知识相结合时，未来的软件技术应该能够最好地提供深入和公正的质量保证。
