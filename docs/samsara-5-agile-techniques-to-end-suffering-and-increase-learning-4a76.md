# 轮回:结束痛苦和增加学习的 5 种敏捷技术

> 原文：<https://dev.to/cubiclebuddha/samsara-5-agile-techniques-to-end-suffering-and-increase-learning-4a76>

循环、冲刺、迭代...这些都是软件过程社区已经使用多年的时髦词汇。这些词中的大部分是为了帮助我们“快速失败和经常失败”，这样我们就可以更多地了解我们的用户对我们编写的东西的感受。但是这些话真的对我们有帮助吗？可悲的是，对于许多软件开发人员来说，这些话都是空话。我们将试图通过分析人类历史上最古老的词汇之一“轮回”来重振这些流行词汇(如“冲刺”或敏捷周期)轮回也指周期(但在这种情况下是生命、死亡和重生的重复)。这个来自梵语的单词有着优美的声音和丰富的哲学、精神甚至数学历史！一些[学者](https://books.google.com/books?id=1ErDrlnNlB4C&pg=PA20&lpg=PA20&dq=india+zero+%22samsara%22+math&source=bl&ots=DMMkER4al5&sig=ACfU3U0XjVsUmuGeKdWz8V6pOESthElXBw&hl=en&sa=X&ved=2ahUKEwispfXn2KzhAhVyS98KHeumAvkQ6AEwFHoECAcQAQ#v=onepage&q=india%20zero%20%22samsara%22%20math&f=false)认为数字零在视觉上代表轮回。所以，让我们比较一下这个有趣的单词的意思；在这个过程中，希望我们能够重新认识为什么我们曾经关心敏捷“周期”我听说有些人不公平地称轮回为“无尽的痛苦之轮”虽然它有一定的活力，但对这个词的理解太简单了。然而，对于那些期待你的团队或组织从其经验中学习的承诺未来的读者来说，这种重复痛苦的形象可能击中了要害。不要气馁！CubicleBuddha.com[的目的是帮助你学会结束这种循环。我创建这个博客是为了帮助你(和我自己)找到一种学习、成长和快乐工作的方法。事实上，禅宗哲学家艾伦·沃茨解释了人类是如何独一无二地学会如何摆脱过去或未来的循环的:](http://cubiclebuddha.com)

> “所以你一圈又一圈，一圈又一圈，永远追逐这样一种幻觉，即在你自己之外，在你此时此地之外，有某种东西可以获得，使事情变得更好。[...[英语背诵文选只有在中间的位置，人类的位置，也就是你可以说的平等的位置，足够平静的位置，才能开始考虑摆脱这场激烈的竞争。只有从那里你看，你才能成佛。所以佛陀的位置可能根本不在轮子上，或者就在轮子的正中央...]所以你看，佛陀是从轮回的幻觉中觉醒的人。也就是说，从生活中可以得到一些东西，明天会带给你，总有一天会好起来。

所以我们看到目标是向内寻找一条脱离轮子的路。对于软件工程师来说,“轮子”指的是“冲刺”(也就是两周的时间增量),一些被误导的经理划分了这些时间，这样他们就可以跟踪我们已经完成了什么。或者至少有时感觉这个过程是强加给我们的。但是如果你从艾伦·沃茨的哲学引语中吸取了什么，请认识到你不是一只轮子上的仓鼠。你没有被束缚在桌子上。你可以为自己回收敏捷过程。就像我们重新分析轮回这个词一样，你可以花时间重新分析最初创造了“敏捷”这个术语的宣言。当你阅读敏捷宣言时，请仔细注意这句话:

> 过程和工具之上的个人和交互

看吧！提出这些敏捷术语的伟大思想家希望你在寻找最适合你和你独特团队的方法时，忽略术语和过程。虽然你的经理可能希望你从一个“冲刺”到下一个，但挑战自己，像禅宗佛教徒一样向内看，找到为什么你的工作不像以前那样令人信服的答案:

*   你的 Scrum sprints 是不是感觉很痛苦，你的团队每两周就会犯同样的错误？
*   你觉得你没有充分了解用户是如何使用你的软件的吗
*   我们真的向用户发布代码并从他们那里得到反馈吗？

如果你不做这些事情，那么你可能根本就不在一个小循环中。你可能在一个很大的轮子上，看起来好像轮子根本不动。如果你没有学习，如果你没有反思你工作的时间，或者如果你只是盲目地跟随这个过程...这就是恶性循环真正导致痛苦的时候。那么我们如何离开方向盘呢？

1.  佛教的方式:你可以(坦白地)承认你不想参与轮子，因此试着通过冥想来呼吸、放松和放慢时间。
2.  *或者...因为你在美国公司工作...*可以让轮子越来越小。把轮子做小一点，每天都学。缩短反馈回路。这就是“快速失败和经常失败”的含义，这样你就可以从错误中学习，并快速整合你的知识。

那么我们来谈谈那些让轮子变小的实用方法。如果我们成功了，也许我们可以将轮回从它的口语含义“无尽的痛苦之轮”转化为它的原始含义“游荡”。以下是一些实用的方法，让你在开发软件时不会漫无目的:

1.  为你的利益相关者做每周演示。记录他们的反馈，并花时间找出你应该实现的部分。你越频繁地演示你的软件，你就越能从你的用户和利益相关者那里学到东西。
2.  花点时间添加指标收集工具(比如 New Relic 或 Google Analytics)，这样你就可以自动了解用户的行为。这使得反馈循环比产品演示要短得多。
3.  举行团队回顾会,这样你可以从他们的过程错误中吸取教训。谁知道呢？也许你的团队可以变得如此善于失败和一起学习，他们也可以提高他们的人际交往技能。应该改进的不仅仅是软件。回顾会议可能是敏捷/Scrum 社区提出的我最喜欢的仪式，因为它真正代表了持续的学习和自省。如果你的团队定期开会，并花时间分享他们在最后一次环行中所学到的东西，那么你就有一个真正的机会来防止你的团队遭受痛苦。我不知道有多少次，一个队友在回顾中不情愿地提出一个问题，却发现团队的其他人也悄悄地有同样的感觉。
4.  创造更小的周期:如果你像我一样，正在与“分析麻痹”作斗争，那么你可能需要更小的周期。有很多方法可以消除这种特殊的痛苦:
    *   允许你的代码至少每 15 分钟中断一次。未来的一篇文章将讨论学习破解自己的代码的乐趣，但是简单地想象一下:
        1.  加载您的项目
        2.  立即看到失败
        3.  快速修复它
        4.  深呼吸
    *   考虑使用红绿重构之类的测试驱动开发(TDD)技术来带给你更多的平静。有许多关于测试驱动开发的技术文章，但是简单地说，TDD 为您提供了一个框架，来确定您的用户现在期望什么，并且在您编写真正的代码之前就验证这一点。这是一种简化你的思维过程，专注于当下必要的事情的方式。对我们大多数人来说，这是一种用心编码的方式——但是如果它对你不起作用，那就停止它。目标是找到帮助你保持冷静、帮助用户和尊重你的同事的技巧。
5.  想象一个微小的成功:在你开车去上班的路上，想象你计划在一天的第一个小时里完成的一件事。选择一个小成就就更好了。选择容易完成的项目(比如喝一杯水或者打开你团队的“待办事项”列表)并完成它会让你习惯于稳步完成事情。
6.  我们将在以后的文章中分享更多的技术。

如果你做了这些事情，你会发现你正在为你自己和你的用户学习和增长价值。如果你做得很好，那么你就是在帮助你的用户和你的社区。你可能会发现，仅仅通过展示自我保健的技巧和展示小的、反复的学习机会的价值，你就在帮助你的同伴。正如我们将在接下来的许多激动人心的博客文章中讨论的那样，只要打破痛苦的循环，你就有能力对你的团队产生影响。如果你能创造一种宽容和学习的文化，那么你就能创造一个每天工作愉快的环境。我听起来不像是在受苦！:)所以回收轮回，把每一天(也许每一刻)都当作学习、成长、快乐的机会。
