# 作为一名开发人员，我在过去的一年中所学到的一切

> 原文：<https://dev.to/jsjoeio/everything-ive-learned-in-this-past-year-as-a-developer-5283>

我在 2016 年 12 月加入了 freeCodeCamp。从那时到 2018 年 3 月，我实习了一段时间，从事数字营销工作，在当地的一个聚会上教书。

2018 年 3 月——开始全职开发(前端工程师)。我觉得从那时到现在我学到了很多。

我写这篇文章的目的是。为了自己的利益反思我所学到的东西
2。就你在第一份工作中可能学到的东西提供见解

## 我学到了什么

以下是我在过去 10 个月里学到的所有重要的东西。

### Git 工作流程

在我开始这份工作之前，我了解 git 和版本控制的基础知识(使用 GitHub，创建 PRs 等。).然而，我从来没有使用过“可靠的”工作流程。

举个例子，我们的是这样的:

1.  从`master`创建`feature-branch`
2.  一旦完成，通过合并到`dev`来测试你的分支
3.  如果看起来不错，提交 PR 将`feature-branch`合并到`master`

现在我也在个人项目中使用这个过程。我甚至提交 PRs(即使我是项目中唯一的一个人),因为这是很好的实践。

此外，编写好的提交消息被大大低估了。阅读[这篇帖子](https://chris.beams.io/posts/git-commit/)。

### 反应过来&回复

在这份工作之前，我已经阅读了 React 文档，但我没有建立多少。有趣的是，在第一周或第二周，我记得一个同事说，“去看看 Redux 文档，构建一些东西，然后给我看。”这提醒了我，这是一个你可以在工作中学到很多东西的行业(这是我喜欢的！).

在我们目前的项目中，我的大部分时间都花在了编写 React 和 Redux 上。

### Next.js

在这里开始之前，我没有听说过 [Next.js](https://github.com/zeit/next.js/) 。我参与的一个最大的项目(也是我现在的主要关注点)使用了 Next.js。

Next.js 功能强大，做得很好，但是文档有时会有所欠缺。然而，不要让这阻止你使用/学习它。它是由我非常尊敬的 ZEIT 团队建造的。

### 单元测试(Jest，enzyme，react-testing-library)

当我第一次开始这项工作时，没有一个前端项目有单元测试。一开始我并没有多想，因为 a)我没有任何经验 b)我不明白单元测试的重要性。

我们的经理和架构师要求我们加入测试。当我们开始我们最近的项目时。我很感激他们这样做，因为我学到了很多关于测试的重要性。

我们刚开始的时候用的是 enzyme，但幸运的是，react-testing-library 出来了，我们转而使用它:)(在我看来，它提供了更好的开发者体验)。

### CICD(持续整合，持续部署)

在这份工作之前，我不知道这是什么。

我的同事教我的。本质上，它似乎可以归结为脚本。我喜欢把它理解为“在本地启动项目时使用的命令，并把它们放到脚本中。”

幸运的是，我有机会使用 Atlassian 的竹子为前端项目建立 CICD 管道。在同事的指导下，我最终写了一些文档。我们现在使用 Bitbucket 管道，但是以前的经历教会了我足够的东西来适应新的平台。

对于部署部分，我们有几个脚本来更新 Kubernetes 中的部署。

### 【码头工+库伯工】

与 CICD 类似，在开始这项工作之前，我没有怎么使用 Docker。我的第一份工作是网页开发，我的老板希望我为 WordPress 开发研究 Docker。他希望这能解决我们在当地建立项目的问题。我想不出来，所以我放弃了。

快进到这项工作——他们在 Docker 容器中运行所有东西，所以我被迫学习。

我学得不多，但我知道如何创建一个 Docker 映像，在 Docker 容器中本地运行它，并将其推送到注册表中。大概就是这样。幸运的是，我只需要知道这些。

我们还使用 Kubernetes 来编排一切。我不怎么摆弄它，但是我已经能够根据我们如何使用它来学习一点点。

### 衔斗、JIRA、汇流、竹子、管道

我是 GitHub 和 GitLab 的忠实粉丝，所以当我听说他们使用 Bitbucket 时，我有点失望。我不像 GitHub 那样喜欢 UI，但是嘿，它完成了工作:)

我们用 JIRA 管理问题/任务，用 Confluence 管理文档，用管道管理 CICD 流程(我们用了一点 Bamboo)。

正如我前面提到的，我们过去使用竹子，现在使用 Bitbucket 管道。Bitbucket 内置了这个功能，这一点非常棒:)

我只是在等待访问 GitHub actions，这样我就可以为我的个人项目设置它。

### 敏捷| Scrum

在这份工作之前，我对敏捷软件开发并不熟悉。我只是根据需要做一些事情，仅此而已。

说到 Scrum，我已经学会真正享受它。我们的做法是这样的:

1.  在我们开始 sprint 之前，我们作为一个团队(后端、前端、经理、架构师、产品)开会，我们选择要添加到 sprint 中的故事。我们根据速度选择一定的数量。

2.  我们开始冲刺(通常持续两周)，然后召集前端和后端，讨论第一个故事。我们通过垂直切片将它们分开——因此我们为前端工作和后端工作创建了一个子任务。然后，我们与后端团队协同工作。

3.  在 sprint 结束时，我们进行演示和检查，以确保我们的工作符合产品的验收标准。如果是这样，我们就赢得了冲刺阶段的分数。

4.  之后，我们会做一个回顾会，分享哪些是有效的，哪些是可以改进的，哪些是值得称赞的。

## 结论

总而言之，我认为在过去的一年里我学到了很多。我觉得我对整个“软件开发生命周期”有了更多的理解。这是一个有趣的行业。

我感谢我的工作和作为开发人员每天学习新东西的机会:)
