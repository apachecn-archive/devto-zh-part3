十年前，当我开始我的开发生涯时，我们有少量的技术/框架可供选择来构建 web 应用程序。PHP、CodeIgniter、CakePHP(仅举几个例子)是开发人员向客户交付定制 web 解决方案的首选工具。

具体谈到为 iOS/Android 开发应用程序，我记得使用 Java 和 Objective C。快进到最近几年，你会看到开发人员为 Android 使用 [Kotlin](https://dev.to/solutelabs/musings-what-the-tech-world-saw-in-kotlin-45me) ，为 iOS 使用 Swift 等。

最近，我们看到新的、更有效的框架出现了。我的意思是它正在爆炸。对于网络应用，你有 VueJS，AngularJS，NodeJS 等等。作为产品所有者，当被要求选择正确的技术组合(无论是网络还是移动)时，你可能会发疯。

> 所以，当一个新的框架/技术出现时，混乱只会增加。

作为产品开发人员，我们必须适应环境，否则就只能选择忍辱负重。我不得不解雇一些不愿意使用 Kotlin 开发 android 应用的开发人员。对我来说这是一个艰难的决定，但是游戏规则不是我制定的。怀着沉重的心情，我不得不与他们分道扬镳。你不能用过时的技术制造东西，对吗？

如果你是一名摄影师或独立建筑师，你会知道像“Squarespace”这样的平台是天赐之物。当你没有那么多钱支付给一个专业的设计师，仍然可以得到一个优雅的网站时，没有什么比这更好的了。

### [](#how-flutter-was-born)旋舞是如何诞生的

*埃里克·塞德尔有丰富的‘渲染’背景。而且，当 Flutter 在三年前作为一个项目开始时，这种背景将帮助 Eric 和他的团队将 Flutter 塑造成今天的样子。他们希望为移动设备做得更好，而“渲染”正是 Eric 的优势所在(他喜欢这样称呼:)。当他曾经在“Chrome”团队工作时，他们试图解决一些棘手的问题——让网络的特定部分变得平滑。这导致他们做了一些实验，这就是有趣的地方。团队开始从不知名的网站移除旧的兼容性支持。他们去掉了很多东西，重新进行了基准测试。所发生的一切，让他们震惊。点击此处观看完整采访— [与谷歌](https://www.youtube.com/watch?v=h7HOt3Jb1Ts&t=252s)共进咖啡。*

著名软件开发人员兰德尔·施瓦茨解释了 flutter 是如何产生的—

Chrome 团队面临的挑战是如何让 Chrome 更快？所以他们从撕掉几乎所有东西开始。最主要的事情是使 JavaScript 和 WebKit 变慢的是 CSS 规则在过去的几年里变得越来越疯狂，以至于多次向下和向上的渲染都必须符合标准的 JavaScript 标准 CSS。

所以 Chrome 团队说，“如果我们可以重新开始，不实现所有的 CSS，会怎么样？不实现所有的布局规则，比如 Flex 和 RenderBox？诸如此类的事情。”所以他们把这些都去掉了，他们得到了更好的体验。好吧。但这仍然不够。所以他们一直在关注 Dart 团队正在做的事情，并说，“如果我们使用像 Dart 这样的语言作为我们的实现语言会怎么样？”

所以他们去找 Dart 的人说，“这是我们需要的。我们想把这个放进去。”他们从 Chrome 那里拿来了渲染引擎，叫做 Skia，S-K-I-A，这是一种控制屏幕上每个像素的方法。这就是 Chrome 用来在屏幕上绘图的，还有一个文本引擎。

他们拿走了 Chrome 的这两个部分，他们说，“让我们从那里开始，让我们从那里开始构建 Dart 来构建 Flutter。”因此，Flutter 本质上是使用 Dart 语言来控制屏幕上的每个像素，使用 Chrome 的原生渲染引擎和 Chrome 的原生文本引擎。事情就是这样发生的。

作为一名开发人员，你知道这种方法的挫折——编译、部署、等待系统启动、等待它执行代码、等待调试器工具——这一切都在等待，等待——这时你意识到你错过了一些重要的东西，不得不再次返回。

### [](#ui-widgets-make-flutter-the-big-deal)UI 小部件让 Flutter 成为“大”交易

对于很多移动开发者来说，这让他们对 Flutter 一见钟情。它有一个内置的 Cupertino 和 Material design 小部件，这使得创建美观的 UI 设计成为可能。从使用丰富的动画到复杂的运动图形，Flutter 将帮助您创建令人惊叹的交互。而且，这是交付卓越用户体验的关键。静态用户界面无助于实现这一点。

我的“设计师”朋友遇到过不得不牺牲他们的视觉/创造力的情况。很多时候，设计师创建了一个布局，做了一些漂亮的动画/过渡来交付一个极好的 UX，却让开发者失望了(从没有足够的时间到专业水平的无尽理由)。有了颤动，不再有妥协。

团队中的每个人，从设计师到开发人员和测试人员，都可以在几秒钟内一起解决问题，而不会引起麻烦。

你可能会问诸如滚动、导航、字体等原生特性怎么样。Flutter 是在牢记平台差异的基础上构建的。也可以根据自己的需要开发定制的小部件。

### [](#hot-reload%E2%80%8A%E2%80%8Abring-your-app-to-life)热重装—让您的应用充满活力

Flutter 有自己的渲染引擎。这使得应用程序开发更快，更容易，更注重性能。从快速构建 ui 到修复 bug， *HotReload 将重新加载时间降低到毫秒级*。

这意味着，当您对代码进行一些更改时，您可以立即在仿真器和硬件上预览这些更改。(*记住，当你对你的 Squarespace 网站进行更改并点击预览按钮*)。就这么快。这实质上意味着您的产品/应用更快上市。如果你正在考虑推出一个 MVP，Flutter 可能是你的正确选择。

### [](#qa-happens-faster)QA 发生得更快

很多专家认为测试人员不喜欢找 bug(与舆论相反)。他们的工作不是发现缺陷，而是提高产品质量。让任何一个测试人员测试相同的特性两次到三次；你会经常看到他/她皱眉。

此外，当我们作为开发人员必须处理客户提出的问题时(如果你有原生应用，由于功能差异，你必须为 iOS 和 Android 分别工作)，事情会变得令人沮丧。并非客户/用户提出的所有问题都是具体的。每一次，你都无法回到产品团队去了解具体情况，你必须自己做出那些小决定。对于两个不同的平台，这意味着要做出相应的改变。对于用户来说，这不是一个问题，因为很少有人会同时使用 iOS 和 Android 设备。有时这些变化并不明显。

然而，对于 UI 来说，情况并非如此。它将是可见的。你怎么知道哪种行为是正确的？有了单一的代码库，这些挫折就不会出现。测试会明显减少。一旦测试人员测试了基于 Flutter 的 iOS 应用程序，他/她就会知道该功能是如何工作的，然后转移到 Android 应用程序会使事情变得更容易。

## [](#can-you-integrate-an-existing-app-with-flutter-and-should-you-convince-your-bosscofounder-to-use-it)能否将现有的一个 app 与 Flutter 整合？你应该说服你的老板/联合创始人使用它吗？

是的。(当然要看你拥有什么类型的 app)。以下案例研究将是一个有用的参考点。

当 flutter 出现的时候，你可能已经为你的业务开发了一个本地应用。现在怎么办？你不能抛弃那个应用程序，然后重新做一遍。嗯，Flutter 可以与现有的应用程序集成。

Groupon Merchant App 就是一个很好的例子。Tomek Polański 设法让他在 Groupon 的团队相信 Flutter 值得一试。

Groupon 有两个应用——面向顾客的和面向商家的。面向客户的应用程序非常庞大，下载量超过 5000 万次。另一方面，商家 app 明显更小(500K+下载量)。因此，对团队来说，在那里集成 Flutter 风险相对较小，也更容易。也因为处理商家 app 的团队规模较小。

在他去年在伦敦 droidcon 的演讲中，他讨论了他用来说服 Groupon 多个团队的方法。了解什么对他们有效，什么对他们无效。

### [](#but-waitflutter-has-its-own-problems-since-its-still-developing)但是，等等……Flutter 有它自己的问题，因为它还在发展中

Flutter 是开发 UI 密集型应用程序的完美选择。当你身边没有设计师或者只是想快速测试市场时，Flutter 可能是显而易见的选择。

然而，由于 Flutter 仍在开发中(尽管它已经过了测试阶段)，库似乎仍然是开发人员的一个问题。例如，当你不得不处理像传感器这样的东西时，你需要为它开发你自己的库。同样，需要与第三方应用程序进行大量通信的应用程序、复杂的数据库以及与 Android/iOS 系统的深度集成，native 可能是正确的选择。你将不得不看看你针对的是哪些人口统计数据和其他因素，并据此做出选择，是先去 iOS 还是先去 Android。另一个例子是需要大量图像处理的应用程序。现在 Flutter 似乎缺少一个合适的库来做这件事。在这种情况下，React Native 可能是合适的选择。

开发人员遇到的另一个限制是集成地图。尽管 Flutter 可以嵌入谷歌地图，但这肯定不是一件容易的事。

另外，热重装也有自己的限制。
