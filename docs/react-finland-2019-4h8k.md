### ~~12~~ 8 个关于更易使用的 React 应用的提示[曼纽尔·马图佐维克](https://twitter.com/mmatuzo)

[幻灯片](https://www.matuzo.at/blog/12-tips-for-more-accessible-react-apps-slides-react-finland-2019/)，[视频](https://www.youtube.com/watch?v=NL6XKcX4sxc)。

1.  创建一个声音文档大纲:从`h1`开始，不要跳级。
2.  正确隐藏内容:`display: none`、`visibility: hidden`和`hidden`对屏幕阅读器隐藏内容，使用[CSS 属性组合](https://medium.com/@matuzo/writing-css-with-accessibility-in-mind-8514a0007939)仅在视觉上隐藏内容，或使用[视觉上隐藏来自 Reach UI 的](https://github.com/reach/reach-ui/tree/master/packages/visually-hidden)组件。
3.  如果你需要一个按钮，使用`<button>`:它们默认是可聚焦的，并且支持键盘事件。
4.  使用片段避免无效 HTML: `<tr><div><td>` → `<tr><><td>`。
5.  关注焦点管理:这可能是模态的一个问题，把焦点放在打开的模态中，不要让它离开模态。
6.  让每个人都可以访问通知:使用`role="alert"`或`role="status"`让屏幕阅读器宣布通知。
7.  宣布页面变化:在单页应用程序中，屏幕阅读器应该阅读导航页面的标题。检查[到达路由器](https://reach.tech/router)。
8.  自动测试你的 React 代码:使用 [React-Axe](https://github.com/dequelabs/react-axe) 和 [eslint-plugin-jsx-a11y](https://github.com/evcohen/eslint-plugin-jsx-a11y) 。

### 自定义 CSS 是 [Artem Sapegin](https://twitter.com/iamsapegin) 不一致 UI 的路径

[幻灯片](https://stopwritingcss.netlify.com/)，[视频](https://www.youtube.com/watch?v=_CsBRkRTzIA)。

*   我们使用自定义样式主要是为了排版、空白和布局。
*   这就导致 UI 不一致，因为 CSS 的表现力太强了。
*   设计令牌是实现 UI 一致性的第一步。
*   但并不是所有设计符号的组合都是好的:例如，你可能会创建不可读的文本。
*   UI 原语(原语组件)是解决方案。
*   [文本](https://github.com/component-driven/component-driven-development/tree/master/src/components/core/Text)和[标题](https://github.com/component-driven/component-driven-development/tree/master/src/components/core/Heading)组件来呈现应用程序中的所有文本。
*   [Box、Flex](https://rebassjs.org/grid/) 和 [Grid (Stack)](https://github.com/sapegin/stack-styled) 组件来定义空白和布局。
*   使用 [styled-system](https://styled-system.com/) 创建这样的组件:适用于 JS 库中的任何 CSS。

### 由[莫妮卡斋](https://twitter.com/monicalent)打造弹性前端架构

[幻灯片](https://monicalent.com/building-resilient-frontend-architecture.pdf)，[视频](https://www.youtube.com/watch?v=brMZLmZ1HR0)。

*   重构只是暂时有助于解决技术债务。
*   第二个系统效应:由于膨胀的期望和过度自信，小型、优雅、成功的系统被过度设计、臃肿的系统所取代的趋势。
*   “‘遗留代码’”通常不同于其建议的替代方案，因为实际工作和缩放。”—比雅尼·斯特劳斯特鲁普。
*   软件的真正成本不是最初的开发，而是长期的维护。
*   架构作为授权约束:关于我们如何使用数据和代码的约束，帮助我们随着时间的推移更快地移动。
*   共享依赖项:将它们添加到设计系统或复制粘贴。
*   解耦代码比干代码好。

如今，您可以利用三个约束来提高前端架构的弹性:

1.  源代码依赖必须指向内部:不要依赖其他团队的代码。
2.  对代码重用持保守态度:避免耦合随着时间推移而出现分歧的代码。
3.  加强你的界限:不要让人们依赖你的代码(使用 [dependency-cruiser](https://github.com/sverweij/dependency-cruiser) )。
