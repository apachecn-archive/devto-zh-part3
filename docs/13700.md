# 测试框架的力量

> 原文：<https://dev.to/everythingfunct/the-power-of-a-testing-framework-2g94>

我现在准备称蔬菜为巨大的成功。在我的上一篇文章中，我讨论了它的设计和构造。我现在正在工作中的一个项目中使用它。它给我们带来的力量和便利是巨大的。

蔬菜的设计是这样的，它的使用高度鼓励用 BDD，或者至少是规范风格来编写测试。这意味着编写测试的行为鼓励你更明确地思考你正在编写的代码的需求。

既然您的测试描述被写成了需求规格，测试框架就可以向您报告这些需求。我们已经将我们的测试和我们的需求作为单一努力的一部分进行了开发；一石二鸟。

接下来，作为运行测试的一部分，测试框架收集了所有信息，并且能够在报告结果时要求它详细，这意味着我们的测试套件可以有效地为我们生成 V&V 报告。现在我们已经从开发测试套件的单一努力中获得了 3 样东西。

最后，我添加了要求框架只运行测试子集的功能。将这个添加到详细输出中，我就可以获得关于我当前正在工作的代码的详细反馈，而不必费力地阅读测试套件其余部分的输出。这缩短并简化了反馈周期，并且使得 TDD，或者至少是在代码之后很短的时间内编写测试更加有用。事实上，我把下面的脚本放到了我们的库中，供每个人在开发或调试时使用。

```
#!/bin/bash

DRIVER_PROGRAM="unit_test_build/vegetable_driver"

./shake.sh "${DRIVER_PROGRAM}" && $DRIVER_PROGRAM -q -v -f "${1}" 
```

我认为从我的测试框架中获得这种级别的能力和灵活性将是未来任何大型项目的必备条件。事实证明，它太有用了，离不开它。如果我想使用的语言中没有这样的语言，我现在有信心创建它，因为我已经为 Fortran 创建了它。
