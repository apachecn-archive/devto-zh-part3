# 因为好像会出现 CakePHP4，所以回顾一下各版本的历史

> 原文:[https://dev.to/dala00/cakephp4-3jfd](https://dev.to/dala00/cakephp4-3jfd)

发布 CakePHP4 的日子似乎很快就要到了。 虽然我现在没有用，但是因为从版本 1 的时代开始一直使用到了 3，所以好不容易才回顾了至今为止的版本。

## [](#cakephp1)CakePHP1

可以简单构筑 CRUD 的截屏视频给人留下了深刻的印象，是一个在看到的瞬间就一下子陷入其中变得非常喜欢的契机。 当时有几个 PHP 框架，但都不显眼，并不是令人满意的容易使用。 给人的印象是，CakePHP 终于在其中出现了容易使用的东西。

参考 Ruby on Rails 做的(？ )虽然如此，但我认为其价值也相当便利，影响了之后的各种框架。

### 代码的自动生成

因为使用 bake 命令自动生成了各种各样的内容，所以通过将其修改后使用，开发速度相当快。

### 简单易懂的文件结构

作为 MVC 基础的各文件本身也很容易理解，用于扩展其他功能的组件和帮助器等也很容易理解，可以简单地进行文件分割，容易开发。

### 细致的版本

不过，我记得 CakePHP1 有 1.1、1.2、1.3 这样细小的版本，添加了稍微大一点的功能。 (添加验证组件或行为等)

## [](#cakephp2)CakePHP2

这是舍弃了 PHP4 对应的版本。 CakePHP1 的项目发生了很大的变化，没有直接动起来。

现在给人的印象是现在也在使用很多。 我觉得 baserCMS 好像是这个版本。 虽然已经忘记了和 1 有什么区别，但不管怎么说，我记得有很多东西一点点变得方便了。 因为我觉得并不是每个都有，而是 1 在经过各种各样的讨论后开发出来的，所以感觉各种各样的事情都很有条理。

## [](#cakephp3)CakePHP3

引入了 Composer，成为了现代的框架。 这里所有的事情都完全变了，CakePHP2 的项目不能直接使用。 大概是因为现在还留有 CakePHP2 的项目。

老实说，这里使用起来相当方便，所以 3 能用了的话，我觉得没有必要用 2。 话虽如此，在业务上无法升级的状况，以及由于 Composer 引进和各种规格变更等的影响，无法迈出的步伐的影响下，给人留下了还存在的印象。 如果可能的话，用 3 吧。

顺便说一下，这个时候我知道了，不过，官方还是做了类似于版本升级用脚本的东西吧。 但是因为没有做什么大不了的事，所以几乎没有意义。

## [](#cakephp4)CakePHP4

支持了 PHP7。 只是这次似乎主要是应对这一点，除了主要原因的破坏性变更以外很少进行，从 3 开始的升级似乎比较容易。 这里是第一次可以进行轻度升级呢。 我觉得能做的人还是不断地做比较好。 PHP5 的支持也到今年结束了。
