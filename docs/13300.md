# 为什么更漂亮

> 原文：<https://dev.to/tibfib/why-prettier-18mj>

在我之前的工作中，我是一家小公司的首席前端开发人员。我也是唯一的前端开发人员。因此，我基本上可以做任何我想做的事。我通过 twitter 听说了 beautiful，决定尝试一下。我立刻喜欢上了它，并将它添加为一个[预提交钩子](https://prettier.io/docs/en/precommit.html)。我不再关心代码的格式。我只是把它打出来，按下保存，就像变魔术一样，漂亮会让它看起来足够好*。*

 *最终，我离开了那份工作，去一家更大的公司尝试更大的团队。我被聘为高级前端开发人员——团队中已经有了另外两个人。我立即向我的同事建议采用更漂亮的，但遭到了很多反对。投诉:

1.  我只是不喜欢它的样子。
2.  它占用了太多的垂直空间。

我不同意这两种抱怨，但是想成为一个好的新同事，我最终减轻了压力；只是在附带评论中提出来(“是的，我真的很喜欢 [typescript hero](https://marketplace.visualstudio.com/items?itemName=rbbit.typescript-hero) 为我处理导入和排序的方式。这也是我非常喜欢 Prettier 的一点——它自动完成了许多我不需要手动完成的事情”)。

随着我对代码库越来越熟悉，我开始在我们的 JIRA 板上为任务创建越来越多的拉请求(不要让我从那里开始)。看到我的代码将如何被接受，我有点紧张——毕竟在我职业生涯的大部分时间里，我主要是作为一名单独的前端开发人员工作。我沮丧地看到，我的前几个公关充满了评论和“需要工作”的邮票。这让我感觉很糟糕。比如，我不是一个好的程序员吗？

95%的评论是关于格式问题的。“我们在这里加了一行。”"我们喜欢在这里用逗号。"“我们这样做。”我最初的几个拉请求，在我最需要验证的时候，被撕成碎片，因为我们没有使用更漂亮的。

丹·阿布拉莫夫在推特上写道，

> 我们需要一些让我们着迷的东西。我们学习机械规则，并对遵守这些规则感觉良好。比修复 bug 容易多了！我们把它们强加给新的团队成员——这是我们做事的方式，你忘了这里的空格和分号。
> 
> 这太疯狂了。打破循环。用漂亮点的。

渐渐地，我了解了他们的编码风格，风格评论也减少了。现在，在我开始工作五个月后，团队同意使用更漂亮的。只是有点...发生了。本周我郑重其事地提出了这个问题，我认为和我的同事共事一段时间后，他们对这个问题更加开放了。前端团队领导决定做一些调查，并最终愿意尝试一下。

我们花了一天的时间争论更漂亮的设置(最初他们想要 180 个字符列！)但最终还是达成了共识。我们各自的编辑都比我们在回购中的`.prettierrc`更漂亮。我有一种很好的感觉，这将是我们最后一次争论代码风格。*
