# 不要像开发者一样思考！

> 原文:[https://dev . to/vip3 rous mango/stop-thinking-like-a-developer-3o3n](https://dev.to/vip3rousmango/stop-thinking-like-a-developer-3o3n)

成为一个组织的开发团队的一员是一个令人耳目一新的变化，这来自于几年的独立自由开发。虽然我喜欢做自由职业者的开发，但是作为一个更大的组织的一部分，有一些东西可以以自由职业者没有的方式激励我，但是作为一个更大的开发团队的一部分，实际上只是一个更大的组织的一部分的一个团队，确实有它自己的一套挑战，我必须学习和吸收。并且很高兴我这样做了，因为我认为从长远来看，这有助于一个更好的开发人员。

随着如此多的新鲜血液涌入这个行业，以及技术日新月异的变化，在我十多年的全栈式 web 开发生涯中，这是我的导师多年来一直灌输给我的东西...

#### 不要只关注“如何”——停止像开发者一样思考！

在我开发“冒险”的早期，我是“维护和保养”团队的一员，经常被叫去帮助企业解决当前运行的遗留系统的一系列特殊问题。

对于分配给我的所有问题，以前的我做了一个简单的发现(或范围)计划会议，尽可能多地学习，然后愉快地启动我的编辑器(当时是 Eclipse...)并开始研究一些难看的代码解决方案。

对于简单的支持票，这很好，我在修复 CSS 问题和简单的 Javascript 交互方面获得了很好的经验，直到我的第一个主要项目落地，这是我们滑块的一个新功能——添加全屏视频幻灯片。

“好，太好了！”我一边想着一边把票分配给自己，并开始思考如何让一个全屏视频播放器滑入滑出，而不是断断续续地过渡到下一个视频。

两天的编码之后，我更新了我的进度，展示了很棒的截图和例子，很快就被我们的项目经理骂了一顿。

**“发生了什么事？!"**我以为。**“我哪里出错了？?"**

我建造了他们所要求的——至少我是这样认为的...

#### [](#focus-on-the-who-why-and-what)关注“谁”、“为什么”和“什么”

作为一名开发人员(包括我自己在内),很多时候我们会被事情的细节所困扰，因为我们知道 web 的内部工作方式，所以在意识到我们忘记问一些关键的项目问题之前，很容易快速深入到我们应该“如何”着手解决一个特定的问题(并很快陷入没有范围的原型领域)。

**谁？**

作为一些最重要的问题，“谁”涵盖了许多主题/角度，取决于观点。

在这种情况下，我从来没有问过业务部门的人，

*   谁将更新或添加幻灯片？
*   “谁”在挑选视频，这些视频是从什么来源提供的(Youtube、Vimeo、自行托管)？
*   谁将更新已经插入的幻灯片并纠正问题？

**为什么？**

帮助组织更大的图片，“为什么”根据观点涵盖许多主题/角度。

在这种情况下，我从来没有问过业务部门的人，

*   为什么要在 Youtube 或 Vimeo 上自主发布视频？
*   为什么我们需要隐藏播放按钮？

**什么？**

“什么”根据不同的观点，涵盖了许多话题/角度。

在这种情况下，我从来没有问过业务部门的人，

*   幻灯片的预期行为是什么？
*   CMS 在模板中创建视频需要什么数据？
*   你有什么视频内容的开始列表？

作为发现过程的一部分，所有这些项目都需要确定范围，但我从来没有这样做过——我只是一头扎进“如何做”中，担心诸如黑色背景闪烁、在移动设备上保持适当的纵横比和自定义静音功能之类的事情。

我从那个项目中得到的最大收获是，我的第一步，实际上是我的前几个步骤，是永远不要开始编码。我应该*永远不要*在项目上马上编码，总是花时间问很多问题，了解更多关于项目、业务、目标的“大画面”——无论什么。

这是给谁的？
“为什么”这样做*这样做*？

所有这些重要的问题都有助于你明确“如何”——你的专长——并且问很多问题有助于让“如何”对你来说更容易。T3】

有一个你永远不会忘记的噩梦项目？它是什么？你下次学到的最重要的东西是什么？我很想听听你的想法！
