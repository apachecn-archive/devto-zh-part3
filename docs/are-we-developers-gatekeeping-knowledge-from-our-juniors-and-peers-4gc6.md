# PSA:对你通用的，对另一个开发者未必。所以在压制这样的内容之前要三思。

作为一个行业，编程相对不成熟，只有不到 200 年的历史(从阿达·洛芙莱斯开始)。网络行业甚至更年轻，还不到 30 岁。

这对于

*   科学医学:~2500 年历史
*   建筑和施工:> 5000 年历史

因此，我们学习的技术似乎在不断变化，几乎没有任何稳定的(想想 12 年以上)长期标准。

根据这个定义:你对你这一代人学到的不值得传播的“常识”所做的任何假设，都是对你的下一代和同龄人的错误假设。

我们还应该摒弃这样的观念:如果一个人不“理解这一点”，那么这个人就不配被称为开发人员。

澄清一下，我的联合创始人绝不是“初级”开发者。我们都有非常不同的专业。

虽然我可能对 ES5 规范和 V8 内部如何工作有很深的了解，但这扭曲了我对 javascript“常识”的看法。

相反，她对 CSS 的各种怪癖以及像`vue.js`这样的现代前端框架是如何工作的有着非常深刻的百科全书般的知识。我承认，直到今天，新手还是会犯一些非常基本的错误。

因此，如果我的假设对于具有不同专业的“高级”开发人员来说是错误的，那么对于一个新的“初级”开发人员来说岂不是更糟。尤其是随着编码训练营的兴起。

因此，直到我们能够为编程找到一个共同的稳定的通用工程标准，我们可以假设这是“常识”，就像建筑师或医生所做的那样(我们可能再过 50 年也不会做到)。我现在沉重地敲响这个钟，因为我们不像其他行业。

永远不要做这样的假设。

# 兄弟语法？我们就不能友善点吗？

即使有合理的批评...

1.  问问你自己，你可能会犯同样的错误吗？
2.  你愿意有人对你或你所爱的人说这样的话吗？
3.  是建设性的批评吗？

而不是以“优越的理由”来守门该写什么不该写什么。相反，我们可以通过批评来鼓励改进。

例如，一次重复的批评。是关于基准测试方法。然而不是像这样的陈述。

> 史上最差基准测试

一个更友好、更有建设性的评论应该是

> 我们可以通过做 ____ 而不是 ___ 来改进基准

坦率地说，对于大多数关于基准测试的建设性批评，你是对的。虽然这可能不会改变最终结果，因为两者之间的差异很大。这是一个有效的改进领域，我和我的联合创始人都为此感谢你。

# 电桥克？

当我说这与性别歧视无关时，我是认真的。

对知识的抑制大概是不分性别的。从高年级到低年级肯定也是如此。

然而,“兄弟语法”这个词是有意使用的。

我肯定有例外(总是有)，但这更多地取决于我们大多数人，发展社区的男性，然后是另一种方式。

虽然我没有任何统计数字来支持它。

我们社区中的少数群体非常了解他们的声音被严重压制的感受。

少数群体是最在意不把“知识”拒之门外的群体。

几乎所有我看到的对我的联合创始人的人身攻击/刻薄批评都来自我们男性发展群体。

我们有责任在这方面提高自己，改善我们的社区。有意识地公开反对。或者在我们的言语和行动中不自觉地表现出来。

我不是要求我们每个人达到不合理的完美主义标准(因为会有失误)，也不是批评你现在是谁。

我是作为一个也犯过错误的人这样说的。在我们个人身上做出微小的、个人的、有意识的改变，这将使我们能够推动社区向前发展。

# 我在其他文章/推文中看到过类似的事情，有什么我能帮忙的吗？

而不是正面作战，和“喂巨魔”。

在非建设性批评的情况下(如更差的基准)，代替作者，礼貌地询问

> 你说的 _____ 是什么意思，内容如何改进？

这给了个人一个澄清的机会，因为他们可能并不总是故意伤害，有时可能只是在翻译中丢失了单词。

只要记住保持**尊重、建设性和清晰**来化解任何紧张

对于非常苛刻的评论或人身攻击，最好是回复评论或作者，而忽略欺负者。让他们知道还有人欣赏他们的工作。一边轻描淡写地指着批评。

> 我个人认为内容是有用的，不管别人怎么说。谢谢你

无论内容类型如何，上述内容都是通用的(它可以应用于开发社区之外)。

最后，对于从事这种批评的开发人员，我写这篇文章的部分原因是，这样我可以在将来为其他人指出它。因为它发生在他们身上。

> 我个人认为你做的很棒，不管别人怎么说。谢谢你。对于持相反观点的开发者，我们应该鼓励而不是压制我们的同行。

沿着这些思路，我是 Recurse 中心的社交规则的忠实粉丝——尤其是在这件事上，关于“不要假装惊讶”的规则。他们对此的解释是:

> 假装惊讶是指当某人不知道某事时你表现出惊讶。在这种情况下用惊讶来回应，会让人因为不知道事情而感觉不好，以后也不太可能问问题，这让他们更难学习。
> 
> 不假装惊讶不是一个好名字。当有人在你不知道某事时表现出惊讶时，他们是假装惊讶还是真的惊讶并不重要。效果是一样的:下一次你有问题时，你更有可能闭上嘴。这条规则的准确名称应该是当某人不知道某事时不要表现出惊讶，但它很拗口，在这一点上，目前的名称已经固定下来了。

假装(或实际上是)惊讶肯定不像你提到的一些例子那样卑鄙，但它仍然会阻止那些对一个地区不熟悉的人大声说话或寻求帮助。我只是想指出这一点，因为我认为人们很容易无意中这样做。

我认为问题是双重的。

一部分是主动看门。人们跑来跑去告诉每个人他们或他们使用的工具不够好。

有 Java 的时候怎么用 PHP？有 C#的时候怎么 Java？当 PHP 最终成为一门*真正的*编程语言时，为什么要转用 JavaScript 呢？

甚至我的编程技术教授都说 JavaScript 是一个玩具，根本无法与 Java 相提并论。

人们为了各种毫无意义的原因在彼此的工作上拉屎，而不是互相帮助。这种情况需要停止，我感觉这个行业中更多的女性和少数族裔将会导致这种情况。CoCs 像野火一样蔓延，人们变得更加善良。我在 dev.to 上看到的几个老派超级书呆子没有呆很久。

另一方面，我们有被动的守门或简单的无知。我猜这就是你所说的。这可能有很多原因，我只能谈谈我遇到的原因。

我经常有这样的印象，我是一个糟糕到平庸的开发人员，所以我不重视我所知道的大部分东西，这让我认为我是这个行业中最后一个学习东西的人。我花了一年理解函数，花了几个月理解闭包，又花了一年理解单子，等等。我总是学得很慢，9 岁时游泳，14 岁时骑车，所以我想这就是为什么别人都已经知道我刚学的东西了。

我还不得不与许多这样的老派超级书呆子一起工作，他们为项目选择技术，并且通常表现得好像他们已经把一切都搞清楚了。我花了几年时间从事自由职业和写博客，发现他们中的大多数人只是在说大话，有一大群人知道的还没有我的一半多，他们愿意和我一起工作或向我学习。
