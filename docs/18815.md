# 年度回顾:同时回顾过去和展望未来

> 原文:[https://dev . to/antjanus/yearly-retrospect-looking-back-and-forward-in-same-time-3pi 1](https://dev.to/antjanus/yearly-retrospectives-looking-backward-and-forward-at-the-same-time-3pi1)

早在 2012 年，我就开始了一个博客系列,在这里我每年都会“检查”自己做过和学过的事情，以及下一年我想做和学的事情。每年，它都为我的学习进度和成就提供了一个无价的视角。[今年](https://antjanus.com/blog/thoughts-and-opinions/yearly-review/how-did-2018-go-and-what-now/)也不例外。

我过去常常在年底感到非常沮丧，感觉自己几乎没有学到任何东西，感觉自己在来年的发展中没有真正的方向。起初，当我开始学习 web 开发时，我感觉到了不断的进步——前一天你正在编写一个小的 PHP 脚本来检索一些博客文章，第二天你就在编写一个带有验证功能的邮件脚本。前一天你还在编写 HTML，第二天你就有了一个成熟的网站。

当你狼吞虎咽地吃基础知识时，学习的速度似乎慢了下来。显然，要学的东西少了，也没必要学了。然而，当我开始写我的回顾时，我对自己了解了很多。

> 当你狼吞虎咽地吃基础知识时，学习的速度似乎慢了下来。

## [](#learning-is-constant-at-a-semiconstant-rate)学习是恒定的——以半恒定的速率

第一件不可思议的事情是年复一年地看到一份长度几乎相同的清单。尽管感觉自己做得不多，但我保持了相当不错的速度。

2012 年，我开始在拉勒维尔开发网站。我也学会了少和淘汰制。

2013 年，我在 Postgres、AngularJS 和其他一些网站上解决了流浪汉、全文搜索等问题。

2014 年是我真正进入 AngularJS 的一年，开始用 Node 编码，写了我的第一个开源代码。

2015 年是我 SASS，一些 C#，学习使用 Babel 的一年。如果我没记错的话，那年我也有一些反应。

每一年都非常忙碌，直到我把这些事情写下来，我觉得我的一年是不够的。我对这一年的看法总是随着我写检查记录而改善。

如果你曾经像那样感到沮丧，那就真正地刮刮桶底，看看会发生什么。今年，我也努力想出一个清单，但又一次，我让自己大吃一惊。

> 每年都非常忙。

## [](#wins-unexpected-progress-and-abandoned-goals)胜利、意想不到的进步和被放弃的目标

回顾过去总是重要的，就像展望未来一样重要。我们从历史中学习----我们都知道注定要重复过去的错误。回顾的概念对技术行业来说非常近，甚至可以说是 SCRUM/Agile 的一部分。

年度回顾的重要部分不仅仅是对一个人的学习“感觉良好”。

我认为将回顾分成讨论这一年的成功(也许是完成的目标)、意外的进展(计划外的成功)和放弃的目标是非常酷的。

对于所有这些，弄清楚*为什么*是很重要的。为什么事情解决了？他们为什么失败了？我们为什么会偏离目标？

例如，去年，我放弃了大部分的副业项目。为什么？我想了想，意识到我把大量的时间花在了自我提升上，而不是兴趣爱好的培养上。我还意识到，我的副业项目常常能满足我在工作中所缺乏的东西，今年的工作非常出色。

这种洞察力有助于引导你去你想去的地方。我对我度过这一年的方式非常满意，所以对于明年，我写下了最小的目标，而不是雄心勃勃的清单，至少在技术方面。

## [](#get-personal)获得个人

这种目标设定和回顾对个人生活也有好处。几年前，我开始在我的技术回顾中添加一个`personal`部分。这些年度检查对我来说比什么都重要。我真的不期望一个随机的开发人员来给我鼓励或有用的反馈。

我倾向于不仅写下我的个人目标，我的成就等等。我还写下我想记住的关于这一年的有趣的事情，这些事情我通常会忘记，让它们随着时间的流逝而消失。所以我一直在记下我听过的音乐，玩过的游戏，读过的书等等。

这实际上对我来说很有趣，因为我经常回来查看我的签到，我记得“哦，是的，在 2017 年，我完成了最初的厄运。这是一个有趣的游戏”或者我调出当年的播放列表，回忆一下。

## [](#checking-in-with-your-checkins)办理入住手续

写下下一年的目标绝不是“写下来就忘了”的情况。我时常会回顾这些回顾，尤其是当我感到有些失落的时候，并仔细阅读它们。我喜欢记住我开始要做什么，更重要的是，为什么要做。

看到我前几年的问题，如果我在重复这些错误，以及我做了什么来应对这些问题，这也很酷。

例如，我的一个大问题是坚持我的目标。这曾经困扰着我，但现在只要我有一个好的理由，我实际上可以放弃目标。

## 目标不是一成不变的——要有适应性

所以我来了这里。不要认为目标是一成不变的，超出目标你就无能为力。

就像技术冲刺一样，放弃不再有意义项目是可以的。你不会为了完成而完成一个不再相关的特性(或者好吧，也许你会)。

我来分享一个工作中的故事。多年来，我一直致力于这个愚蠢的 bash 脚本，它可以限制备份服务器上硬备份的数量。偶尔，我们会收到来自虚拟机的警告，说驱动器快满了，我会 SSH 登录，删除备份，然后我会在脚本上做一点工作，因为我浪费了 30 分钟来查找服务器 IP，尝试 SSH 登录，尝试记住备份在哪里，等等。我最终放弃了它，因为我不能让脚本正常工作。

几个月后，我们改用 S3 作为我们的备份存储。今天完成这个脚本将是完全无用的。尽管这个项目已经接近完成，但是因为它不再相关，所以放弃它是可以的。我想我知道当我开始写那个脚本的时候会发生这种情况——它是为一个错误的解决方案解决一个问题，而我们正计划修复这个问题。

## [](#start-in-december)十二月开始

最重要的是，12 月开始！新的一年充满了活力，你会发现自己兴奋地投入进去，做你的项目，把所有的时间花在你的新目标上。

然后在二月耗尽。去当地的健身房看看吧。

相反，做我做的，在 12 月开始你的目标。为什么？

因为就在你到达想要放弃的阶段时，又是新的一年，你会有新的能量帮助你度过戒烟的难关。如果你去健身房，到那时你会成为常客的一部分，对所有加入但几周后就会离开的人感到恼火。
