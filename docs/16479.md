### 完成挑战的提示

*   建立切实的目标
*   让你的目标*灵活:*尝试新事物是可以的！
*   转发你的推文
*   共享您的项目

我完成挑战的最大秘诀是建立目标。无论是在第 30 天之前用 Sass 建立一个产品登陆页面，还是在第 70 天之前从 Udemy 获得一个证书——都要让你的目标具体化。尽管你应该坚定自己的决心和奉献精神，但如果你不能及时实现目标，或者即使你决定要专注于完全不同的事情，也要记得对自己温和一点。

作为一个初学者，当似乎有太多的*要学的时候，写下目标和创建课程可能会很棘手。有时候做一个项目或学习新东西最困难的部分...实际上是开始了。我发现自己浪费了很多时间去阅读关于该学什么的评论，而实际上我只是在拖延投入工作和发现我感兴趣的东西。*

 *在我的挑战中，在决定坚持使用 Javascript 之前，我涉猎了一点 Java。当然，这可能会让我在#100DaysOfCode 挑战中走一段弯路，但更重要的是，它坚定了我一定要继续学习 Javascript 的决心。尝试一些我后来意识到我不喜欢的事情并不是浪费时间；这是一种很好的探索方式，尤其是在一个有这么多领域可以尝试新事物的领域。此外，知道你不想做什么和知道你想做什么一样重要。

我建议当你在推特上谈论你的#100DaysOfCode 挑战时，把你的推文串联起来。当一切都在一个地方时，更容易看到你在整个挑战中完成了什么。例如，如果你在第 68 天感觉没有动力，你可以滚动到第 32 天，意识到你已经学到了很多。这总是让我觉得自己在进步。

在你的推文中分享你的项目。将您的 CodePen 或 GitHub 链接到您最新的项目。分享可能看起来很可怕，但这是一个从希望你成长为开发者的社区获得反馈的好机会。分享你的工作也是展示你一路进步的好方法。讨论你在创建项目时学到了什么新的框架。

如果你目前没有在一个可以展示你写的代码的项目中工作，截屏你对教程的回答或者分享一个帮助你学习一个困难概念的资源。有时候我没有“编码”很多，但是我通过阅读这里和其他网站上的文章学到了很多...这些天我没有在推特上发布我的代码，而是发布了文章的链接。

我的最后一个建议是把一个小时分成更小的部分，让时间紧迫的人更容易管理。你不需要为了挑战而一口气写很长时间的代码。一天中 15-20 分钟的突发编码也可以，当陷入工作中时，呼吸新鲜空气后回到项目中通常会有所帮助。我通常会在工作日开始前(我精力最充沛的时候)和在客户之间工作时做很多辅导。根据你的学习方式，如果你是分部分学习的话，消化材料会更容易。

对于那些正在挑战或开始挑战的人，让我们一起编码吧！我很乐意在他们的旅程中与更多的初学者互动。如果你已经完成了#100DaysOfCode 挑战，有什么小技巧可以分享？*
