# 轻松的前端开发

> 原文:[https://dev.to/rmnvsl/easy-frontend-development-8gh](https://dev.to/rmnvsl/easy-frontend-development-8gh)

## [](#background)背景

在我的开发职业生涯的开始和这些年里，我主要是在前端进行修补，具体来说，我使用 HTML 和 CSS 将 Photoshop 中创建的网站设计转换为模板，JavaScript (jQuery)提供了一点帮助！).这些模板后来由客户的后端程序员与一些 CMS 一起实现。尽管那时的技术、流程和用例已经发生了变化，有时是彻底的变化，但我们总是会回到根本——普通的`*.html`文件。

那时候，*很多工作都是重复的(或者也许我不知道更好的做事方式):将设计中常见的部分复制粘贴到每个模板上，比如页眉或页脚。想象十几个不同的子页面——主页、文章、产品、购物车等的模板。，这时你意识到你忘了在页脚的某个地方添加一个网站的标志。你必须打开每一个模板，并将其添加到位置。当然，PHP 后来帮助我生成了结果模板，但我发现它有点麻烦。*

现在，当我完成时，我必须分享我的工作，让它被检查是否一切正常。这意味着创建一个包含所有文件的 ZIP 文件，通过电子邮件发送，然后等待响应。发现错误？设计师意识到他想改变一些东西，即使是一小块？整个过程又开始了。

有了免费的主机，在 FTP 时代，这变得容易了一点，但是随着设计的不断变化，仍然非常容易出错和费力。

## [](#what-i-was-lookinglook-for-now)我现在在寻找什么？

没那么多:

*   **共享模板的公共部分**——不再复制——将相同的东西粘贴到几个地方
*   为设计师或客户准备一些更新的预览图，这样他们就可以对变化立即做出反应(或者当他们有时间的时候)
*   一个简单的**更新的可能性** -消除另一个通过 FTP 粘贴的拷贝
*   最终解决方案的更好的**发布方式——ZIP 文件，真的吗？**
*   更快的**反馈处理**——电子邮件或聊天交流会变得非常混乱，人们很容易忽略重要的事情
*   给文档留一些空间**——不仅是为了最终的后端实现者，也是为了我未来的自己**

## [](#todays-possibilities)今天的可能性

正如我在文章开头所描述的，一切都在不断变化，但是经过几次尝试和失败之后，我目前的技术组合包括:

*   Nunjucks - JavaScript 模板语言
*   [吞咽](https://gulpjs.com/)——处理重复性的任务
*   [Browsersync](https://www.browsersync.io/) -提供实时重新加载，在开发阶段非常有益
*   为你的文件提供(私人)存储空间
*   [Netlify](https://www.netlify.com/) -主持你的预览

### [](#nunjucks)双节棍

Nunjucks 作为一种模板语言提供了我正在寻找的一切，它支持变量，模板继承，这对于定义全局布局，宏，这意味着自定义函数，过滤器，修改输出等等。它允许你将部分模板分割成多个文件，以保持干燥(不要重复自己)的原则。

```
{% extends "base.html" %}

{% block header %}
<h1>{{ title }}</h1> {% endblock %}

{% block content %}
<ul>
  {% for name, item in items %}
  <li>{{ name }}: {{ item }}</li>
  {% endfor %}
</ul> {% endblock %} 
```

问题来了——为什么不直接使用这些静态站点生成器？或者 React，默认情况下是基于组件的？我认为用这些创建虚拟模板是多余的。我必须记住，我不知道谁会使用我的作品，无论是它的输出还是源代码，所以我试图让最终用户尽可能简单，让它成为任何人。最近我还发现了[嵌入式 JavaScript 模板](https://ejs.co/)，这似乎是一个不错的选择。

### [](#gulp-amp-browsersync)一饮而尽&浏览器同步

众所周知的任务自动化管理器 gulp 仍然在今天的 web 开发中占有一席之地，即使我们现在有了强大的 Webpack。通过普通 JavaScript 函数配置的任务对于广大受众来说是可以理解的。Browsersync 通过自动同步在同一时刻反映在浏览器、多个浏览器甚至移动设备中的更改来加速开发过程。节省了很多时间。

这里有一个[的例子](https://gist.github.com/crazko/d59053a5f3c236a20d27eacbb8a5e920)一个`gulpfile.js`可能是什么样子。这有点简化，因为没有项目有相同的设置。

### [](#bitbucket)Bitbucket

Bitbucket 是 Git 代码管理，它存储你的代码，当你把他们添加为贡献者时，你可以和其他人分享这些代码。它提供了无限的私有存储库，这简直太棒了。有了每个项目的存储库，您可以将简单的文档保存在`README.md`文件中，或者使用嵌入式 wiki 页面获得更复杂的帮助。

但是真正有价值的是问题跟踪器。您可以在特定的标签中单独引导对每个 bug 或新功能的讨论，并在完成后将它们标记为完成。你不再迷失在无数的电子邮件链中。这个很厉害。

### [](#netlify)Netlify

最后，Netlify 是一个静态托管。你不能在后台运行 PHP 或其他语言，这很好，因为我们想要的只是呈现虚拟的 HTML 页面。它连接到您的 Bitbucket 的存储库，并可以在每次更改(每次提交)时运行特定的任务。

你记得 gulp 吗？

当你把所有东西连接在一起时，Netlify 可以运行 gulp 任务，构建你用 Nunjucks(或任何其他模板语言)编写的模板，并在一个定制的子域上免费托管它们，比如[https://metropole-33b4eb.netlify.com/](https://metropole-33b4eb.netlify.com/)。它会在您将源文件中的每一个更改推送到存储库时都这样做。

现在你只需要和你的客户共享一个域名，他或她就可以随时看到你作品的最新版本。自动化很强。

## [](#summary)总结

每个项目都是不同的，所以我没有指定过程中可能用到的任何 CSS 预处理程序或 JavaScript 库。可能性大概是数不清的。我目前的设置由前面提到的 Nunjucks+Gulp+browser sync+bit bucket+Netlify 组成，现在对我有效，但将来可能会改变。

请记住，并不是每个人都习惯使用问题跟踪器或吞咽任务/npm 脚本之类的东西。但是教你的客户或同事正确使用它们真的会有回报。对双方来说。
