# 澄清 GDPR

> 原文：<https://dev.to/xowap/clarifying-gdpr-1gld>

几个月前，GDPR 席卷了全世界。结果，一切仍然照常发生，但是你得到了更令人讨厌和讨厌的 cookie 同意机制。我绝不是律师，但在与各种客户讨论过之后，让我们假设我对需要做的事情有所了解。

# 我们有饼干

首先，每个人都应该记住一件非常重要的事情。**不是饼干的问题**。从来都不是关于饼干的。没人关心你的饼干。

问题是:

1.  你收集哪种个人资料？
2.  你用它做什么？

简而言之，是的，你可以收集某人的电子邮件作为身份验证，但这并不意味着你有权向该地址发送推广内容。这也意味着没人在乎你该死的会议饼干，所以请不要再为“必要的饼干”征求同意了，这完全是胡说八道。

同样值得强调的是，不是数据本身，而是你用它做的 T2 过程。您可以出于多种目的处理相同的数据，每一个目的都需要自己的理由。

# 同意培养

人们很容易产生这样的印象:从现在开始，你做任何事情都要征得同意，但事实并非如此。处理个人信息实际上有六个法律依据。以下是最臭名昭著的例子:

1.  **合法利益**。你需要权衡这个过程对用户和公司的利益。如果用户的利益超过了你公司的利益，那么你可以合法地处理这些数据。

2.  **合同履行**。有时候你需要一些数据来执行你和用户之间的契约。通常情况下，如果用户购物并用信用卡支付，你就可以使用他们的信用卡号码进行支付。

3.  **同意**。著名的那个。基本上，如果其他 5 个原因都不适用，那么你需要征得用户的同意。

还有 3 个其他的基础，它们当然非常重要，但使用频率却不那么高

1.  **遵守法律**。如果你需要数据符合法律，那么你就可以自动处理它。这是有道理的，对不对？

2.  **切身利益**。如果收集这些数据会直接挽救某人的生命，那么你可以用它来达到这个目的。主要用于紧急服务。

3.  **公共利益**。也许处理这些数据对公众有帮助，或者你代表政府行事。

我在每一点上都很快，你可以在这篇文章中得到更多的细节。

# 特征等级

我看到一些网站基本上给你两个选择:要么安装一百个追踪器，要么放弃 CSS 和功能。很简单，你不能这么做。你没有理由惩罚一个想要保护自己隐私的用户，特别是因为上面列出的原因:

*   如果您需要某个特定功能的数据，它通常属于*合同履行*或*合法利益*案例【请注意“再销售”不是一个功能】
*   即使它没有，也有很多不需要用户同意的情况
*   除非是广告，在这种情况下，它不是一个功能，这取决于你说服用户给你他们的同意

简而言之，没有削减功能，由于 GDPR 的限制。

# 保持清洁

每个数据处理都必须有截止日期。也许是瞬间，也许是永远，也许是 3 年(律师往往倾向于同意是 3 年)。但是您需要确保每个进程将数据保留最少的时间，并在不再需要时清除数据。

举几个例子:

*   最好在 3 年后删除不活动的账户(除非你需要数据来履行合同)
*   如果您导入一个联系人文件来邀请用户的朋友，最好立即删除它

# 便携性

您应该为您的用户提供一种方法，以某种方式从您的公司提取他们的数据，这将允许其他公司创建一个导入程序并使用您的数据。

但是请注意，这仅适用于您的用户提供的数据，不适用于您之后生成的数据。

Tinder exports 就是一个很好的例子:你会看到你的个人资料、描述和你发送的所有信息，但你看不到你的*热度*评分和你收到的信息。

# 保持跟踪

基本上，以上所述的一切都已经以某种方式就位。GDPR 只是提供了应该做什么的明确指示，并加大了对不遵守者的制裁。

一件事后看来显而易见但之前没有人想到的事情是，你需要跟踪所有的数据处理过程，以便了解你所做的一切，并告知用户你对他们数据的使用情况。如果没有一个中央**寄存器**,你怎么知道数据会发生什么，甚至你自己都不知道？

GDPR 现在清楚地表明。你必须在一个文件中记录所有的事情。为了这种用法，我起草了一个 [YAML 模板](https://gist.github.com/Xowap/d74bdedeca627e1778aef3196d99fac1),我把它包含在我工作的每个库中。这个想法是，对于每个 pull 请求，这个过程都要分析新的数据流，并确保它们出现在寄存器中。这是一个很好的方法，可以让它保持最新，同时还可以处理合理数量的代码进行分析。

目前还没有工具，但使它成为 YAML 文件的想法显然是要有一个寄存器“收集器”，它将能够轮询和合并所有存储库中的所有寄存器文件。在我的公司，我们按客户对存储库进行分组，这是自动向客户提供其 GDPR 过程文档的好方法。

# 谁需要遵守？

欧盟要求合规以保护其公民。这意味着世界上任何与欧洲公民打交道或在欧洲运营的公司都必须遵守 GDPR。

这对从外部进口服务的欧洲公司来说很重要。例如，如果你从一家美国公司购买服务(作为一家科技公司，你最终总是会这么做)，你需要确保他们符合 GDPR 标准。否则你没有权利向他们导出数据。

# 检查表

最终，GDPR 更多的是关于责任，而不是真正深刻的变革。我们已经了解了 GDPR 最重要的方面，这篇文章当然不是详尽无遗的。然而，我相信这已经是一个更好的理解，你可以在野外找到什么。

让我们用一个清单来简化事情:

*   确保你对你的过程有一个记录，这是所有数据过程及其合理性的文件
*   如果其中一些理由是基于用户的同意，请确保您提出要求并且能够证明这一点
*   在你的用户界面中包含一个让用户删除他们所有数据的方法(并定期检查)
*   还包括对所有用户生成的数据进行 JSON 转储的方法
*   确保不再有用的数据被正确删除
*   确保所有导出的数据都发送到欧洲公司或声明符合 GDPR 标准的公司
*   还有，拜托，拜托，不要再问什么需要 cookie 和无数未记录的广告 cookie 了

一旦你这样做了，你将远离罚款和 5 年监禁。
