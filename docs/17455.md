# 我在 2018 年读过的科技文章

> 原文:[https://dev . to/design frontier/tech-things-I-read-in-2018-9 C2](https://dev.to/designfrontier/tech-things-i-read-in-2018-9c2)

这将是我今年阅读的科技相关书籍的一个非常简单的列表。还不包括数百篇博文和论坛讨论。我会把它分成两类，值得读的和不值得读的。希望这能节省你一些时间。我也会在看起来合理的地方加入一些关于一本书的笔记或想法。

## [](#things-worth-reading)值得一读的东西

首先，在一位同事的建议下，我今年加入了计算机协会。他们收集的论文多得令人难以置信，其价值相当于门票的价格。此外，他们有一个合理的道德准则，为你制定自己的道德准则提供了一个良好的开端。我的意思是...如果你倾向于阅读学术论文，加入是一个相当可靠的决定。

### [](#a-philosophy-of-software-design-by-john-ousterhout)[一种软件设计哲学](https://www.amazon.com/gp/product/1732102201)作者约翰·奥斯特豪特

这个很好。它有一些好东西。它有一些愚蠢的东西。但是关于 API 优先设计和适当调整模块大小的讨论提供了一个很好的框架。

### [](#it-doesnt-have-to-be-crazy-at-work-by-jason-fried-and-david-heinemeier-hansson)[工作中不一定要疯狂](https://www.amazon.com/gp/product/B079WV79TK)杰森·弗里德和大卫·海涅迈尔·汉森

又一个好消息。感觉自以为是又爱说教。几乎把这个放在不值得的名单上，但是如果你读了它并且想，“该死的，我的公司不是这样的”，你可能要开始寻找一个更好的公司。有理由不完全像他们概述的那样(比如不选择和他们一样的商业模式)，但是你的公司不应该和他们写的完全相反。而且值得看一个对比的观点。

### [](#verbal-judo-second-edition-the-gentle-art-of-persuasion-by-george-thompson)[言语柔道，第二版:温柔的说服艺术](https://www.amazon.com/gp/product/B00FJ3CMI6)乔治·汤普森著

到目前为止，一切都很棒。关于有效沟通和口头化解紧张局势的伟大著作。

### [](#business-model-generation-a-handbook-for-visionaries-game-changers-and-challengers-by-alexander-osterwalder)[商业模式生成:远见者、游戏规则改变者和挑战者手册](https://www.amazon.com/dp/B06X426D4F)亚历山大·奥斯特瓦尔德著

关于开发商业模式的优秀书籍。

### [](#radical-candor-by-kim-scott)[激进坦率](https://www.amazon.com/Radical-Candor-Kim-Scott/dp/B01KTIEFEE)作者金·斯科特

如果你想更好地成为团队成员或团队领导，请阅读这篇文章。优秀的书。

### [](#designing-dataintensive-applications-the-big-ideas-behind-reliable-scalable-and-maintainable-systems-by-martin-kleppmann)[设计数据密集型应用程序:可靠、可伸缩和可维护系统背后的伟大理念](https://www.amazon.com/gp/product/B06XPJML5D)马丁·克莱普曼著

读这本书。它有时非常重。不过，完全值得。如果您想了解在数据密集型应用程序中必须做出的权衡和决策，这是无可替代的。精彩的阅读。但是要准备好面对很多激烈的理论。真的不能高度推荐这一款。读一下。现在读吧。

### 霍尔格·哈姆斯、科林·罗戈夫斯基和路易吉·洛·雅科诺的《在基于微服务的系统中采用前端架构和模式的指南》

概述各种 FE 架构如何使用微服务的优秀论文。每种方法的利弊。(在 ACM 中提供)

### Markus Ast 和 Martin Gaedke 通过无服务器计算实现独立的 Web 组件

本文提出了一些非常有趣的想法来创建由无服务器功能支持的全功能 web 组件。为未来创造了一个有趣的潜在架构。(在 ACM 中提供)

## [](#things-not-worth-reading)不值得一读的东西

这些书没有宣传的那么好...我很可能没有读完它们，把它们留在这里作为对你的警告。希望能帮你省下一些钱和时间。

### 超越遗留代码:延长软件生命(和价值)的九种实践，作者大卫·斯科特·伯恩斯坦

这本书很糟糕。不值得。

### Adam Tornhill 的《你的代码作为犯罪现场:使用法庭技术来逮捕你的程序中的缺陷、瓶颈和糟糕的设计》

这个有一些好的部分，但是你可以跳过它。阅读微软关于错误产生原因和预防的论文，你会比阅读这本书更有收获。(说真的，微软已经就常见错误的根源以及如何防止它们做了大量的学术研究
