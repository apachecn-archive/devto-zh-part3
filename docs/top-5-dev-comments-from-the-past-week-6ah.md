我认为术语“全栈”和大多数人有不同的含义(我猜)。我认为，当你掌握了足够多的知识，能够不依赖技术，并使用最合适的技术来解决任务时，即使这意味着学习新的东西，你也可以被认为是全面的。

在你深入了解一些前端技术和后端技术后，你开始考虑设计/架构而不是语言限制，这允许你根据需要轻松地学习任何其他语言

谢谢你的报道！以下是我的想法。

开箱比较 Webstorm 和 VSCode 是不公平的。两者都使用许多扩展，你总是在两者中安装更多的扩展(或者卸载一些默认的)。如果最终结果是相似的功能，那么没有实际的区别。

尽管如此，就编码而言，Webstorm 是一个远优于 VSCode 的 IDE，原因有很多，但有一个主要原因你也提到了:重构。

在重构方面，VSCode 比不上 Webstorm，而重构是编码中如此重要的一部分。它的重构是我为什么花了一个月的时间只使用 VSCode 给它一个公平的机会，又回到 Webstorm 的首要原因。

你没有提到的一件事，我认为是“在文件中查找/替换”模式，如果不是最大的区别特征的话。几年前发布时，它改变了游戏规则。没有它我简直活不下去。我从未见过这样的工具。VSCode 的“在文件中查找/替换”面板明显比 Webstorm 之前的“在文件中查找/替换”面板差，后者本身就很棒。

在 Webstorm 中，您可以重命名文件夹和文件，它们将在整个项目中更新。您可以将文件和文件夹拖到新的位置，您的整个代码库将更新对它们的每个引用。在 VSCode 中有一个扩展应该可以做到这一点，但是它有很多问题，并且不能像预期的那样工作。它的一个最大的错误是，它把所有的 node_modules 导入都变成了它们的完整路径，就像“../../../node _ modules/path/to/the-libs/file . js "。不能用了。

许多其他好的特性也让 Webstorm 的编码体验变得更好。未使用的功能变暗，粘贴时自动导入注入，导入自动完成，粘贴时自动将 HTML 类转换为 React 的类名，以及许多小而有用的功能，使在 Webstorm 中工作成为一种乐趣。

然而，公平地说，VSCode 有一些 Webstorm 没有的非常棒的扩展。

最大的特色是 VS Live Share，每一个版本都越来越好。但是，我可以使用它，并且仍然可以在 Webstorm 中编码。我主持了一个实时共享会话，更改会在 VSCode 中自动更新，以便另一端的人可以看到这些更改。它并不完美，因为您不能利用“跟随”功能，但它仍然很有用。

Git 扩展更好。Gitlens 和 Git History 充满了 Webstorm 所没有的伟大特性。不过，我主要在命令行上使用 Git，所以我不会太想念它们。

这可能看起来微不足道，但是您可以对应用程序框架进行的定制要比 Webstorm 好得多。我试着安装了 Webstorm Material 插件，它让 Webstorm 像糖蜜一样运行，而且它花了相当多的谷歌来找出如何完全卸载它，因为它留下了各种各样的变化。Webstorm 被 Darcula 颜色框架卡住了。不是交易破坏者，但 VSCode 的主题看起来更有凝聚力。

明显的赢家是网络风暴。我只是没有看到 VSCode 在功能上与 Webstorm 的一些最重要的差异化优势相提并论，所以我坚持使用它。我唯一的功能要求是类似 VS Live Share 的东西，但老实说，我认识的其他人似乎都使用 VSCode，所以没有人可以分享。#孤独的网络风暴#沃斯

您不应该忽略 package.json，它是所有定义包依赖关系的开发人员共享的关键文件。您应该忽略构建工件和已安装的依赖项以及特定于用户的配置文件，例如特定于单个用户的 IDE 配置。请注意，有些 IDE 配置文件是您不想忽略的，比如构建配置，因为它们对整个团队来说是相同的。
