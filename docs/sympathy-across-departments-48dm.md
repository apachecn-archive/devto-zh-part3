# 跨部门的同情

> 原文：<https://dev.to/jessekphillips/sympathy-across-departments-48dm>

很晚了，我睡着了，但我醒来后开始思考这个问题，我相信我已经与我的沟通。

我的头衔是软件测试工程师(STE)，我每天做的工作是测试和软件开发工程师(SDET)的混合。我曾以专注的方式直接与一些开发人员合作。最近，我不仅一直在推动测试软件设计和方法，而且还整合了 CI/CD 管道的 CD 部分(让我们现在不要进入我所指的连续交付)。

这是我的错误:我一直在假设每个人都知道我同情他们面临的挑战，并且我正在做的工作(并不总是理想的)实际上是我作为一个开发人员想要的，而不是我作为一个 STE 想要开发人员为我做的。我想详细介绍一下我得到的反馈，从而为这篇文章提供相关的背景，说明为什么开发人员会反对 CI/CD...但那会深入到公司的历史运作中，如果做错了，你会误解阅读它。无论如何让我试试。

历史过程是，开发人员开始他们的工作，他们完成最初的工作，包括他们认为必要的任何数量的测试，并将其发送给团队以安装到环境中供 QA 测试。对于基于 CI/CD 的管道，有一个环境安装步骤以及由 QA 团队管理的集成测试。当一个新项目开始时，有一组修改来为新客户项目的环境实现流水线功能(每个客户从一个基础平台得到一个高度定制的产品)。这是一个挑战，开发者面临着发布的压力，但是管道不是免费的，没有额外的工作就不能完全执行。

请记住沟通错误，我的期望是管道是有用的，开发人员将投入时间和资源来使管道发挥作用...我没有想到的是，该管道的 CD 部分被归类为一组有益于 QA 的步骤，而与开发工作无关。结果是，删除那些管道步骤以获得发布。记住我让你做好准备，如果我没有做好工作，你会误解最后这句话，当他们的管道失败时，开发人员怎么能说发布已经准备好了呢？是的，这也是我的第一印象...为什么 STE 要建一个不通过管道的建筑？但是，CD 部分不是开发人员的管道部分，它被认为是一些测试工程师为了使他们的生活更容易而添加的步骤。

现在让我跳回到“不理想”你听说过分布式版本控制系统吗？那些使用集中控制版本系统的人批评它没有必要，因为一般来说，你不需要管理同样合法的相同源代码库的主干，最好有一个单一的真实源代码。我们的产品就是这样，在一个基础状态下，许多主干在树林中分叉，我们都单独维护它们，这是分布式版本控制系统帮助管理共性的理想情况。

继续，我们还有一个老产品。是的，我们每天都在开发新产品和新功能，但我们不是从零开始，我们生活在微软的代码库中，因此不准备直接进入 Docker 和 Kerbernetes 的新 CI/CD 世界。这意味着在开发管道时，我无法访问 GitLab 期望的环境管理系统，但我仍然可以构建它并获得很大的好处，即使这种[方法不受支持](https://stackoverflow.com/questions/50278800/where-does-gitlab-execute-its-environment-job)。因此，开发人员可以从基础上派生并获得一个工作管道，但要建立一个新环境并运行完整的管道，需要进行大量的配置和软件安装。这也意味着环境并不干净，部署和测试步骤可以在相同的环境中并行运行。记住我说的不理想，但从我的开发者的角度来看，这仍然是值得的。
