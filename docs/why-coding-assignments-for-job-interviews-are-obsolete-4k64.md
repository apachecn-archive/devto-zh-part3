# 工作面试的编码任务已经过时了

> 原文：<https://dev.to/andreirusu_/why-coding-assignments-for-job-interviews-are-obsolete-4k64>

作为过去 5 年来在 Github 上维护一个相对受欢迎的开源项目的人，我越来越怀疑那些要求我在面试过程中完成编码任务的公司。出于这样或那样的原因，我经常接受采访。

如果成功的话，大多数公司会在最初的电话/skype 聊天后让我做编码工作。其他人会在审查完申请后，甚至在电话筛选之前，马上给你分配编码任务。有些组织声称他们收到了大量的申请，因此这是一个实用的解决方案。在申请欧洲粒子物理研究所的全栈开发者职位后，我得到了这样一个任务，这显然是不成功的。

然而，以我的经验来看，最常见的情况是在电话面试后接到任务。并且所述作业通常不需要花费超过 3 或 4 小时。甚至有一个平台来促进这一点——coder pad . io，一个旨在提供一个专门为工作面试任务设计的现场编码环境的网站。很公平，公司需要一个合适有效的工具来雇佣他们能找到的最好的候选人，而且要更快。对吗？不对。

我们不再需要编码任务了。编码任务已经是过去的事情了，在大多数情况下，它们并不能证明什么，因为它们很普通，而且大多数缺乏创新。就创造力而言，即使是人们寄予厚望的欧洲粒子物理研究所的任务也相当令人失望。虽然我还是没通过，但这不是重点。

现在我们有了一个更好的工具。这对每个人都有好处，不仅仅是对渴望人才的公司。这是开源社区。组织只需要在 Github 上挑选一个开源项目，并要求受访者提交一个 pull 请求，而不是让程序员去做一个很有可能被拒绝从而浪费资源的任务。更好的是，它可能是公司使用的一个项目。显然，如果面试不成功，这对受访者也更有利。在常规的赋值情况下，编码者将最多接收几行作为反馈，这不太可能非常有用。

编码任务也是疏远的，有压力的，不确定的。我们的生活中不需要这些。拉式请求是有用的、授权的、包容的和透明的。像 Coderpad 这样的平台在这里可以很好地促进事情的发展。作为维护者，我很乐意帮助他们指出哪些 Github 问题最适合作为作业。

最近，我接到了一个 3 小时的任务。这是我的回答:

您好{受访者姓名}，

恕我直言，要求受访者花 3 个小时进行编码测试是相当长的时间。我的意思是，在过去的 5 年里，我一直在开发这个开源项目。所有的源代码都在 Github 的[…]上，坦率地说，我认为这是一个很好的迹象，表明我能够用 Javascript 编码。你可以检查{项目}的源代码，它也有大约 85%的测试覆盖率，大部分代码是由我写的(或审查)。此外，你有我的简历，想必有足够的经验和学历来胜任这个职位。

所以现在由你来决定是否要继续下去。就我个人而言，我不认为我花 3 个小时去做一个工作面试的编码测试是很有成效的，因为我完全可以花这些时间去修复一个开源项目的一些 bug，而这个开源项目可以让社区受益。我希望你能理解。

安德烈，问候
