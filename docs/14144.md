## [](#the-problem-with-tech-tests)科技测试的问题

在某个时候，有人决定将技术测试纳入开发人员招聘流程，作为对候选人用相关语言编写代码的能力的一个很好的衡量——以衡量他们的技术知识。

问题是，大多数技术测试似乎没有经过深思熟虑，可能会遇到许多问题:

*   它们被用作招聘决策中的唯一(或非常大的偏差)衡量标准
*   他们没有很好的技能覆盖或突出语言深度
*   他们最多是粗略的
*   他们的要求太高了
*   他们不认可也不衡量其他重要技能，比如同理心、沟通能力，甚至(有时)逻辑和解决问题的能力。
*   它们非常严格，不能代表现实世界中的工作方式

## 当好的技术测试变坏时

当然，任何过程或其组成部分都存在其他问题。但是在前面的列表中有几点我想强调一下，因为我认为它们从根本上破坏了技术测试试图实现的目标。

### [](#cursory-technical-tests)粗略的技术测试

我见过在全栈开发角色中使用的测试，测试是向 C#类添加一些属性，将它们连接到前端的几个文本框，并让它们提交到某种形式。

这个特殊的测试被限制在大约 40 分钟。时间绰绰有余。但是，什么时间绰绰有余呢？把表格上的按钮塞进去。Net，把值贴出来塞进一些类属性里？！

有了这样一个简单、轻量级的测试，你可能会说没有必要进行它。除了初学者 C#教程，它几乎没有触及任何级别的技能，它占用了 40 分钟的时间，你本可以花在带候选人四处看看，聊聊他们解决大问题的方法或在团队中工作。

如果你要进行一次测试，让它与工作申请相关，并且足够提供你想要的数据。

### [](#largescale-unpaid-projects)大规模的、无报酬的项目

在过于肤浅的另一面，那些测试是如此的单一，以至于它们实际上是独立的项目。但是，如果一个更大、更具体的项目式测试能够消除一个表面问题，那就更好了，对吗？

嗯，是也不是...

是的，你应该能够看到候选人能力的更多方面，但是这些考试类型的问题不在于它们的规模。问题是，期望申请人花很多时间免费做这些事情是很常见的。

是的，他们向你申请*，向*申请*你的*公司的工作，但是这个过程仍然是双向的。此外，他们可能会申请几个不同的职位，五六个小时的测试很快就会累积起来——尤其是当你要求他们基本上免费为你写一个迷你应用程序的时候*。*

设计师经常被潜在的客户置于这种境地，他们很快(无论多么无意地)低估了他们的技能和时间，要求大量的具体工作；在投机的基础上进行的工作，他们可能“赢得投标”、“得到工作”、“敲定项目”。

设计行业对此进行了反击，并拒绝了！Spec 网站完美的封装了这个动作。

观察某人是如何工作的没有错，但是不要仅仅为了最后一次面试而要求他们付出太多的时间。

### [](#focus-placed-on-raw-technical-skills-alone)专注于单纯的原始技术技能

同样，虽然技术测试的字面目标是解决一个给定的问题，但潜在的目标应该总是看某人如何应对挑战，他们在解决问题时如何工作。

这些知识应该主要从围绕这些主题的讨论中获得:他们所做的项目，他们发现自己所处的场景，等等。然后，技术测试可以自由地支持和巩固这些发现，此外，还可以展示对相关事物的某种程度的能力。

对我来说，招聘决定与以下两个品质密切相关:

1.  这个人与我们公司的契合度如何——我们能一起工作吗？
2.  这个人能学习、发展、成长吗？

这显然不能适用于所有行业。我不希望有人仅仅因为勇敢的态度和敢做的精神就被聘为心脏外科医生。但是一个有分析头脑的经验丰富的开发人员不应该仅仅因为她不了解 ReactJS 而被忽略。

我在 Twitter 上找到的一段伟大的(转述的)引言总结了这一点:

> 你可以学习一门新的语言/框架/无论什么，但是你不能抛弃一个人

### [](#strict-test-conditions-exclude-realworld-practices)严格的测试条件排除了真实世界的做法

我也见过(也经历过)遵循非常严格规则的测试。这些问题从“不能上网”，到与测试要求的工作水平不匹配的时间限制，再到令人瘫痪的指令性编码标准。

是的，你需要某种限制——例如，给考试设定时间框——但是通过对考试设置不切实际的限制，否则在现实世界中就不会(或者*不应该*)存在，我们缩小了我们在现实世界中对申请人的看法。

我的意思是，谁没有在[栈溢出](https://www.stackoverflow.com)上寻求帮助或者参考一组文档，因为他们不可能把整个 JestJS api 都记在脑子里？

## 科技测试能帮上什么忙

他们*可以*给予极大帮助的地方是给出一个概述，一个对候选人解决问题和任务管理方法的客观评估*超越*他们的代码技能的字面属性——他们如何注释他们的代码，或者他们在编写函数中的效率如何，他们的编码风格，等等。

他们可以突出一门语言或一个框架的优势和劣势，甚至是一种设计模式的方法(坚实的，干燥的)，但是技术测试的结果不应该决定候选人的唯一成功。

好的技术测试应该是:

*   深入到足以收集它们存在的相关数据
*   报销考生的时间和费用(对于大型考试)
*   作为候选人个人资料的部分衡量标准，而非最终结论
*   给予公平的工作界限，而不是局限于不切实际的规则
*   不被视为特定技能水平的唯一标志

## [](#tech-tests-still-have-a-place-but)理工考试还是有一席之地的，*但是...*

技术测试有它们的位置，可以很好地完善一个人的整体形象，但它们是一种微妙的平衡:足够大，值得努力；小到不会消耗招聘流程；旨在作为大局的一部分发挥作用。

不要忘了，现在有一大堆评估技术技能的方法，无论是 GitHub 作品集、NPM 贡献，还是可以用来强调决策和开发过程的网站。
