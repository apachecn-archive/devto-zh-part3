### FAQ # 1:JWT(Java Web Tokens)&Oauth 之间的区别

[jwt-vs-oauth](https://community.apigee.com/questions/21139/jwt-vs-oauth.html)

[深度副本 vs 吞副本](https://dzone.com/articles/java-copy-shallow-vs-deep-in-which-you-will-swim)

[RFC7519](https://tools.ietf.org/html/rfc7519)

| JWT | OAuth |
| --- | --- |
| 它是一个令牌，JSON Web Token (JWT)是一个紧凑的、URL 安全的方式来表示在双方之间传输的声明 | 这是一个分发令牌的框架 |
| JWT 是一种不同的 OAuth 令牌 | OAuth 令牌不是唯一的令牌 |
| 不是指向信息的指针 | 信息指南 |
| JWT 包含真实的信息，它可以很大 | 可以很小，因为它只是一个指针 |
| JWT 是自我验证的,“他们的意思是，任何 JWT 的持有者都可以打开它，验证它，然后根据其中提出的声明做出授权决定。 | Oauth 不是 |
| JWT 支持“联合”——任何人都可以生成令牌，任何人都可以读取和验证令牌。 | 当没有联盟时使用 |
| 当需要异步时使用 |  |
| 安全。两者都是不记名代币。两者都需要作为秘密来保护。 | 相同的 |
| 对于自我验证令牌非常有用 |  |
| 有更多关于用户、会话的上下文，包括签名。 | 较少上下文 |
| 性能更好，因为不需要服务器调用 | 需要服务器声明 |
| 强制这些令牌具有更短的到期时间，因为它们在预定义的规则重新生效之前一直有效 |  |

## [](#faq2-difference-between-container-and-vm)常见问题 2:容器和虚拟机的区别

[差异](https://www.docker.com/what-container#/package_software)

| 容器 | 伏特计 |
| --- | --- |
| 共享操作系统和资源 | 一个完全虚拟化的系统获得分配给它的自己的一组资源，并进行最少的共享。 |
| 如果你只是想把进程彼此隔离开来，并想在一个规模合理的主机上运行大量的进程，那么 Docker/LXC/runC 似乎是一个不错的选择。 | 如果您想要完全隔离并保证资源，那么完整的虚拟机是一个不错的选择 |
| 容器在底层是如何工作的？每个容器都在自己的名称空间中运行，但是使用与所有其他容器完全相同的*内核。发生隔离是因为内核知道分配给进程的名称空间，并且在 API 调用期间，它确保进程只能访问自己名称空间中的资源。* | **虚拟化在底层如何工作？**实际效果是，虚拟化允许您在相同的硬件上运行两个完全不同的操作系统。每个客户操作系统都要经历引导、加载内核等过程。你可以有非常严格的安全性，例如，客户操作系统不能完全访问主机操作系统或其他客户操作系统，把事情弄糟。 |
| 虚拟机是一种打包虚拟硬件、内核和用户空间的便捷方式。 | 另一方面，容器只打包用户空间；没有内核或虚拟硬件。 |
| 由于虚拟机的大小和格式，与其他开发人员共享整个虚拟机可能不太方便 | 容器的优点是为开发人员、架构师、质量工程师、发布工程师和系统管理员提供了一种协作的货币，它将整个用户空间以一种方便易用的格式打包和运输。 |
