# 少写代码

> 原文:[https://dev.to/mokagio/write-less-code-1bbh](https://dev.to/mokagio/write-less-code-1bbh)

*这是我的博客[mokacoding.com](https://mokacoding.com/blog/wirte-less-code/)的交叉帖子。*

在我开始软件开发职业生涯后不久，一个更有经验的人——不幸的是我不记得他的名字了——告诉了我一些多年来一直困扰我的事情。我见过的最好的开发人员花在删除代码上的时间比添加代码的时间还要多。我在这个行业工作得越多，就越能体会到少写代码的价值。

我喜欢写代码。你可以用代码让事情发生，就像成为一个巫师。我也喜欢打字这个实际的行为。按键的咔嗒声，手指的感觉——即使是在 2016 年的 TouchBar MacBook Pro 上。给我一台电脑和一杯热饮来陪伴我，我就可以把一天的时间打出来。

作为软件开发人员，我们很容易迷失在编写代码的愉悦体验中。这是我们擅长的，也是我们觉得舒服的。这就是问题所在，**太多时候，我们没有质疑就陷入了写作的行为中，并以过多的代码而告终。**

代码是一种负担。我们拥有的代码越多，维护它就越困难。更多的活动部件意味着更多的东西会坏掉。代码库越大，理解它就越有挑战性，你会被精神负担压垮。

从效率的角度来看，如果我们可以用更少的代码获得同样的结果，我们可能会更好。硬件、编译器、语言，它们将以我们大脑无法比拟的速度不断进步。为易于理解而优化是有价值的。

如果我们想称自己为软件工程师，而不是代码猴子，我们需要做出改变。我们需要从编写代码使事情发生的机械行为，转向精心构建小型、高效且易于使用的系统。我们的工作就是解决问题，而不是写代码。

花更多的时间删除代码而不是添加代码不仅仅是重构的问题。是的，重构是控制代码库的复杂性和规模的最好方法，总是使其易于使用。然而还有更多，写更少的代码是关于**抵制开始打字的冲动，并且首先考虑事情**。

问问自己，是否有办法用更少的活动部件达到同样的效果。除了第三次复制粘贴，是否有可能在一个专用组件中提取一些功能？你能写一个封装了操作序列的函数吗，这样读者不用深入每一步就能理解它做了什么？

我们可以更进一步。作为技术专家，我们喜欢用技术来解决问题，但有时技术并不是正确的答案。质疑每一个新的功能需求，每一个产品的新增功能。你真的需要吗？这对你的用户和客户来说会更好吗？你会惊讶地发现，用减法而不是加法可以解决多少问题。

这种方法是无聊的解药，是持续改进的良方。对于给定的框架、语言、编程风格，不管你有多有经验，你总是可以挑战自己，找到更好的工作方式。即使您非常优秀，已经达到了技术和物理极限，可以编写的代码很少(很可能您还没有，很抱歉打断您)，您也可以提升一个层次，将这些过滤器应用到产品设计领域。

**养成对长代码的不容忍**。每当你发现自己为了完成某件事而写了很多行的时候，停下来问问自己，你是否正在构建太复杂的东西。可以简化吗？能不能少写点代码？
