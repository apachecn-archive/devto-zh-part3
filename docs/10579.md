# 停下来。为“回归”而自动化一切是浪费时间

> 原文:[https://dev . to/dowenb/stop-automating-everything-for-regression-is-a-waste-time-1b 19](https://dev.to/dowenb/stop-automating-everything-for-regression-is-a-waste-of-time-1b19)

## [](#before-you-automate)在你自动化之前

[![Gears](../Images/6804ff5a69f9fcde16c7a014ed28b0f2.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--oEYkM-Ue--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/4z91y0jzhpsg1e6qeciy.jpeg)

做你的分析，确定你最大的风险，并选择如何减轻这些风险。这是抽象的，那么你能识别什么样的风险呢？

*   允许无效输入会使应用程序崩溃，或者允许记录损坏的数据
*   用户不再能做他们以前能做的事情
*   用户现在可以做一些他们以前被禁止做的事情，也就是不允许他们做的事情
*   可用性受到损害，因此现在使用该应用程序更加困难或混乱
*   应用程序根据需要执行更长时间
*   应用程序不再显示隐私或安全性

对于您自己的应用程序，您可以识别更多的风险，但归根结底是这样的:
应用程序不再按预期运行，阻止了客户或企业获得想要的价值。

## [](#be-selective)有选择性

[![Doors](../Images/94ca629b88b4dc72301460f1a933fca0.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--c7er68DP--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/yx3jv0mrgf1uer4lom4f.jpeg)

假设你已经确定了你的关键风险，问问你自己，自动化你的 UI 会给你减轻这些风险的信心吗？

如果是——停止阅读，去实现自动化吧！

如果像我一样，答案几乎总是“不完全”，那么在你将太多时间沉入 UI 和前端自动化之前，请仔细思考。并开始问这样一个问题“那么我如何减轻已识别的风险？”。

在某些时候，没有单一的答案，但坚持下去。与您的企业和/或客户合作，您将实现这一目标。

那么你有什么选择呢？

*   API 自动化与 UI 自动化相结合
*   混乱或“猴子”测试
*   探索性测试，由“按需”执行的工具和脚本支持
*   数据驱动的自动化

如果你对此有什么好的想法，请告诉我！

## [](#if-it-doesnt-bring-you-joy-throw-it-away)如果它不能给你带来欢乐，就扔掉它

[![Joy](../Images/c1366a48492863aa914732ca2b732826.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--oVd-v6G_--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/1mpb7m2knuscufwo5a03.jpeg)

那么-你已经有了一个自动化回归套件？它给你带来价值了吗？它给你带来快乐了吗？

记住，重要的是不要让你的回归测试过时。确保他们仍然给你价值，他们不会变得古怪。如果他们花了很长时间去跑步，他们还值得吗？

如果回归测试不再给你带来快乐，不要害怕停止运行回归测试！快乐的话，读一读“做出改变的信心”。
