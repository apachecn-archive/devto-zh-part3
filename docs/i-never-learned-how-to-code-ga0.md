# 我从未学过如何编码

> 原文：<https://dev.to/coda/i-never-learned-how-to-code-ga0>

部分是受到了 [Arik 关于](https://dev.to/acoh3n)[如何充分利用 dev.to](https://dev.to/acoh3n/how-to-make-the-most-out-of-devto-ag0) 的帖子的启发，我决定写我的第一篇帖子，关于我如何从未学会如何编码。

## 我不属于这里

我从来不认为自己是一名程序员。我一直认为编码太“难”了，涉及软件和网络应用的问题应该留给研究计算机科学的人去解决。每当我遇到 coders IRL 时，我从来没有感觉到我属于那里，我总是从外面寻找。我对 dev.to 也有同样的保留，直到我看到 Arik 的帖子。

当我看到 GitHub 和 HN 上的讨论时，我仍然觉得自己是局外人，并认为我不属于这里，因为我不能贡献任何与技术相关的东西。我总是想:“一个商业分析师或营销人员能为这些社区做出什么贡献？”

## 潜入 Excel

作为一名商业分析师，我整天被困在 Excel 中。在我加入的每个团队中，关于 Excel 的问题都会被抛给我。我成了工具大师，第一次觉得这是程序员掌握 Java 或 Python 时一定有的感觉。但是 Excel 不是编码语言。充其量，它是一种声明性语言。

我深入研究了自动化重复性工作流程，并开始在 VBA 编写宏脚本。这是“编码”吗？我复制和粘贴单元格和格式，在工作表中循环，并使用第三方 API 运行自定义函数来连接我们的 Oracle 数据库。输出仍然是 Excel 内部的东西，这是一个供“商业”用户使用的工具。我觉得这是我在编码方面得到的最技术性的东西。

## 做自己擅长的事情

也许我没有学会如何编码的最大原因是，我要花 1 个小时才能做出一个“真正的”编码员或程序员需要 5 分钟才能完成的东西。回想亚当·斯密关于比较优势的理论，我应该只关注我最擅长的工具，把代码委托给编码员。

不幸的是，我总有一部分想更深入地了解事物是如何运作的。我加入 dev.to 的部分原因是为了在一个我不认为自己会因为不知道如何编码而被批评的环境中学习。Dev.to 看起来像是一个拥有分析师思维的人可以感到自在并发展他或她的技能的地方。

## 当前工作

我认为像我这样的人跨越两个世界，很难找到一个合适的社区。在一家初创公司工作了几年后，我幸运地在目前的公司找到了一个可以施展才华的地方。我不是构建 Excel 和电子表格，而是为我们的用户构建实际的应用程序。用我们一位用户的话说:“我不是软件工程师，但这是我没写任何代码就完成的最多的软件工程。”

仍然有令人惊叹的软件工程师在构建 Coda 背后的机房，但该平台解锁了我用 Excel 未能解锁的创造性的一面。我仍然想为 Coda 后面的引擎室做贡献，并尝试到处提交 PRs，但直到我的提交进入生产还有很长的路要走。
