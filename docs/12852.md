# 轮班团队，而不是测试人员

> 原文:[https://dev.to/angelariggs/shift-teams-not-testers-4979](https://dev.to/angelariggs/shift-teams-not-testers-4979)

我已经写了测试人员所戴的许多帽子，以及测试人员和工程师之间合作的重要性。我在那些文章中的想法主要集中在测试人员为他们的团队所做的工作上，但是最近我开始考虑另一方面——团队和公司为他们的 QA 工程师和测试人员所做的努力。

“左移”的说法很流行，我认为其背后的概念很重要。但是，如果我们不关注测试人员向左移动，而是关注团队和公司改变他们的思维模式和他们的过程，会发生什么呢？

我所在的团队很早就理解并欣赏我的参与，并热情地让我参与发现、架构规划、客户会议和演示。我曾经和一些公司一起工作过，他们理解我在特性测试和发布管理之外所做的“额外”工作的好处，但是并不欣赏它，也没有提供我需要的支持来工作而不精疲力竭。我所在的团队将我的角色隔离为“评审和发布”,当我试图做文档、过程改进或特性计划时，他们会推后。

这并不是鼓励测试人员更早地参与 SDLC。我认识的大多数测试人员已经想要这样，并且理解它的重要性。我们知道，当我们从一开始就被包括在内时，我们会带来有益于产品和团队的技能和见解。作为测试人员，更早地参与进来也有利于我们，因为它给了我们一个关于我们正在测试的东西的更好的知识基础，并且我们能够更好地理解和减轻风险。

当我成为一个团队或组织中 QA 角色定义有限的一员时，我发现有一个发起人或倡导者真的很重要。当我试图改变人们的心态和期望时，有人支持团队中的测试人员是团队的一部分，并且需要被包括在内，这很有帮助。如果你的支持者是团队领导或高级工程师，那会很有帮助，因为他们隐含的领导角色会给他们的观点带来可信度。倡导者可以号召“粘合工作”,并帮助团队的其他成员看到它。他们可以从工程的角度解释为什么在计划会议时让一个有测试思维的人是有用的。他们可以支持质量流程建议，并就如何推广它向您提供反馈。

我很想知道其他测试人员是如何解决这个问题的——你是如何从你的团队那里获得改变他们对你的角色的期望的支持的？你是给某人贴上倡导者的标签，还是另辟蹊径？
