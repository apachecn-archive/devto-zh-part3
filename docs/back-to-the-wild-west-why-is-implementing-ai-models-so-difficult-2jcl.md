# 回到狂野西部:为什么实现人工智能模型如此困难？

> 原文：<https://dev.to/dataroots/back-to-the-wild-west-why-is-implementing-ai-models-so-difficult-2jcl>

<sub>**原作者:**乔纳斯·通多</sub>

人工智能的崛起势不可挡。企业不仅比以前更频繁地求助于人工智能来促进他们的数字化形成，他们还拥有数据、资源以及他们需要的技术和合作伙伴的访问权，从而使这一切成为可能。尽管许多人渴望开始，但他们甚至在事情实际启动和运行之前就陷入了困境。由于缺乏耐心，他们迷失了最初的方向，或者遇到了障碍，导致他们的概念证明不断被拖到众所周知的堆栈的底部:实现的道路突然显得比最初设想的长得多。

越来越多的企业开始意识到人工智能提供的潜力。人工智能为促进您业务的创新方面提供了卓越的可能性，但它能带来的好处更重要。投资回报是必须的。有三种方法可以做到这一点:

*   您可以通过销售丰富的数据或创建交叉销售和追加销售等营销模式来创造收入流。

*   例如，您可以利用预测性维护或自动化手动流程来优化运营效率。

*   机器学习和人工智能可以用来检测欺诈和消除风险，从而降低成本。

然而，一些企业仍然对人工智能不感兴趣，例如因为他们没有数据。那些拥有大量数据的人有时会被诱惑立即开始工作，雄心勃勃而鲁莽地绘制各种模型。不幸的是，在这个阶段，事情可能会出错，最终，工作方式中没有实现任何东西。

## 菜鸟的失误

最重要的原因是，他们从技术角度出发，忘记了倾听他们的业务需求。有一个案例，一个问题，数据团队开始尝试使用人工智能来解决这个问题。然而，它根本不是这样的。人工智能无法改变你的业务，除非你开始从业务角度开发东西，而不是从 IT 角度。即使是最有希望的概念证明(PoC)在实施之前也没有任何价值。当经销商的营销部门建立一个模型来预测潜在客户将购买哪辆车时，只有销售部门实际使用它，这才会导致收入增加。

没有终端用户的参与是新手经常犯的错误。你不能简单地将人工智能视为灵丹妙药，立即建立一个庞大的基础设施，而没有一个有前途的基础，也没有立即成功的希望。

## 意大利面代码

在许多情况下，概念验证是由缺乏编程知识的人开发的。根据定义，数据科学家擅长处理数据，但这并不意味着这个人会写出最好的代码。许多数据科学家在进入数据科学领域时都有商业智能方面的背景，并以一种相当特别的方式处理问题，使用可能在 PoC 阶段有效但不适合实施的东西。

例如，他们将在不考虑基础架构的情况下开发 PoC，其中一切都在云中开发，而客户希望在内部运行该模型。当交付结果和代码的时候，客户通常会得到一盘他无法解开或实现的代码。

## 狂野西部

当谈到人工智能时，我们的现代商业景观和西部大开发时代的淘金热有很多相似之处。每个人都加入了人工智能的行列，并开始努力挖掘自己的数据，寻找承诺的黄金。没有标准，也没有黄金法则来规定如何将机器学习或人工智能项目集成到现有系统中。

即使是拥有已经实现了几个模型的庞大团队的公司也困在他们的工作方式中，继续寻找一个框架。有人可能会说他们制造了一个瓶颈，这使得他们很难移交项目。

## 良好的开端是成功的一半

那么，你如何明智地应对这些挑战呢？一个人工智能模型的成功实现始于一个经过深思熟虑的发布。从一开始，最终用户和企业就必须参与选择正确的案例。你会选择一个有大量可见性的案例，还是会优化一个可能在公司看不到，但有高收益的流程？重要的是要平衡这些考虑。他们必须交付一个成功的、有希望的概念证明，说服最终用户释放足够的预算并实际实现模型。除此之外，相关 IT 部门的现场支持也是必不可少的，这样工作方式就可以与当地的技术系统充分协调。

## 框架

最重要的是，你需要一个带有标准化参数的设计方法框架。你会选择哪个流程？你如何以最佳方式参与业务？交付成果是什么？您将使用哪些数据源，哪些将是您的编码标准？

当您的解决方案基于这样的模型环境时，您将能够推出项目并更快地实现它们，以标准化的方式工作，并且仍然以相同的方式集成各种解决方案。不仅如此，您将能够提供一些功能，使得通过 API 推出模型更加容易，保证可伸缩性(考虑到代码的可读性),并且能够顺利地促进迁移。

这种结构化的工作方式已经在 Dataroots 转化为一个编码框架，作为贯穿每个项目的一个公共线程。这种被称为代码的方法为团队和客户提供了一个强大的立足点，可以立即保证交付的项目将满足特定的质量标准，并保证它们的无错误实现，无论是在内部还是在云中。
