# 本周我学会了:尽早问正确的问题

> 原文:[https://dev . to/rachelsoderberg/this-week-I-learned-ask-the-right-questions-early-5ip](https://dev.to/rachelsoderberg/this-week-i-learned-to-ask-the-right-questions-early-5ip)

本周的课程在本质上并不是真正的技术，但在我作为一名专业软件开发人员的成长过程中仍然意义重大。在我构建的订单处理器准备投入生产之前，我正处于开发的最后阶段，但不幸的是，由于我的疏忽，这个工作周的大部分时间都花在了重构其代码基础的重要部分上，而不是清理和进行任何最终的“生产就绪”调整。

首先，我的公司没有遵循任何严格的开发标准(我正在努力改变这一点)。我们目前不做 TDD、单元测试、敏捷、瀑布，甚至定期的代码审查。哎哟。但是，再说一次，我正在采取措施改变这种情况。

当我负责订单处理器应用程序时，我并没有得到太多关于如何使用它的信息，只是被派去比较遗留代码并“对其进行改进”对于我所需要的大部分内容来说，这很好，但导致了本周困扰我的事情:我还没有必要的经验来了解你的客户的输入有多有价值。无论他们是付费客户、您的销售团队、负责订单的经理还是其他实体，他们都知道应用程序的日常使用方式。

如果我在项目开始时遵循了敏捷原则，我会向我的“客户”询问关于项目的信息(例如:它是如何日常运行的),并有希望被告知它位于服务器上，每五秒钟扫描一次状态为 0 的新订单，并通过从它们停止的地方继续处理状态为 0 的订单。不幸的是，由于不知道这一点，我在构建应用程序时假设处理器只在下新订单时运行一次，并且只以线性方式处理订单(从状态 0 到完成，中间没有提货)。

这一周我花了很少的时间来梳理我的代码库，提取每个处理状态下必须处理的部分，并确保它仍然按预期运行。因为订单可能以任何状态出现，所以我不能再使用我在处理开始时在变量中设置的值——我还必须整理出我的数据，并利用我最初使用这些 getters 和 setters 的数据库表。第三，我引入了一个简单的计时器，它每五秒钟运行一次应用程序，检查任何未处理的订单，并相应地处理它们。

总的来说，虽然这是令人讨厌的一周，但它对我的个人发展至关重要，并且给我上了一课，将永远伴随我。在这个订单处理器投入生产后，我被分配了一个新任务，我将更好地准备提出正确的问题*在*之前，我已经写了数千行代码。
