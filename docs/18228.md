# 冒名顶替综合症还是勉强度日

> 原文：<https://dev.to/jessekphillips/impostor-syndrome-or-just-getting-by-4n6o>

我看过一些文章，谈到了冒名顶替综合症的挑战。我从来不觉得这是对我工作方式的恰当描述。我很擅长我所做的事情，但实际上我很难写出来，说出来，呈现给别人，因为我刚刚开始相信。我从不担心我的行为超出了我所做的，这实际上可能会反映出我的环境不好，因为这意味着没有人会注意到(幸运的是有一些)。

促使我写这篇文章的是，我开始意识到我的期望太高了。我不知道这个行业是怎么回事，但它制造了这么多废话。我只是稍微开个玩笑。有如此多的模式、范例、实践等等。它们都植根于一些有效的观点，甚至为行业和做事的方式提供了巨大的好处。BS 来自于框架的构建，以及这些模式需要在任何地方使用的想法。

(我将坚持使用编程示例，如果有人问我，我更有可能专门发布关于测试的内容)。考虑持续集成，历史上长期存在的分支和将它们引入主线的努力确实存在问题。CI 就这样诞生了，学会做较小的改变，经常把它们带入主线。好的，代码应该在之后编译，编写的单元测试也应该通过，哦，它应该可以和整个应用程序套件一起工作。CI 是帮助长期合作的一个非常基本的原则，现在你不能声称做 CI，除非仙女们用偷来的仙尘祝福它。如果您是唯一的团队成员，并且在一个分支上做所有的工作，那么即使有 CI 框架管理构建，您也不是在做 CI，因为 CI 是关于协作的(注意，我确实相信您可以与自己协作)。抱歉，我的观点是，如果你觉得自己不擅长 CI，因为有太多的工具和框架可以帮你做这件事...你可能没有看到它真正的含义。

另一种被框架滥用的技术是依赖注入。它又一次有了卑微的开始，事物从无到有的出现(字面意思是你调用一个函数，它会创建一个新的实例并拥有它)。它有一些非常好的应用程序，有时可能有点乏味，这就是框架的由来。所有这些创造事物并通过长时间的调用传递它们，这样一个需要它的地方可以利用它而不用创造它，所有这些都随着一个框架而消失，因为那个框架可以用来指定在一个地方凭空创造什么。如果你发现你喜欢在你的 OOP 语言中使用静态类，或者喜欢传入一个参数而不是在函数中创建它，那么你就在做 DI。

我的期望太高了。我不得不不停地查找 IoC，我应该使用 MVC 还是 MVP，或者我的函数应该有多少行代码。这些模式被创建、定义并转化为框架，因此它们更容易被访问。但是对于那些不知道它们是什么或者不知道如何在正确的时间使用它们的人来说，它们变得触手可及。当你有一个用 DI 编写的 TDD 的应用程序，没有人想碰它，因为代码库是一团乱麻，这个使用 TDD 和 IoC 的新系统被吹捧为简化一切，因为它遵循这些原则；不是我不知道这些东西是什么，而是我对自己有一种期望，认为我可以把它们拿出来并很好地使用它们是错误的...没人能做到(反正不够)。
