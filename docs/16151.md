# 软件中的质量属性

> 原文:[https://dev.to/frosnerd/quality-attributes-in-software-1ha9](https://dev.to/frosnerd/quality-attributes-in-software-1ha9)

# [](#introduction)简介

设计系统架构时，你必须做出决策。这些决定将影响你的系统在不同场景下的行为。行为会以这样或那样的方式影响系统或产品的功能。

例如，面向服务的架构(SOA)将复杂的功能实现为松散耦合的服务的组合。每项服务都是或多或少独立开发、部署和运营的。与整体架构相反，松耦合应该给表带来某些好处。

但是我们如何讨论、测量和评估架构决策的影响呢？你可能听到人们谈论“A 是一个可伸缩的、容错的数据库”或者“B 比 C 更容易维护”。这些概念的常用术语是*非功能需求* (NFR)。对于每个架构师来说，nfr 都是一个重要的话题。该名称是作为对受业务利益相关者严重影响的*功能需求* (FR】的补充而产生的。

我个人更喜欢质量属性这个术语，而不是 NFR。“非功能性”中的“非”意味着需求和功能之间的脱节，这在大多数情况下是不正确的。如果您的系统不可用，它也不会运行。

FRs 和质量属性之间的联系可以通过识别架构上重要的 FRs 来建立[1]。架构上重要的需求需要特别注意，因为在架构方面的错误决策可能会导致需求无法实现。

你如何识别与涉众和你的团队相关的质量属性？在您的系统或服务环境中，相关的质量属性有何不同？在这篇博文中，我们将介绍一种叫做迷你质量属性研讨会的技术，它有助于回答这些问题。之后，我们将详细解释一些常见的质量属性。

# [](#miniquality-attributes-workshop)迷你质量属性工作坊

## [](#overview)概述

质量属性用于评估系统的质量。维基百科列出了 [82](https://en.wikipedia.org/wiki/List_of_system_quality_attributes) 不同的质量属性。哪些属性对您很重要，这在很大程度上取决于您的情况和系统的不同涉众。

迈克尔·基林将迷你质量属性研讨会描述为传统质量属性的替代方法[2]。本研讨会的目标是识别对系统利益相关者重要的质量属性。涉众通常是有代表性的用户、业务专家、项目经理、IT 部门和开发团队。

研讨会的结果应该是质量属性场景的列表。这些场景可能会被优化，可能已经有了某种优先级。活动应该有时间限制，公开点应该制定为后续行动项目。

车间的主要工具是*系统属性网页*，或*质量属性网页*。它允许对质量属性场景进行聚类，也用于在整个研讨会中对属性和/或场景进行点投票。

## [](#agenda)议程

研讨会的议程包括以下几点:

1.  **介绍格式。**这应该陈述议程，解释会议的目的和方法。
2.  **介绍质量属性。**第二点要引入质量属性的概念。为了便于讨论和节省时间，提前准备一个质量属性分类法是很有用的，它可以作为一个基线。
3.  **生成场景。**下一步应该通过利用头脑风暴/头脑写作技术或沿着分类法移动的更结构化的方法来创建尽可能多的场景。
4.  确定场景和质量属性的优先级。在生成场景后，必须对其进行优先排序，以确定哪些场景需要改进和处理。区分质量属性的优先级也给出了一个关于优先级的总体概述，这些优先级可以用来做出与质量需求一致的架构决策。可以使用点投票来确定优先级。
5.  **提炼场景。**生成的场景通常以非结构化格式创建。在细化过程中，目标是将原始场景转换成结构化的格式。
6.  **回顾。**细化后的场景呈现给利益相关方。

在研讨会期间，至少完成优先排序是非常有用的。细化应该有时间限制，从最高优先级开始，如果需要更多时间，可以离线进行。如果你没有时间了，复习可以在以后进行。

## [](#scenarios)场景

质量属性场景是研讨会的核心组成部分。原始场景是一种灵活的、非正式的描述质量需求的方式。原始场景通常由一个句子组成，并通过将其放在 web 中来赋予质量属性。这里有几个例子:

*   "向购物篮中添加产品应该总是有效的."(可用性)
*   "浏览作品集应该会有反应."(性能)

在细化步骤中，原始场景被转换成正式场景。正式场景具有以下属性:

# [](#generic-quality-attribute-taxonomy)通用质量属性分类法

下面的分类法是受 O'Brien 等人在软件架构技术倡议[3]下发表的技术说明的启发。你可以把它作为你第一次研讨会的基础。我将只提到每个属性并给出一个快速定义。请参考其他来源的详细解释。

*   **互操作性**描述了一个服务与其他服务通信并允许其他服务与之通信的能力。它衡量信息交流的自由程度。像编程语言无关的数据格式、内容协商、向后兼容的 API 等措施。可以支持服务之间的互操作性。
*   **可靠性**反映了服务正确运行的能力。支持回滚和恢复的自动化可以减少平均故障间隔时间(MTBF)。
*   **可用性**是服务响应请求/可访问的能力。可以通过增加容错措施和弹性(例如通过冗余)来增加正常运行时间。
*   **可用性**衡量服务提供的用户体验质量(UX)。它可以通过拥有一个以 UX 为中心的开发工作流程来增加。由于不可用或速度慢的服务也是不可用的，因此可用性和其他属性之间有很强的相关性。
*   **安全性**有两大方面:保密性(只授予授权用户访问)和真实性(信任所提供的信息)。在密码学领域，例如加密和数字签名，可以找到有用的技术。此外，您应该实施安全的流程，例如定期使凭据失效或轮换凭据，或者强制实施双因素身份认证。通过定期执行渗透测试来检查你做得如何。
*   **性能**是一个宽泛的话题。在服务的上下文中，人们通常将响应时间或延迟作为一种性能度量。这可以通过为问题选择正确的算法和数据结构，并根据负载调整系统来实现。添加自动化性能或负载测试来检测回归。
*   **可伸缩性**描述了处理变化的能力。当谈到可伸缩性时，定义系统对什么样的变化做出反应是很重要的，例如，用户数量的增加，商店中提供的新产品，更多的请求进入，甚至更多的开发人员加入公司。可伸缩性通常是通过解耦和分离关注点，结合选择算法和数据结构来实现的，这些算法和数据结构允许通过添加更多资源来提高性能。
*   **可扩展性**表示在不影响系统其他组件或部分的情况下向组件添加功能的能力。涉及松散耦合、通信标准和进化友好接口和模式的架构促进了可扩展性。
*   **适应性**影响需求改变时改变系统的容易程度。与可扩展性类似，这可以通过松散耦合来实现，但也可以通过抽象来实现，例如，在数据库和应用程序之间放置一个层，以便可以交换数据库技术。适应性也受可配置性的影响。
*   当涉及到对单个组件、组件之间的交互以及系统整体的构建和自动化测试时，可测试性很重要。除此之外，了解这些测试能够多好地检测错误也是至关重要的。在分布式系统或面向服务的架构中，可测试性尤其困难，因为组件是通过不可靠的网络连接的，可能存在不同版本的服务，并且没有一个实体知道所有组件的内部状态。还要注意，涉及自动伸缩和服务发现的动态环境的行为可能很难预测。
*   **可审计性**捕获执行系统审计的能力。例如，出于法律、安全或财务原因，可能需要进行审计。可审计性是一个艰难的目标，因为它要求流程中涉及的所有服务都是可审计的。一般来说，拥有事件和数据的不可变存储和版本控制已经为可审计系统做出了很大贡献。
*   **可观察性**表示如果可能的话，系统中的变化是否以定量的方式反映出来。推广 DevOps 文化可以增加可观察性，因为负责变更的团队也负责运营，这极大地受益于丰富的可用度量。
*   **可操作性**描述了系统在运行期间部署和操作的难易程度。除了可观察性之外，操作可以由自动化来支持。像混沌工程这样的技术可以通过故意引入错误来测试你的可操作性。

请注意，许多场景可能适合多个属性，并且场景也可以相互关联。在我看来，这不是一个问题，而是促进了关于质量的讨论。还要记住，还有更多可能的质量属性要包括在内。如果在研讨会期间，您觉得其他的比这里提到的更重要，只需根据需要扩展、替换或删除选择。

# [](#summary)总结

在这篇文章中，我们已经看到了你的软件架构如何影响你的应用程序的质量和功能需求。没有正确的解决方案，相反，它总是不同质量属性之间的权衡。

迷你质量属性研讨会是一种轻量级的形式，用于收集利益相关者的质量属性场景并对其进行优先级排序。从收集尽可能多的原始场景开始，之后你将对最重要的场景进行优先排序和提炼。质量属性本身的优先级使您能够挑选架构，并做出促进利益相关者优先级的选择。

你有没有遇到过人们根本不谈论质量的项目？你的团队是否曾经在软件架构方面做出一个决定，而这个决定最终成为你的一个功能需求的障碍？如果你回想一下你的上一个项目，你认为最重要的两个质量属性是什么，为什么？欢迎留言评论！
