# 对旧的已知问题寻求新的解决方案的努力

> 原文:[https://dev . to/jesperreiche/the-push-for-new-and-shiny-solutions-to-old-known-problems-70l](https://dev.to/jesperreiche/the-push-for-new-and-shiny-solutions-to-old-known-problems-70l)

作为一个物种，我们总是被新奇所吸引。这可能对反馈循环短的事情很有帮助，因为我们很快就知道“新的闪亮的”东西实际上是帮助了我们还是伤害了我们。

在软件开发中，JavaScript 社区因“新颖闪亮”的框架而臭名昭著。这已经被一次又一次地嘲笑了，这是有充分理由的。但更令人担忧的是，我发现在 JavaScript 领域之外的大型组织和软件项目中，也有很多同样令人担忧的趋势。以谨慎、保守和稳定为准则的行业或组织。但是由于某种原因，一个也许是错误的“保守主义”的尝试最终却没有成功。

我和许多其他人讨论过这个问题，或者在和其他人的讨论和谈话中观察到这个模式，这些模式只是证实和支持我看到的画面。

我主要担心的是，我看到许多项目和组织根据排名前 5-10 的技术公司的最新趋势做出技术决策。对于客观上更简单的解决方案会更好的问题，似乎有一种“新的和闪亮的”解决方案的趋势。似乎许多架构师和开发人员从脸书、亚马逊和谷歌那里寻找解决方案，并得出结论:“如果我们使用相同的技术、结构和建筑，那么我们就是安全的。如果它对脸书、亚马逊和谷歌有效，那么对我们来说肯定也是正确的可扩展方法。”

这种对待技术和软件的“懒惰”方式不仅会导致很多问题，而且会导致比坚持基本原则更糟糕的结果。当然，这并不是说模仿巨人或查看已建立和验证的解决方案来寻找灵感从来都不是正确的方法，但许多人似乎错过了一个重要的步骤。

### **哪些问题是*我们*试图解决的**

从别人那里找到好的和相关的灵感的关键是首先找出你试图解决什么问题。这可以是查看整个解决方案的顶层视图，也可以放大到解决方案的任何较小的特定部分。

如果你不知道你要去哪里，任何路都会带你去那里

一旦你发现你的解决方案试图解决哪个(或哪些)问题，*然后*你可以去看看这个问题以前是否被解决过(很可能已经被解决了)。

**但是，**在考虑其他解决方案时，有一点非常重要。我认为这是许多人和组织重视不够的地方。首先你必须弄清楚你在试图解决什么问题*。我不认为很多人会在这里出错。但是当看到其他解决方案时，你也需要做一些复杂的发现或思考。因为为了让他们的解决方案很好地适合你，你需要知道或弄清楚；当他们找到解决方案时，*他们*试图解决的问题是什么。*

我遇到过有人推荐 NoSQL 数据库，因为它们非常智能，你可以把任何数据放入其中，而无需结构化。这很聪明。**如果**您正在处理非结构化数据。如果您正在处理非常精细的结构化数据，那么您将获得的可能只是复杂性和性能问题。此外，事实上，在我听到他们提出的案例中，团队中没有一个开发人员对该技术有任何经验。所以你可以在此基础上增加成本和复杂性。这根本说不通。

如前所述，最令人担忧的是，推荐这些解决方案的人不一定是年轻渴望的黑客，而是在大型组织中工作的经验丰富的架构师，我本以为他们会更加保守。

### **为什么会这样？**

所以最大的问题当然是。为什么会这样？

我们可以称之为无能，并以此结束。但那将是相当无知和无益的。

我认为做出这些决定有几个潜在的解释。我相当确信，人们做出这些决定是基于他们是正确的假设，甚至可能是一个“安全”的选择。如前所述；“如果它对谷歌或脸书有效，那么它也应该对我们有效”。

因此，问题的一部分就像之前强调的那样，这种懒惰的思维实际上弊大于利。这可能不是一个“安全”的选择，如果掉以轻心，这可能是一个错误和危险的选择。

但我在与人们谈论类似经历时也发现，可能有其他因素推动这一趋势。最近的一个例子是一个较大的丹麦政府项目，其中一家较大的管理咨询公司担任顾问，非常努力地推动基于亚马逊的基础设施和解决方案。这不一定是坏事，因为它可能是正确的解决方案。

但在这种情况下，例如他们推动亚马逊 DynamoDB 真的很难，因为它是未来，可以容纳任何东西。又是真的。但这并不是他们试图解决的问题的核心。他们所有的数据都是结构化的——几乎是尽可能结构化的。这是房地产领域的政府数据。它被仔细审查，审计和组织得非常好。更复杂的是，团队中没有一个人有使用 DynamoDB 的经验，所以每个人都必须参加几周的课程。再次增加了成本和复杂性。

是什么推动了这种决策？你是否应该戴上锡纸帽子，声称在上述案例中，管理咨询公司可能在推动某项技术时获得了经济利益？也许吧。但是为什么为政府工作的人没有提供足够的推进力呢？可能是，如果他们有能力剖析决策是糟糕的，那么他们一开始就不需要管理咨询公司。或者可能是一个非常糟糕的沉没成本谬误的例子，如果你反对管理咨询公司的建议，他们很难接受昂贵的账单。

我不知道这个问题的最终答案，但我觉得这种趋势有点令人担忧和悲伤。这看起来像是“快乐跑步机”的软件开发版本，一点也没有真正的进步。

在我看来，简单永远是赢家。你赢在减少复杂性，而不是增加复杂性。从解决问题的最简单的方法开始，看看它能给你带来什么。简单使它更便宜。简单的解决方案更容易实现，更容易测试，更快发布，从而更快获得反馈。一旦你有了这个反馈，不管是来自单元测试、概念验证还是用户测试，那么你就可以决定如果你的简单解决方案被证明太慢或者太初级的话，就增加复杂性。

总是从简单的解决方案开始。
