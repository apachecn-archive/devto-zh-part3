# 关于软件开发的三个神话，被经验增长打破

> 原文：<https://dev.to/psmyrdek/three-myths-about-software-development-busted-by-experience-growth-eid>

让我问你一个问题——成功的软件开发人员最重要的特质是什么？有经验的人可能会说“看情况”。确实如此，这一切都取决于我们的工作和生活经历。回到过去，当我开始我的软件开发冒险时，我会说成功的人必须知道一切，他们在“一个人的军队”模式下工作，他们可以解决你摆在他们面前的每一个问题。该死…我甚至没有接近正确的答案。

## 不是要知道所有的事情

知道所有的事情是关于应该被认为是成功的人的最流行的神话之一。在我开始软件开发的最初几天，所有的书籍、教程和指南都让我感到压力很大。“看起来成功的开发人员在他们的职业生涯中都吃这些东西”——这对我来说是非常明显的。这真的意味着成功的开发人员是多种编程语言的专家，深刻理解所有的模式和实践，参与所有类型的项目，并且对所有事情都有自己的看法吗？肯定有人符合所有这些要求，但这是一个规则吗？

我不这么认为。我有机会与各种类型的公司合作——初创公司、软件公司、外包公司——但没有一家公司会雇佣上述那种人。这是否意味着他们对聘请专家不感兴趣？当然不是！但是他们对专家的定义与无所不知的人大相径庭。它看起来怎么样？

我们经常称某人为“专家”,因为他们以“T 型”方式发展了他们的技能。在软件开发的一个非常具体的领域，这样的人被认为是专家，并且他们知道关于最重要的、最新的事实的广泛的主题。被认为是专家的人们正在考虑不依赖于特定编程语言或技术的通用实践和解决方案，而不是最流行的框架的品牌和标志。这种技能的分配帮助他们非常快速地转换技术环境，并以比经验较少的同事更抽象的方式思考。谈到同事，专家不能忽视那些经验不足的人——这就是为什么他们能够与来自不同部门的人讨论各种话题，在整个组织的完全不同的层面上工作。专家们绝对可以进入交流。

那么这里的规则是什么？我会说，专家不是知道所有的事情，而是非常了解一件事情，而其他事情都停留在他们感兴趣的领域。

就这么简单。

## 这不是单人游戏

如果你认为软件开发是一个单人游戏…那么，你可能需要重新考虑一下。但是好吧——找到这种信念的来源很容易。

对于我们来说，通过类似于关于超级英雄或战场上的士兵的电影来解释未知的活动是很自然的。好莱坞经常给我们举一个“xyz-man”拥有超能力的例子，由于这个例子，世界再次被拯救了。我们并不真正理解超级英雄力量的机制，但我们知道反派将在电影的结尾被打败。我们中的一些人认为开发人员拥有同样的超能力，由于这种超能力，他们能够独立解决各种问题。

这种观点有两个主要问题。首先，我们工作的“外部品牌”充满了错误的假设。其次，这可能会给开发者带来很多内部问题，他们觉得自己不够好，无法处理像好莱坞大片中主要角色这样的问题。

事实是，有经验的开发人员可以很容易地说“我不知道”，而不用担心这种说法会损害他们的声誉。

我不知道如何解决这个问题，这是我第一次看到这样的事情。你能看着它试着帮我吗？

我不知道如何解决这个问题，因为这不是我所熟悉的系统的一部分。你能帮我理解一下吗？

我不知道我的想法是否涵盖了每一个细节——我们能否尝试逐步解决和发现这个问题？

说“我不知道”并邀请其他人来帮助你通常被认为是成功的软件开发人员最重要的技能之一。没错——通过说你需要帮助，你证明了有比你自己的自我更重要的事情——例如，成功交付你目前参与的这个至关重要的项目。执行良好的团队合作可以减少解决给定问题所花费的时间，还可以帮助在团队中更平均地分配知识，这可以为利益相关者节省大量的金钱。

我们玩的是多人游戏，说“不知道”也没什么不好。

## 不是所有的事情都值得你关注

涵盖了两个非常常见的误区——成功的开发人员不需要知道所有的事情，并且他们不是在“一个人的军队”模式下工作。第三种情况更有趣——它是关于忽略不值得你注意的事情。听起来很自大，不是吗？别担心——这确实有道理。

对于一些开发人员来说，总是有足够的时间从他们的“待办事项列表”中勾选大部分项目，而对于其他人来说却是不可撤销的，这怎么可能呢？当然，我们都同意，所谓的“缺少时间”是我们周围的人最常见的问题之一，但这真的与缺少时间有关吗？我们不是都配备了同样的 8 小时工作日吗？

“缺少时间”问题的解决方案被称为“明确的优先级”。

就是这样。没有优先级列表，所有的事情都一样重要。这是知识工作者一直在努力解决的问题——我有一些知识，我能做各种事情，但我下一步该做什么？也许我应该做尽可能多的事情？也许我应该回复每一封电子邮件，因为这有助于我在公司赢得声誉？也许我应该处理我们积压的每一张票，而不需要根据我们的优先顺序来安排事情？不，这不是专家的工作方式。

你知道“帕累托原理”吗？观察表明，80%的结果来自 20%的原因。例如，公司 80%的利润来自 20%的客户。80%的销售归功于 20%的销售人员。80%最重要的问题由 20%的员工解决，等等。

对于编程的上下文，如何翻译这个规则？嗯，以这种方式考虑一下——知道我们在时间、预算和目标压力下运作的事实，试着找出这一件可以将你的项目向前推进 80%的事情。您正在使用的应用程序的哪个核心领域可以改进，从而为用户带来最大的价值？从项目整体成功的角度来看，你接下来要处理的哪个任务是最重要的？有没有可以推迟、委派给别人甚至放弃的任务？

时间是宝贵的，所以以最有效的方式使用它。这就是成功的开发人员尽可能做的事情。

几年前我在这里提到的神话被认为是理所当然的。对我来说很明显，我必须知道每一个与编程相关的问题的答案——否则人们会把我当成一个平庸的开发人员。同样的事情还有不知道的事情，承认这个事实，或者邀请其他人来帮助我。

这是一个健康的观点吗？当然不是。我们的环境真的是这样吗？谢天谢地没有！得益于不断增长的经验，我开始注意到我的期望与现实有多远。

谷歌首席执行官桑德尔·皮帅曾说，与我们自身的变化速度相比，IT 行业的发展速度令他担忧。很难说我们是否能够以与 GPU、硬件和人工智能相关技术相同的速度发展自己，但肯定的是，我们应该尽可能灵活。在变化如此迅速的市场条件下，我们自己的价值观和信念应该随着时间的推移而演变，这没有什么错。前一段时间，我们决定在一个敏捷的环境中工作，所以我们不能忘记敏捷也与我们自己有关。
