# 你的原型变成了你的产品

> 原文：<https://dev.to/checkgit/your-prototype-just-became-your-product-2c10>

## 现在，你如何处理科技债务？

> “我们以后再修。”

这不在计划之中。当项目开始时，每个人都同意我们从一个快速原型开始，然后清理成一个更易于维护的解决方案。以惊人的凶猛，你的团队在开发的每一个阶段都全力以赴，迅速做出了一个前所未见的原型，让你既骄傲又不安。每个人都分享了他们对精彩功能的兴奋之情，他们认为这些功能将会更上一层楼。该产品是成功的。等等，产品？这不只是一个原型吗？哦不。

让我们先澄清一些事情——这里的问题不是原型。你所感受到的是恐惧。你突然痛苦地意识到你积累的所有技术债务，而你却没有策略来处理。

> “很好。我们现在就去解决。”

现在问题已经清楚了，是时候解决它了。首先——沟通。代码质量是每个人的问题。真诚地行动，把这些问题公之于众是你的责任。然而，仅仅说有问题是不够的。你需要量化和交流它。为什么技术债务甚至是一个问题？这对你和你的团队有什么影响？用户肯定没有意识到这一点，那么如何传达这一代码质量问题不仅仅是表面现象呢？

代码分析工具对于洞察项目的健康状况是不可或缺的。诸如特性周转时间和回归频率之类的度量标准也可以传达问题，甚至可能更有效，因为它们对您与涉众的关系有直接的影响。

> "让自己参与到这个问题中去。"

与你的团队一起制定偿还项目债务的计划。一个策略是将每个周期的时间划分为特色工作和偿还技术债务。理想的比例取决于你的团队。挑一个，看效果如何，每个周期调整。这种策略的一个好处是，通过用管理代码质量的时间来填充周期，你给了自己一个缓冲区，以防有紧急的事情需要你立即注意。

当你创建债务时，记录你的债务。当你写代码的时候，避免债务是困难的，甚至是不可能的。一看到债务就承认:提出问题，记录下来。您可以从该目录中直接找到某个问题，而不是通过筛选代码来识别某个问题，从而节省时间。

技术债务并不总是显而易见的。如果你还没有，和你的团队一起建立一个代码评审过程。这些额外的眼睛会暴露出一些可能会被忽略的问题。你不可能一个人抓住每一个问题。

一个可靠的代码评审过程有一个额外的好处，那就是在整个团队中传播知识，否则这些知识可能会被个人所孤立。你的团队是你最宝贵的资源。利用这一点，你就会飞起来。

科技债务的影响是真实且可衡量的。即使是最好的工程团队也在挣扎。非常成功的团队和一般的团队的区别在于，最好的团队在过去都受过重创，并从中吸取了教训。也许你已经感受到了那种痛苦，也许这对你来说仍然只是一种假设。无论是哪种情况，只要你诚实地面对你在这个过程中所做的技术决策的影响，这个问题是不可避免的，也是可控的。
