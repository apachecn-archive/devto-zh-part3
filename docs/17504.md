# [](#why-im-running-for-tag)我为什么要竞选 TAG

(如果你想知道这个标签业务是什么，可以看看布莱恩的《大忙人初级读本。Alex 的[帖子支持我的提名](https://infrequently.org/2018/12/alice-boxhall-for-w3c-tag/)也有一点历史。)

2011 年初，正当我准备开始在悉尼的 Chrome 团队工作时，我碰巧听到了[杰森·怀特](https://www.jasonjgw.net/)的[演讲，讲述了他通过屏幕阅读器使用 Linux 和网络的经历。](https://www.youtube.com/watch?v=uJH16eBJEHc)

虽然不是我第一次刷 <sup id="fnref1">[1](#fn1)</sup> 与辅助技术用户的经验，但它确实让我有所触动。事实上，我已经准备好彻底离开科技行业了(我甚至被另一个领域的非全日制学位录取了，但那是另一回事了)，但杰森对通过屏幕阅读器使用网络浏览器的描述，以及艾瑞亚的工作方式，说服了我再试一次。我看到了一个对技术持乐观态度的理由，也看到了一个与残疾人合作的机会，让网络成为获得更大权力和自由的工具。

## [](#chrome-accessibility)铬无障碍

所以 8 年前，我带着或多或少的冷漠来到了无障碍环境。一开始我很幸运地被介绍给多米尼克·马佐尼(无畏号的创造者)！)，他已经在 Chromium 代码库中做了大量的工作来集成 WebKit 的可访问性实现。Dominic 的可访问性哲学给我留下了深刻的印象——从根本上讲是务实的，对有可访问性需求的开发者和用户都有同情心，并且总是面向未来。

随着 Chrome 和 Chrome OS 可访问性团队的成长，Dominic 在确保我们的团队代表大部分用户方面发挥了巨大作用。David Tseng 很早就加入了我们，他是当前版本 ChromeVox 的首席开发人员，现在是我们 Chrome OS 可访问性工作的技术主管。

我们的整个团队现在包括使用屏幕阅读器的高级开发人员、放大和高对比度用户，以及强烈喜欢使用替代基于指针的输入的人，还有没有特定的可访问性需求但对团队使命有强烈信念的队友。我每天都从与如此多样化和有才华的一群人一起工作中学习。

## [](#learning-the-ropes)学习诀窍

我最早的项目之一是 Accessibility Developer Tools 扩展和库，旨在将可访问性检查和调试集成到 Chrome DevTools 中。这迫使我真正投入到 WCAG 2.0 中，开始理解网络上的可访问性这一广阔的、偶尔有点零零碎碎的领域。(事实上，我已经给出了一个关于对比度、颜色表示和可访问性开发工具中的“建议颜色”特性的[整体演讲](https://docs.google.com/presentation/d/1wNR8g-mhAcbU_g-5S4FJ749b4Tn5KhIw3zA-RFunylo/edit#slide=id.g63d390087_1_136)。)

大约在同一时间，Web 组件项目真正开始了，我能够与 [Polymer](https://www.polymer-project.org/) 团队一起工作，并帮助他们确保他们的原始组件套件满足可访问性的高标准。这项工作让我直接看到了 web 作者面临的一些挑战，尤其是 web 组件作者。

在谷歌之外的世界里，我很幸运地从许多为网络标准做出贡献的易访问性专家那里学到了东西，也从其他通过演讲和文章分享他们知识的人那里学到了东西。我是莱奥妮·沃森(Léonie Watson)的特别粉丝，她长期以来一直为 web 标准做出贡献，并一直就前沿的可访问性主题发表一流的演讲。詹姆斯·克莱格是另一个杰出的盟友，他的实用主义和解决方案重点帮助我形成了可访问性的方法。

## [](#developer-experience-and-potholes)开发者经历与坑洼

我认为我的首要任务是改善开发者对网页可访问性的体验——将工具、API 和教育结合起来，帮助实现网页“默认可访问”的理念。正如大多数易访问性专家所知，“易访问性”是一个不可及的移动目标——没有什么是“完全易访问的”。随着用户体验和前端开发模式的不断发展，我们需要与它们一起发展易访问性技术。同时，可访问性领域的研究和开发推动了新的可访问性最佳实践的创建。

在过去的几年里，我越来越多地参与到 Web 标准中。我希望它能成为一个有效的杠杆来“提升”网页的可访问性。

在我的[标签选举声明](https://www.w3.org/2018/12/03-tag-nominations#ab)中，我写了我的信念“大多数 web 开发人员都想创建可访问的体验，但不幸的是，编写代码的最简单的方法可能会导致不可访问的体验”。我一遍又一遍地看到这种情况，比如 flex order、floats、禁用焦点轮廓的重置样式表、单击和按键需要冗余事件侦听器、手动滚动模式 UI、阴影 DOM breaking label 和 IDREF 关联、tabindex 灾难、display: contents 等。等。等。关于可访问性的教育有时感觉就像分发标有坑洞的路线图，以及如何用觅食的物品从头开始建造小桥的说明。

我认为我们可以做得更好。我认为易访问性专家会喜欢停止帮助人们绕过坑坑洼洼，并开始帮助人们创新真正令人愉快的、包容性的用户体验。

## [](#doing-better)做得更好

在我正在从事的项目中，[可访问性对象模型](https://github.com/WICG/aom/blob/gh-pages/explainer.md)项目旨在修补 Web 组件故事中的一些可访问性漏洞，并为开发人员提供更具表现力的 API 来声明其内容的语义。 [`inert`](https://github.com/wicg/inert) 旨在给开发者更简单的工具来管理焦点。 [`focus-visible`](https://github.com/wicg/focus-visible) 旨在鼓励设计师采用大胆、醒目的焦点风格，而不是隐藏或最小化它们。

但我认为我们也需要避免制造新的坑洞，这就是为什么我希望能够在标签上工作。我作为浏览器开发人员和可访问性倡导者的经验给了我一个难得的视角，我想把它带到新生的 Web 标准中，并帮助标准工程师理解他们自己工作中的可访问性考虑。

[未来可及](https://www.bonfire.com/thefutureisaccessible/)。包容残疾人对每个人都有好处。我们越能改善网络的基础设施，我们就越能推动残疾人参与网络标准，并为可访问性和可用性创造良性循环。我希望有机会帮助推动标准朝着这个方向发展。
