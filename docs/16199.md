# 我跟随其他开发人员一周所学到的

> 原文：<https://dev.to/matmooredev/what-i-learned-shadowing-other-developers-for-a-week-5eeh>

这是新的一年，我刚刚开始一份新的工作，在公务员系统的一个不同的部分。

我是周一加入的，在我见到我的新经理后，他建议我在本周余下的时间跟随不同的团队，以此来更多地了解这个部门。

我以前从未正式跟踪过某人，所以当我弄清楚这个地方是如何运作的时候，我也在弄清楚如何最大限度地利用跟踪。这些是我一周后的外卖。

# 蔽是辛苦的工作

跟踪有很多好处。这给了我一点支持网络，否则我不会有，并减少了跨团队合作的障碍。我看到了多个团队如何开发和操作他们运行的服务，这给了我一个向我自己的团队介绍新实践的公共参考点。我还很好地了解了各个团队的不同之处以及我们的共同问题。

也就是说，这是非常紧张的一周，结束时我筋疲力尽——作为一个天生的内向者，我发现在如此短的时间内会见如此多的新人是非常令人疲惫的。如果我再次遇到这种情况，我宁愿先花一些时间和我的指定团队在一起，这样我就可以适应我的角色，并对什么是有用的信息有更多的了解，而不是试图吸收尽可能多的信息。

# 准备好提问

这周我问了很多问题。对我来说，一个巨大的文化冲击是发现这里人们日常使用的大量缩略语，这超出了我习惯的标准公务员行话。这几乎就像学习一种不同的语言。显然我不是唯一一个有这个问题的人，因为[有人为它做了一个应用](https://bigbookofacronyms.herokuapp.com/)。多次问东西是什么意思感觉有点尴尬，但当你是新人时，我想你必须习惯它。

除了首字母缩写词之外，还有很多人们认为理所当然的领域知识，所以我需要问很多看似愚蠢的问题。我被介绍给许多不同的系统，它们有着复杂的交互，我发现很难跟踪什么是什么。看到东西如何组合在一起的图表是很有用的。对我来说，这也有助于提出关于用户和正在完成的任务(输入和输出)的澄清性问题，而不是一些数据在中间是如何处理的。

# 不同的人可以教你不同的东西

我不是以同样的方式和每个人一起工作。我交谈过的一些人比其他人更健谈，他们有自己的想法给我看。在其他情况下，我与开发人员一起完成他们正在进行的任务(尽管我更多地扮演了导航员的角色，而不是编写代码)。我认为这两种方法都很好——价值在于理解不同的团队做什么以及为什么做。

有一件事很有帮助，那就是当我跟随的人问我想从这次经历中得到什么——这让我们达成了共识，也更容易提出我感兴趣的其他话题。你也可以转而问“如果你是我，了解/研究什么是有用的？”

我交谈过的许多人能够向我展示系统如何配合的图表，我还看到了一个呼叫中心操作员实际使用其中一种服务的视频，这非常有趣。

# 找事情做单干

我发现给自己设定几个低优先级的任务很有帮助。这意味着，如果我跟踪的人有会议或其他需要做的事情，他们仍然可以做，而不会觉得他们必须整天照顾我。周五，我跟踪的开发人员有 20%的时间学习，所以我花了一些时间学习 Spring，这是一个我以前没有使用过的框架。

我还试图尽可能地避免管理方面的事情，并编制了一份新的起始任务列表，试图为下一个人改进这个过程。

当我加入我的新团队时，我将专注于使软件更容易更改和部署，所以我本周做的另一件事是获得最佳实践的电子表格，并根据我从其他团队学到的知识开始优先考虑一些事情。
