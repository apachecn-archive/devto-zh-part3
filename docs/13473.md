# 合作代码评审

> 原文:[https://dev.to/jessekphillips/partnered-code-reviews-3pi8](https://dev.to/jessekphillips/partnered-code-reviews-3pi8)

你熟悉代码审查吗？您是亲自进行评审还是通过变更请求工具(合并/拉动)进行评审？当你讲述你所做的改变时，有人站在旁边对你有什么好处？

Git 常规提交有助于在没有合作伙伴的情况下进行合作代码评审。当决定如何组织你的变化时，考虑这些问题/考虑事项

*   我在改变什么，为什么
*   这会帮助我告诉别人我刚才做了什么吗
*   这是否包括所有需要的更改，以便审查可以确认所有需要的组件(尤其是那些相互通信的组件)已经被正确修改。
*   如果恢复，这有用吗
*   为了恢复，我需要将许多提交分组在一起吗
*   这个提交应该上行吗

第一个问题是代码评审让你问什么，以便向你的伙伴解释。

# [](#reality)现实

你所犯下的错误的实际结果将会很大程度上来自于违纪、责任和懒惰。我相信有很好的理由做出例外，因为开发工作可能会很混乱，但有一个接近的目标将是有益的，因为你首先要问自己，为什么这是一个例外，并对这个答案感到满意。
