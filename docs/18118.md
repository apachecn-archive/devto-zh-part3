# 个人开发的动力持续不了，做不完。 试着考虑了原因和对策。

> 原文:[https://dev.to/nabettu/-20lm](https://dev.to/nabettu/-20lm)

[有一个叫运营者公会](https://twitter.com/Neko_Inu_/status/1063308508718755840)的，个人或者是用少数人团队制作 Web 服务和 APP 的人的灌篮团队，在那里和[furuta](https://twitter.com/furuta_app) 和[naichi](https://twitter.com/naichilab) 聊天

个人开发是**可以用**【自己想做的东西】**制作成**【自己的喜欢】**的**最好的舞台**。 如果是创作者的话，就没有办法不做了吧。**

 **但是，如果什么也没想就踏入去的话，就会发现是比想象中更崎岖的道路。

明明不是工作而是“自己想做而做的”，却不知不觉间有了逃到某个地方的动力。 过了几个月也做不完的现代沙拉家族。 只是因为工作忙得一个星期没照顾好，就荒废到可以说“什么，这该死的代码”的 master 分支。 那个时候的目标是“中了一枪，平时白天就用温暖的被炉顶着 lady bouden 的自己的样子”到底要去哪里呢？

那样令人目不忍睹的状况，我也有。 回顾一下**“为什么会变成这样。 ”为了不成为**，我们考虑了原因和对策，希望能作为大家的参考。

# 构思太大了，做不完

首先是这个。

“我想到了一个好主意！ ！ ！ ”他情绪高涨，提出“既然如此，那就租一台昂贵的租赁服务器吧”(本来租就在公开之前就可以了)，这是一个没有注意到的陷阱。

尺度很大是件好事，但真的是**自己一个人就能做好的尺寸感**吗？

## 一个月就做好吧

如果是职业程序员的话，总觉得应该能计算自己的工时。

如果是将一个月的空闲时间全部投入进去也做不出来的话，还是重新考虑一下比较好。 几乎肯定是高迪。

## 要削的地方要削，先打出 MVP

能做的人很好，但不是的人先考虑一下一个月就能做好的 MVP 吧。

例如，如果要制作 Twitter 这样的服务的话，先把登录、主页和关注功能都放在一边，只制作投稿画面及其一览画面，看看会是什么样的感觉。

登录后，在主页上编辑自我介绍文，制作可以更改用户图像的图像上传功能，在这样的过程中大概会有**的动力**。

先把这个服务中最核心的部分做清楚吧！

## 切成可以公开的部分，只制作该部分进行用户测试

像这样做了 MVP 之后，给熟人看看吧。

因为没有首页的说明，所以请自己粗略说明后再触摸投稿画面吧。

“不太清楚啊。 hahaha”或“这和 twitter 有什么不同？ 可能会被这么说而消沉。

但是“咦，也可以这样使用呢”“相反，如果只能嘟囔[的话](https://nyaaan.haramishio.xyz/)？ ”之类的意见可能会得到。

为了不让动力溜走，一点一点也可以，所以听到这些积极的声音，不要停下脚步。

## 计划好制造时间

接下来，为了防止我们继续前进，为了防止懈怠，“一个人开发集训！ ”或者一边喊着“在商务酒店只带电脑住一晚”之类的，也许也可以试试。

总之，不确保时间是不会有进展的，**这不是为了向上司献媚而举行公司忘年会的时候**哦！

# 技术上很累

接下来试着做了一下，发现比想象中要难，“结束 Progate 和 Rails 教程的我应该没有做不到的东西。 。 。 会产生”这样的心情的家伙。

请把这里写的东西记在脑子里，防止心碎于未然。

## 尽量不使用没有经验的技术，即使使用也只使用一个吧

因为基本上好像全部都没有经历过，所以想要做的是个人开发的反模式。 不是特别想做的东西是做不完的。

这里聚焦目标，定为稍微挺直一点就能做出的水平，宏伟的构想以后再说吧。 因为“千里之行也一步”。

## 技术掌握和新开发尽量分离吧

即便如此，如果最后还是做不出自己能做的范围内的东西也没用的了不起的东西的话，**技术学习和新开发尽量分离**吧。

这是为了防止“如果不把学习作为里程碑的话，在开发中学习新技术的过程中，开发看起来完全没有进展，内心会生病”。

关于逐渐掌握的技术，我想最好另外制作技术掌握检查表，可以看到与开发的 todo 不同的积累。

另外，如果是“为了学习技术而制作”的模式的话就不太担心了。 因为即使做不到，只要能掌握技术，目标就能达成。

## 即使是肮脏的代码，也要先制造活动的东西再重构

“与其追求完美，不如先结束它。”

是啊。 先写下能动的东西吧。

## 动态部分也是静态的，暂时制作

先写一下之后可能会输入的数字什么的，先写下来然后再修改吧。

与其拘泥于细节，不如先将其全部使用！

如果是另外做设计的话，那个部分已经贴上图片结束，之后再用代码写也不错。

## 暂时只对应 PC，只对应 iOS

积极应对之类的还早着呢。

在这个时代，一旦制作了能用智能手机看到的东西就好了！

## 完全不写 CSS，只先做核心部分

举起“写 CSS 的辣手”的人手

先吐槽一下 bootstrap 或者 CSS 框架，然后再写吧。 是吗？

## 外协

我一定会做的！ 在这种情况下，自己不擅长的部分就外包吧。

还有比如[谁都可以做猜谜的服务](https://quiz-maker.site/)之类的，在做 CGM 的时候，试着把内容外包，即使有 10 个问题，人做的数据也会有那种感觉。

# 在制作期间先出现了类似的服务

还有这个。 不管是个人还是工作都有吧。

因为 Web 服务一般都是“公告栏”。 牡蛎的话就一起了哦。

## 因为有很多人想同样的事情，所以我不介意

“谷歌是第 18 个搜索引擎”

今天就记住这些吧，回去吧。

如果用 Web 服务这样狭窄的框架来考虑的话，那就太糟糕了 w

## 即便如此也不能成为完全相同的服务

我想即使功能相似，思想也不会一致吧。

facebook 和 Twitter 终究只是写下来作为时间表，但不知什么时候却有了这么大的不同。

思想不一样的话，方向也会改变，所以我觉得好好想想自己的想法再回头看就好了。

# 在队伍里做，其他成员不动的 or 自己的负担很大

虽然这不是个人的情况，但是每个人能调动动力的时间、成本和风险都不一样。

## 缩小队伍

觉得麻烦的话就快点解散吧。 这是浪费时间。 再想到什么的时候聚聚不就好了吗？

如果有想继续下去的成员，不要只考虑届时谁会带回某项资产。

# 虽然做了，但是没人用，很伤心

虽然这是这次做完之后，但是虽然努力制作了很多东西但是没有用，很伤心。

但是把一个东西全部做出来是很厉害的，所以如果有干劲的话，为了不浪费至今为止的努力，再努力一点吧。

## 自己总之要用

首先不是“想要自己才做的”吗？ 试着用一下，如果有一点不错的地方可以再给朋友看看，试着用了很多之后也会想“啊，也许可以这样使用”或者“横向展开可以这样做”。

那个在 twitter 潮流中偶尔出现的[APP☆制造商](http://appli-maker.jp/)，在最初流行之前好像也花了好几个月一直自己制作内容哦。

## 总之做自己想要的部分

一直使用的话，下次应该会变成“我也想要这个”“那个也想要”。

即使做自己不能成为第一用户的服务也完全没有意思，所以不断地为自己制作功能吧 w

最坏也可以成为为自己提供的最好的服务不是吗？

# 总结

就这样，我写了我的想法，就是防止个人服务的积极性问题于未然，即使发生也要想办法。

虽然是个人能做的东西的范围逐年变大的网络行业，但是华丽地装饰出道的服务的背后有很多折断的心。

能做和动力无关的人很好，但“我想做 Web 服务！ ”我希望人们在挑战之前能稍微用心一点，不要折断心，把它做好！
