# 内容网的兴起

> 原文:[https://dev.to/robmarshall/the-rise-of-the-content-mesh-igf](https://dev.to/robmarshall/the-rise-of-the-content-mesh-igf)

2018 年 10 月 4 日，Sam Bhagwat 在 gatsbyjs.org 上发表了一篇名为“[交付现代网站体验:内容网格之旅](https://www.gatsbyjs.org/blog/2018-10-04-journey-to-the-content-mesh)”的文章。Sam 介绍了内容网格的概念，并谈到了所涉及的不同部分。

实质上，内容网格允许您(来自 Sams 文章):

> 选择适合电子商务或博客等特定使用案例的最佳解决方案；它为快速迭代提供了一个现代的框架，并且它生成了开箱即用的闪电般的网站。

那是一句激动人心的话。

在不同的项目中，我曾多次想要使用这个平台的一部分和那个服务的一部分，但都被捆绑到一个没有灵活性的预设 CMS 上。看起来我们有解决方案了！

## [](#why-hasnt-content-mesh-been-possible-before)为什么以前内容网格还没有可能？

说实话。一直都是。

REST APIs 已经存在多年，web 开发人员已经能够将第三方服务集成到网站中。Google Analytics 是一个很好的例子，它是一个可以放入网站并在其他地方管理的脚本。我不认为这是一个“不可能”的案例，而是它没有被广泛考虑。它现在受到关注的原因是因为以下几点。

### [](#dedicated-web-services-for-specific-usecases)针对特定用例的专用 web 服务

完全成熟的 CMS 服务仍然存在，它们在不断地被改进和建立。然而，微服务是为了做一件事而建立的，而且要做对。这意味着正确的工具总是可以用于这项工作，而不是破解一个已有的功能来为你做这件事(你必须热爱 WordPress 开发…)。

以电子商务业务为例。产品需要管理，支付需要进行，还有博客、CRM、用户数据存储和分析。

### [](#one-size-fits-all)一刀切

在过去，我会引导客户使用单一的方法，比如万磁王。它处理所有的事情(这在一开始是很棒的),但是开发和改变受到软件的限制。当不断推出最终违背内置功能的变更时，这就成了一个问题。

### [](#so-microservices)这么微服务？

现在情况不同了。每个需求都有服务！

*   产品管理:[语义 3](https://www.semantics3.com)
*   支付:[条纹](https://stripe.com)
*   用户数据: [Auth0](https://auth0.com/)
*   博客:
*   CRM/分析:[细分市场](https://segment.com/)

这些都可以通过各自的 API 调用，并显示在中央站点上。

这些服务中的每一个都做特定的工作，而且做得很好。他们有全面的文档，和一船已经写好的代码，可以放到网站上。例如，管理产品的人很可能不是写博客的人。将这些区域分开意味着更少的干扰，以及更干净、更精简的网站管理方法。

这种微服务方法有几个好处:

*   如果一项服务出现故障，其余的仍然可以运行，这就允许故障隔离
*   更强的安全性–每个服务都有自己的凭证
*   易于集成和部署–NPM、Netlify 等
*   不局限于特定的堆栈–可以尝试其他服务
*   网站可以很容易地进行扩展——如果一项服务不能充分发挥作用，就移除并替换它
*   更容易理解——每个工具都解释了它的作用，并且只解释了它的作用

### [](#an-evolution-of-workflow-and-techniques)工作流程和技术的演变

旧的整体服务通常建立在旧技术之上，开发过程较慢。需要添加功能？那么看起来你正在编写一个大的 PHP 函数，这违背了软件的原始设计。需要维护本地环境？让团队成员做出改变需要时间。

这已经被更现代的技术解决了，例如 React 和 Vue。这些 JS 框架实现了快速启动时间，启动一个新项目非常容易(想想 create-react-app)，以及一个难以置信的大社区一直在推动改进。JS 的开源方面很棒，问题被解决、打包并存储在 NPM 上。

这些技术与敏捷方法(或类似的、敏感的主题)相结合，意味着想法可以在眨眼之间变成现实(取决于互联网速度)。web 开发社区已经接受了这个“打包”的概念，并且正在真正地使用它。越来越多的企业开始理解推出 MVP 并以此为基础的想法。在工作流程中引入 Sprints 和 weekly roll outs。这种思维模式需要一套更灵活、更可扩展的工具。生产速度取决于它。

正因为如此，微服务能够很容易地到位，这意味着内容网格的到来。由于被视为 NPM 上的另一个包，它就像在控制台中运行一个简短的命令并包含到代码库中一样简单。大多数复杂的事情都由 Webpack 处理。一切都非常接近即插即用。

## [](#so-what-next)那么接下来呢？

显然，每个项目都有自己的一套要求和限制，但我认为内容网格策略可以适用于几乎所有情况。一旦为您和您的团队创建了一个基本框架，就可以很容易地引入服务。您可以替换“一种尺寸不太合适”的解决方案，使用最有效的解决方案。目前似乎只有 GatsbyJS 提供了开箱即用的集成，但用不了多久，其他公司也会加入竞争。内容网格的概念意味着最好的解决方案都可以使用。这绝对是正确的方向。
