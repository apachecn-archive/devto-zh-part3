# 自我评估的技巧

> 原文:[https://dev.to/annarankin/tips-for-self-reviews-29ce](https://dev.to/annarankin/tips-for-self-reviews-29ce)

对我和大会上的其他人来说，这是一个绩效评估的季节——一年中自我反省的时候(伴随着许多其他的情绪)。每个人对待评论的方式都不一样:战战兢兢，兴奋，烦恼，冷漠。我努力期待与我的经理和队友分享反馈——有一个过程，以及定期、专门的时间来反思个人进步和表现，以对我有意义的方式来衡量，这很好。

虽然不是所有的回顾过程都包括我在下面谈到的步骤或方法，但我相信一般的观念(诚实、善良和专一)仍然适用。我不打算谈论给别人反馈(这篇文章已经够长了😂)，而是把重点放在自我评估、目标设定和充分利用你得到的反馈上。以下是我多年来在这个过程中发现的一些有用的技巧——包括什么，在哪里寻找关于你表现的重要信息，以及如何以有价值的方式使用你得到的反馈。

在 GA，我们的绩效评估周期如下:

1.  员工写下自我评估和对经理的评估
2.  同事之间互相评估(匿名)
3.  经理评估他们的直接下属(同时撰写他们自己的自我评估)
4.  经理和员工开会讨论所有这些反馈，并讨论下一步措施

## [](#self-review)自我检讨

可怕的自我检讨！😱即使有具体的问题要我回答，这对我来说也是这个过程中最难的部分。我通常会创建一个要点大纲，并在进行过程中填充相关的支持细节，而不是将我的回答写成段落。

### [](#calling-out-the-good-stuff)召唤出好东西

像许多开发人员一样，我每天都在处理冒名顶替综合症。最初，我很难不焦虑地说出我的优势和成就(或者在下一句话中立即减少它们)。当我列出我引以为豪的事情时，我仍然害怕被认为傲慢或过于自信。一些真正帮助我的事情是:

#### [](#find-concrete-records-of-your-past-work)找到你过去工作的具体记录

*   如果你使用像 JIRA 或特雷罗这样的系统来跟踪你做了什么，你可以用它来反思你已经完成的工作。六个月内会发生很多事情——很有可能有一些你忘记藏在单子里的成就！

*   大约一年半前，我开始在一个 markdown 文件中跟踪我的日常待办事项(我使用 [Bear](https://bear.app/) ，但我曾与使用谷歌文档、苹果笔记应用程序和 Evernote 的人交谈过，取得了巨大的成功)。我也用这个文件来跟踪“肩拍”(每天要求帮助或澄清)、会议和其他任务，以及我花在这些事情上的大概时间。当我思考我花了大部分时间在做什么的时候，这种低层次的细节对我来说是有趣和有用的(并且是查看模式的好地方！).

*   如果你没有一个强大的工作跟踪系统，试着查看与工作相关的电子邮件和过期文件。有可能那里有足够的东西来唤起你的记忆！

#### [](#review-past-work-artifacts)回顾过去的工作神器

*   这主要适用于您创建文档或整理提案/项目简介的情况，但也适用于您创建的 pr 和您帮助实现的功能。

#### [](#look-at-new-skills-youve-picked-up)看看你学到的新技能

*   也许你转换了你项目的前端从 jQuery 反应过来或者调试了一个毛数据库问题；也许你第一次开始做服务器逻辑。你第一次做的任何事情都应该被记录为一次胜利！

*   超越技术成就。任何影响你作为员工的表现和价值的事情都是公平的——领导一个项目、发表演讲、参与导师计划或写博客(😉)都是影响你整体表现的重要因素。

### 对需要增长的领域保持诚实

审视自己需要改变的部分可能是一种痛苦和谦卑的经历，但对你的个人成长绝对是必不可少的。知道什么需要改进是至关重要的，这样你就可以更好地将你的努力集中到下一个层次(个人的和专业的)。丹·阿布拉莫夫(Dan Abramov)最近的帖子[“2018 年我不知道的事情”](https://overreacted.io/things-i-dont-know-as-of-2018/)是一个非常酷的例子，说明一个非常成功的开发人员评估他的技术知识差距。有了你想做的事情的具体清单，选择目标就变得容易多了。

不去纠结自己的缺点也同样重要。像任何 bug 一样，应该承认它们，对它们进行分类，最终应该制定一个计划来解决它们。以下是我过去关注的几个类别:

#### [](#technical-skills)技术技能

*   你希望对代码库的某个部分了解更多吗？也许您对您团队的部署设置感兴趣，但是对发生的事情只有粗略的了解。您想学习应用程序架构技能，还是觉得自己对日常使用的工具缺乏了解？你想升级的任何东西在这里都是不错的选择。
*   请记住*要非常具体*——“我不擅长 Ruby”或“我想在部署方面做得更好”不如“我想在高层次上提高我对 OOP 的理解，并在实践中做出更好的设计决策(这里的例子)”或“我想了解我们如何自动化部署以及将新代码推向生产的步骤。”

#### [](#workflow-improvements)工作流程改进

*   每个人(嗯，几乎每个人)都说他们需要在“时间管理”方面做得更好深入探究并质疑这是否真的是*的问题。也许这实际上是分心的问题，你如何跟踪你的时间，或者你如何计划你的一天。*

*   思考开发过程中你经常感到摩擦的部分。例如，我花了一段时间来及时地审查和合并我的 PRs。我承诺不要让超过两天的时间在公关上没有任何进展，我很自豪地说，我在提醒我的同事我需要他们的反馈时变得更勇敢了，在迅速采取行动方面也变得更好了。

#### [](#leadership-amp-mentorship)领导&师友

*   如果你对领导团队或项目感兴趣，考虑在你的评估中提出来。承担哪怕是一个很小的领导角色，也是向你的同事学习和展示能力的有力途径。例如，我承担了在工作中发起和领导我们双月一次的 JavaScript“委员会”会议的任务。起初我感到害怕和焦虑，但是在一些格式失败和实验之后，我们设法找到了我们的最佳状态！我喜欢知道这些会议对我的同事有真正的价值(这也是让他们有价值的巨大动力)。

*   导师对我来说是一个非常个人化的目标。如果你对教育感兴趣，或者想帮助指导这个领域的新手，你绝对应该设定一个目标。也不需要很正式！我从“非官方”的指导和被指导中学到了很多东西，回馈技术社区也是非常令人满足的。

## [](#setting-goals)设定目标

所有这些反思(结合你从别人那里得到的反馈)毫无意义，除非你用它做点什么！一旦你有时间处理(并仔细检查，如果适用的话)你收到的任何反馈，就和你的经理谈谈你未来六个月的工作重点。根据您、您的同事和您的经理在评估中列出的内容，设定三到五个具体目标。好的目标是:

*   **具体。**模糊的目标比有具体验收标准的目标更难完成(很像工作票🤔).考虑将“更好地学习 Python”(举例来说)替换为你想朝这个方向迈出的一步，比如“每月参加两次 Python 聚会”或“完成我三个月前购买的高级 Python 课程。”

*   **专注。为自己的成功做准备，而不是试图改善你能想到的每一件事。“如果一切都很重要，*没有什么是重要的”这句话在这里很适用。给你选择的目标以它们应有的重要性，给你自己足够的带宽去实现每一个目标。***

*   **个人有意义。**避免仅仅根据你的经理/另一方认为你应该做的事情，或者你认为你*应该*选择的事情来选择目标。如果你的目标不是真正属于你的，你将缺乏完成它们的动力(并且它们更有可能因此而失败)。更糟糕的是，你选择把时间投入到你可能不关心的事情上，而不是你真正感兴趣的事情上。(这里有一个警告:如果你的经理告诉你你*需要*去做某件事，你绝对应该去做——并且努力理解它对你日常工作的重要性。)

*   走出你的舒适区。在设定目标时，不要害怕给自己一点压力。如果你设定了雄心勃勃的目标，你可能会惊讶地发现你在一个回顾期内可以完成什么！不过，一定要和你的经理一起确保这些目标是可以实现的。例如，对你来说，在接下来的四个月里做四次演讲，或者阅读十本新的编程书籍(也许是这样的！).只要确保你有足够的时间和支持去实现你的目标。

## [](#at-the-end-of-the-day)在一天的最后

无论你的绩效评估经历如何，或者你公司的流程是什么样的，我希望这些建议对你有所帮助。值得注意的是，这个建议在拥有健康、坦诚的反馈文化的组织中最有效。与任何建议一样，请谨慎使用，并注意您所处的环境。
