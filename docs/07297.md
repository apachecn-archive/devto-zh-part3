# 领先程序员的有点傻的历史——第二部分

> 原文：<https://dev.to/justzeros/a-slightly-silly-history-of-leading-programmers-part-2-1eil>

**电脑，进入左舞台……**

*[(错过 part 1？在这里阅读。)](https://dev.to/justzeros/a-slightly-silly-history-of-leading-programmers-15d8)*

当计算机首次亮相时，数学家们被请来处理这些复杂的机器。但是很快数学家们就厌倦了，回去做数学家们做的事情，把计算机留给电气工程师去处理。没过多久，这些人就决定这不适合他们，于是“电脑人”就被叫了进来。

当然，除了花时间在电脑上，没有人知道一个“电脑人”做了什么(我们现在将省去愚蠢的引语，你会明白的)。他们热爱数学、逻辑和电学，但既不是数学家、逻辑学家，也不是电气工程师。这些计算机人员是一群奇怪的人，虽然你可能认为他们会相处得很好，但随着时间的推移，裂痕出现了。但那是以后的事了。

现在，当计算机出现在办公室时，经理们被期望管理机器和伴随它们的计算机人员。然而，经理们发现，与电脑人交谈几乎难以理解，缺乏基本的社交技能，而且气味难闻。更糟糕的是，计算机人员没有正确地尊重和认可工人->经理->管理阶层！他们以为做一个“电脑人”就胜过了一切！

经理们试图通过他们的标准策略让计算机人员遵守规则:威胁、截止日期、告诉他们做什么，以及大喊大叫。虽然出现了一些暂时的改进，但这些策略并没有很好地发挥作用。经理的工作受到了阻碍，因为计算机人员所做的工作看起来像是秘书工作(打字)、维护工作(打开昂贵的设备，上面贴着“打开这个盒子，保修失效”之类的标签)之间的交叉)，还有建筑工作(用铁丝爬过阁楼。)

经理们意识到他们工具箱里的工具对计算机人员来说没什么用处。一些经理继续使用这些旧工具，导致计算机人员离开去不同的公司。这不是问题，因为计算机人才在这个时候需求量很大，而且找新工作很容易。公司对计算机的依赖无助于问题的解决，而且似乎膨胀了计算机人的自尊心，让经理和主管们都很沮丧。

一些经理试图忽略计算机人员，希望他们会做正确的事情，而不是把事情搞砸。至少这种方法没有赶走计算机人，并且取得了一些有限的成功。但是，随着他们的业务越来越依赖这些人，简单忽视他们的方法承担了越来越多的风险。

高管们认为经理应该能够管理计算机人员，因为说实话，这有多难？他们让经理们去上数据处理课程，学习穿孔卡片、BASIC 和 COBOL 语言，以及其他一些杂费。管理人员不需要做任何工作，只需要得到足够的行话来正确地告诉计算机人员该做什么。这将恢复事物的自然秩序。

这产生了两个令人惊讶的效果:

1)一些经理讨厌培训，这让他们觉得自己很愚蠢，因此更加讨厌计算机人员。

2)一些经理喜欢培训，并开始同情计算机人员，看到他们的工作有多么困难(和不同)。他们仍然没有足够的知识来理解计算机人员在谈论什么，但现在他们更加犹豫要不要做上层管理人员所要求的大喊大叫和恫吓。

不出所料，这两种影响都没有导致对计算机人员的适当管理。

最后，一位恼怒的主管想到了一个绝妙的主意！如果他们不能教经理计算机，他们会教计算机人员管理！毕竟，管理是一门科学！

他们推断:

1)电脑人知道如何与其他电脑人交谈

2)这些新经理应该能够准确地告诉计算机人员该做什么

3)新经理应该能够判断他们是否做了，并在必要时适当地大喊

现在的问题是，哪一个计算机人员应该被提升？执行官和经理们调查了计算机人员，他们像未洗过的群众一样挤在一起，紧紧抓住那台巨大的灰色机器，就像他们的个人自由女神像。

高管和经理们认为，他们应该提拔“最好的”计算机人员，不管“最好”是什么意思。观察了一会儿后，他们注意到一个计算机人员在机器前花的时间比其他人多，而且其他人经常问他们问题。他们有了新经理！

他们带着这个幸运的人离开了电脑，躲在紧闭的门和天鹅绒的绳子后面。他们给他们买了一条领带，教他们秘密握手，并让他们去参加经理培训，学习如何用经过实践检验的真正的命令-控制策略进行正确的管理。

深深地舒了一口气，经理们和高管们回到各自的会议室庆祝他们的胜利。他们完成了不可能的任务。

或者……他们有吗？
