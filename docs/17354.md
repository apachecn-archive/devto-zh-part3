## [](#my-product)我的产品📷

我的想法是建立“Skok”，一个(桌面)照片管理应用程序，可以扫描照片目录，将结果保存在数据库中，并以不同的方式进行查询。最重要的是，它将显示这些照片，并为更大更好的功能奠定基础，包括:

1.  将照片移入更好的目录结构
2.  基于文件外 md5 哈希的照片重复数据删除
3.  能够选择照片并复制它们
4.  基于 EXIF 数据在地图上显示照片
5.  “在这一天”功能，显示在这一天全年拍摄的照片

这个应用程序的想法源于一个我觉得市场没有满足的真实需求:以一致的方式自动组织照片，重复数据删除，以及移动/查询照片数据。最重要的是(也是我产生这个想法的原因)，桌面没有“时间跳跃”,我真的希望如此。

## [](#the-best)最好🎉

*   最小范围
*   事先对产品进行规划/思考
*   使用您知道或至少已经测试过的堆栈。

在一个超级紧张的期限内，最大的困难之一就是范围。范围蔓延是真实的，错误地估计你能做什么或不能做什么是完全可以预料的，并且在这个过程中的任何打嗝都比常规工作传播得更快。

我已经想了 Skok 差不多两年了。我把瞄准镜放下了，因为我知道它很容易爆炸。`the Shippening`的范围是 UI +扫描+显示数据。不要太花哨。这是一个可以实现的目标。

在过去的几年里，我也花了很多时间在项目评估和缩减范围上。我每天工作的一部分就是思考这些事情，调整时间表，找出(细节)需要做什么，适应这些事情，等等。

对于这一部分，我是在我的元素。

斯塔克也不算太差。我花了两年时间调查此事。我使用了几个构建工具(Rollup，Webpack，Parcel)来测试它们，所有主要的前端框架(Vue，React，Angular，甚至 tried Cycle)，并且我研究了为像我这样没有桌面经验的人创建桌面应用程序的最佳方法。所以我最终在 electronic 中构建了几个小应用程序。之前的研究是这个项目成功的关键。

最终堆栈:

1.  带打字稿的电子
2.  使用宗地编译器进行反应(JS ES2015+)
3.  SCSS 风格-自定义网格和自定义一切
4.  用于主<=>渲染进程通信的从头编写的库
5.  用于移动目录/文件的“walker”NPM 包(我为了工作目的对这个进行了大量测试)

我说“最终堆栈”是因为...好吧，继续读下去。

## [](#the-good)好人🙌

*   启动产品并将其置于“可工作”的状态是一项艰巨的任务
*   其他人做同样的事情会非常鼓舞人心

我想说好的事情是最终的结果。考虑这个应用程序已经 2 年了，这意味着我没有为它写一行代码。现在我有了一个完整的设置和开发工作流程，还不算太糟糕。其实挺享受的。另外，我正在学习 React(再次，第六次)，所以我有一个很好的出口。我也对 Parcel 的未来发展感兴趣，同样，这是一个很好的出路。

有时候编写第一行代码是最难的，但我认为对于一个开发人员来说，最难的事情是建立合适的开发环境，并直接为产品编写第一行代码。我已经过了那个阶段，尽管是以一种麻烦的方式。我知道对我来说，开发环境可能会让我马上离开这个项目。

其次，成为社区的一员是我开始这个项目的原因，也是我坚持下去的原因。Jevakallio 的帖子令人鼓舞。我想和大家分享我的成就，我也这样做了。我甚至制作了一个快速的 YT 视频，解释我如何设置 Electron 的 main/renderer 线程进行通信(不会链接，不是真正的好视频哈哈)。

## [](#the-bad)坏了🙁

*   在默认使用计划堆栈之前，花太多时间让“省时”工具工作
*   设计可能是你最不想关注的事情，但是拥有一个“丑陋”的应用程序并且没有设计准备会非常令人沮丧
*   这个堆栈仍然不理想，但据我所知，它是最好的一个

因此，手里拿着一叠文件，我遇到了第一个问题。我试图回避我自己在电子、包裹和反应方面的经验，并使用另一个我从未测试过的工具，因为我希望它能进一步简化我的工作流程，并帮助我避免未来的复杂性。

不幸的是，在过去的几个月里，这个“一体化”的电子工具在 Windows 上坏掉了。随着 Electron 4 的发布，它“过时了”(不是真的，但它只是还不支持 Electron 4)，很快就有了更好的版本，彻底改变了工具的工作方式。

我有很好的先见之明，在发货开始前测试了工具，不幸的是，如果我坚持使用我所熟知的工具，那么时间本可以更好地用于设置所有的初始工具。

另一个糟糕的是 CSS。我喜欢 CSS。CSS 很棒，但我不擅长把 CSS 放在一起，让默认风格的东西看起来更好。我建立了一个坚实的网格，我建立了一个漂亮的按钮，就是这样。

我再也不能使用 Illustrator 了，所以模仿这些东西(我已经习惯了)不是一个选项。这是一个巨大的问题。我所能想到的就是“他妈的，这个应用程序太丑了”，这令人沮丧。Illustrator 一直是我勾画网站布局和设计的重要工具。

当我看到哪怕是最简单的设计时，我也更容易编写风格代码，并对应用程序的外观感到满意。这可能不是世界上最重要的事情，但我讨厌做难看的产品。

最后，堆栈仍然不理想。或者说，开发环境不是。包裹很棒...但是要让它和电子一起工作还需要一点努力。电子仍然需要在后端更改时完全重启。我仍然不能只是“运行”一些东西，并期望它编译和工作。

package 非常接近，是目前为止我最喜欢的工具，但我仍然不知道如何让它同时编译前端和后端(这可能吗？).加...据我所知，没有人支持。对于电子前端来说，Live-reload 似乎也不存在。:/

## [](#the-ugly)丑陋😱😱😱

*   时机

时机。真是糟糕的时机。圣诞节和新年之间我超级忙。工作繁忙，个人生活更是如此。我用俄罗斯的方式庆祝新年，这意味着礼物、计划和压力。我在包装礼物，而不是在开发应用程序。我正忙于一个项目，累得后来都不想写应用程序了。

我很高兴没有其他丑八怪:)
