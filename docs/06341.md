# 当没人关心时，如何保持代码质量和自我发展的 5 条建议

2017 年 10 月 2 日，我拿到了作为 Rails 开发人员的第一份薪水。我在一家只使用 Rails 的 web 开发公司工作了一个半月。我很高兴在面试中听到这样的方法。但从那以后我冷静下来了。我试图在这篇文章中解释为什么，分析原因。

### [](#looking-back)回首往事

我从事网络开发已经有一段时间了。但是今年年初我的处境很奇怪。我远程工作，有体面的薪水和灵活的时间表，但我对此并不满意。我职业生涯的最后三年并不快乐。我在跳槽，寻找一份薪水最高、囚禁时间少的工作。我想，如果我的工作能让我有足够的时间用于个人生活，那对我来说就好了。的确有很多人在工作之外寻找快乐:在他们的爱好、旅行、家庭等等。

但是我从那里去不了任何地方。我仍然对我的工作感到压抑，因为我想把每件事都做好，挣大钱，享受这个过程。今年 2 月，我偶然看到了维克多·谢佩利夫的文章[三种类型的程序员](https://mkdev.me/en/posts/the-three-types-of-programmers)，读了这篇文章后，我彻底扭转了局面。

> 一种编程语言(或者更确切地说是语言+工具+基础设施+社区)强加了一种特定的思维模式，一个程序员的幸福和生产力在很大程度上取决于他的模式是否与他必须工作的语言和环境的模式相一致。

也许是这样？我喜欢我的文本看起来不错，无论是一篇文章还是一个程序代码。当我在大学的时候，我带着绝对的钦佩阅读代码。

但是当我开始工作时，我发现对于一个客户来说，代码是否好看并不重要，只要它能工作。他也不关心一切是如何运作的。他需要的是:

1.  它工作正常；
2.  这是在适当的时间(这通常意味着“昨天”)；
3.  从外面看很好看(界面)。

因此，我得到的只有单调和腐朽，而不是创造好看的文本和完美代码的快乐。这就是我们得到的:

*   如果它崩溃了，你修复它；
*   我们必须在周一开始促销活动，而你整个周末都坐着完成它；
*   “我们已经转移到一个新的服务器，集成将会工作，对不对？”“给我权限，我要检查所有的设置，”

而你在 PHP 中做所有这些，这让你紧张和怀疑，最重要的是，也是重复的。

### 我想成为一名 Rails 开发者！

所以受 Victor 文章的启发，我开始探索 Ruby。他帮我找回了编程的乐趣。

学习曲线很陡，这意味着一方面，空缺职位更少，另一方面，坏工作更少，薪水更高。然后我熟悉了 Rails，这被认为是所有创业者的最爱。事实证明，像正常人一样做一些快速有效的事情是可能的。

我意识到我应该开始使用 Rails 而不是 PHP。我有点想从一张白纸开始我的程序员之路，并最终停止牺牲代码质量来支持艰难的业务需求。换到新的语言，我决定养成使用最佳实践的习惯。这意味着总是测试代码，投入足够的时间来考虑任何决策和使用先进的技术。因此，我将最终开始磨练我的技能，而不仅仅是一个劳动者。随着技能的提高，我也会变得更快乐，更有价值。那是我的计划。

在研究过程中，许多经验丰富的人都怀有我的希望。我们从维克多开始，他小心翼翼地训斥了我 10 周，并向我展示了 Ruby 的美丽。然后[安东](https://mkdev.me/en/mentors/davetoxa)向我介绍了 Rails，事实证明，如果我有一个任务，我可以自己在合理的时间内创建一个可以工作的东西。

嗯，是我重新工作的时候了。我花了一段时间才找到工作，我不得不将就于比我预期的低的薪水。我不得不忘记我过去做的所有好事和项目。但是我确信有一份梦寐以求的工作就在我面前！

我在周三晚上和周四上午 10 点有一个面试，我应该带着笔记本电脑出现在办公室。有人告诉我，我的知识“对一个初级职位来说只是有点不够”，并邀请我参加这个项目，并说“我们人手不够，如果你能管理好，那就太酷了。”

然后，我所有的想法开始被一个个毁掉，就像海滩上的沙堡被海啸摧毁一样。

### [](#working-life)工作年限

我的第一个工作日充满了令人不快的惊喜。原来我受邀的项目早在 2014 年就开始了。你认为这意味着它又大又复杂吗？那你是对的。它也很老了。我看了一下项目代码，发现有 Rails 4.1 和 Ruby 2.2.3。

首先，我创建了一个空数据库。但我试图摆脱数据迁移是一个错误。当数据库从转储中恢复时，我试图安装 gems。我又失败了。在 Gemfile 中有一些到这个特定提交的链接，带有注释“提交到主分支时进行更正”这是一年前犯下的错误，但没有得到纠正。当我打开测试文件夹时，我发现已经很长时间没有人测试任何东西了。有一些测试，但只有几个，很明显，他们是在这个项目的早期阶段完成的。

第一天结束时，我终于在浏览器中看到了登录页面。应该提到的是，从那以后我们已经更新到了
Rails 4.2.7。不过，也不是没有困难。

除了技术问题，管理上也有一些问题。开发过程被分成几个团队，这导致了截止日期和需求忽略了真正的程序员的可能性的情况。开发人员只有时间编码。战略思考阶段，测试和调试被搁置，以利于最后期限。因此，任务完成得很差，或者根本没有经过测试。而且，所有旧的部分从来不重构。它们是临时修补的，因为‘我们没有时间，有最后期限。'

这样的问题很可能不是独一无二的，你们中的大多数人不时会面临同样的问题。问题是我们作为开发者应该怎么做？或者，换句话说...

### 如何忍受这一切而不成为一只代码猴？

肯定有两个层面的问题:

1.  公司/项目层面的问题；
2.  技能水平上的个人问题以及工作流程和时间管理。

从自己做起当然更容易。公司也承担了很大的责任，但是我认为开发者应该更关心他或她自己的自我发展。

事情是，大脑从任何重复的行为中创造出模式，并且它会尽可能地使用这种模式。这就是为什么当登山者因为肌肉疲劳而开始移动不充分时就会停止锻炼。不得不在多个场合对代码质量做出妥协的程序员，过一会儿就会因为同样的原因默认开始写脏代码。

像“为自己写一个质量代码，只完成工作中的任务”这样的想法也会导致同样的情况。坏的行为模式将开始形成，因为无论如何，工作时间超过了个人时间。这就是为什么我们需要考虑一些事情。

我发现了五个关键问题，它们可能会出现并把我变成一只代码猴子。当我解决这些问题时，我掌握了我的技能，没有放弃，享受这个过程，并且总体上坚持我为自己制定的从 PHP 转换到 Ruby 的计划。我就是这么做的。

#### [](#1-big-unfamiliar-project)1。陌生的大项目

很多时间通常花在搜索信息上，试图理解一个项目是什么以及它是如何工作的。你可能认为在熟悉这个项目后，你会摆脱这个问题。但这基本上是个例外。检查一些陌生和不熟悉的项目的代码是很常见的。

所以最重要的是，我或多或少探索了主题领域。这有助于了解我在每个任务中肯定会遇到的基本术语。然后，我搜索与实体(如数据库中的模型和表)完全相同的术语的匹配项。然后我可以分析这些实体，寻找它们之间的任何联系。

第二阶段是我接到一个具体任务的那一刻。首先，我要求经理或测试人员向我展示它在界面中的位置。在那里可以找到所有需要的代码部分。如果我在理解代码中发生的事情上有问题，我会咨询那些已经参与这个项目一段时间的人。

#### [](#2-not-enough-time-for-testing)2。没有足够的时间测试

即使客户推迟了截止日期，测试对于开发来说也是非常必要的，在评估任务的时候应该给它留出时间。即使雇主说“你需要写的不是测试而是无错误的代码”，我也不会听他们的，而是写测试。当我估计完成手头任务的时间时，我也会提供一些时间进行测试。我只是不说这个时间也包括测试时间。这个比较容易:)

#### [](#3-insufficient-task-specifications)3。任务规格不足

有时候一些需求只在开发阶段被提及。此外，界面的外观和实际操作之间存在巨大差距。如果经理没有意识到这种差距有多大，他要么会设定不可能完成的任务，要么会抓狂为什么“这么小的事情花了这么长时间”真的让我很烦。

所以为了防止一些解释和重写，最好在任务评估过程中澄清一切。除此之外，我还会在任务开始前检查是否所有的材料都准备好了，尽管这不是我工作的一部分。我认为最好把一切都搞清楚，而不是以后再去责备。当任务没有按时完成的时候尤其如此，因为我们在等待一些界面或者文本。

#### [](#4-code-review-at-the-end-of-the-task)4。任务结束时的代码审查

这通常是在截止日期的最后完成的。如果发现了任何问题，你必须马上重做。此外，你没有机会讨论所做的决定。

我用一种相当粗糙的方法解决了这个问题。我要求对我推给 PR 的每个部分进行代码审查。因此，当有一个截止日期时，我可以确保所有的东西都被检查过了，我不应该重写任何东西。

#### [](#5-i-dont-see-a-development-path-for-me-as-a-programmer)5。作为一名程序员，我看不到一条适合我的发展道路

这里我的意思是，我的工作过程更多的是持久而不是发展。没有可以理解的导师。当我解决小问题时，我学会了打补丁并通过类比来工作(实际上，使用类比是一种通常的做法，但前提是你面前有一个优质的例子。)

这个问题可以在公司层面解决。例如，你可以向你的雇主提出这个问题，或者找一份对员工的职业发展更重要的工作。但是这些对我来说都还没有发生，所以我自己解决一切。

你需要在你的工作时间内提高你的工作效率和质量。周末和空闲时间都不够，更别指望了。

你还应该有一个全球性的自我发展计划。如果在项目范围内无法付诸实践，那就把空闲时间花在这上面。但是你必须记住，任何与主要项目无关的技能可能永远只是一种学术技能。

我做了一个决定:我需要一个导师，这样我才能提高工作效率。你可能会问，如果我身边有很多同事，为什么我会选择一个导师？问题是我的同事们通常都忙于他们自己的任务，没有时间陪我。此外，导师可以承担你自我发展的战略任务，而同事会给你如何创造有用的东西的建议。

当工作任务以适当的人工成本完成时，我将能够提高我在项目之外的整体水平。我想导师可以告诉我我的盲点以及如何在职业上进步。之后，我可能会和我的雇主谈谈加薪的事。

#### [](#problems-on-the-level-of-the-company)公司层面上的问题

似乎一些开发定制软件的公司把他们的程序员变成了装配线上的工人。他们在自己的网站上做一些简单的程序，而首席执行官们认为，只需增加工人数量就可以提高效率，而不是发展已经工作的人。

我还不是一个经理，作为一个业余爱好者来做判断，但在我看来，任何开发公司都应该注意这些事情:

*   定期建立人才库；
*   勾勒出公司内部的职业发展规划，对员工和雇主都清晰(不仅指职业阶梯，还包括技能素质)；
*   不要将有问题的项目的责任仅仅转移到开发人员身上。

如果对这些问题视而不见，后果将不堪设想。那些极其认真地选择工作的开发人员最终会离开。那些更关心稳定性或具体公司的人会留下来，但以粗心的态度工作。这样，不管他们雇佣多少人，效率都会很低。

目前，我可以看到人们如何通过雇用学生来建立人才库。一方面，教育那些没有被商业生活腐蚀的人更容易。但另一方面，学生并不是能形成公司中坚力量的劳动力。

当开发人员和学生的比例变成 50/50 时，一切都变得很奇怪。有足够的劳动力，但他们经常落后于计划，因为有大量的重写工作。年长的开发人员忙于其他项目，所以他们也没有时间教这些学生。

这个问题没有普遍适用的答案:“我们应该为一个开发者做些什么来进步？”有些人通过与他人交流取得了很大的成就，但却日以继夜地用问题轰炸他们。这对这样的人来说已经足够了。有时这个问题相当特殊，但是不可能问同事，因为他们可能很忙或者在做其他项目。如果这个人坚持不懈，他或她会谷歌，问朋友，通过 Github 搜索，但会独自解决问题。

但并不是每个人都是那样的:有些人可以征服一座高峰，而有些人只是放弃了。对一些人来说，付费并从专家那里获得快速和高质量的答案更容易。有些人在空闲时间解决他们在工作日无法解决的问题。

总会有一个答案，但每个人都应该根据自己的情况来寻找答案。雇主应该有兴趣与开发商一起处理问题
*。*

### [](#who-is-responsible)谁负责？

我认为这里“双方都有责任”，就像任何其他关系一样。但是，我们也应该考虑到双方的承诺程度。当你开始在一个团队中工作时，你期望过度保护。你希望他们能照顾你，激励你，所以你要全力以赴。但这种想法是 na？非常幼稚。而与此同时，如果我什么都自己做，为什么还要职场关系？如果有可能成为自由职业者，我们为什么还要组建团队？在很大程度上，因为我们不希望自己的技能在自由职业市场上做单调乏味的工作时走下坡路。

在我看来，伙伴关系是唯一可能的互动类型。当我们谈论工作关系时，我们可能有不同的职责，但有一个共同的目标。在每个公司里，每个开发人员都可能阐明他们提供什么样的职业发展。公司提供的选择越少，员工的自我发展就越有希望。

但如果管理层有一点点灵活性，并准备好沟通，你们可以一起思考如何帮助员工进步，以及他们需要多少种方式。也许他们需要从第三方聘请一些导师，或者让有才华的员工放松一点，这样他们就可以教其他人了？也许他们需要给工人一些自我教育的自由时间？

值得注意的事实是，还有成熟开发人员的晋升问题，因为在这种情况下，在公司内部找到导师更加困难。但即使这样，也可以通过共同努力来解决。
