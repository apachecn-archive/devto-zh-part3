# [](#lets-finally-git-along-github)让我们最终混过去吧，GitHub

Git 和 Github 被误解得太多了，但是现在是我们更好地了解它们的时候了。

我想分享一下我以前是怎么理解他们的，现在又是怎么理解他们的。如果你不想读我的故事和咆哮，你可以[跳到](#collaborate)我分享我对 Git 和 GitHub 工作流的了解的部分。

### [](#first-date-impressions)初次约会印象

以下是我第一次使用 Git 和 Github 之前对它们的初步印象。

Github 的网站上说 Git 和 Github 很容易学。他们甚至说不用任何代码就能学会！那就放心了！

*我听说 Git 在命令行上展示了它最大的潜力，它本身看似简单明了，毫不张扬。听起来简单，但相当强大！*

看起来 GitHub 是一个简单的网站，导航起来比 Git 更直观。嗯，这似乎是可以管理的！

“Git”是一个简单的单音节名字，听起来很活泼。Github 有他们可爱的吉祥物 Octocat。太可爱了。所有这些。

[![](../Images/b53b32ff5b460b89e764fa647fd5e665.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--HkyWJ6Vq--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/bdnxqr1dluk8q65tm0on.jpg)

###### 看着 Octocat 的眼睛，我真诚地相信 GitHub 是一个可爱而无害的平台，它的唯一目的就是让我的生活更轻松。

Git 是一个*版本控制系统*，它是一个工具，你可以在你自己的计算机上本地使用它来管理你的项目的开发历史。**另一方面，GitHub** 就像带有在线社交网络的 Git，你可以远程分享工作。它们都是为了让我们安心地构建和协作项目，所以你自然会觉得它们应该简单易用。

但是如果是这样的话，为什么人们在使用它们的时候会恐慌呢？

### [](#why-didnt-i-get-git)为什么我没有得到 Git

由于许多原因，每当我不得不接近 Git 的时候，我都会有许多内心的恐惧。它在命令行上工作，使用起来很简单，但是对于许多没有经验的用户来说也是晦涩难懂的。当事情出错时，它会用一行行不熟悉的单词对你大喊大叫，当事情顺利时，它会很生硬，或者你没有得到任何认可，这让你怀疑你到底有没有做任何事情！

我不知道你怎么想，但对我来说，这听起来让人想起一段糟糕的、不愉快的关系。

我是在开玩笑，但是对于像我这样的初学者来说，当我只能看到文本时，很难理解发生了什么，但是我无法想象我正在处理的结构的变化或更大的画面。

再加上 GitHub 和一些同样困惑的人，这就升级成了一次更加困惑、高风险和紧张的经历。你意识到你的决定实际上是有后果的，它们会影响其他人和你的项目！

### [](#learning-from-the-past)吸取过去的教训

我和我的几个新兵训练营同学在上周我们的第一次合作项目经历后艰难地认识到，如果不试图理解 Git 如何工作并在 Github 上建立一个良好的工作流程，你不仅可能会毁了自己的努力和项目，而且还会严重挫伤你的关系和信心。

尽管由于 Git 和 GitHub，我的项目经历并不尽如人意，但这是必要的，因为现在我变得非常积极，不想重复我的错误。因为我现在知道在一个项目上合作是什么样的，所以我可以更清晰地阅读 Git 文档，并且我可以有效地研究我遇到的并且不想重复的问题的解决方案。我有更强烈的感觉(我觉得！)Git 和 Github 如何帮助我与我的团队和谐地工作，我认为分享我所学到的东西是正确的，这样我们才能更好地维护和保持我们的关系(和我们的理智)。

# [](#a-collaborative-workflow-guide)协同工作流程指南

在我对简单工作流的解释中，我将跳过对克隆、添加、提交或推送等基本例程步骤的深入解释。相反，我会花更多的时间来解释我们在与合作者合作时需要采取的额外步骤。此外，为了简洁起见，我将举例说明一个理想的情况。我认为像合并冲突这样的问题应该有自己的帖子。

如果你不熟悉 Git 基础或者你有点生疏，你可以先看看 [GitHub 自己的指南](https://guides.github.com/activities/hello-world/)或者 [git-the simple guide](https://rogerdudler.github.io/git-guide/) ，这两个我觉得都够清楚了。

### [](#1-set-your-team-up-for-collaboration)1。让您的团队进行协作

这是假装项目周！我们是一个由多名成员组成的团队，我们都在为一个全新的项目而努力。为了让事情有一个好的开始，我们将清晰地构建我们的项目，建立我们都同意的规则，并坚持一个计划。

### [](#2-create-a-master-branch)2。创建主分支

首先，我们需要指定一个人，让我们把她命名为 Aliya，并要求 Aliya 在 Github 上为我们的项目创建一个新的存储库。或者在我们从另一个存储库的模板开始的情况下，她可以派生存储库。无论是一个全新的存储库还是一个分支，它都是一个新的主分支，供我们所有人使用！干得好，阿利亚！

### [](#3-add-collaborators)3。添加协作者

但是等等！在我们进入代码之前，Aliya 仍然必须将我们添加为项目的合作者！

她点击靠近顶部的新存储库的**设置**标签，然后点击左侧菜单上的**合作者**。然后，她使用搜索栏搜索每个团队成员，并点击**添加合作者**按钮，将他们添加到我们的项目中。轻松搞定！

既然不碍事，大家都可以克隆(但是**不能**叉！)将存储库的副本复制到它们的本地目录中，并独立地对它们进行操作。

### [](#4-establish-a-game-plan)4。建立游戏计划

理想情况下，在我们开始我们的项目和打字之前，我们应该有一个清晰的计划，如何一起解决它，并建立期望和规则。我们同意我们不应该在同一个文件上工作，我们*尤其是*不应该在同一个代码上工作。

我们决定各自发挥自己的优势和兴趣，并给每个成员分配一个项目特色。我们将从主分支中创建**特性分支**，并且我们将独立地在我们自己的分支上工作。规则很简单:*呆在你自己的分支上，不要写不属于你的代码*。

### [](#5-create-a-feature-branch)5。创建特征分支

我们每个人都可以使用命令行创建自己的分支:

<center>`git checkout -b sweet-feature-im-working-on`</center>

`-b`标志将创建我们的新分支，唯一的名称是`sweet-feature-im-working-on`，`checkout`会在分支创建后立即将我们移至该分支。*

所以每个人都有自己的分支，用不同的名字描述他们在做什么。我们每个人都可以离开我们的分支，开始改变我们自己的特征。只要记住专注于分配给你的任务，不要弄乱其他代码！

[![](../Images/797608962eed55a70bf9fd351df1af15.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--_wI3kcUR--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/o5r8g3oemvp3htln7jbu.jpg)

### [](#6-im-finally-done-with-my-feature-time-to-push)6。我终于完成了我的特写！该用力了！

一旦我们完成了对我们特性的工作，我们可以使用命令行将它推送到 GitHub 上的远程存储库分支:

<center>`git push origin sweet-feature-im-working-on`</center>

那很容易！

### [](#7-make-a-pull-request)7。提出拉取请求

然后，我们将跳转到 Github 上的远程存储库，点击绿色按钮**比较和拉取请求**。

在 pull request 页面上，我们需要比较主存储库的分支和基本存储库的主分支。

世界卫生大会....什么？这么多大词...

停下来，深呼吸，仔细阅读继续。 **HEAD** 就是我们现在的分公司所在的地方。因此，我们希望确保在选择要比较的内容时，我们的基本库**主**分支与我们的主库和我们一直在处理的分支**sweet-feature-im-work-on**进行比较。

做出这些选择，如果你得到批准，你就可以开始并准备合并了！点击表格右下角的绿色按钮**创建拉动请求**，结束流程。

### [](#8-merge)8。合并！

有许多方法可以着手做这件事。默认情况下，任何参与者都可以批准一个拉请求，甚至是他们自己的请求，并将更改合并到主分支中。如果你独自工作，这没什么，但是在一个团队中，如果有人在合并代码之前没有仔细检查他们自己的代码，事情就会变得有问题。

我喜欢民主的方法，在这种方法中，我们可以选择团队中的一个人来审查和批准所有的拉请求，或者我们可以要求至少有一个我们个人之外的其他合作者来做这件事。无论哪种方式，同行都在仔细检查我们的工作，确保它不会破坏项目。如果一个同行决定我们的代码是可接受的，他们会批准合并，但是如果他们认为它需要一些额外的工作，他们会要求我们在合并被批准之前将其改进到一个可接受的水平。

我们的工作已经被批准合并到总支了！现在我们已经正式完成了我们的特性，我们可以安全地删除我们的特性分支了。

### [](#9-pull-all-updated-changes)9。提取所有更新的更改

在每个人都成功地合并了他们的更改之后，我们将从头再来一遍大多数步骤。

但在我们再次这么做之前，我们需要重新开始。也许每个人都应该在转向另一个新特性之前获得完整的更新文件集。

让我们回到命令类型的主分支...

<center>`git checkout master`</center>

...and pull all the updated changes into your local repository...

<center>`git pull origin master`</center>

现在我们有了所有更新的文件，我们准备再次工作！

### [](#10-repeat-until-done)10。重复直到完成

从这里开始，步骤是一样的，

1.  创建另一个新的工作分支。请记住，在您自己的分支上，只对您自己的文件执行您自己的功能。同时，自由地、经常地承诺吧！
2.  当你有完整和功能代码时，按下
3.  在 GitHub 上，比较分支并发出拉取请求
4.  同行评审代码并批准拉请求，然后将更改合并到主分支。
5.  返回主分支以提取所有更新的更改。

我们将重复这个循环，直到我们完成我们的项目！
