# 真理的单一来源:设计系统的未来

> 原文：<https://dev.to/inthepocket/a-single-source-of-truth-the-future-of-design-systems-35ib>

去年在 Shift 上，我们讨论了设计系统入门:协作和代码连接的生活方式指南。这个概念正在逐渐融入我们的工作流程，并且已经有了很好的例子。在口袋里，我们不断努力为我们的合作伙伴在现有的和新的项目中实施设计系统，但在这样做的同时，新的挑战出现了。通常设计系统是由设计师或设计团队发起和拥有的。在实施所设计的屏幕的过程中，与原始设计的差异迟早会开始出现。

为每个屏幕进行交互和视觉设计的方法长期以来一直是移动应用程序设计的标准。这种方法对客户和开发人员来说都很清楚，适合生命周期短的小型产品的需求。如今，我们构建的产品是持久的，需要更多的可维护性，需要可伸缩性，并且在一个组织中的各种平台和其他产品上是一致的。这些产品需要不同的、更有效的设计和加工方法。

我们现在面临的主要挑战是工具之间不能以理想的方式进行通信。细节被遗漏了，重要的细微差别被忽略了。设计和工程之间有巨大的差距；为了确保我们处于不断变化的顶端，仍然需要做大量耗时的手工工作。当一个新的设计师或开发人员加入团队时，需要一个快速的入职流程和良好的结论性文档来开始项目。设计师应该能够快速地从构思转移到生产，开发人员应该能够直接进入代码。

> 我们今天的工作方式可能会扩大工程和设计之间的差距，设计和建造之间的许多层是一种负担。我们可以做得更好。我们可以更好地工作。- *亚历克斯·施莱弗，Airbnb*

当我们审视行业现状时，我们发现其他公司也面临着这些挑战。像 Airbnb 这样的公司已经投入了大量资源来开发特定的工具，以维护设计语言，同时使他们能够弥合设计和开发之间的差距——这种差距在过去几年中变得越来越大。由于大量工具的可用性，软件开发变得更加协作化，使得工程越来越多产。相比之下，设计工具已经落后了。

当谈到设计系统时，这些公司面临的主要问题是，他们通常专注于一个工作流程，一种技术，这使得该系统很难匹配或集成到其他系统中。理想情况下，他们希望他们的系统与某个特定的技术和工具包完全分离，使其对于任何给定的环境都是“即插即用”的。从长远来看，这可以通过创建一种通用文件格式来实现，任何设计工具或新技术都可以从这种格式导入和导出到一个真实的来源。

为了支持这个基本上是设计和开发的一个真实来源的想法，我们开始与一个跨能力团队一起构建我们自己的系统。使用我们的原子设计方法，我们分析了我们当前拥有的组件和模式，并创建了新的组件和模式来满足构建完整产品的需求。
