# 什么是测试？

> 原文：<https://dev.to/codetips/what-is-testing-3mip>

> 不管你已经写了五分钟还是五十年的代码，你永远也写不出完美的软件。

这并不意味着你写不出好的软件，但是它会有错误(缺陷),尽早接受这一点很重要。

计算机会做我们让它们做的事情，但它们还不够聪明，不能做我们想做的事情。

在某些时候，你会犯一个(希望如此)小的疏忽，或者一个错误的假设，你会创建一个 bug。它可能永远不会实现；有些错误只在奇怪的情况下发生，所以你可能会很幸运，但有一天你可能会开始得到一些奇怪的结果，你必须找出原因。

一个简单的例子是一个接受两个数并把它们相除的函数。

```
function divide(num1, num2) {
  num1 / num2
} 
```

似乎很直接。这三行代码不会出错，对吗？不对。

如果`num2`为零呢？在[数学](https://www.mathsisfun.com/numbers/dividing-by-zero.html)中——除以零没有意义。试试看——打开计算器，试着用零除任何一个数。

在编程中——除以零可能会产生意想不到的后果。它可能导致意想不到的或错误的结果，或者，在某些情况下/语言中，它甚至可能使您的应用程序崩溃。

现在想象你的应用程序有成千上万行代码，而不仅仅是三行，你应该开始理解错误是如何被引入的；不管你有多小心。

这就是开发人员测试的原因。我们提出“测试用例”(比如用零运行我们的`multiply`函数)，并观察它如何反应。如果它没有像我们期望的那样反应，我们就修改代码。

在六个月的时间里，当我们忘记了不应该被零除，我们决定改变`multiply`函数时，我们再次运行我们的“测试用例”,我们将看到它产生了一个意想不到的结果，我们避免了破坏我们的应用程序。

测试用例通常也写成代码，所以它们可以自动化。我们将在后面的文章中更详细地介绍这个过程——这只是对什么是测试以及为什么它如此重要的介绍。

测试并不能减少出现错误的机会，它只是帮助我们增加对自己编写的代码的信心。你应该总是寻找机会来改进你的测试或者你的代码，并且你应该**总是**记住，不管你有多少测试，也不管你做了多久的开发人员:

> 不管你已经写了五分钟还是五十年的代码，你永远也写不出完美的软件。
