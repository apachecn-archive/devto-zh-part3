# 循环连续部署-第 2 部分

> 原文:# t0]https://dev . to/jeansmaug/dploizione-continu-with-circus-part-2-6 op

[第一部分](https://dev.to/deploiement-continu-avec-circleci-partie-1)有助于发现循环的基本概念以及如何应用这些概念。在这一部分中，我们将探讨如何通过各种方式在工作之间共享信息。

## [](#%C3%A9tat-des-lieux)场所状况

```
version: 2.1
jobs:
  build:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn build

  lint:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn lint

  test:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn test:unit

  deploy:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn build
      - run: yarn firebase deploy --token "$FIREBASE_TOKEN" --only hosting

workflows:
  version: 2
  integration:
    jobs:
      - build
      - lint
      - test
    - deploy:
          requires:
            - build
            - lint
            - test
          filters:
            branches:
              only: master 
```

我们的配置是有效的，但我们可以看到它是重复的。每个作业都包含这些行:

```
docker:
  - image: circleci/node:11.10.1
steps:
  - checkout
  - run: yarn install 
```

前两行(运行时环境配置中的行)只是配置文件中的重复项，不会影响性能。但是，为了确保可持续性，将使用执行者和命令对这些碎片进行测量。

第二个问题是，每个作业都必须重新安装相关性，因为它们在隔离环境中执行其语句。
要解决此问题，我们将安装依赖关系，缓存它们，以便工作流作业可以共享它们。高速缓存的相关性也可用于将来的作业。

最后一个问题是需要在建置工作中重新建置实际环境执行应用程式，而建置工作已完成此步骤。此外，问题还在于作业是孤立的。默认情况下，部署作业无法读取生成作业所生成的内容。我们将利用工作区来解决这个问题。

## [](#cacher-les-d%C3%A9pendances)隐藏依赖关系

要使用缓存，我们将定义一个新作业来安装我们的依赖项:

```
install:
  docker:
    - image: circleci/node:11.10.1
  steps:
    - checkout
    - restore_cache:
        key: v1-dependencies-{{ checksum "yarn.lock" }}
    - run: yarn install
    - save_cache:
        key: v1-dependencies-{{ checksum "yarn.lock" }}
        paths:
          - ./node_modules 
```

*   `restore_cache`:恢复给定密钥的缓存
    *   `key`:缓存恢复密钥
*   `save_cache`:备份缓存文件
    *   `key`:缓存存储密钥
    *   `paths`要储存的项目，此处为 node_modules

此步骤将尝试恢复包含“`node_modules`”的缓存，以便跳过依赖项安装步骤。

注意:快取金钥包含由 checksum 函数产生的`yarn.lock`档案杂凑。因此，如果文件更改，散列将更改，缓存将重新生成。

我们可以更新配置中的其他作业以使用缓存:

```
version: 2
jobs:
  install:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - restore_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}
      - run: yarn install
      - save_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}
          paths:
            - ./node_modules

  build:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - restore_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}
      - run: yarn build

  # autres jobs...

workflows:
  version: 2
  integration:
    jobs:
      - install
      - build:
          requires:
            - install
      - lint:
          requires:
            - install
      - test:
          requires:
            - install
    - deploy:
          requires:
            - build
            - lint
            - test
          filters:
            branches:
              only: master 
```

现在只需安装一次，即可在不同作业之间共享依赖关系。请记住，每个作业都需要完成 install 作业。

使用缓存可以节省大约 20 秒的部署时间。

## [](#les-workspaces)Les 工作区

工作区是一个允许作业与其后续作业共享数据的空间。
在本例中，我们将在`build`作业中构建应用程序，将静态文件存储在工作区中，以便`deploy`作业在必要时可以检索它们。

```
build:
  docker:
    - image: circleci/node:11.10.1
  steps:
    - checkout
    - restore_cache:
        key: v1-dependencies-{{ checksum "yarn.lock" }}
    - run: yarn build
    - persist_to_workspace:
        root: ~/project
        paths:
          - ./dist 
```

*   `persist_to_workspace`表示要写入工作区
    *   `~/project`:要在其中获取信息的文件夹，默认为`~/project`
    *   `paths`要写入工作区的档案/资料夹路径清单

```
deploy:
  docker:
    - image: circleci/node:11.10.1
  steps:
    - checkout
    - restore_cache:
        key: v1-dependencies-{{ checksum "yarn.lock" }}
    - attach_workspace:
        at: ~/project
    - run: yarn firebase deploy --token "$FIREBASE_TOKEN" --only hosting 
```

与第一部分的配置相比，使用工作区作为高速缓存的补充可以节省大约 30 秒🎉。

## [](#sur-le-les-executors-et-les-commandes)🍒在上🍰:执行者和命令

在此，我们将优化配置，以最大限度地避免在配置文件中出现重复。
Les `commands`使我们能够对我们工作的阶段进行系数计算。
使用`executors`可以对运行我们的脚本的环境进行系数计算。

```
commands:
  checkout_and_restore_cache:
    steps:
      - checkout
      - restore_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}

executors:
  node:
    docker:
      - image: circleci/node:11.10.1 
```

我们可以更新我们的工作:

```
install:
  executor: node
  steps:
    - checkout_and_restore_cache
    - run: yarn install
    - save_cache:
        key: v1-dependencies-{{ checksum "yarn.lock" }}
        paths:
          - ./node_modules 
```

## [](#la-configuration-finale)拉配置压轴

```
version: 2.1

commands:
  checkout_and_restore_cache:
    steps:
      - checkout
      - restore_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}

executors:
  node:
    docker:
      - image: circleci/node:11.10.1

jobs:
  install:
    executor: node
    steps:
      - checkout_and_restore_cache
      - run: yarn install
      - save_cache:
          key: v1-dependencies-{{ checksum "yarn.lock" }}
          paths:
            - ./node_modules

  build:
    executor: node
    steps:
      - checkout_and_restore_cache
      - run: yarn build
      - persist_to_workspace:
          root: ~/project
          paths:
            - dist/*

  lint:
    executor: node
    steps:
      - checkout_and_restore_cache
      - run: yarn lint

  test:
    executor: node
    steps:
      - checkout_and_restore_cache
      - run: yarn test:unit

  deploy:
    executor: node
    steps:
      - checkout_and_restore_cache
      - attach_workspace:
          at: ~/project
      - run: yarn firebase deploy --token "$FIREBASE_TOKEN" --only hosting

workflows:
  version: 2
  integration:
    jobs:
      - install
      - build:
          requires:
            - install
      - lint:
          requires:
            - install
      - test:
          requires:
            - install
      - deploy:
          requires:
            - build
            - lint
            - test
          filters:
            branches:
              only: master 
```
