# 充分利用快速销售代表的十大技巧

> 原文：<https://dev.to/joshhadik/ten-tips-to-get-the-most-out-of-rapid-reps-480n>

这是我用来每周学习新技能的一个策略系列的第二部分，我称之为快速代表。如果你还没有读过第一篇文章，你应该读一下第一篇文章。

在这篇文章中，我列出了我的十大秘诀，告诉你如何最大限度地利用你的日常快速练习！

## 挑选时间、地点和持续时间。

Rapid Reps 的美妙之处在于，你可以设置他们按照你的时间表工作。如果你目前失业，试图学习新技能来打动潜在雇主，你可以选择花费一个小时或更长时间的任务。如果你从 8 点工作到 6 点，只是想要一个小方法让你保持在游戏中，你可以选择一些你下班回家后能在 30 分钟或更短时间内完成的小任务。

我个人？我喜欢把我的时间控制在 45 分钟或更少，并且在我早上喝咖啡的时候做。

## 不要过度伸展自己。

现实一点。如果你每天只有 30 分钟来完成一个快速 rep，不要选择一个你以前从未使用过的八种不同新技术的任务(例如:用 Express GraphQL API 构建一个 React/Redux 应用程序，并用 Travis CI 将其部署到 Google 的 Kubernetes 引擎)。找到一项你想学的技术，坚持一周，每天坚持，下周继续下一项。

## 不要害怕周中的变化。

有时候，我们作为开发人员高估了自己的能力，或者低估了任务的难度。我发现自己不止一次违背了最后一条建议，过度扩张了自己。一项我认为需要 45 分钟的任务最终花了两个多小时。

在这种情况下，不要害怕在一周中间改变路线。试着把这个大任务分成两到三个小任务，你可以在接下来的几周内完成。快速代表并不意味着要花一整天的时间，你应该尽你所能把他们控制在一定的时间内。

## 休息一天。

就我个人而言，我尽量每周从任何形式的编码中抽出一整天的时间。这给了我一个机会，让我稍微清醒一下头脑，刷新一下思维，这反过来帮助我在第二天以更多的注意力和精力回到编码工作中。这意味着你的理想目标应该是每周做六次，并有一个指定的休息日。

## 尽可能多地凭记忆做。

每天你都应该努力尽可能多地记忆，而不是复制和粘贴你在之前的演示中编写的代码。尽管这样做的目的不是记住解决特定问题所需的逐行语法，但强迫自己练习一下也无妨！

## 在你第一次代表之前做一次任务。

快速重复有助于你重复一个过程足够多的次数，使核心概念和工作流程深入你的记忆，但是你可能想考虑事先走一遍这个过程，这样你就知道你在做什么了！

每当我试图处理一项新任务时，我喜欢在周六花一个小时左右的时间浏览一遍教程，真正搞清楚这个过程。这样，在周一，当我做第一次报告时，我已经知道我需要做什么，而不是花额外的两个小时去学习。

## 一周内做一些小的补充。

起初，仅仅是完成任务就已经够难了，但是通常到第 3 或第 4 天，你就可以用第一天一半的时间快速完成任务。在这一点上，我喜欢寻找一个小的额外任务，我可以添加到每天的过程中！

以下是我如何将此付诸实践的一些例子:

**例 1** - >当我花了一周时间集中学习如何在 Phoenix 应用程序中实现 OAuth 时，我开始只是通过一个提供商 Github 设置 OAuth。大约在第三天，我添加了脸书，然后是谷歌，到最后一天，我有了一个你可以用 Github、脸书、谷歌或 Twitter 登录的应用，我甚至建立了一个[非常酷的系统](https://github.com/JoshHadik/RapidReps/blob/master/Phoenix/phoenix_oauth/authy/lib/authy_web/views/layout_view.ex)，可以很容易地以一种简单的方式添加新的 OAuth 链接

**示例 2** - >当我用 Travis CI 花了一周时间专注于设置到 Elastic Beanstalk 的多容器部署时，我从一个部署到单个环境的超级基本工作流开始，到周末，我有了完全自主的系统，用于[构建 docker 映像](https://github.com/JoshHadik/RapidReps/blob/master/AWS/multi_container_elastic_beanstalk/thunder_wind_sprint/.build/build_and_push_images.sh)和[部署到基于 git 分支的两个独立环境](https://github.com/JoshHadik/RapidReps/blob/master/AWS/multi_container_elastic_beanstalk/thunder_wind_sprint/.travis.yml)，一个用于生产，一个用于暂存。

## 跨周任务使用连续性。

这一个和上一个有点类似，但是不是集中在一个星期的过程中对一个任务的增加，这是关于在你从一个星期到下一个星期完成的任务的性质之间创造一个连续的流动。

因此，如果你真的有兴趣学习某项技能，**把它分解成一个非常简单的“开始”任务**，在一周内每天完成这项任务，然后在新的一周快速重复。

例如，几个月前我感兴趣学习的技能之一是如何设置 CI/CD 工作流，使我能够轻松地部署多容器应用程序。我决定先小试牛刀，用一周时间构建一个本地运行的 Docker 映像，并以此为基础进行构建。以下是我遵循的基本流程:

**第 1 周** - >从 docker 映像构建一个工作 docker 容器。
**第二周** - >构建一个双容器环境，在我的本地环境中使用 docker-compose 将两者连接在一起。
**第 3 周** - >用 Travis CI 构建一个单容器应用并推送到 Elastic Beanstalk。
**第 4 周** - >构建一个双容器应用，用 Travis CI 推送到 Elastic Beanstalk。
**第 5 周** - >构建一个多容器应用，用 Travis CI 推送到 Elastic Beanstalk，设置 RDS 为数据库层。

请记住，在开始的时候，我几乎不知道 Docker 是什么，在我的生活中从未使用过 CI 工具，除了简单的 S3 桶存储之外，没有使用过 AWS。我得说从那以后我已经走了很长一段路。

## 让清洁成为你工作的一部分。

如果你正在使用像 AWS 这样的按秒付费的云服务，并且想尽可能少付费(如果你是免费试用，最好什么都不要付),这一点尤其重要。)

我喜欢在开始每次工作的时候，列出一张清单，上面列有我完成任何工作需要完成的所有步骤，清单上的最后一项总是“清理干净”!这意味着浏览每一个在线服务，清理我在代表期间做的任何事情，无论是删除 Github Repo，关闭弹性 beanstalk 环境，还是删除 Docker hub 上的图像，我都喜欢清理除了我电脑上的代码之外的所有东西。

你最不希望看到的就是上百张无用的图片和存储库塞满了你的在线工作环境，如果你不小心的话，当你开始每天快速重复时，它们会很快增加。

## 最后一天重构并展示你的代码。

当我完成一周的最后一次报告时，我会试着花额外的 30 分钟左右来检查和清理代码。我还删除了任何可能混入的“易受攻击的”信息(环境变量、api 键等)。)

一旦我完成了，我就把代码复制到一个单独的 [git 库](https://github.com/JoshHadik/RapidReps)并把它推送到 GitHub。这有两个主要目的:

1.  它确保你永远不会丢失你在那周快速重复中写的“示例代码”,以防你想回头看它作为参考。
2.  它给了你一个很好的代码库，你可以用它在将来向潜在的雇主展示你的技能！

## 额外提示——使用条纹应用程序！

*我与条纹应用没有任何关系，我只是认为它是一个有用的工具，可以很好地与快速代表合作*

[Streaks](https://streaksapp.com) 是一款 iPhone 应用程序，可以让你设定一些你想在日常生活中实施的习惯。每天当你完成任务时，你可以打开 Streaks 应用程序并将其标记为完成，Streaks 会显示你连续完成该任务的天数，也就是你的“streak”

我有一个任务设置为快速代表，截至 2019 年 2 月 5 日，我已经进行了 80 天！我发现使用 streaks 应用程序每天都给我一点动力来完成我的记录，因为我不想打破这个记录！

*Streaks 只在 iPhone 上可用，如果 Android 有更好的替代品，请在评论中告诉我。*
