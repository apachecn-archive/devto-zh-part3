# 为什么在软件测试中要选择质量而不是数量？

> 原文:[https://dev . to/lambdatest/why-to-choose-quality-over-quantity-in-software-testing-pk8](https://dev.to/lambdatest/why-to-choose-quality-over-quantity-in-software-testing-pk8)

作为质量保证的一部分，在软件行业中，你总是被期望携带一根质量棒，以确保质量维持到“T”。我们总是被要求设身处地为客户着想，确保产品/项目满足客户的期望，达到最高的质量。

但真正具有讽刺意味的是，我们所有的质量指标都归结为量化的数字和术语，如记录的 bug、编写的测试用例数量、执行的测试用例数量、花费在测试上的时间、测试的 URL、为[跨浏览器测试](https://www.lambdatest.com/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=webpage)检查的浏览器、缺陷泄漏等等。

我们设计了我们的工作系统，在这个系统中，我们被要求将质量置于数量之上，但最终却被分析在定量方法上。我相信专注于测试的定量方法对你的软件测试团队是不公平的，即使我们遵循定量方法，也必须有一个系统的方法来判断基于我们的软件测试度量的个人努力。

## 仅仅用数字来证明我们的软件测试指标可以吗？

我们需要问自己如何证明我们的测试方法是正确的，每条路径都是定量可视化的？这是测试质量急剧下降的原因之一。一个简单的例子是当你用记录的 bug 的数量来衡量你的团队的效率或效能时。

你的每个团队成员会采取的第一个方法是在应用程序的任何地方找到尽可能多的 bug。我知道很多人会争辩说，直到我们在 web 应用程序中找到 bug，这有什么关系。但是最终，这是测试质量到位的地方。

看看我们最近正在研究的[敏捷软件开发方法](https://www.lambdatest.com/blog/automation-testing-important-agile-development/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog),随着周期的缩短和测试被推到每个周期的末尾，我们剩下的就是在最短的时间内测试应用程序的看似高压。我们为每一个版本执行所有这些基于风险的测试和冒烟测试，以确保应用程序为其最终用户提供无缝体验。

在如此关键的情况下，用这种编号系统来驱动团队是没有用的。**重要的不是数量，而是每个错误的本质，如果让客户来处理，会造成多大程度的破坏**。因此，即使我们可能会将这些“x”个错误放入我们的桶中，我们也可能会在交付产品的质量方面犯下巨大的错误，因为我们在寻找应用程序时已经制定了思维方式。这是在定义我们的软件测试度量标准时要考虑的最大原因之一，在这些度量标准中，质量在任何时候都应该超过数量。

今天，保持质量的方法是完全不同的，并且只在利益相关者的满意度方面说话。**质量完全由客户驱动**。利益相关者认为质量等于利润。**质量越高，软件的可预测性就越高，这意味着人们可以在市场定价方面冒风险**。

利益相关者会知道他在产品稳定性方面的立场，以及他可以在市场上推广产品的深度。但是，由于我们在启动项目时开始工作的方式，即收集需求、定义测试范围、团队协调和分配、测试活动等，这一点完全丢失了，我们往往会忘记我们作为测试人员的真正使命，这是'**构建项目**的主要目标。这是促使这个项目帮助最终用户解决问题的唯一原因。

然而，重要的问题应该是，我们正在测试以确保这些问题得到解决吗？如果没有，我们是否经常向我们的利益相关者提供反馈，以帮助他们更好地了解项目。不断问自己“这是客户期望的吗？”这一点很重要或者“有没有更好的方法来解决同样的问题”。仅仅从客户那里获取需求并构建它们并不能让我们的工作圆满完成。当我们开始项目工作时，我们需要和客户坐在一起，了解他们对项目的期望是什么，以及他们如何看待项目的质量方面。

例如，如果你的客户更关注品牌方面，即使是像素化的 logo 问题对你来说也是非常严重的，如果不是对开发者来说的话。如果他的目标是构建一个金融应用程序，那么与用户数据的安全性相比，用户界面和 UX 对他来说可能不是那么重要。这里**‘目标’是关键**。这是我们作为测试人员需要考虑的事情。这应该是你的团队 OKR( **目标和关键结果**)，而不是那些数字驱动的指标。

**OKR 是一个受欢迎的领导过程，它帮助个人、团队和组织在一个统一的方向上共同努力实现他们的目标**。跨团队和组织设定目标。这种 OKRs 有助于专注于生产力和推动公司文化。

## [](#quality-is-subjective-amp-may-change-from-customer-to-customer)质量是主观的&可能因客户而异。

重要的是我们如何根据这些目标来安排我们的测试工作。事实上，这有助于我们决定什么该修补，什么不该修补。因此，归根结底，你对项目涉众的观点和任务的了解越清楚，就越有利于构建和优先化你的测试工作。通过询问你自己来分析你的风险**你的客户想要什么**将帮助你提高质量。这听起来更像是一个业务分析师的角色和职责，但是我们不要忘记我们作为测试人员也需要有这个基本的技能。我们的测试策略是由这些分析本身驱动的。

因此，让我们专注于编写足够数量的驱动您的客户和项目目标的测试用例，而不是专注于编写大量没有主要症结的测试用例。强调严重程度高的问题，而不是仅仅在你的桶里装满无数的小错误。**优先考虑会拖垮客户的风险，而不是你的评估矩阵**。

质量过去是、将来也将是无可争议的赢家。无论如何量化你的测试过程是不够的，但是当然，这个问题对于所有那些需要度量的组织来说是没有答案的'**那么我们如何度量质量呢？**“制定这些指标的主要目的是关注如何提高/降低这些数字，或者使这些数字处于相同的水平以实现质量，但因为人就是人，我们更认真地进入这个数字业务并推动它们，因为它们已被标记为我们的增长评估。因此，重要的是要记住，是什么驱使我们作为测试人员，以及如何构建我们的评估矩阵。如果您关心浏览器兼容性测试，那么这里有一篇文章可以帮助您评估测试工作流程的[跨浏览器兼容性矩阵。](https://www.lambdatest.com/blog/creating-browser-compatibility-matrix-for-testing-workflow/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)

所以我们知道，到目前为止，质量测试比数量测试更好。确保您步入正确方向的方法是在您的团队中招募正确的软件测试人员，并在您已经拥有的软件测试团队中吸收质量测试的概念**而不是数量测试的概念**。

## [](#how-to-judge-a-quality-tester-apart-from-the-others)如何判断一个质量测试人员与众不同？

我在这个行业已经七年了，在指导了这么多初露头角的测试专业人员之后，我关于在质量的基础上衡量个人的整个想法一直来源于他们分析业务需求的能力，将他们分解成更小级别的块，并确保那些是按照他们的意图构建和工作的。

对我来说，测试人员的意图比他/她给我的测试用例或 bug 的数量更重要。我一直更喜欢提出问题并理解优先含义的人，而不是‘只是测试’的人。

我在这么多测试人员中观察到的最常见的行为是，一旦故事/需求被分配给他们，他们就开始编写测试用例。他们往往会忘记基本的基础步骤，即坐下来分析故事中提到的需求。他们忘记了通过站在终端用户的立场上思考问题，意识到终端用户可能会使用的工作流。找出受影响的区域，并查看用户在流程中可能通过的所有验证。

我总是坚持在开始[编写有效的测试用例](https://www.lambdatest.com/blog/17-lessons-i-learned-for-writing-effective-test-cases/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)之前做一份清单，这有助于正确的测试覆盖。另一个重要的方面是回溯，不管是需求还是出现的 bug。这有助于确保没有遗漏需求，并且能够找到错误的根本原因，这有助于减少此类错误的发生。[好的错误报告](https://www.lambdatest.com/blog/advanced-guide-on-writing-a-bug-report/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)和积极的态度也有助于成为好的测试人员。

这些是我通常在团队中推行的一些质量方面。至于他们的测量要求，我把他们放入技能部分，并标记他们之间的个人，而不是那些指标。

优秀测试人员的一些品质包括:

*   了解业务的优先级和严重性。

*   能够深入系统并全面思考。

*   遵循质量流程，如有需要，采取纠正措施以进一步改进。

*   快速而持续的学习者。

*   对测试的热情。

*   良好的沟通技巧。

*   分析能力。

*   与其他团队成员合作并协调工作。

要成为一名成功的软件测试人员，可能有更多非常有效的技能。

由于这种度量马拉松，我们作为测试人员倾向于吸收一些作为测试人员的'**不需要的**或'**不好的**'品质，相信我，这些是目前非常普遍的事情，例如:

*   基于假设执行测试。

*   没有分析就报告 bug。

*   商业分析能力差。

*   缺乏客户洞察力。

*   沟通能力差。

*   没有能力遵循流程。

*   害怕拒绝工作或思想。

关键是在你的团队中注入这些品质，并带来积极的品质。以下是我如何鼓励他们发挥自己最佳品质的方法:

1.  定期举办研讨会，帮助他们熟练编写 bug 报告。此外，告诉他们如何在不陷入预先假设的情况下做更好的测试。

2.  为了增强糟糕的沟通技巧，我坚持让他们与开发人员进行循环和内部通话。这有助于增强他们对 web 应用程序的入站和出站流程的信心和理解。辅助[连续测试](https://www.lambdatest.com/blog/what-is-holding-a-software-tester-from-finding-bugs/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)和软件测试度量的质量。

3.  对拒绝的恐惧总是困扰着新生或年轻的软件测试人员。这可以通过合作管理轻松解决。我确保从不批评他们的错误，而是给他们提供一些建议，告诉他们如何更好地测试 web 应用程序或产品。通过这种方式，我确保消除任何阻碍我的软件测试人员发现 bug 的障碍。

4.  为了保持高昂的情绪，我确保每月或每季度举行颁奖/礼物仪式，以表彰非凡的努力。这样做，我让测试人员在彼此之间变得更有能力，当然是以一种健康的方式。

5.  我相信[左移测试有助于提高产品质量](https://www.lambdatest.com/blog/how-shift-left-testing-can-help-your-product-quality/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)。将左移测试与连续测试结合起来，可以在时间、资源和金钱方面创造奇迹。我鼓励年轻的软件测试人员也成为左移测试的一部分。这有助于他们从客户需求收集阶段就理解测试场景。对 SRS 文档的良好理解有助于他们在软件测试度量方面坚持质量。

## 不要偏离你的目标！

我已经多次看到这种情况发生，软件测试人员会过多地关注提高他们的 bug 数量，结果，他们最终会偏离他们要测试的功能的目标。我敢肯定，你一定也经历过同样的事情！

嗯，质量测试用例是非常关键的，如果我们不坚持我们的目标，并在增加每日错误日志计数的基础上报告错误，那么我们可能最终掩盖了关键的质量测试用例。

可以把它看作是为测试部门设置正确的 OKRs(客观关键&结果)。如果你是 QA 领导或经理，负责调整发布管理中的测试团队，那么[为你的测试部门](https://www.lambdatest.com/blog/how-professional-qa-lead-set-goals-for-a-test-department/?utm_source=devto&utm_medium=organic&utm_campaign=sep09_sd&utm_term=sd&utm_content=blog)设定正确的目标就变得非常关键。

您可以将上述积极技能作为您的主要目标，并在此基础上评估您的团队。这有助于团队成员的进步和进一步发展，这将对你的项目/产品产生直接影响。**我们的 OKRs 或评估矩阵应该能够回答像**这样的问题:

1.  我们想要什么，重视什么？

2.  我们感知到了哪些问题，如何识别？

有了这些明确定义的 okr，我们可以帮助团队交付高质量的产品，而不是在团队成员之间比较和分析毫无价值的数字(指标)。话虽如此，收集这些数字来提高你的质量目标应该是唯一的本质。信不信由你，数字驱动着人们的心理，因此我们如何设计和利用它们是很重要的。

所以我们不要以数量来推动质量。质量应该是唯一的驱动力，应该被认为是实现顾客满意的唯一主要方面。
