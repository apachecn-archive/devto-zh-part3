# 无头 CMS 的局限性？

> 原文:[https://dev.to/jameesy/limits-of-a-headless-cms-41fb](https://dev.to/jameesy/limits-of-a-headless-cms-41fb)

嘿戴夫托！

长话短说，我是一名开发全栈团队管理应用程序的前端开发人员。该应用内置于 Node.js 和 MySQL 中，允许团队领导邀请他们的团队成员，他们将在那里接受评估，然后在公司内进行相应的评级和组织。

它将在几个月后推出测试版，我已经对它的稳定性有所了解，但展望未来，我担心应用程序在当前状态下的可扩展性。

我很想从头开始，或者至少重新构建应用程序的某些方面。

这个想法是有团队管理功能，但通过他们的 API 从第三方人力资源应用程序中引入数据，然后在应用程序中对用户数据进行排序。我想在某个时候集成一些应用内的培训功能(视频/材料等)，以及建立一个(轻)测试部分，用户可以解决问题，并获得更高的评级。

我最初的直觉是直接进入 Node.js/React 堆栈，但是，我的技能主要在于前端，我担心我缺乏后端知识会让我对应用程序日益增长的复杂性感到失望。

对于那些实际访问应用程序并注册的用户来说，像 Sanity 或 Strapi 这样的无头 CMS 似乎是一个完美的解决方案，但我不确定这些是否适合支持更复杂的功能。我猜如果所有的东西都通过 GraphQL 层，那么 headless CMS 可以与从独立的第三方 API 中提取的数据并行执行。或者它们只是适合我在所有例子中看到的标准网站/博客/电子商务用例？对于我的应用程序来说，这是一个好的解决方案吗？或者我只是提出了新的问题？
