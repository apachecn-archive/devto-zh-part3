# 进/出三个朋友

> 原文：<https://dev.to/dev3l/entryexit-three-amigos-2hjk>

**进入/退出三个朋友**是敏捷惯例，有助于对流入和流出团队价值流的工作单元达成最低程度的共识。

敏捷联盟对[三个朋友](https://www.agilealliance.org/glossary/three-amigos/)的定义解释了什么和为什么，但没有对过程做出规定。实现细节留给从业者来填写。

### 敏捷联盟定义:

> 三个朋友指的是在开发之前、期间和之后检查工作增量的主要视角。这些观点是:
> 
> *   **业务** -我们试图解决什么问题？
> *   **开发** -我们如何构建解决方案？
> *   测试 -这个呢，可能会发生什么？

### 进入/退出三友流程

> 因此，一个产品负责人、一个开发人员和一个测试人员走进一个酒吧，坐下来，开始谈论正在开发的系统应该做的事情。产品负责人描述用户故事。开发人员和测试人员问问题(并提出建议)，直到他们认为他们可以回答这个基本问题，“我如何知道这个故事已经完成了？”-

* * *

牢记精益创业(持续交付、开发运维、产品驱动等)的思维模式，对话应该围绕一个假设，完成的**看起来像什么，以及确定提取学习成果的方法。它们应该至少出现*两次，进入和退出，每个待办事项*通过价值流。当工作细节或客户需要必要的支点时，应召开额外的特别朋友会议。**

我更喜欢朋友角色被命名为:*产品、工程师、质量*。不同的术语创造了一幅略有不同的画面。在最好的情况下，这将是产品所有者、软件开发人员和质量/自动化专家之间关于特性结果的两分钟会议。在另一个极端，它变成了一个扩展的对话，让位于一个细化的会议。团队不应从事范围和愿景无法达成一致的项目。

当主题专家不在时，如果他/她有必要的领域知识来戴上那顶“帽子”，团队成员可以介入并填充角色。戴上“质量”帽子的工程师。

## 建议流程

清单是适应新流程的一个很好的起点。下面的清单是我用来帮助最初促进这些会议的团队工件。这是一种团队技能。从相对良好定义的工作项目开始！

### 录入三个好友清单

[ ] -产品待办事项满足团队 **[准备就绪的定义](https://dev.to/dev3l/definition-of-ready--1jfb)**
[ ] -代表的每个朋友: **PO / DEV / QA**
[ ] -时间限制在 **15 分钟**或更短
[ ] -目标结果(**假设**)呈现
[ ] - **由主题专家
[ ]介绍**的功能-确定类似的系统功能
[ ] -满足条件( **[小黄瓜](https://dev.to/dev3l/behavior-driven-development-bdd-crash-course-170)** )最终确定
[ ] -三个朋友**同意[完成](https://www.scrumalliance.org/community/articles/2008/september/definition-of-done-a-reference)** 的定义

### 退出三个朋友清单

[ ] -产品待办事项符合团队 **[完成的定义](https://softwaredev3loper.wordpress.com/2016/03/27/definition-of-done/)**
[ ] -代表的每个朋友: **PO / DEV / QA**
[ ] -时间限制在 **15 分钟**或更少
[ ] - **由开发人员
[ ]引入**的功能-开发人员演示的满意条件
[ ] - QA / PO 接受或拒绝用户故事过渡
