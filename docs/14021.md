# 瑞典人正在毁灭 SRE

> 原文：<https://dev.to/sudolecture/swes-are-ruining-sre-3oom>

**前言**:这是匿名账号写的。为什么？在某种程度上，我害怕报复。我愿意认为我们的行业比这更好，但情况并非总是如此。

我是一名系统工程师，住在一个科技环境不断发展的城市。在我的职业生涯中，我有机会在传统的网络工程领域工作，也有机会帮助构建相当大规模的自动化系统。我的目标是为开发人员解决难题，这样他们可以更快、更有信心地交付产品。我喜欢认为我在这方面已经成功了，即使我的想法没有立即被采纳，它们通常最终会被组织采纳。

也就是说，我被降级到我的城市。我所在的城市缺乏强大的创业文化，因此也缺乏许多股票期权和奖金，而这些东西在科技领域更普遍地交给了工程师。这些是科技工作者退休能力的重要关键，简单的 401k 通常是不够的，特别是如果你在 30 岁左右开始创造市场平均水平。在这最后一轮面试中，这是我的目标。

在我开始申请公司之前，我就知道站点可靠性工程(SRE)的软件工程师是个问题。当我在一家大型专业社交网络面试时，我第一次意识到了这一点，当我不知道如何将 Python 应用程序后台化时，我失败了。现在我每天都在 Go 中写作，我不知道为什么一个有能力的软件工程师会问这个问题，但这就是让我失望的问题。

当我在 DevOps 的日子里，目睹一个 SRE 瑞典人谈论他们在一家大型比特币公司的 SRE 之旅时，引起了进一步的关注。当被问到时，SRE 说系统工程师(SE)在 SRE 没有一席之地，因为他认为不会编码是不合格的。

对于那些不知道的人，系统工程师当然知道如何编码，但是我们可能对 TDD 有点笨拙，我们可能不能为你设计一个巨大的应用程序。我花了很多时间编写 CLI，我甚至为微服务添加了功能，老实说，最近我做了大量的后端开发。也就是说，这不是我擅长的。我喜欢为事物建立自动化，通常那个领域的软件机制要简单得多。我可以监听事件循环，或者运行一个作业，它会补偿任何给定堆栈中的弱点。

虽然我想告诉大家，事情已经发生了变化，但情况只会变得更糟。许多公司已经明白了这样一句话:“SRE 是通过软件工程师的视角进行操作的”，而没有附加任何上下文。一家公司告诉我，我执行 TDD 是错误的。他们是对的。在为代码编写测试之前，我让代码可读——这对软件工程师来说是显而易见的。我不知道我所做的是否会改变代码的行为。对于一个系统工程师来说，这并不明显，但可以通过一些轻微的推动来纠正。我没有参加系统工程测试。

下一家公司让我的葡萄变酸了不少。我真的得到了一个连接四(它是连接三)的算法来玩。后来我参加了一个架构白板会议，在面试中我没有得到任何反馈。后来，我得到的反馈是我把一些事情复杂化了，编程部分失败了。在现实世界中，我们在哪里玩基础设施中的连接 4，或者建筑不仅仅是工程师之间的对话？建筑总是提议、拒绝和接受之间的来回对话。没有实时反馈的建筑只是在偏见的回音室里的白板上玩工艺品。

这里的重要性是 SRE-SE 如何对担忧做出反应(讽刺的是，这有助于 SRE 保持主动)-他们是忽略这些担忧，还是制定一个计划？SRE 通常不是解决方案的一站式商店，相反，在问题完全解决之前，它通常需要多次迭代。

系统工程师需要知道如何编码。他们需要了解测试实践。我完全支持这一点。但是因为他们不能完美地制作 Java 应用程序的内部结构，包括完美的 TDD、OOP、工厂和项目结构而放弃一个系统工程职位候选人是愚蠢的。

虽然对一些人来说可能不明显，但这些都是俱乐部心态的症状。SRE 只由软件工程师组成的想法需要在一场大火中迅速消亡。我从未参加过试图用一个他们在工作过程中永远不会看到的测试来难倒软件工程师的面试；你能想象如果我让一些 CS 毕业生在 Linux 上改变一个用户的外壳吗？或许还能让他们给我解释一下《谍影重重》和《谍影重重》的区别？请一位高级软件工程师解释 Linux 容器由什么组成或者每个名称空间是如何工作的怎么样？

那将是愚蠢和狭隘的。所以，不要再对系统工程师做同样的事情。

面试的目标是让候选人尽可能地光彩照人。因此，你现在要在两个你见过最好的一面的候选人之间进行选择，而不是一个坐着上完算法课的候选人和一个当你让他们在你面前第一次现场处理和解决算法练习时反击恐慌的候选人，而你却在你面前晃着薪酬和福利的胡萝卜。

这里的长期影响是，我和其他人都被挡在了拿 RSU 奖金和红利的工作岗位之外。这是在保护各种各样的象牙塔，在这种情况下，这也是倡导多样性的团体一直在大声疾呼的。你的面试和招聘方法不仅无效，而且非常不称职。

从短期来看，公司会失去那些理解为什么正确命名空间很重要的人。有些人可以向您展示为什么在某些情况下使用较重的基本容器图像但较小的应用程序层比使用整体较小的图像尺寸更有效，这些人现在被关于矩阵和假想线绘制的问题淘汰了。尝试让您的顶级软件工程师调试两个节点无法通过网络通信的问题，然后让一个普通的系统工程师做同样的事情。现在用 IPTables 联网的容器来做这件事。告诉我你的发现。

软件和系统工程技能集建立在相同的基础上，但在专业上有所不同。给我们足够的时间，我们能完成我的工作。给我足够的时间，我能做好他们的工作。我在平衡质量和生产力方面最好的经历是作为一名系统工程师和一名瑞典人一起工作。在那种情况下，我们表演了魔术，我们的职业缺陷被共同的目标和一个新的、更大的工具箱所取代。一个没有另一个，只不过是我们以前用过的那台破机器的新版本。
