# 建议上的建议

> 原文:[https://dev.to/kaydacode/advice-on-advice-4b6l](https://dev.to/kaydacode/advice-on-advice-4b6l)

网上流传着许多来自那些有过旅行经历的人的建议，他们希望分享自己的经历。这是我喜欢互联网的主要原因之一...技术社区和(有时)社交媒体为分享这些知识提供了一个很好的平台。但是，每当我遇到一些“建议”——似乎总是有人被冒犯或发誓说这个建议是错误的。

### 当有人对某个话题提出建议时，通常是基于他们的经验和从经验中获得的洞察力。每个人都有不同的完全独特的经历！

# [](#advice-on-advice)劝告上劝告

1.  建议不是非黑即白的
    这与第二点密切相关，但我想特别强调一下。通常，当我在开发过程中遇到让我畏缩的事情时，我会在 Twitter-verse 中把它作为建议或嘿，不要这样做。这是否意味着你永远不应该这样做？没有。每件事都有闪光的地方和时间，但要考虑为什么会出现这个建议，并在你的实施过程中考虑它，看看你能否改善痛点。

2.  每当我遇到一个不使用故事板的 iOS 团队时，我都会退缩。毕竟，它们确实使事情变得更容易，开发时间更快，我只是不明白为什么有人不想要它！说实话，这对我参加过的每个小团队都很有效。有一次我加入了一个大团队——很合得来。由于故事板会引起的所有合并冲突，它会消耗很多时间来解决所有的问题。即使有多个故事板，这也是不可行的，并且会导致一个非常混乱的解决方案。每当我遇到有不同意见的人时，我是不是很粗鲁？没有，但是我听了他们为什么有不同的意见。这让我想到...

3.  这适用于双方，建议者和接受者。在我上面的例子中，你应该对为什么有人有不同的观点感兴趣。可能是你还没到达的一个捕捉点，也可能他们的情况有点不一样。但是，对于现在和未来的决策来说，这仍然是很好的洞察力。
    至于对建议者的同情——把自己放在那里通常是非常可怕的。我这样做是因为我喜欢帮助别人避免我犯过的错误，并帮助人们在工作中取得成功。但时不时会有人引用我的推文，羞辱我太“非黑即白”。当然，它可以在 140 个字符中以这种方式出现，但任何关注我的博客或推特的人都应该知道，这从来不是我的意图。谈话会更有建设性地利用每个人的时间。
    感同身受大有裨益。

4.  如果有人启发了你，感谢他们！让他们知道他们的洞察力已经帮助你做出更好的决策。
    如果你不同意所说的话，你可以采用第三种方法来解释你对这个话题的看法，或者忽略它，继续你的一天。否定和贬低那些试图帮助别人的人是没有意义的。
