别担心。我知道我不是第一个指出这一点的人(比如:[这里](https://www.forbes.com/sites/lizryan/2018/04/12/no-youre-not-crazy-the-hiring-process-is-broken)和[这里](https://medium.com/racial-equity-in-tech/real-talk-the-technical-interview-is-broken-b84b8375dccb)和[这里](https://daedtech.com/hiring-is-broken/)和[这里](https://medium.com/@evnowandforever/f-you-i-quit-hiring-is-broken-bb8f3a48d324)——geesh，4.3K 拍手！[这里](https://be.helpful.com/https-medium-com-fnthawar-helpful-technical-interviews-are-garbage-dc5d9aee5acd)和[这里](https://rejected.us)和[这里](http://blog.interviewing.io/technical-interview-performance-is-kind-of-arbitrary-heres-the-data/)和[这里](https://zachholman.com/posts/startup-interviewing-is-fucked/)和[这里](https://blog.usejournal.com/rethinking-how-we-interview-in-microsofts-developer-division-8f404cfd075a)和[这里](https://sockpuppet.org/blog/2015/03/06/the-hiring-post/))。这篇文章既来自我多年来作为被采访者的个人经历，也来自我 15 年来雇佣其他技术人员的经历。

首先，请允许我尝试建立至少一些可信度。虽然我不是“著名的”开发人员，但在我超过 18 年的职业生涯中，我已经取得了相当多的成就。不是每个人都会被打动，我对此完全没意见。在这个行业中，绝对有很多更令人印象深刻的人，但我认为，意识到我从何而来以及为什么我认为我至少有一些资格做出这样的评论是有帮助的。

*   我做过开发人员、架构师、技术产品经理、开发人员交互设计师、团队领导和总监。
*   我在七个行业工作过(取决于你怎么数，更多)，从小公司到大到威瑞森。
*   我在现场和远程都做过顾问、个体承包商和雇员。
*   我管理和雇佣过开发人员/工程师、建筑师、图形/视觉设计师、交互/UX 设计师、开发倡导者/布道者和开发工具产品经理。作为其中的一部分，我从零开始设计了许多版本的招聘流程，并不断重复，在这个过程中尽我所能向他人学习。
*   我负责过数百万美元的预算和大型全球分布的团队。
*   我与人合著了四本开发人员书籍，写了比我记忆中更多的开发文章和博客，并在许多开发人员活动中发言。
*   我已经获得了三项专利，八次获得微软 MVP 奖，并入选了 INETA 演讲局。

信不信由你，我不是一个极端的自我主义者，我希望我的大多数同事都可以证明这一点。我和其他人一样定期遭受冒名顶替综合症的折磨。我只是根据上下文列出这些条件。我知道很多关于制作软件的事情，但我知道还有很多很多我不知道的事情。我写以上内容主要是为了提醒自己，是的，也许我确实能在这个持续的对话中提供一些有意义的东西。我不止一次决定不写或不发表这篇文章，但我正在做！

我有欧洲历史学位，辅修人文学科。我上过几门 CS 课程，但我所知道的绝大多数来自于在职学习和课外自学。总的来说，我不会贬低 CS(或相关)学位，也不会建议如果你想进入软件开发行业，没有 CS 学位是更好的选择。我试图弥补我在正规培训中的不足，我敢说我在这方面已经相对成功了，但在这个行业中没有 CS 学位会更难。

## [](#so-what-is-broken)那么什么是破？

我已经谈到了这一点，并在这里提供了一些更好的方法的建议。但我就直说了。尽管如此，在过去的一年里，我还是被面试官告知“技术不够强”。说什么？

请记住，我谈论的或多或少是我所熟悉的技术堆栈范围内的平均完整堆栈和/或前端工作。我知道我的极限。我不会到处申请嵌入式开发或操作系统开发或几乎任何低级语言/领域的开发。我没有申请像数据可视化、机器学习等高度专业化的领域。我不会申请我没有经验的工作。不，这些都是普通的工作，我有大量的证据证明我完全有能力胜任。毕竟，我为什么要找一份我知道马上就会被发现是冒名顶替的工作呢？

这些人是如何确定我在技术上不够强这一确定的事实的呢？想猜猜吗？如果你说，“在 codewars.com 上找到的带有随机脑筋急转弯类型问题的现场编码测试”，你就对了。现在不要误解我，我也通过了这些测试的面试。嗯（表示踌躇等）..这不是很奇怪吗？

你知道还有什么奇怪吗？我从来没有，一次也没有，因为技术能力不足而被解雇。我知道一个事实，那就是我十五年前写的代码仍然在生产中活跃着。我写的代码至少被成千上万的人日常使用。我从来没有，一次也没有，被一个经理告诉我真的需要提高我的技术实力。

然而，经理和同事告诉我的是，我写清晰的代码。我*已经*因为我的深思熟虑的、可用的架构、我对在必要的地方记录我的代码的关注、以及我把露营地收拾得比我发现它的时候还要干净而受到称赞。我*被告知*我的代码比我的一些同事有更少的缺陷，无论是在测试中还是在生产中。我*被其他高级开发人员、高级架构师、首席执行官、首席技术官和高级副总裁*告知，我在架构和技术方面的贡献和建议非常有价值。

然而，这个愚蠢的小编码挑战(我想)他们认为我做得不够好，或者花了太长时间来解决，或者任何明确的证据证明我在技术上不够强大，无法做我已经成功做了十多年的基本相同的工作？我说有东西发臭了，而且不是我家猫的猫砂盒。

重要的是，记住我只是这个问题的一个例子。就像上面那篇 4.3K 拍手的文章里说的，我绝对不是唯一一个！我只想把我的故事加入其中。如果你想要比我更值得关注的人的例子，请查看 [rejected.us](https://rejected.us) 。

## [](#the-stifling-strangling-elitism)令人窒息、令人窒息的精英主义

当我思考这一切的时候，我想到问我的母亲(纯粹是巧合，因为我们正从机场开车过来),在她的工作中，他们是如何进行采访的。她拥有理学学士、MSW 和 MPH 学位，为退伍军人服务，但也在其他场合工作过。她开始向我解释 PBI，我就像，“是啊！我们也是这样做的，但我们也有这些小的武断和基本上不相关的技术挑战……”

你知道她问我什么吗？她问这是不是我们行业最近出现的问题。她问是否有什么变化导致了这种功能障碍。我其实不知道这是怎么开始的，但我有一些猜测。我认为它很大程度上是从我们巨大的性别差距问题的源头开始的。说白了，就是来自一群从智力优越感中获得自我价值的书呆子。公平地说，我把自己算在那一类人里，但是随着我越来越意识到这个缺点，我已经试着在思想上变得更好。

沿着这条线的某个地方，我们开始了这个计算机怪胎的神话，它基本上是一个关于一个不完全具有社交能力的男性用他非凡的天才来弥补这一缺陷的故事。我几乎不认为这种刻板印象需要我来为它提供证据。去 HN 或 Reddit 上逛逛，或者，嗯，去任何有大量想要成为头号男性极客的地方。沿着这条线的某个地方，这个(几乎完全)男孩俱乐部决定保持排他性。很早就开始了。如果你读史蒂文·利维的《黑客》一书，你会发现它的萌芽。

值得庆幸的是，我们的行业已经越来越意识到这个问题，但我们的实践仍在追赶，我认为这种面试策略是这个问题的一种表现。从某种意义上来说，这并不是工作表现的预测指标，但这是一个很好的方法:1)让面试官感觉自己很聪明，2)剔除“痞子”，保持俱乐部的独特性。毕竟，成为团队中技术强大、富有成效的成员的唯一方法是你能“以每小时 100 英里的速度快速完成简单的事情”

你可能想知道最后一句话是从哪里来的。我要说的是，正是这种形式的面试给了它最大的推动力:不是别人，正是 StackOverflow 的创始人乔尔·斯波尔斯基和他的“[游击式面试指南](https://www.joelonsoftware.com/2006/10/25/the-guerrilla-guide-to-interviewing-version-30/)”这份指南通过他的“乔尔测试”在开发社区得到了推广(其他方面都很好)，公司可以根据这份测试对自己的 StackOverflow 招聘信息进行评级。十二个问题中的大部分都是很好的指标，但是第 11 个是我在这里呼吁的:“新的候选人在他们的面试中写代码吗？”

当然，如果你跟着面包屑走，你会找到他的向导。与测试一样，他的指南也有优点，但在“编写代码”一节中，他不仅阐述了大量精英主义，甚至还提倡备受指责的白板编码技术。根据指南的创作时间，也许白板编码这种过时的概念可以被原谅，尽管即使在早期，他也没有理由不把候选人带到本地桌面。

但是我跑题了。我做出了精英主义的断言，所以我应该支持它。请注意他的故事，关于“好”程序员是多么罕见，那么少人能通过类中的指针和递归，仅仅学习 Java 是多么可怕。让我们看看。

“如果基本概念不是那么简单，以至于你甚至不需要考虑它们，你就不会理解大的概念。”

这方面的问题是什么是“基本概念”的模糊概念。我的意思是，如果它们真的是基本概念，那么这个陈述就是真理。问题在于，每个开发人员、每个采访者似乎对“基本”的含义都有自己的看法。最近有人向我断言，使用按位 XOR 来查找重复奇数次的数组中的数字是“简单的”(即基本的)。

乔尔继续说道，嘲笑那些他认为无法获得计算机科学学位的人...他们什么都不懂了。班上 90%的人都去了政治科学专业，然后他们告诉他们的朋友，在他们的政治科学课上没有足够多的漂亮的合适性别的成员，这就是为什么他们换了。出于某种原因，大多数人似乎天生就没有理解指针的那部分大脑。”你听到谦虚滴声了吗？只有少数特别指定的人才拥有这部分大脑。只有一些人有一种神经机制，这是一个非常不寻常的说法，当你仔细想想，除了他的个人轶事，基本上没有证据支持它。

然后，他嘲笑地使用了“脚本专家”这个术语，并说，“所有优秀的程序员都应该能够处理递归和指针，这是判断一个人是否是优秀程序员的一个很好的方法。”他甚至说，“我想让程序员知道 CPU 级别的编程。”我真的很想知道，在无服务器、容器和巨大虚拟化的今天，这有什么关系。但即使在 2006 年，它也几乎与 90%的软件应用程序开发(他的公司做的那种事情)无关。我甚至认为这在某些方面不太相关，但这是另一篇文章的主题。

尚不完全清楚他是想说递归、指针和 CPU 级知识是基础，还是仅仅是造就“优秀程序员”的要素很明显，即使他不认为这些是“基本的”,他也认为这些是决定好的和坏的程序员的重要和陈词滥调。

现在我知道这个行业中会有一部分人会对乔尔在这里写的东西热切点头。他们会用怀疑的眼光看着我，同意知道`const circleArea = r => Math.PI*Math.pow(r, 2)`是你在这个行业里不知道的事情。他们认为`const startsWithCap = s => /^[A-Z]/.test(s)`和`const sumArray = arr => arr.reduce((acc, curr) => acc + curr, 0)`一样，牢记在心是至关重要的。

(声明一下，我不得不提醒自己数学。PI 存在于 JS 中，而不是作为近似值硬编码 3.14159，我不得不查找`pow`函数——我在想也许有一个像^一样的指数运算符，但我现在记得那是用于按位异或的。我想这告诉了你，尽管我多年来一直是一名卓有成效的全职 JS 开发人员，但我不得不经常使用这些东西。)

老实说，在某种程度上，我不反对这些问题对我来说似乎是基本的，但这些问题本身几乎没有告诉我关于候选人的一般软件开发能力的任何有意义的东西。我完全不同意递归和指针是优秀开发人员的试金石。我认为它们是理解起来有用的东西，但充其量是相对平凡的细节。许多开发人员都很有效率，并且可以通过简单的理解来应付。学习如何遍历一棵树是很容易学会的，同样，对于大多数应用程序开发来说，你真的不需要死记硬背。我真的不确定搜索指针有什么实际意义，但我完全支持比严格必要的理解更深入的理解。我并不是说知道这些没有帮助，而是说它们很难成为一个“好”程序员的指标(不管这意味着什么)。

坦率地说，如果一家公司所做的只是问我 Joel 的例题，甚至是测试我的递归和指针，我会觉得这是一种解脱！但根据我的经验，2018 年科技面试的现状更倾向于聪明的脑筋急转弯。例如，有一次我遇到了从两个数组中重新组合二叉树的问题。在另一个例子中，我被要求找出二维数组中 N 个矩形的坐标和边界(其中 0 是背景，1 是矩形的一部分)。一家公司的面试协议甚至建议，一个人应该去 codewars.com 实习做准备。我这样做了，这也是我遇到上述解决方案的地方`const findOdd = (xs) => xs.reduce((a, b) => a ^ b)`被高度评为“最佳实践”,如果你能相信的话。

关键是，这些问题与如何成为一名优秀的开发人员无关，就像他们可能选择如何煮咖啡或系鞋带一样，也就是说，根本无关。正如拉兹洛·博克在他的巨著《工作法则》中指出的，能够快速解决这类问题是一种可以学习的独立技能！(我*高度推荐*！).这些脑筋急转弯并不能预测未来的表现；他们只是告诉你这个人擅长脑筋急转弯(也就是说，他们已经练习过了)。

我也曾经有一个同事(他面试过我并通过了)告诉我，他只是想看到一个人可以编码的证据。他特别选择的工具是相对简单的反转绳子的挑战。好吧，好吧，你想看到有人真的会编码，但是真的吗？这真的是一个真正的问题吗？我想知道是否有人雇佣过声称有制作软件的实际经验(或正规培训)但实际上不会编码的人。说真的！如果是这样的话，那就很容易终止雇佣对话了。

给我 10 分钟时间和候选人在一起，我可以问一些结构化的面试问题，这些问题比 100 个这些荒谬的测试给你的更多地让我了解他们作为开发人员的能力。我可以很自信地告诉你，这个人是否能写代码，*而不用看他们写代码。有一次，一位经理让我很为难，因为他让我“研究”一位候选人，而我花时间向 PBI 提问。我告诉他，根据他们的回答，我会非常清楚他们是否真的能胜任这份工作。我是对的。这个特别的 dev 非常能干，尽管在他的一些解决方案中有点太聪明了(在我看来)。*

我在这里离题了一点，但只是为了举例说明。典型的“观察某人编写代码”面试问题只不过是一种筛选掉那些在匆忙、人为、压力大的情况下表现不佳的人的方法，这些问题与软件制作业务没有什么关系。

因此，如果这些测试有(而且[已经被证明有](https://be.helpful.com/https-medium-com-fnthawar-helpful-technical-interviews-are-garbage-dc5d9aee5acd))如此小的预测能力，为什么在这么多年之后，它们*仍然是去筛选工具？我怀疑有一个答案。我认为懒惰是其中的一部分，或者更慷慨地说，优先权。在较小的商店里，开发人员(以及几乎没有受过培训的经理)被期望进行技术面试，这使得它成为一件临时的事情。他们可能记得他们是如何被训练的，在网上快速搜索技术面试问题，抓几个，然后去镇上。这种事我见过不止一次了！*

但是，即使在招聘机构人员充足的软件公司，他们也会使用这些。甚至有一个家庭手工业，声称通过为公司提供审查服务来为公司“解决”这个问题！我上面提到的鼓励人们实践 codewars 的公司就是这样一家公司——众多公司中的一家。有预算和员工做得更好的人只是*而不是*。

自然地，我们这些书呆子式的工程师喜欢尝试用“数据”来解决问题我们认为大多数人类问题可以通过某种软件干预得到更好的处理。所以我们把这个问题框定在诸如“我们如何消除偏见？”(这肯定是有价值的)，但我们似乎从来没有缩小这个额外的水平，并质疑这些类型的评估是否实际上有意义和预测性。这相当于试图弄清楚如何让一个钻头成为一个更好的锤子。

我认为任何人都很难证明在这类问题上的表现和实际工作能力之间的真正联系。这在很大程度上是因为能力的衡量本身并不是一项客观的活动。就像流程一样，每个开发团队都有自己的想法，知道哪些指标是重要的，每个公司都有不同的价值观，增加这些价值观是成功和竞争力的组成部分。就像大多数复杂的人类努力一样，一旦你将事情分离到足以客观地衡量它们的程度，它们就会过于专注，远离整体，从而失去真正的意义。

然而，我们仍然坚持使用这些看似更加客观的工具来评估候选人。我们为自己使用“数据”并利用我们的特殊工程技能来解决非常困难的以人为中心的问题而感到自豪，这个问题就是预测在为他人提供产品或服务的团队中的角色是否成功。当数据不一致时，我们当然会做人类的事情，忽略它或重新解释它或质疑它对这个或那个的有效性——因为我们的本性是证明我们已经相信的东西，而不是面对与之矛盾并要求改变我们信念的证据。

## [](#a-persistent-dangerous-mythology)一个持久的、危险的神话

所以这充其量是一个持续的神话。但这是一个危险的神话。因为除非你是一家顶尖的大型科技公司，有大量高素质的求职者登门造访，否则错过那些恰好在这类测试中表现不佳的优秀求职者的可能性非常高。尽管顶级科技公司能够承受(由于申请者数量庞大)许多假阴性，但业内大多数其他公司真的承受不起。众所周知，对人才的需求远远超过了供给，而制造更多障碍的公司——尤其是那些没有告诉你任何有意义的东西的公司——最终会遭受损失。

也有可能使多样性差距永久化，因为这类测试中隐含着一种“让我们看看你是否像我一样”的心态。在这种情况下,“我”是特殊的雪花天才开发人员，他们可以很容易地解决这类问题。女性已经在我们的行业中受苦了，因为错误的概括仍然非常普遍，以至于她们在这种工作中能力不足。多亏了詹姆斯·达莫尔的《宣言》和捍卫它的人们，我们在如此公开的展览上看到了这一点。无论社会文化和生物因素如何影响 T2 兴趣的平均差异，更有问题的观点是这种性别化的兴趣显示了个人的 T4 能力。换句话说，这些刻板印象导致实际上有能力并对 STEM 感兴趣的女性在制度上普遍被劝阻不要追求它。

正如本文所指出的，“例如，在工作面试中，工程师被期望站起来在白板上写代码，这是一个神圣的传统，这种高压局面对那些感到格格不入的人不利。”换句话说，这些面试技巧对那些已经有了归属感的人来说是非常不利的(也就是说，他们属于极小的、被人看不起的少数群体，约占 18%)。再加上相对较高的可能性，即(最有可能的)男性阿尔法极客面试遭受偏见(无意识或无意识)，这进一步使此类评估的结果偏向负面。

最重要的是，正如 Triplebyte 的这篇文章[指出的，还有其他*承认并声明*对开发人员的感知“类型”有偏见。所谓的“企业”和“学术”开发人员在这些公司遭受着强烈的负面偏见，我个人在反对 C#的精英主义中看到了这一点(我碰巧有很强的 c#背景)和。总的来说是网络。对许多人来说，我被微软授予 MVP 奇怪地是我能力的反面指标。谢天谢地，我对我深厚的 JS 经验有了一些信任(考虑到 JS 已经被“真正的”开发人员耗尽了多长时间，这对我来说是相当讽刺的)。无论如何，很明显，如果你不了解“酷”的技术和背景，这种偏见会导致面试偏向负面。](https://triplebyte.com/blog/who-y-combinator-companies-want#.8dspa9u3p)

所谓的企业开发人员也更有可能没有正式的 CS 背景。他们更倾向于用有用的和有能力的抽象来解决实际的业务问题(比如。NET 或 Java 或诸如此类的东西)，而不是修补较低级的算法问题，或者例如编写自己的编译器。因此，他们不太可能在这类考试中表现出色。然而，他们是团队中绝对有生产力和能力的成员，为企业创造有利可图的软件。

重申一下，公司面临的危险是，由于很大程度上的任意性、偏见性和非预测性的技术评估，错过了原本非常合格的个人。他们还倾向于强化部落的、阿尔法极客的单一文化，这种文化排斥那些背景、才能和资质更加多样化的人的观点和经历。

## 我们都是特别的雪花独角兽

我认为这类评估的流行也是由对我们所做工作的难度的过度膨胀感所驱动的。每个公司都喜欢想象自己很特别，有独特的需求，只有“最好的”才能满足。我从未见过一家公司不以某种方式宣称自己“只雇佣最优秀的员工”当然，我们都知道这一点都不可行，但我们喜欢说出来，也喜欢相信它。

这种对自己部落优越性的信念导致我们高估了面试评估的重要性。我们的行业更加关注那些最终对我们团队的成功不太重要的事情，从而加剧了这个问题。这些技术采访说明了这一点。当我们“筛选”人员时，我们会根据技术列表和/或候选人的技术水平进行筛选。

相反，我们经常忽视或只是口头上说说更重要的方面，如“与他人相处融洽”、“工作努力”、“乐于助人”和“不是一个傲慢的人”我们喜欢抛出“热情”，但为什么这不是初步筛选的一部分？对于一个公司的成功来说，团队协作和人际/软技能与特定的技术能力一样重要。我认为，平均而言，情况更是如此。正如我在这里[所说的](https://medium.com/software-developer/the-dystopian-world-of-metrics-gone-wild-6d7347adbbae)，“一起工作不仅关乎技能，也关乎关系。”

此外，技术总是在变化和发展。我们今天构建应用程序的方式与 10 年前大不相同。在过去的五年中，我们使用的技术体系在前端和后端都有了很大的发展。这种流失没有减少的迹象。除非你只是雇佣一个短期合同工，否则有人有证据证明有效地学习新技术和新模式同样重要，如果不是更重要的话。为什么我们不提前筛选出那些呢？(为什么公司不更加重视持续学习？为什么没有更多的公司投资培训那些甚至没有经验的人呢？)

**事实是，除了一些特殊领域的例外，我们所做的大部分并不特别，与下一任雇主(或你的候选人的前任/现任雇主)也没有什么不同**。我们真的认为我们所做的比候选人工作过的最后一个地方(或 10 个地方)需要更多的审查吗？我们真的认为我们的审查过程如此特别，以至于我们会神奇地得到比行业其他人更好的结果吗？(提示:正确的答案应该是响亮的“不”。如果你怀疑我，[去浏览一下这个网站](https://rejected.us)。)

## [](#so-how-can-we-fix-it)那么如何才能修复呢？

首先，如果你做过这类面试，你不应该觉得愚蠢或糟糕。它们基本上仍然是“最先进的”，尽管有大量证据表明它们不具有预测性，但大多数人不知道这一点，因为他们太忙了，不知道这一点。我写这个的很大一部分原因是为了帮助提高意识。但是现在你知道了。现在你可以做得更好。

除非你雇佣一个几乎没有经验的人，否则他们很有可能已经在其他地方成功开发了软件。那么你的目标不是确定基本的编程能力。我认为你的目标主要是了解他们，并让他们了解你的公司和/或团队。开诚布公地谈论你的环境、你的实践、你当前的计划和未来的抱负。希望他们会有问题问你。使用基于表现的面试(查阅)深入了解他们过去是如何做事情的。在进行这些结构化面试时，仔细阅读消除偏见的技巧。我向你保证，这几个小时是值得的。

一种选择是[完全放弃技术面试](https://be.helpful.com/https-medium-com-fnthawar-helpful-technical-interviews-are-garbage-dc5d9aee5acd)。作者提出了一些好的观点。最终，*是*关于橡胶何时遇到路面。[这家公司](https://actionverb.com)甚至在你的头 90 天内提供“7500 美元的离职奖金”，我认为这是一个小聪明，因为它在万一失败的情况下更安全——你和他们都不需要对分离“感到难过”。类似这样的东西也可以减少招聘人员的费用和奖金，这些通常在 60-90 天后支付。这些方法确实要求在最初的日子里不要无精打采；你可能想准备好一种方法来确定是否每个人都满意这种合身。

但是我认为我们仍然可以进行某种程度的面试——只要它能给申请人一个展示他们实力的机会。这就是 PBI 方法的帮助之处，通过关注他们实际做了什么，而不是对假设或任意评价的提问。我发现这些都是很好的工具，可以更好地了解人们，很好地了解他们的优缺点。深入研究人们实际上做了什么，他们为什么这样做，并围绕这一点提出深刻的技术问题，会提供更多的见解——未来表现的最大预测因素是过去的表现。

但是如果你真的，真的必须看某人写代码，至少让它看起来像是他们每天要做的事情。如果你愿意给他们空间去学习“你的方式”做事，就像我希望的那样，那么也许不要根据你做了什么来要求一项任务，而是根据他们说他们做了什么来要求一项任务。与让他们匆忙猜测/了解你的期望相比，你能更好地将他们所做的与你的需求对应起来。

我[以前提倡](https://dev.to/ambroselittle/the-top-3-ways-to-fail-at-interviewing-software-developers-17me)(并使用)带回家的编码任务方法。这仍然比典型的亲自编码挑战好得多，但它很繁重，并且在个人投资方面提高了申请人的门槛，这在像我们这样的竞争激烈的市场中并不好。这也不利于那些在生活中不允许他们在主要工作之外投入太多时间的人(例如，单亲父母或大家庭的父母、有第二职业的人等)。).即使你提供补偿，这仍然是一个很大的要求。我在使用这种方法的过程中发现，带人到现场检查他们的解决方案，我获得了更多的价值。在设计和代码评审期间，我们从他们那里学到了比任务实现本身更多的东西。这让我想到..

我认为，更好的选择是我现在的雇主是如何做的。他们要求人们带来一个展示他们工作的项目，带领团队完成解决方案，并对其进行评审。未来表现的一个更好的指标是相关的工作样本，没有什么比他们实际做的真正工作更相关的了。我们喜欢鼓励人们带一些他们引以为豪的东西，但就个人而言，我会很高兴回顾一些人们不引以为豪的东西，并让他们解释为什么以及他们下次会如何做得不同。除了作为一个比随机编码挑战好得多、好得多的指示器之外，它还 1)大大减少了人为的压力，并且 2)经常模仿您在现实生活中在同行代码评审期间可能会做的事情。

如果有人没有一个他们可以随时和你一起回顾的项目，你仍然可以提供一个任务作为选择——甚至合作创建它。

显然，还有更多，但我希望，至少，这个评论让那些使用刻板的编码挑战方法的人重新考虑。我希望我们中有足够多的人继续分享为什么它们如此成问题，最终将我们行业的潮流转向更有效的方法，使其成为规范而不是例外。

为美好的新年干杯！
