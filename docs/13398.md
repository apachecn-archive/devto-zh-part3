# [](#introdu%C3%A7%C3%A3o)导言

Git 是一种允许您控制文件版本的版本控制系统。忘记通过将文件命名为:project-version 1、project-version 2、project-version-final 来创建目录。git 不再需要手动控制文件版本，也就是说，您可以在目录中找到所需的版本。

使用小命令可以进行此版本，我将介绍主要内容。对于最感兴趣的[文档](https://git-scm.com/docs)是一个例外，它用英语和葡萄牙语解释了系统的其他各种功能。

# [](#principais-comandos)主要命令

*开始之前，请确定您的电脑上已安装 git。*

命令:`git init`—启动所需文件夹中的版本控制

## [](#stages)阶段

我们在 git 有几个阶段:

*   Working Directory -这是我们修改文件时的初始阶段
*   临时-将文件添加到控件时，文件将进入此阶段
*   History -当我们使用控件保存此文件时，文件会到达此阶段

> `git add .`

*OBS:有这一点，是一个命令，表示所有文件都被添加到暂存阶段。*

`git add`因此负责将工作目录文件传递到暂存。

> `git add nome_do_arquivo`

*只传递一个文件。*

> `git commit -m "Mensagem"`

*备注:-m 标志是在提交时放置消息的命令。如果不使用标记，将打开一个编辑器来放置所需的消息。*
此命令保存历史上处于暂存状态的文件，即如果处于暂存状态的文件不存在，`git commit`不会保存任何内容。

> `git status`

*如果你碰巧不知道档案的阶段，git 状态会通知你。*

# [【Github】](https://github.com/)

现在想象一下，在一个公司中，有几个开发人员这样工作:一个需要更新一个代码，另一个 dev 正在开发一个新功能；当他们完成后，他们如何在不丢失任何代码行的情况下控制更改？

## [](#s%C3%B3-o-git-resolve)只有 Git 解决吗？

是的，只有 git 能解决，这是一个完整的系统。我们将使用另一个名为 remote 的功能。远程是一种添加服务器的功能，可为其版本控制的“*history*”阶段提供服务。而我们要使用的服务器是‘T2’github。如今有多种服务器，如 BitBucket 和 Gitlab，但 Github 是最著名的服务器，因此了解 Github 非常重要，因为它主要用于创建您的产品组合、协作进行开源项目以供企业查看(github 相当于名片)。

在 [github](//github.com) 上创建用户。创建一个新的存储库，这将是正在进行版本控制的代码所在的位置。
*有必要强调 github 只为处于历史阶段的文件提供服务*

> `git remote add "nome curto" "url"`

*加入您储存库的 URL*

> `git push -u origin master`

将“历史”阶段的资源复制到存储库

但到目前为止，我们只是把历史阶段换到了存储库，并没有解决问题。

魔法发生的地方:

> `git pull`
> 我们寻求从我们的存储库到我们控制的变化，即我们带来 github 的文件，在我们控制的范围内变化。

这就解决了最初的问题:当开发人员完成代码更改时，开发新功能的 dev 将执行“t0”pull 过程，并将代码升级到较新的版本，不会出现任何问题，也不会担心丢失其已完成的所有工作很漂亮吧？

不要浪费时间使用这个系统，明智地使用！
