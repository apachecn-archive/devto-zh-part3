# DevOps 是一种不断发展的文化，而不是一个团队

> 原文:[https://dev . to/Kyle Galbraith/devo PS-is-a-evolving-culture-not-a-team-2 EDI](https://dev.to/kylegalbraith/devops-is-an-evolving-culture-not-a-team-2edi)

关于 DevOps 的误传似乎越来越多。有时它被表示为工程结构中的另一个团队。其他时候是现有团队中的一个人(即 DevOps 工程师)。但在我看来，这些观点是错误的，没有抓住 DevOps 的要点。

DevOps 是一种文化，而不是一个团队或个人。它由团队培育的哲学、过程和工具的集合组成。它在发展，而不是一成不变的，因为团队和技术都在发展。

它打破团队之间的壁垒，而不是建立新的壁垒。有工程头脑的人变得精通运营，反之亦然。两者成为同质的一个。如果做得好，它可以改变一个组织和产品。

通过成为一个同质的单位，开发可以进行得更快。他们可以迭代他们的产品或服务，以更快的速度更好地适应用户需求。同时保持产品或服务的质量、交付和可靠性。

### 它是如何工作的呢？

从外面看，这听起来很复杂。

坦率地说，在“孤岛”团队很常见的组织中，这可能非常复杂。在这些文化中，运营团队感觉他们的工作处于危险之中。开发团队感觉好像他们被要求在他们已经做的基础上做更多的事情。

这些文化存在比表面看起来更深层次的问题。通常，这些感觉或信念源于对组织缺乏信任。其他时候，它们源于团队感觉与他们的工作目标脱节。

无论是什么情况，这些文化都需要重新设置，以便 DevOps 文化在它的位置上发展。

最简单的方法就是撕掉创可贴。在 DevOps 文化中，孤岛被拆除，开发不再与运营隔离。在很多情况下，这两个团队合二为一。这意味着团队处理整个应用程序生命周期。从开发到部署及以后，该团队对每一步都负有责任。

这可能会让人感觉非常混乱，就像发生重大流程变更时的情况一样。但是，对抗混乱的一个好方法是相信团队会为他们做出正确的决定。没有什么比不信任和不信任进行转变的人更能让这种转变失败得更快。

通过将这两个团队结合起来，可以在整个团队中形成一个强大的知识库。运营人员可以分享他们在部署和监控方面的专业知识。开发人员可以分享他们在应用程序开发中的专业知识。

这种知识共享通常会导致流程中大量重复、手动和缓慢的任务实现自动化。这种自动化加速了产品或服务的交付。自动化还降低了与部署相关的摩擦。

DevOps 文化需要改变思维方式。这些团队需要获得对他们所负责的系统和应用程序的控制权。团队拥有他们系统的整个生命周期。因此，他们专注于快速迭代、经常协作和持续改进。

这种完全的所有权导致效率降低，反馈循环更紧密，可靠性提高。

### [](#what-devops-is-not)什么叫 DevOps 不是...

我发现看到组织和团队实现与 DevOps 背道而驰的过程是令人沮丧的。团队或个人不对 DevOps 负责。

对我来说，这完全忽略了 DevOps 的意义。这是整个开发团队相互支持的文化。这种文化不是静态的，它是流动的，随着成员加入团队和技术的变化而发展。

此外，DevOps 的文化在不同的组织之间，甚至在不同的团队之间也会有所不同。这是因为一个团队或组织不是另一个的翻版。所以你可以从别人那里学到新的想法和概念，但是这些会融入你的团队或组织。

在 DevOps 原则支持的文化中，可以实现真正的业务影响。紧密的反馈循环和相关的自动化允许业务更快地迭代。更快的迭代意味着更快的决策和更快的用户价值。这些好处是非常真实的，但要获得这些好处，必须做好实施工作。

关注 DevOps 文化创造的利益和成果。

*   开发速度-更快地创新以推动业务成果。
*   快速交付——频繁可靠地发布。
*   可靠性——放心地发布你的产品或服务。
*   可扩展性——自动化和可靠性允许您相对容易地扩展您的架构。
*   协作——没有更多的墙，更多的交流，更多的所有权。

文化在进化，实现文化的过程也在进化，但是这些好处是你的北极星。这是一种旨在团结团队交付业务成果的合作。它推倒了围墙，加速了你的产品或服务的交付。

### [](#conclusion)结论

我们给这种快速可靠的迭代文化起了个名字，增加了协作和大量的自动化。那个名字是 DevOps，但实际上我们可以叫它任何我们想叫的名字。

使用 DevOps 要记住的重要事情是原则和它可以帮助您实现的切实好处。它不是另一个团队或一个人。这是一个团队共享的共同文化，它促进所有权、责任、增加的协作、更快的迭代和拆除阻碍这些的障碍。
