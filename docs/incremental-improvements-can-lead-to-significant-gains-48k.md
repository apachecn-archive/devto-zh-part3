# 渐进式改进可以带来显著的收益

> 原文：<https://dev.to/shrysr/incremental-improvements-can-lead-to-significant-gains-48k>

在阅读 James Clear 的《原子习惯》这本书时，我在思考我选择拥抱[Emacs](https://www.gnu.org/software/emacs/)并逐渐掌握它与书中宣扬的哲学密切相关。

我的努力最初是从渴望一个系统开始的，这个系统可以在一个定制的环境中量化和管理我的任务、习惯、笔记、博客写作、工作申请和项目，并且能够构建代码工具包来执行重复的任务。正如在之前的一篇博文中提到的，在选择 Emacs 之前，我尝试了几种方法。我的想法是找到或创建一个单一的系统来跟踪我生活中所有重要的事情(简单而高效)。这取代了使用多种工具和技术的零散方法，例如，将 Sublime Text / Atom 作为文本编辑器，将 [Todoist](https://todoist.com/?lang=en) 作为任务管理工具。

我从 Emacs 的普通配置开始，并煞费苦心地借用(并最终修改)lisp 片段来实现期望的“特性”或行为。每周只有几个专题，最初只关注 Org mode 的行为。那是将近 3 年前的事了。截至目前，我能够管理我的博客[hugo]，查看我的电子邮件[mu4e]，浏览网页[w3m]，从(几乎)任何地方无缝捕捉笔记/想法/任务[Org 模式]，在 IRC 上聊天，轻松构建多语言代码笔记本[Org babel]。所有这些都为我在速度和效率上提供了显著的优势，但仍有很大的改进空间。

当然，我今天看起来离我的目标更近了..然而，当我开始时，我不知道这是否是一个白日梦。这通常非常令人沮丧，从记忆 Emacs 中“疯狂”的按键，到努力让 lisp 代码片段按预期工作。

选择 Emacs 也有意想不到的回报。例如，将我的 notes 和 Emacs 配置与多台机器同步的需求引导我使用 Git。Magit 的易于访问的命令和相对可视化的界面对 git 的工作提供了巨大的帮助，尽管对 Git 的工作方式没有任何深入的技术知识。

我的 Emacs 之旅证明，随着时间的推移，一个递增的、复合的改进最终会带来显著的收益。承认我正站在巨人的肩膀上也很重要，令人敬畏的 Scimax 是我工具箱的基石。
