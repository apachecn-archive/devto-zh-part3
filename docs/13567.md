# 推动更深入的理解

> 原文:[https://dev.to/goyo/push-for-a-deeper-understanding-1fo6](https://dev.to/goyo/push-for-a-deeper-understanding-1fo6)

不久前，我和我的一位大学同事进行了一次奇怪的对话。他正努力在 Angular 中获得一些基本的东西，他向我寻求帮助。事情是这样的:

> 他:你知道为什么这可能不起作用吗？
> **我:**哦，对了，你调用东西的顺序不对。把这个移到这里，把这个移到那里。
> *...编码的声音...*
> **他:**是啊，现在工作了，谢谢！
> **我:**没问题，这就是自举的工作原理。看，当有棱角的时候...
> **他:**我不管角自举是怎么运作的。我想做一个 app，而不是学习 Angular 实现细节。
> **我:** Meh。

听起来很熟悉？迄今为止，在我的职业生涯中，至少有十几次这样的对话。

手头的问题，如果这还不明显的话，是我们经常太专注于解决手头的问题，以至于忘记问一个最重要的问题:*为什么？*

[![](../Images/9cf39f64d943aeb91facf19f8943a5c0.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--qzFvLoJN--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/vyuzsxwoxnq9strreojk.jpg)

这样做的一个后果很简单——下次我们遇到类似的问题时，我们会再次碰壁，不得不重新解决它。在“最好”的情况下，新问题将与旧问题足够相似，我们将能够复制以前找到的解决方案。

另一个后果更为深远。每当我们忘记(或有意识地拒绝)去推动更深的理解，我们就剥夺了自己一点点的成长。我们只是稍微老了一点，而不是学到了一些东西，成为了一个稍微好一点的程序员。

对于一个单独的例子，整体差异可能是微不足道的，几乎不明显。然而，随着时间的推移，小胜利(或他们的缺乏)会复合，差别是巨大的。

因此，这里是我的建议，你现在已经明白了:**当你解决一个问题时，问问你自己:为什么这个解决方案有效？允许我不再面对这个问题的基本原则是什么？**

祝您愉快！
