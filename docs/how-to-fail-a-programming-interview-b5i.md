# 如何在编程面试中失败

> 原文：<https://dev.to/mortoray/how-to-fail-a-programming-interview-b5i>

我进行过多次采访。我拒绝人的原因有很多，但是有反复出现的主题。我想列出最常见的问题。这些事情中有一些我曾经认为是显而易见的，而另一些可能不那么明显。

## 不知道编码的基本知识

这个让我很困扰。

创建列表、拆分列表和遍历列表应该不是一个挑战。函数、类、变量、语言的基本结构对你来说应该一点都不困难。被这些绊倒是我被拒绝的最快方式。

我不期望你了解整个语言，并且记住了它的标准库。不过，我说的是基础。你没有理由在面试中查找循环语法。我还发现，一些人知道的越多，他们编码的速度就越快。流利的语言可以减轻压力，解放你的大脑去思考问题而不是语法。

也许你使用很多语言和技术。这个我懂。我也是，而且我经常忘记如何用一种特定的语言做某事。但这不是借口。如果你在为面试选择语言，你需要在开始前复习一下基础知识。我知道 C++的基础比 Python 难，但是我希望你了解你选择的语言。

当我进行实时编码时，我为这种语言创建了一个备忘单。我可能需要的所有东西的快速参考——我不想在观众面前被 for 循环绊倒！你也可以这么做，然后在面试前背下来。只要提前了解语言，就没有借口。没有。

## 不说话，说话难听，或者不感兴趣

面试是关于交谈的，你如何交谈是至关重要的，这一点也不奇怪。

如果我问一个开放式的问题，那是因为我想要的不仅仅是几句话作为答案。抓住机会说话。在一些句子之间提供短暂的停顿；这给了我时间去理解，如果我想打断的话。

当你编码的时候，我希望你能谈谈你在做什么。我不想看屏幕上的代码。告诉我你为什么选择这个代码，或者你现在的目标是什么。如果我不知道你在想什么，我就帮不了你。我不想坐在那里看你挣扎。如果你说出来，我可以帮助你集中注意力，并引导你找到正确的答案。如果你什么都不说，你就冒着把时间浪费在不相关的事情上，或者浪费在对问题的误解上的风险。

学会如何说好。有些人经常说“嗯”和“啊”，以至于我很难理解他们。没有人期待世界级的演讲，但是干净的演讲对听众来说更容易。你希望面试官思考你的话的意思，而不是试图解读你在说什么。实践是你的朋友。不需要太多努力就能有明显的改善。

与不善言辞密切相关的是听起来不感兴趣。听到一个内心已死的人，而他的工作就像监狱一样，真是令人倒胃口。也可以被认为是不尊重。我正在花时间采访你，我希望你在这个过程中表现出一点兴趣。

说话清晰而有兴趣。

## 把问题想多了，复杂化了

面试不会很长，你需要做的所有工作都应该在分配的时间内轻松完成。如果你有一个小时的编码面试，你可能只有 35-40 分钟的编码时间。我们需要时间来交谈，而不仅仅是代码。

我不会给你一个耗时太长的问题。的确，我给出的初始问题只需要 10-20 分钟的编码时间。我更喜欢问后续问题，而不是一个大问题。然而，即使给你一个实质性的问题，你也必须假设这个问题在简短的面试中是可以解决的。

把问题复杂化，问太多如果，跑题，都会消耗你宝贵的时间。专注于手头的问题。不要寻找隐藏的诡计，也不要试图设计完美的系统。这也是为什么说出你的想法是至关重要的。这给面试官一个机会让你保持在正轨上。

解决给定的问题。

## 对上一份工作一无所知

我总是问以前的工作经历。不仅仅是通史，我还想听听你的具体作用。我想听你说你知道如何融入这个团队。我会根据你对产品和开发过程的描述来判断你的兴趣程度。

我在听具体的技术。很好，你是全栈开发者，但这意味着什么？有没有用 NodeJS 和 Angular？你是如何与服务器通信的？你用的是哪个数据库？此外，你实际上做了哪些工作？我想了解你的参与程度。

如果你不能提供细节，我会认为这是因为你要么夸大了你的职位，要么你没有参与这个项目。无论哪种方式，都会对你产生负面影响。

我也用这些信息来指导接下来的面试。如果你是一个顶尖的开发人员，我会保留有用的提示，给你更多的空间自己工作。如果你是新来的，我会更大方地提供帮助，以避免不必要的压力。

## 写马虎的代码

是面试。我并不指望你能获得完全干净的代码。然而，可接受的代码和草率的代码之间有很大的区别。

我最大的不满是糟糕的变量名。为什么要调用变量`n`？你在现场采访，想和我交流。清晰的名字有助于我理解你的代码。

如果这是一个在线面试，或者我已经给了你设备，我倾向于原谅一些格式问题。一些采访编辑让正确缩进或修复东西变得很有挑战性。但是我确实看到了一致性问题，比如大括号和分号在哪里。

我很少仅仅因为马虎而让别人失望。然而，在很多情况下，我看到糟糕的代码拖慢了面试:要么候选人努力理解他们的代码，要么我不得不问这是怎么回事。

## 侮辱某物或某人

就是不做！

不管你的观点或经历如何，不要对任何事情说任何负面的话。你完全不了解我、面试官或者我的背景。你不知道我认为什么概念是好是坏。你不知道我和谁一起工作过，也不知道我经常和谁聊天。

与面试官世界观相悖的负面点，立马伤害了他们对你的印象。即使我同意你的观点，现在也不是时候。我是一个思想开放的人，但仅仅是你选择这个时候抱怨一些事情的事实就会对你产生负面影响。我不想和消极的人一起工作。你是来推销自己的，不是来贬低别人的。

如果你被要求谈论一些不好的事情呢？这些问题会直接出现，比如“X 怎么了”，或者愚蠢的“你最糟糕的经历是什么？”这是你咽下你的骄傲，谈论学习经历，差异，和群体斗争的时候。你谦虚地说话，关注积极的一面。我喜欢人们把糟糕的情况变成积极的故事。我喜欢和这些人一起工作。

## 结论:大量软技能

编码面试是一个互动的过程。你需要和面试官一起工作。即使你的面试官不好，这也是真的。即使是没有受过训练的面试官也会倾向于说话，试图帮助你。此外，由于清单上的许多标准都是标准社交技能，每个人都在评判它们。对于没有接受过面试训练的人来说，这可能更糟糕，因为他们不太可能忽视面试压力。

如果你读过我的书，你会知道我非常重视人际交往能力。你可以看到这些失败中有多少与软技能有关。但是，如果你希望通过面试，你仍然需要有解决问题的代码。

本文中的一切都是为了得到一个好的解决方案。通过确保面试顺利，你可以充分利用你的时间。您得到了有用的反馈，可以专注于编写正确的代码。我知道不管你做什么，有些面试都会很艰难。但是，尽你最大的努力，绝对不要让他们变得更糟。

如果你碰巧犯了这样的错误？修好它。这些都是可以练习和提高的技巧。
