# 与人交谈比与机器交谈重要得多

编程是一项团队运动。在极少数情况下，你可能会看到一个人从零开始打造一个出色的产品，但在绝大多数情况下，你需要一个团队。

沟通技巧可以成就或毁掉一个项目。别担心，不仅仅是你和你的团队，美国国家航空航天局也在努力应对这个问题。

对项目的成功来说，软专业技能可能比纯技术技能更重要。如果你雇佣了 5 个世界上最好的数据库专家，但他们拒绝相互交流，那么谁会在乎你最终拥有 5 个不同的 MySQL、Aurora 和 MongoDB 实例。

# [](#have-a-deep-understanding-of-what-you-are-building-and-why)深刻理解你在建设什么，为什么要建设

大多数人在有目标感时会更快乐。这也适用于工作。

作为一名软件开发人员，你的目标不是将 JIRA 翻译成 JavaScript，将 Trello 翻译成 C#等等。

你的目标是用代码解决问题。

如果你对你正在构建/维护的系统有深刻的理解，那么你可以在纯技术之外做决定。这个特性有必要吗？它解决什么问题？我们能用别的方法解决这个问题吗？我们**希望**首先解决这个问题吗？

这种思路有时被称为*业务环境*，但是如果你想做好你的工作，你不仅要理解环境，还要能够塑造和影响环境。你不必在你的组织中有一个 C 级职位来影响你的产品。或者至少理解它。

# 如果你团队中的代码审查是一种有压力的经历，那么你这样做是非常非常错误的

哦天啊。代码审查。

我们真的没有想到这一点，但是将我们的作品公之于众并让其他人审阅的行为对我们的职业来说有点独特。难怪人们会对整个经历感到焦虑。

我个人见过有人在 X 不在办公室，或者 Y 出差的时候提交代码评审。x 是一个杰出的程序员，但是忍受他的代码审查过程是一件苦差事。如果你留下 50 个吹毛求疵(那是一个词吗？)，在一个初级程序员的人的 PR 下刻薄的评论，你不是在证明你作为开发人员的优越感。你在证明你不是一个好人。

> 好吧，但是当我看到这个特性完全被破坏了，我该怎么办？

站起来。私下接触那个人**。与他们交谈，找出他们为什么以这种方式实现代码。**

 **大多数人不想写糟糕的代码。如果他们这样做了，他们可能正在处理你没有意识到的约束。他们也可能不擅长编程，这是你作为导师大放异彩的机会。

# [](#something-will-go-wrong-be-prepared)会出事，做好准备

根据维基百科:

> 墨菲定律是一句格言或警句，通常表述为:“任何可能出错的事情都会出错”。

这是千真万确的事情之一。在设计一个系统的时候，总是假设有些东西可能会坏掉。

如果您正在构建一个登录表单，假设人们会将整本书复制并粘贴到密码字段中。

如果你正在建立一个所见即所得的窗口，假设有人会试图打破它，他们很可能会成功。

如果你有一个数据库，它会在某个时候关闭。如果您没有测试过从备份中恢复数据库，那么它就不是备份。

如果你在观众面前做现场演示，确保演示在线、离线、倒置和水下都可以进行。

# [](#dont-be-afraid-to-say-i-dont-know)不要害怕说“我不知道”

在我的职位旁边有一个**高级**最棒的部分是，我可以*最终*回答一个问题说:

> 我不知道，没试过。我去看看，然后给你回复。

当我大三的时候，我很害怕有人发现我是个骗子。作为一名开发人员，几年后——如果我没有看到一些东西，那可能是它直到现在都不相关。或者我又有一项很酷的技术要学。终身学习不是软件开发中的时髦词，而是现实。

或者我只是一个不可思议的骗子，设法愚弄所有的人，让他们相信我可以做好我的工作。世事难料。

# [](#learn-in-public)当众学习

一旦你从“我不知道”变成“好吧，这很有趣”——与他人分享。写一篇博客，录一段视频，在公司知识分享活动上做一次演讲，或者只是...告诉别人。如果你认为某件事对每个人来说都是显而易见的，事实并非如此。即使是最资深的人也有向初学者学习的地方，反之亦然。

教学是确保你真正理解所讨论的主题的一种不可思议的方式。

俗话说:

> 一个人教，两个人学——有人很聪明
