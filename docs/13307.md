# 书评:学习图表

> 原文:[https://dev . to/raymond Camden/book-review-learning-graph QL-2d 67](https://dev.to/raymondcamden/book-review-learning-graphql-2d67)

在过去一周左右的时间里，我一直在尽可能多地学习 GraphQL。我已经参加了一些关于这个主题的会议，对它有了一个基本的了解，但我想真正开始巩固我对这个主题的知识，为我们工作中的一些事情做准备。我在 Twitter 上偶然看到一个关于[“学习 graph QL”](http://shop.oreilly.com/product/0636920137269.do)的话题，所以我决定读一读。这本书是由亚历克斯·班克斯和伊芙·波尔切洛写的，不到 200 页，所以可以很快阅读，但包含了很多材料。我将在下面详细介绍，但要点是我绝对推荐它，我将购买一份硬拷贝，在接下来的几个月里放在我的办公桌上。

正如我所说，这本书正好在 200 页(198 页)以下，由以下几个部分组成:

*   “欢迎使用 GraphQL”很好地介绍了 GraphQL 的“为什么”。正如我所说的，我已经很好地理解了为什么，但这一部分做得真的很好，可以成为说服管理层为什么应该采用它的好方法。即使你已经参加过一些入门课程，我还是建议你阅读这篇文章，巩固这些原则。

*   “图论”涵盖了 GraphQL 背后的一些学术方面，老实说，我本打算跳过这一节。我真的很高兴我没有。它为图的概念提供了一个伟大的背景和一些迷人的历史背景。我很欣赏这些例子中有多少是与现有的社交网络和应用程序联系在一起的，这确实有助于它们变得有意义。我无法告诉你有多少次我在面向对象相关的书籍中使用了与实际开发毫无关联的例子，所以在野外使用现有的例子真的有助于理解它。

*   “GraphQL 查询语言”深入到了 GraphQL 的语法中，尽管我以前已经看过很多，但我学到了很多，并且对 GraphQL 的强大功能印象更加深刻。正如我所说的，我以前参加过 GraphQL 的演示，我完全理解您只能涉及这么多，但是我真的对查询语言的一些更高级的方面感到惊讶。

*   “设计模式”是上一节的一个很好的反面。它描述了如何定义将使用查询语言的模式。与前一部分一样，我以前也看到过这一部分，但是看到您可以深入到什么程度是令人大开眼界的。基本上，这一节和上一节把我从“是的，GraphQL 看起来很整洁”变成了“我再也不想用 REST 了”的 kool-aid 饮者。

*   “创建 GraphQL API”将带您构建一个真实的、简单的应用程序。总的来说，这部分做得很好。我能够理解这一章 75%的内容，和文本一起构建应用程序。尽管在某个时候这样做变得更加困难。要明确的是，有一个 GitHub repo，其中包含每个部分的完整代码，您可以随时抓取。此外，我不知道读者的意图是否是在阅读时实际构建应用程序。但是我非常相信输入代码是帮助我学习的一种方式，感觉作者可以做更多的工作来实现这一点。尽管我有这些问题，但我确实喜欢这一章以如何向应用程序添加登录和授权结束的事实，这是我以前在 GraphQL 中没有见过的。

*   “GraphQL 客户端”详细介绍了与 GraphQL APIs 的集成。它涵盖了从纯 cUrl 调用到优秀的节点库和前端实用程序的所有内容。这里的例子是基于 React 的，所以我没有尝试自己构建它，但我计划很快看看 Vue 支持。

*   “真实世界中的 GraphQL”涵盖了订阅——一种使用 websockets 监听数据变化的方式，然后深入到一些很棒的主题，如文件上传、安全性和性能。我以前从未见过这些话题(再说一次，你在会议陈述中只有这么多时间)，这本书很好地介绍了它们。

总的来说，非常好的一本书，我推荐。正如我所说的，我计划买一本纸质的，这是我很少为技术书籍做的事。不过，我想指出两个问题。

首先，本书假设读者熟悉 ES6 语法。如果您还不习惯粗箭头函数、展开操作符和模板字符串，那么您可能会对下面的代码有一些问题。我并没有指出这是一个问题，只是作为一个警告。如果你在这方面需要帮助，我会在 [MDN](https://developer.mozilla.org/en-US/) 查看优秀的教程，或者在评论中告诉我，我会尽力回答。
