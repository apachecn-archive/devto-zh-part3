# 为什么不写测试？

> 原文:[https://dev.to/pbeekums/why-are-tests-not-written-1lo8](https://dev.to/pbeekums/why-are-tests-not-written-1lo8)

很难反对编写测试。相对于编写它们所花费的时间，它们节省了大量的时间。产品质量的提高也给用户留下了良好的印象。然而很多时候，测试并没有被写出来。即使对于像我这样的自动化测试的拥护者来说，也有不写测试“就这一次”或者“这个特性不需要测试”的诱惑。

这是为什么呢？有了这么明确的奖励，为什么借口来得这么容易？

虽然我怀疑一篇博文可能有太多可能的答案，但我认为我已经找到了一些更常见的答案。

最明显的一点是，测试无助于软件工作。用户不在乎你有没有测试。他们关心你的产品是否有效。虽然从长远来看测试可以帮助实现这个目标，但是如果你看到你的产品在工作，那么就很容易说它已经完成了。

有一些过程黑客可以尝试将编写测试作为“完成定义”的一部分。然而，当事情到了紧要关头，必须在截止日期前完成时，首先被削减的是考试。截止日期是为了让工作软件到达用户手中，他们再一次不关心你是否有测试。

我们喜欢说“哦，我们将在发布后编写测试。”问题是，许多功能也会在截止日期前被删除。发布后会做些什么？特色。不是测试。测试总是被推后。

另一个问题是，当涉及到自动化测试时，有一个很大的延迟。仅仅编写一个测试并让它在以后捕获一个 bug 是不够的。需要有足够多的测试让你感觉舒服，不需要手工测试每一个可能的回归。如果在有几十个案例的情况下，你只有少数几个测试，那么这些测试就不能提供那种信心。您仍然需要进行大量的手工测试，而这些测试为您节省了很少的时间。进行一些测试实际上非常接近于没有测试。不幸的是，构建足够多的测试来达到可以信任测试套件的临界点需要时间。

类似地，编写测试最困难的部分是开始。一旦有人用用于编写测试的模式建立了初始测试套件，添加另一个测试就相当容易了。首先，创造这一切需要大量的努力。弄清楚什么样的测试策略实际上会提供高度的可信度是需要时间的。单元测试应该包含什么？集成测试应该包括什么？我们应该有单独的前端和后端集成测试吗？我们会使用 Selenium 进行更全面的测试吗？我们如何着手为这些类型的测试播种数据库？

所有这些问题都很容易由以前回答过的人来解决。但并不是每个团队都可以接触到有这方面知识的人。即使有了这些知识，所有的软件系统都有自己的怪癖，需要对测试策略进行一些调整。搞清楚这一点需要大量的前期时间，并且会增加延迟满足感。

克服这些问题没有简单的答案。在某种程度上，这归结为一个想做测试的意愿问题。Maleega 最初的 MVP 是零测试的，很多 MVP 可能都是这样。尽管我很想继续下去，但我知道我必须在某个时候咬紧牙关，花上几周时间进行测试。看到以前公司的好处，无疑有助于克服延迟满足的诅咒。
