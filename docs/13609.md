# 套路，弱势 vs 强势

> 原文:[https://dev.to/jjsantos/routine-weakness-vs-strength-3pin](https://dev.to/jjsantos/routine-weakness-vs-strength-3pin)

你好。

你们大多数人可能都读过或听过这句话:“权力越大，责任越大”，即使你不知道这句话是谁说的，但你已经明白这是什么意思了。

我想告诉你常规的力量。常规有好有坏，这就是力量所在，如果你知道如何使用它们，常规会是一个很好的工具。

一个好的例程可以帮助你比以前更快地完成测试，因为你获得的关于系统的知识可以让你准确地到达你想要的地方或准确地完成任何任务，但随着时间的推移，这种优势可能会成为弱点。每当你的团队试图理解某个特定的功能时，他们都会对系统的丰富知识感到欣慰；这里的问题是，这些知识会让你对系统可能存在的 bug 视而不见，主要是在系统快速增长的时候。

[![Image from pixabay](../Images/d395b28fb72046f4a59332ef5f8d16c2.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--wJc-_33m--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/ry6dc5mfgwqrye9nrrhg.jpg) 这是惊人的，因为我们可以了解一个真正的用户将如何使用我们的系统；但是一旦你理解了一切是如何工作的，你总是会从 A 点直接到 B 点，然而有一些测试用例，比如从 A 点到 C 点，或者从 B 点到 A 点，这些都是可选的测试用例，必须进行测试以避免最终产品中的意外行为。这就是我所说的“弱例行程序”，我这样叫是因为久而久之，重复这种类型的例行程序会使我们的测试结果更加“弱”，因为作为人类，我们往往会忽略或忘记某些测试案例...

为了避免这种“弱例程”的情况，您必须开始自动化测试。测试自动化允许您将时间投入到对您的项目以及将来要测试的新功能最重要的任务中。

[![Image from pexels](../Images/726eb7e0f88b559c0ab17e0bdebb5226.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--4jF50B0p--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/wbx0z0ogr686tdk7gb1w.jpg)

当我开始从事测试自动化的工作时，我没有意识到我的工作的重要性。经理要求所有的测试都必须手动完成，所以我做了同样的事情两次(LOL！).在一天结束的时候，我做了手动和自动测试，这让我觉得我没有取得任何进展，或者我没有生产力。当我改变了我的职位并开始领导一个 QA 项目时，我才真正明白我的技能是重要的，他们低估了测试自动化；我开始自动化系统的主要流程，仅在几个月内，我就完成了系统主要核心的自动化，并涵盖了最关键的“备选测试用例”，因此现在我可以完全专注于正确审查新功能，然后将它们添加到自动化框架中。

我只是想与你分享，希望它会有用或激励。测试自动化通过能够自动执行那些常规的过程，使我的生活更有效率。
