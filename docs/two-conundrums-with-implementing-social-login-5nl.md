# 实现社交登录的两个难题

> 原文：<https://dev.to/ryan/two-conundrums-with-implementing-social-login-5nl>

实现社交登录一开始很简单，但是有很多奇怪的情况需要考虑，有很多不同的潜在方法来处理它们。

例如，我正在开发一个 web 应用程序，用户可以注册一个电子邮件地址或某些社交账户。用户应该能够自由链接或取消链接任何社交账户，对不对？我正在努力思考一种方法来处理一些情况，而不会让用户觉得太复杂。

# 例 1

假设一个用户注册了一个电子邮件地址，现在他们想链接他们的脸书帐户。然而，该脸书帐户已被另一个帐户使用(很可能是同一个用户点击了一次脸书登录，无意中创建了一个新帐户。)

在这种情况下，您会提示用户合并他们的帐户吗？(实现起来很棘手，而且具有破坏性——也许他们会合并错误的帐户，而你无法修复这个错误)或者你会告诉用户“你必须登录到另一个帐户并先取消链接”吗？(对用户来说困难/耗时)

# 例 2

用户首先注册了一个社交帐户，但现在想要取消该帐户的链接。必须有一种方法来验证用户的身份，所以他们需要提供一个电子邮件地址。为了删除其他信息而要求用户放弃个人信息似乎是错误的。但是我能看到的唯一选择是让用户选择完全删除他们的账户，这看起来也是有敌意的(你想删除脸书？给我你的电子邮件或吻别你的帐户😨)或者我可以提示他们设置一个仅用于登录的用户名，但这也可能会令人困惑，他们可能会被锁定在自己的帐户之外，而且没有办法恢复。他们将需要提供一个有效的电子邮件进行验证，所以这似乎是一个毫无意义的额外步骤。

似乎没有好的答案！社交登录应该会让事情变得更简单，不是吗？怎么办？
