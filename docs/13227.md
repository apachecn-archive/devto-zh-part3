# 关于评论的评论

> 原文:[https://dev.to/recursivefaults/a-comment-on-commenting-3c97](https://dev.to/recursivefaults/a-comment-on-commenting-3c97)

史蒂夫是软件新手，他们花大量时间阅读文档，编写代码，注释掉代码，然后再试。

劳拉的工作方式有点不同。她不会注释掉代码块，但她在工作时会使用分支和堆栈来保存自己的想法。

Ralph 不会留下注释代码，也很少使用分支或隐藏。他会经常把自己的工作重置回已知状态。他扔掉所有不用的东西。

你很可能是这三个开发人员中的一员。史蒂夫和劳拉都希望保留他们工作和想法的旧版本。拉尔夫只保留有用的东西。史蒂夫和劳拉想知道，“如果我需要它怎么办。”拉尔夫说，“你不会需要它的。”

在我继续说下去之前，让我说一下。这三种工作模式都是有效的。它们各有各的后果。考虑以不同的方式工作看起来是什么样子可能会暴露你当前工作流程中你喜欢或不喜欢的某些事情。

## [](#about-steve)关于史蒂夫

史蒂夫将完成他的专题，并开始清理代码。史蒂夫的清理工作主要是移除他保留的所有注释掉的代码。当他做完这些后，他还有多少精力去重构呢？当他进行重构时，他会在试图清理代码的同时开始另一个注释代码的循环吗？他很可能会肤浅地重构。他将重命名一些变量和函数。他已经累了，删除注释的行为是有意清理代码的行为。

## [](#about-laura)关于劳拉

劳拉开始她的工作，构建她的代码，保留有用的东西，但是频繁地隐藏和分支，以探索没有想法。她留下了一堆堆的东西和树枝。也许她会更深入地重构。毕竟，她还没有花时间收拾东西。她将通过重命名事物或者提取代码到新方法来重构她的代码。当她回到她的下一部电影时，她看到了一堆留下的树枝和树枝。她不确定里面有什么，但她很快就会把它们扔掉。也许她会不小心合并一个或者删错一个。她有一些额外的工作来维护这些仓库和分支。

## [](#about-ralph)关于拉尔夫

拉尔夫除了工作软件什么都没有。他很少分支，什么都不评论。他能做到这一点而不丢失太多工作的唯一方法是通过微小的提交。他做一些小的，有时是一行，修改并提交它们。如果他错了，他会回去。他使用源代码控制和他的编辑器来为他保存状态。当他清理他的代码并重构时，他重命名、提取，甚至可能有精力构建新的类并查看类之间的耦合和内聚。他可能会看到其他重构的机会。他不需要在工作后的清理上投入任何东西。他不需要处理仓库或分支。他不必删除注释过的代码。

## [](#walk-in-their-shoes)步其后尘

我建议任何想要进步的人，想想你在这个故事中是谁。试着站在别人的立场上走几天。会很有挑战性。你甚至可能会得出这样的结论:这种新方式行不通。请记住，这些人中的每一个人都从事软件行业。每一种都有效。

我建议像拉尔夫那样工作。他在他的工作流程中是有效的，因为他的工作规模很小，并且一直维护工作代码。对你来说那看起来像什么？
