# 不是 Dev，不是 Ops，我们是什么？

> 原文：<https://dev.to/sloan/not-dev-not-ops-what-are-we--1kkc>

这是一个不愿意透露姓名的会员发来的匿名帖子。请仔细考虑你的回答，因为这些通常是很难写的帖子。如果你想留下匿名评论，请发电子邮件至。

一些背景供参考:

我是一个长期从事“运营”的人，为了乐趣而编写 web 应用程序。我职业生涯的大部分时间都是在一个运营团队担任经理，该团队管理应用，但不管理主机。我们与 infra 合作，根据我们的规格设置服务器，安装存储并开放端口。我们被授予了配置机器的证书。我们处理应用程序的日常支持操作。在 DevOps 成为流行语之前，我很幸运地成为了实施它的草根运动的一部分。

从那以后，我做了一些咨询工作，并在那里工作了几年。然后我得到了管理一个站点可靠性工程团队的机会。这对我来说是一个新的概念，但在做了一些研究后，对我来说并不新鲜。

在那里呆几年是一个不争的事实。我们不是 SRE 队。我为一家拥有微服务架构的大公司工作，总共大约有 300 项服务。运营团队没有与开发团队整合。我现在正试图重组我的团队，以更好地适应公司的需求。

我的建议是我们的八人团队与产品负责人合作，了解他们领域下的应用程序组合。我们将帮助确保像优雅的退化和可观察性这样的事情与功能特性一起被考虑。我们将帮助建立监视器和警报，并与我们的 24/7 noc 紧密合作。我们将为 noc 建立和管理警报和事件管理工具，成为 2 级支持和严重级事件经理。我们还将负责计划外的生产变更、湖功能标记、缓存刷新和临时批处理作业。

我们不打算参与基础设施架构或服务器管理之类的事情。我们更倾向于面对基础设施。那里有需求，也有巨大的机会。我在上面第一段中看到了很多我曾经管理的团队的相似之处。有人知道类似的团队结构或概念吗？我在寻找例子或资源来帮助设定方向。

TL；有没有开发人员知道有一个团队在生产中“管理”和支持他们的应用程序，但不是传统的运营团队？
