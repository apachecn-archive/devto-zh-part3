# 探索代码库

> 原文:[https://dev.to/busypeoples/exploring-a-codebase-5ef9](https://dev.to/busypeoples/exploring-a-codebase-5ef9)

在许多情况下，我们可能会接触到现有的代码库。无论是加入一个现有的产品，与一个团队合作来帮助解决一个问题，还是被引入来进行代码评审，或者从一个不同的团队接管一个现有的代码库。

我们作为开发者看到的是其他开发者一开始写的东西。我们试图浏览代码库，试图更好地理解整体结构。我们寻找能帮助我们形成想法的东西，比如函数，比如文件结构。我们寻找快速指标来帮助我们评估质量。我们可能会尝试识别模式，我们可能会搜索一致性或一个自述文件。我们可能会尝试查找文档、通读文档块或搜索其他指标。

我们的第一步主要是通过尝试识别架构来快速获得更好的**高层次概述**,同时查看代码是如何构造的，以形成关于质量等更具体的想法。

与了解代码库的人相比，我们有一个优势:我们可以从一个全新的角度来看待它。另一方面，我们也缺乏关于历史背景、工作流程、时间限制、不断变化的团队结构等方面的信息和知识。我们无法判断产品是由一个简明的计划驱动的，还是由任意的和不断变化的需求驱动的。至少在最开始的时候是很难理解的。

到目前为止，这并不是什么新鲜事。每个人进入现有代码库的方式不同，但我们主要做的是关注代码，并且我们主要假设它是在完美的条件下编写的。当我们问自己:“当它可以这样实现时，为什么它是这样构造的”，那么我们是在假设**理想条件**，但也可能缺乏一些背景。

但是，如果我们退一步，从不同的角度思考，我们还能从现有的代码库中学到什么呢？也许问我们自己一些问题是好的。

*   写代码时的条件是什么？

*   这些情况会随着时间而改变吗？

*   产品管理或一般管理扮演了什么角色？

*   有时间讨论技术债务吗？

*   最初的设计是由唯一的开发商推动的吗？

*   是否存在导致过度工程化或欠工程化的约束或驱动因素？

*   有时间做一些重构吗？

*   范围是否随时间而变化？

*   谁推动决策？

*   谁推动时间表和范围？

代码库不会提供上述所有问题的所有答案。但它所做的，是帮助我们摆脱认为这只是开发者的问题的想法。这有助于我们摆脱开发者是唯一对代码负责的人的想法。代码是不同过程、人员、协作、驱动因素和约束的结果。
