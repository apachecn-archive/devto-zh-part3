# 使用左移测试方法的连续测试

> 原文：<https://dev.to/lambdatest/continuous-testing-using-shift-left-testing-approach-31i4>

软件行业正以极快的速度变化，而“上市时间”变得比以往任何时候都更加重要。“瀑布模型”是组织使用的最突出的软件模型之一，但它有一个主要缺点——在软件开发的每个阶段都要进行测试，因此在软件进入生产阶段之前，高度关键的错误不会被注意到。如果“有问题的软件”被发送给最终客户，这种情况也可能导致业务损失，这是每个传统的“先开发后测试”模型中都存在的问题。这产生了更多进化的模型，如“[左移测试](https://goo.gl/7ypqdR)”&“连续交付测试”，其中测试被认为是开发功能不可分割的一部分。

## 什么是左移测试？

在传统的开发模型中，测试仅在开发周期的末尾执行，即需求收集(计划)、分析、设计、开发、测试和部署。测试、识别和分类问题，以及将开发变更推进到产品中是有成本的。如果测试/验证团队在[生命周期](https://goo.gl/uF2Nm2)结束时遇到“非常严重的问题”(发布阻止问题)，成本可能会上升。这样的问题甚至会导致整个发布的停滞。

有了左移测试，测试不再由测试/验证团队孤立地执行；但它与产品开发同步进行。有了这样的方法，测试团队更有权力，因为他们与项目的其他利益相关者紧密合作，例如产品计划团队、项目计划团队、开发团队、发布团队等。因为测试与开发过程紧密相关；可以更快地识别和解决 bug。尽管开发和测试团队从项目开始就同步工作，但是使用右移方法向 [DevOps](https://goo.gl/vn4ogd) (开发人员+运营)的过渡也同样重要。组合的方法(左移测试&右移测试)被称为“[连续测试](https://goo.gl/Ptm4ro)”。

## 连续测试

如前所述，左移测试更多的是关于以更加同步的方式执行开发和测试工作；另一方面，持续测试将测试与开发和运营(即 DevOps)同步，以便实现必要的开发和业务目标。持续测试通过集成 DevOps 带来了左移测试的最佳实践。

质量保证(QA)团队在持续交付的过程中发挥着至关重要的作用，通过这一过程，新的变更将以更可持续的方式发布给最终客户。他们在持续交付中的工作是通过确保开发团队解决 bug 来加速部署&变更被推送到相关分支，如生产环境或[试运行环境](https://goo.gl/oMakZZ)。持续的测试确保了被发现的错误在正确的时间被修复，这样一个高质量的产品才能被运送到最终客户手中。

简单来说，左移测试强调在 SDLC(软件开发生命周期)的早期阶段进行测试的重要性，即集成 QA 和 Dev 持续测试将 DevOps 的因素添加到整体等式中。这一有价值的附加功能使得以更快的速度解决问题并将更改发送给最终客户成为可能。

## 连续测试和左移测试的区别

现在我们知道了左移测试和连续测试，让我们看看两者之间的一些关键区别

1.  **UI 之外的测试**:在 Shift-Left 测试中，在发布之前对 [UI & UX 测试](https://goo.gl/KG1Go6)的关注增加了。然而，web 应用程序或网站不仅仅是令人印象深刻的 UI。除了有效的用户界面，性能在塑造产品的过程中也扮演着重要的角色。作为一名开发人员，您必须确保页面之间的过渡是平滑/无故障的，web 本地组件的正确组合(即使用混合方法)，CPU 消耗最少，散热要求得到满足，等等。在持续测试过程中，这些超出网站/网络应用程序“外观”&感觉的因素都被考虑在内。因此，连续测试是更好的，因为它考虑了“用户界面、UX 和性能指标”，这些都是任何产品(网站/网络应用)的关键要素
2.  测试中的更多实时场景:众所周知，产品开发的任何阶段都会出现 bug。这使得“产品测试”成为开发生命周期不可或缺的一部分。左移测试侧重于早期阶段的测试，即开发的生产前阶段，而连续测试侧重于开发周期中从生产前到生产后的部分。只有在极端条件下测试您的系统时，才会出现一些错误，而且在早期发现此类问题的可能性不大，因为“组件/模块”仍处于设计/开发阶段。
    这就是持续测试在对产品进行全面测试时发挥关键作用的地方。考虑一个移动应用(app)；当应用在“理想条件”下执行时，即 CPU 内核以最高频率执行，GPU 以高频率执行等，它可能会表现良好。；而如果在具有低配置的移动设备上执行测试，则性能会显著变化。除了性能之外，网络应用的外观&感觉可能会因设备而异，例如，当在安装了 Android-O 的“平板设备”上呈现相同的页面时，在安装了 Android-O 的“移动设备”上呈现网络应用的“页面/组件”的方式&可能会有很大的不同。就移动&平板设备而言，存在大量碎片，并且随着设备上操作系统版本的不同而变得更加复杂。为了确保你的网络应用能在指定的设备和各自的浏览器上完美运行，必须进行彻底的[跨浏览器测试](https://goo.gl/1pEJTT)；这在仍处于前期制作阶段的网络应用上是做不到的。这就是连续测试发挥关键作用的地方，因为它考虑了实时场景。

3.  **增加学习，更积极**:左移测试增强了测试/验证团队的能力，因为它为测试人员提供了一个与产品经理、项目经理、开发人员、其他测试工程师等密切合作[的机会](https://www.lambdatest.com/blog/top-19-collaboration-tools-for-your-software-testing-team/)。这有助于在测试团队中灌输‘开发思维’。这种方法有助于他们在产品规划&开发阶段将问题可视化。因此，测试人员可以在产品开发周期的早期阶段报告问题。他们感觉更有动力，因为每次互动都有学习的机会。在某些情况下，测试团队的思维过程会变得“线性”。
    持续测试是在一个更加“完美”的产品上进行的，因为它是在产品开发的每个阶段进行的&测试。因此，测试人员在连续测试过程中遇到的严重程度&类型的问题可能与左移测试中发现的问题非常不同。这使得测试团队更加了解“关键问题”,并且他们可以在每个测试周期的开始就结合这些知识。测试团队的整体动机&更加积极主动，因为有了更多的知识&他们可以利用这些知识来改进产品。

4.  **持续集成&持续部署**:今天的现代开发实践利用了 [CI(持续集成)和持续部署(CD)](https://goo.gl/7FhofN) 。CI 是一种用于简化整个“软件发布”过程的常见实践。另一方面，CD 确保重要的特性&错误修复定期&及时地发送给客户。通过实施 CI，重要的变更被定期合并到“生产部门”&相同的变更通过 CD 发送给客户。

在将变更发送给最终客户之前，必须对它们进行测试，这就是连续测试在验证这些变更时发挥重要作用的地方。随着产品开发达到更加“成熟的阶段”，CI & CD 周期的数量也会增加。为了确保将高质量的产品运送给客户，还应进行连续测试的连续[测试周期](https://goo.gl/uEnfVy)(在 CI & CD 的每个阶段)。

## 左移测试——成功的连续测试的子集

所以我们研究了左移测试和连续测试之间的区别。但仍然存在的问题是，我们是否应该将它们视为两种完全不同的测试方法？答案是否定的！它们提供了差异，但是正如我们前面所看到的，连续测试包括从左移到右移的所有测试。因此，认为它们相互排斥是错误的。相反，左移测试可以作为连续测试的一个子集。当结合得好的时候，左移测试和连续测试可以在市场上推出更快更健壮的产品。

## 结论

越来越多的组织职能正在向“持续测试”这样的实践过渡，因为测试过程更加严格、广泛和彻底。然而，为了获得更好的 ROI，将左移测试与连续测试相结合是至关重要的。左移测试应该作为你连续测试循环的基础。此外，在您的组织职能过渡到 [DevOps](https://goo.gl/e1rsUh) 即持续集成(CI) &持续部署(CD)之前，请确保所有“相关职能”，如营销、业务开发、支持、文档等。准备好迎接变化，因为每个“构件”在这个过程中都至关重要。
