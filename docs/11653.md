# 为什么解决问题一定是开发者最强的技能？

> 原文:[https://dev . to/Cesar _ mostacero/why-problem-solution-must-be the strong-skill-for-developers-582 a](https://dev.to/cesar_mostacero/why-problem-solving-must-be-the-strongest-skill-for-developers-582a)

在我看来，解决问题是开发人员最好的资产，不仅能够编码，而且能够设计解决方案，这是开发人员中的一个与众不同的因素。

精通一个框架，一门特定的编程语言或技术，这很重要。但是解题应该是这些知识的补充。

解决方案应该独立于语言。根据编程语言的不同，您可能需要重新编写一个算法，或者使用一个已经构建的具有相同行为的函数。但是主要的知识应该是我们如何为每个问题设计一个最优的解决方案。

**大脑是一块肌肉，肌肉应该得到锻炼**

如果一个足球运动员在周末的比赛中假装表现良好，他应该在一周内进行训练，以便在身体和精神上为比赛做好准备。开发者也是如此。

我们不能总是只在网上搜索来找到某个问题的答案。开发人员应该能够:识别、设计解决方案并解决这类问题，然后从技术上来说，你可以在网上或甚至与你的团队一起获得资源，但每个开发人员都应该有创造解决方案的能力。

真的需要吗？

根据我的经验，我只参加过一次技术面试，在面试中我被要求编码并解决 3 个问题。所以我们可以说，要被录用，将取决于面试官；但是在实际工作中，开发人员每天都会面临逻辑问题，所以最好做好准备。那么，我对问题的看法和回答是:是的，肯定的。

**定义明确的流程**

并非所有的问题都是相似的，但是对于所有的问题，我们可以将过程分为 3 个基本步骤:

*   识别问题:我们无法修复/解决我们不知道的事情。第一步必须始终理解问题，能够自己描述它。一个很好的练习是把它画在纸上，试着自己想象，代码看起来是什么样子，识别输入/输出/规则等等。这个问题涉及到。然后，试着描述它，如果你和别人一起工作，试着解释这个问题，如果他/她能理解，你可能对这个问题有足够的知识和理解。

*   设计多种解决方案:不考虑代码本身，更多地考虑伪代码和算法。如果我们实现的算法/方法的本质对于特定问题来说不是理想的，那么只设计一个解决方案并试图让它为问题工作可能是危险的。这里是在进行编码之前的重要性，认为...好好想想。

*   选择并实现最佳解决方案:这是最后一步(理想情况下),我们没有多少解决方案可供选择，我们只需要选择一个并实现。选择理想的解决方案有什么重要性？两个字:码一次。

代码应该永远是解决问题的最后一步，在识别问题、思考和设计解决方案之后，我们最终来到了代码。

那么，如果代码是最后一步，就不需要测试了？在我看来，第一个“测试阶段”是在第二步，我们设计解决方案，但每个阶段都必须测试，避免假设。

假设我们想根据分数和其他一些规则为学生确定一个排名函数。当我们开始设计解决方案时(大部分是在纸上)，应该考虑所有可能的情况，分数、规则、规则的例外、数据类型、错误处理、数据大小等。不仅仅是排序功能。

有时我们发现临时的解决方案，我们知道对于某些特定的(有效的)场景，我们的代码会失败，但是如果我们试图覆盖它，代码可能会更复杂。这是最后一步的真正重要性，同意所有的场景都可能发生在生产中(甚至是你没有想象到的)，然后，如果你已经知道有些东西可能会破坏你的代码，不要选择那个解决方案，在第二步投入更多的时间，设计一个最佳的解决方案，只有在这之后，编码它。

**时间是开发人员最宝贵的资源**

[![Thinks twice (or more), code once.](../Images/d90a815c6866ce7c2b2f5aef5d76a210.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--3QAuXGhU--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/f3ilehcoxrp6rz0pueab.jpeg)

通常，作为开发人员，我们没有足够的时间，我们试图更快地编码。事实也是这样，如果你编码的东西不够好，你将需要投入(更多的)时间，当“修复”在一段时间后带着新的 bug 回来时。
正如最好的编程名言之一提到的:三思而后行(甚至更多)，编码一次。

尽可能多的练习
也许在简历中不会像某些框架中的大师那样有价值，但是相信我，在工作中它真的很有帮助。
