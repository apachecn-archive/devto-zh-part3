## 为什么有效的 bug 报告是必要的？T3】

写一份准确的错误报告是至关重要的，因为这样你会增加修复你发现的错误的机会。写一份有效的错误报告并不困难，你只需要一个**好模式和一些练习**。记住一份好的错误报告是一个成功的软件测试人员的主要职责。

**有效而准确的报告的两大优势:**

-指定的开发人员拥有开始工作的所有必要信息
-团队中的任何其他测试人员可以**重新测试**它，即使记者不在工作

让我们开始看看你如何能写出最好的报告！

## **4 撰写准确的缺陷(bug)报告的一般规则**

1.  清晰准确:良好的写作和[沟通技巧](https://bugs-bunny.net/best-non-technical-skills-for-successful-software-tester/)在这里是必不可少的，因为你必须避免混乱和误解。不要错过关键点和重要信息，因为这将导致开发过程的放缓。也很有可能你的报告会被忽略，错误也不会被修复。
2.  **在提交报告之前进行研究**:当你发现一个 bug 时，在提交报告之前，对重复的问题进行快速研究。在许多情况下，有很多未解决的 bug 报告，所以很可能已经报告了一个 bug。确保您找到的问题不是旧问题的重复。
3.  简单:用简单易懂的方式写你的报告。使用简短但有意义的句子，避免过于复杂的解释和难以理解的词语，否则只会混淆或误导读者。
4.  **为不同的 bug 创建不同的报告**:将每个 bug 作为一个单独的问题进行报告。通过这种方式，您可以更有效地组织和跟踪您的问题，在相应的 bug 修复后关闭每个问题。此外，检查和解决问题的速度会更快。

## **如何写一个基本的 bug 报告(模板)**

现在来看看实践吧！检查这个简单的报告模板，并根据您的需要进行调整。在大多数情况下，您将会使用一个 bug 跟踪工具，它使得编写缺陷报告的过程更加容易，因为它会自动填充一些信息，比如报告者或者问题编号。如果你不使用这样的工具，你可能需要手动写这个信息。

**描述**又名**总结**:保持简短但信息丰富

**重现步骤**:你是怎么产生这个 bug 的？写下你遵循的所有步骤

**预期结果**:在你的案例中，预期的行为是什么？预期的结果通常在相关的需求中预定义。

**实际结果**:在你的案例中，实际上是什么行为？写出 bug 的实际行为。

额外的，但是**非常重要的信息**你必须提供 **:**

*   **与您产生 bug 的用户的凭证**
*   问题区域/ **网址**
*   产品的**版本**(如果适用)
*   您使用的 [**浏览器**](https://en.wikipedia.org/wiki/List_of_web_browsers) ，尤其是在设计问题的情况下
*   **优先级**——这取决于 bug 的严重程度。大多数**严重的**bug 应该先修复**。**
***   **严重性**——显示 bug 的影响。有 5 种严重性类型:**

 **1.  **拦截器:** 无法进行更多测试
2.  **危急:** 应用程序崩溃，有数据丢失
3.  **重大:** 重大功能丧失
4.  **轻微:** 轻微功能丧失
5.  **琐碎:** 小 UI 错误

*   **截图**或/和**截屏** ( [截屏](https://icecreamapps.com/Screen-Recorder/))。一张图等于千言万语。采取适当的截图或截屏，捕捉这个恼人的错误在行动中！

## 额外提示

1.  一旦发现错误**,立即报告。如果延迟，你可能会忘记一些重要的信息。**
2.  尝试再次重现该问题。确保你的 bug**是可重现的**。
3.  **提交报告前仔细检查**。再次阅读，找出并纠正任何问题。毕竟你是一个测试者，你甚至应该测试你自己的工件！
4.  注意你使用的语言。**不要骚扰**或批评犯错误的开发者。
