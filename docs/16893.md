# 为什么代码所有权对于敏捷开发是必须的

TL；博士:

*   了解一个代码库需要一段时间
*   越多人知道并编辑一段代码，效率就越低，因为会出现混乱和错误
*   然而，强代码所有权也带来了另一个瓶颈
*   打击中庸之道！争取功能所有权，以减少效率低下和瓶颈

如果你想拥有最佳的开发速度，那么把一段代码分配给一个人是最佳的。这也是 Communify 发展如此迅速的原因之一。我拥有整个代码库的所有权。我知道每个剧本的作用。寻找哪里从来都不是一件事。必须弄清楚某事如何工作或某事做什么，从来没有发生过。

如果你在一个团队中，有时能够修改别人的代码可能是有用的。这被称为弱代码所有权。您对自己的代码库负责，但其他人可以对其进行更改。您审查这些更改，并确保代码库保持您喜欢的方式。我认为这是最好的，因为等待其他开发者为你做出改变也是一件很烦人的事情。它会让事情慢下来，迫使你去做别的事情。这也是为什么我不太喜欢后端和前端的区别。我知道 python 和 react native，但是，在我目前的日常工作中，我不被允许在后端工作并在那里进行更改。很多时候，我被迫开始做其他事情，并停止我现在正在做的事情，只是因为后端人员没有时间做出微小的改变。如果我也能处理后端，我可以在几秒钟内完成这个改变。现在我得等几个小时。

一个应用程序中的许多新功能涉及到代码和其他工作的所有层的变化。即 UI/UX、前端、后端和测试。因此，如果你有一个前端后端分离的团队，一个新的功能必须由两个人来完成。如果你也有单独的 UX/用户界面和测试，这个数字变成 4。通常，一个在另一个之前完成。因此，一个人必须等待另一个人来完成它！依我拙见，这是开发速度和敏捷的一个巨大瓶颈...现在我甚至还没有谈到 UX 和测试！这些功能通常也由不同的团队来完成。如果这是你作为一个团队做事的方式，每个人都必须一直等待对方，那么一个新功能可能需要几天，甚至几周。瓶颈越来越大。

因此，在一个想要真正敏捷的公司里，我认为你必须采取中庸之道，让每个人对代码库和过程的一部分负责，但允许每个人访问和编辑所有代码，最后，通过团队领导适当的问题/特性分配，努力自己开发代码。这也说明了为什么全栈开发者如此有价值。尤其是当他们还能处理一些 UI、UX 和测试的时候。理想情况下，你不应该根据创建代码和任务的技术和技能来分离它们，而应该根据它们的特性来分离。这确保了代码的所有权，同时使得一个人从头到尾自己添加一个新特性成为可能。你甚至可以更进一步:让设计该特性的人也评估该特性在产品中的引入，并观察其对产品的影响。看客户怎么看，效果不好就改进。我认为这很好，原因有很多。一是因为您提高了敏捷度，二是因为您看到了自己所做事情的结果。我自己作为一个前端开发者，有时候会很无聊。我从来没有机会去思考某样东西应该是什么样子，我只需要去创造它。我从来不知道我做的事情会有什么样的结果，我只是必须去做。太无聊了....

所以，总结一下:

*   敏捷
*   争取整个生产线的特性所有权，以减少瓶颈、低效和等待
