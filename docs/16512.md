# 如何准确估计项目/任务的时间

> 原文:[https://dev . to/luminous men/how-to-estimate-time for a-project task-accurally-27mn](https://dev.to/luminousmen/how-to-estimate-time-for-a-projecttask-accurately--27mn)

大多数人不知道如何充分评估任务的时间安排。哦，这有时会让他们紧张...这里，还有“最后期限悄悄溜走”而且他们是 500%以上的再保险以防万一(还是不够)。经理们试图挤压“故意膨胀的期限”，以便他们可以承诺一些更容易接受的东西(常见的管理疾病)。还有含糊不清的嘟囔而不是具体的数字(更多的是开发者的问题)。

我个人，我相信几乎你们所有人，都讨厌像“什么时候能准备好”这样的问题但是在被认为是受控的项目中，它们在任何时候都是必不可少的。

因此，为了实际估计你或你的团队的一些工作的时间，你需要做一些准备。

# [](#step-1-understand-customer-requirements)第一步。了解客户需求

如果你不知道船的目的地，你怎么能说旅行需要多长时间。

首先确定项目中需要完成的所有工作。这包括，例如，功能性和非功能性需求(有时很难一次完成，也许这个过程需要迭代地完成)。作为其中的一部分，确保你留出时间开会、汇报、沟通、测试和其他对项目成功至关重要的活动。

# [](#step-2-order-these-activities)第二步。订购这些活动

现在，按照需要发生的顺序列出所有你确定的活动。

在这个阶段，你不需要加上你认为活动需要多长时间。然而，你可能想记下任何重要的截止日期。例如，在开始集成之前，您可能需要获得系统的另一个组件的文档。

# [](#step-3-risk-management-step)第三步。风险管理步骤

列出所有相关的假设、排除和限制。

在项目的初始阶段，需要集思广益来确定风险。你的项目没有任何风险的假设是错误的——每个项目都有风险，如果你看不到它们，那么你就看错了方向。当没有识别和度量风险时，这意味着您不再能够控制项目和完成项目所需的时间。你依赖于运气——那些你认为我无法实现的风险会增加项目实施的时间或者毁掉项目

# [](#step-4-make-the-estimates)第四步。做出估计

有几种估计时间的方法:

## [](#expert-judgement)专家判断

实际上，这是在该领域专家的参与下形成的评估。这可能是一个古鲁，或者你可以吸引一群专业知识的携带者。

专家对评估做出他们的假设(时间或美元)。之后，你可以把所有的提议平均一下，在讨论中你可以试着得出一个统一的解决方案。当然，让专家参与讨论各种方案更有效，而且会给出更准确、更合理和经过检验的评估。

## [](#threepoint-estimating)三点估算

最常见最简单的方法之一。在这个框架内，首先确定乐观(O =乐观)、悲观(P =悲观)和现实\平均(M =中间)估计。

P、M 和 O 的值由专家确定(以小时、天、美元为单位)，例如，在项目团队讨论期间。诸如此类的问题都是针对这个问的:“项目需要多长时间，如果一切顺利，就不会有风险和问题？”，“最消极的情况可能是什么，需要花费多少时间/精力？”等。此外，将获得的 P、M 和 O 的值代入公式:(O + 4 M + P) / 6

计算结果给出了平均估计值。该公式一方面可以考虑可能的正面和负面情况，另一方面可以“消除”它们的影响，获得更现实的评估值。

## [](#cost-of-quality)质量成本

相当有趣的技术，起初，时间或预算只是为功能的开发而估算的，没有考虑到错误和问题，好像我们立即得到了完美的没有缺陷的软件(从未见过这样的独角兽)。此外，为了使软件更接近“理想”状态，估计需要多少额外的时间和预算来处理现实中的错误和问题。

在评估软件质量保证的成本时，您可以分析和考虑以下方面:

*   缺陷预防活动的费用
*   测试成本
*   内部错误的纠正
*   修复外部集成问题
*   根据实际环境和数据安装和配置软件的成本

## [](#analogous-estimation)类比估算

在这种方法的框架内，我们可以利用过去解决这类问题或项目的经验。这里的主要步骤是找到可能的相似之处，突出相似的任务。

要找和以前经历相似的朋友或者任务，可以做一个分解。

## [](#parametric-model)参数化模型

这是最准确、最灵活的评估方法之一。它的本质是建立一个确定的参数化模型——基于过去的经验、可用的数据和度量、统计的预测。

事实上，建立了一个特殊的数学模型，它允许您跟踪最终等级如何根据初始参数而变化。这种依赖性甚至可以被可视化。这将有助于分析估计值与平均值的偏差界限，并了解什么会对此产生影响。

## [](#bottomup-estimation)自下而上的估算

这种方法类似于专家评估，只是在这种情况下，不是对整个项目作为一个整体进行预测，而是单独对其组成任务进行预测。这是对项目不同阶段的某种分解。看起来像什么:我们收集专家的意见，例如，从分析，开发，测试，软件支持的专家那里。我们一起总结他们的评估，加上他们在互动上花费的时间，并制定一个总体预测。

换句话说，我们收集部分评估，知道软件开发过程中的每个参与者需要多少时间，并考虑到额外的风险，将所有的东西放在一起。

## [](#conclusion)结论

我认为为了得到更准确的结果，混合几种技术可能更好，例如，自底向上的评估和三点评估来评估项目的各个部分。

我们需要教人们做评估，尤其是在 IT 领域。这对于项目的成功，对于参与其中的人员的成功，当然还有客户的满意，都是非常重要的。

并不总是能够正确地解释客户的需求，并总是期望他会提出一些新的东西或者修改规范中已经存在的东西。当一个人不想或没有时间理解需求或将项目分解成部分时，您总是可以听到乘法支持的不同因素或者通过 pi(3.14..)或由 e(2.71..).乘数不小，它们的分布可能很大。但是这一切——对小事不注意的问题或者懒惰去理解以上所有的问题。很明显，评估需要资源，有时是严肃的，但是对琐事越关注，对执行时间的预测就越准确，尤其是如果我们有一个已经制定并执行了此类项目的团队。

不幸的是，教一个人关注小事既困难又费时。但这是必要的，即使有阻力。
