# 我指导初级团队成员的方法

> 原文：<https://dev.to/pretzelhands/my-approach-to-mentoring-junior-team-members-1nd2>

> 这篇文章最初发表在我的个人博客上。

这篇文章可能有点奇怪。我一般不谈论我的软技能，因为我认为我是一个笨拙的书呆子，有时甚至无法在脑海中区分英语和德语，从而使可笑的糟糕的代码转换成为现实。

另一方面，我最近发现自己在`$BIGCO`工作期间做了很多指导，我在[上一篇关于自由职业者](https://dev.to/posts/how-i-got-into-freelancing)的帖子中提到过，和我交谈过的人似乎对此很感兴趣。

我在这里写的东西可能完全是胡说八道，但是嘿，再读一遍第一段，你就会明白为什么了。我想这在一定程度上是为了让我好好想想。

## 对每个人都要有耐心

我不知道你是否在你的团队中的其他成员中注意到了这一点，但是当有人向许多人提出问题时，他们似乎会奇怪地感到恼火。可能是因为他们不喜欢在工作时被别人打断，可能是因为他们不喜欢被别人的问题困扰。但是，当有人有问题时，当他们从座位上站起来时，看到他们问的人翻白眼或叹气，总是有点令人难过。

因此，我总是，总是，总是努力保持耐心。有很多人来问我一个问题，我花了一分钟才给他们答案。他们摆脱了困境，可以愉快地继续他们的工作，而我的工作没有损失任何重要的生产力。如果你的团队经常出现一个问题，把它写在 FAQ 页面上。

如果你发现自己迷失了思路，去看看 Swizec 的这篇文章。如果你花几秒钟写下一些东西，没有人会因此而对你不满。

## 给他们解决问题的工具

有一件事很好做，那就是不要直接向某人指出解决方案，除非它对你的最后期限至关重要。但是如果你在你的项目中有回旋余地，试着向他们展示他们如何自己找到解决方案。他们不知道如何开始检查他们程序中哪里出错了？教他们如何使用断点来单步调试他们的程序！他们不明白为什么他们的`flexbox`行为古怪？向他们展示一个类似于 [flexbox.help](https://flexbox.help) 的网站，让他们向你描述他们的目标，让他们在向你解释的同时点击这个网站。

如果他们真的遇到了更大的问题，他们会回来再次寻求帮助，但如果他们自己能够解决，那感觉就像是他们独立于你取得了巨大的胜利。它给人一种自信和独立的感觉。

所以告诉他们如何到达那里，但不要只是指着屏幕告诉他们，他们应该已经，明显使用了`justify-content: space-between`而不是`justify-content: space-around`。让他们有自己的胜利。

## 结对编程能帮上一吨

这当然取决于你正在交往的人的类型，但有时人们在结对编程时会感到更加自信。有时，如果你是他们调试橡皮鸭的鸭子，这已经很有帮助了。当他们大声地向你解释代码时，很多时候事情会在他们的脑海中出现。

所以，如果他们愿意，而你有几分钟的时间，和他们坐在一起。让他们解释他们的代码，和他们一起走一遍。有时候一想到有人会支持你就够了。

## 倾听个人

当然，这一切都因人而异。一周前加入你的团队的初级开发人员很可能比两个月前加入的人需要更多的帮助。有些人不喜欢有人在他们编码的时候监视他们，这也没什么。不管怎样，如果你站在他们旁边看着我，我会摸索着完成一个`docker run`命令。

每个人都是不同的，每个人都需要不同的指导方法。但这些是我发现对帮助他人学习有用的一些东西。也许你能为自己找到一些用处。

最重要的是，当一个问题对你来说微不足道，但对你的下级来说却无法解决时，请永远记住:从前你也是编程新手。；)
