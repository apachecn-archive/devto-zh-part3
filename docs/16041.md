# 演示过程中的实时编码

> 原文:[https://dev . to/craignicol/live-coding-during-a-presentation-41 A8](https://dev.to/craignicol/live-coding-during-a-presentation-41a8)

我做过一些演讲，其中一些涉及到现场编码，以帮助听众理解我想教的课程。这是一个我很少使用的技巧，但是它很容易出问题。

为一个演示文稿进行现场编码与为你的工作或一个乌合之众会议进行编码不是一回事。演示的现场编码必须和演讲的其他部分一样流畅和优雅。不用谷歌 API 文档，不用被语法错误绊倒(尽管如果你没有发现它们也不用担心，有人会发现的。)

如果你对幻灯片感到紧张，现场编码可以帮助减轻一些压力。当我开始的时候，它确实帮助了我，因为我可以假装我正在和我的计算机进行对话。幸运的是，这是一个有麦克风的会议。

现场编码也非常适合展示一些新颖的东西(或者至少对大多数观众来说是新颖的)，因为它可以证明它是真实的，并且是可行的。在一次演示中，我用它来演示 TDD。

不过要小心。技术可能会让你失望，所以想一想你可能会以另一种方式展示。你有可以在别人的笔记本电脑上播放的视频吗？可以用截图演示吗？您可以用白板演示解决方案吗？

如果你觉得在观众面前写代码不舒服，那就不要写。代码可以(我使用 [VS 代码片段](https://code.visualstudio.com/docs/editor/userdefinedsnippets)，但是有很多宏选择)，所以你可以逐渐地揭示它。或者预先构建并取消注释。

如果你事先准备好代码，以便和幻灯片一起分发，那么就要对代码进行大量的注释，这样当人们后来下载它的时候，他们就能理解上下文了。

我确实喜欢看到现场编码在演示中做得很好，但是除非你精心准备，否则它不会做得很好。思考这个故事，以及你如何和何时引入新概念。不要害怕回头再去找一些东西(“这里是如何连接，但我想先给你看看搜索。一旦你看到搜索结果，我会回到这里的参数。”)参考你的笔记是可以的，尤其是如果你刚刚抛出了满屏的代码让观众阅读。让每一次互动都有用。

回溯是可以的。展示你犯的错误，你看到的错误信息，以及你是如何改正的。你不会是最后一个犯这种错误的人。

放松并享受它。如果代码是你所做的事情，那就对它感到舒服，即使神经正试图把你从胃里翻出来。
