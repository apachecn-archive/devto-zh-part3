# 我第一次单独参加 React Amsterdam

> 原文:[https://dev . to/Darren Vong/my-first-react-Amsterdam-experience-as-a-solo-attendee-2d 14](https://dev.to/darrenvong/my-first-react-amsterdam-experience-as-a-solo-attendee-2d14)

第一次参加会议，尤其是独自参加，可能会是一次令人生畏的经历。但是，如果你想提升专业和技术水平，参加这种课程的好处是值得的，比如从你所在领域的领导者那里找到新的想法和灵感，以及本文中详细介绍的其他内容。

通常，参加会议的主要障碍是昂贵的门票。然而，由于成功申请了 React Amsterdam 提供的[多样性奖学金](https://medium.com/@ReactAmsterdam/apply-for-a-diversity-scholarship-attend-react-amsterdam-2019-for-free-65cd27d17a33)，我得以免费参加活动本身！

申请奖学金无疑是我今年做出的最好的决定之一，所以如果你有资格参加他们即将举行的姐妹会议之一，我强烈建议你去申请，因为这个过程只需要填写一份简短的申请表。

以下是我第一次参加 React Amsterdam 的经历，以一个单独参加者的角度。在会议的每个方面，我都留下了我从这次经历中学到的一些小技巧。

我希望这将有助于如果你想在未来几年去，并鼓励你这样做，即使你不得不自己去！

# [](#slack-channels)松弛的渠道

作为第一次单独出席会议，官方 React Amsterdam Slack 是了解活动最新情况和与其他与会者联系的宝贵方式。活动的实时更新自始至终都在那里提供，官方组织者会迅速回答人们的任何问题。

它的采用似乎很自然，因为大多数开发人员可能已经在工作中使用该平台进行日常交流了。这是显而易见的，因为在整个活动期间，每天都有消息发送，参与者组织非正式的聚会和会面。

就我个人而言，我能够见到来自澳大利亚和丹麦的开发人员，只是通过给其他独自参加的人留下邀请信息，让他们在午餐时出去玩。在另一个场合，我和几个来自加拿大的开发人员和招聘人员共进晚餐，寻找与我发送的信息相同的信息。

总的来说，我认为这使得会见其他与会者更容易，这意味着自己去的想法少了很多令人生畏。这让我的经历变得更加有趣！

提示 1:如果你是独自参加下一个，加入官方的 Slack，积极参与！这可以是安排一些事情，也可以是回应别人提出的想法。

# [](#getting-to-the-conference)参加会议

阿姆斯特丹反应堆(React Amsterdam)所在地 Krom southal 位于阿姆斯特丹北部，紧挨着 IJ 河。这个位置很容易到达——对我来说，离我住的附近的一家旅馆步行 20 分钟就到了。

由于会议组织了专门的摆渡班车服务(往返于阿姆斯特丹中央车站和会场之间)，只要您靠近地铁或电车站，就可以很容易地到达会场，这要归功于城市中出色的公共交通网络。如果你是一个自信的自行车手，你甚至可以骑自行车去那里，因为那里有现成的自行车出租。

## [](#kent-c-dodds-requisite-react)[肯特·c·多兹](https://kentcdodds.com/)——压惊反应

如果有人想找一个好的理由来感受阿姆斯特丹，这个开场白就是一个非常强烈的回应。在这个演讲中，Kent 经历了一个提升 React 开发人员(或者只是一般)的关键方法——更好地理解我们日常使用的抽象概念！

从一个他通过理解下面发生的事情来固定他的 tap 的个人例子，到实现类似于在进行中的工作的悬念特性的抽象下的代码，它很好地演示了我们如何通过更好地掌握幕后的事情来减少我们的代码。

## [](#siddharth-kshetrapal-refactoring-react)[西达尔特](https://sid.studio/) -重构反应

对于那些正在寻找如何在他们现有的 React 代码库中发现代码味道的可靠提示的人来说，Sid 为您提供了这 30 分钟的纯现场编码精彩内容。涵盖的技巧包括:

*   有一个最小的 API 表面积，这样开发者只需要学习众所周知的行为就可以使用一个组件；
*   遵循单一责任原则，其中一个组件应该做好它的一项工作，并将其余工作委托给其他组件；
*   避免特性嫉妒——例如，当父组件对应该写在子组件中的特性太感兴趣时…

…还有更多！特别是对于那些纯粹对代码感兴趣，而对传统演示不太感兴趣的人，我强烈推荐给这个一个手表！

## [](#max-stoiber-tech-regrets-at-spectrum)[麦克斯·斯托伊伯](https://mxstbr.com/) -理工大感慨看谱

这是一个反思性的谈话，关于 Max 在构建 Spectrum(一个面向开发者社区的实时消息应用)时对一些技术选择的遗憾。在一天结束时，我们都会犯错误，在我看来，他分享经验教训的残酷诚实的方式令人耳目一新。

我个人最喜欢他分享的遗憾是没有使用 [Next.js](https://nextjs.org/) ，因为他以“服务器端渲染”开始。这能有多难？我要建自己的服务器端渲染(SSR)服务器！”。他很快发现做好这件事对 T2 来说极其困难，事实上，他已经为拥有一个可靠的服务器环境奋斗了两年。

马克斯接着说，他保证没有人比蒂姆·诺特肯斯更了解 SSR，所以他建议任何需要 SSR 的人使用它，而不是(糟糕地)重新发明轮子。我是一名成功的初创公司创始人，已经被 GitHub 收购，在未来可能需要 SSR 的项目中，我肯定会三思而行，如果有人认为构建自己的 SSR 解决方案很容易，我会让他们参考 Max 的演讲！

## [](#max-gallo-scaling-applications-with-microfrontends)[马克斯·加洛](https://twitter.com/_maxgallo) -用微前端扩展应用

在这个充满活力的演讲中，Max 谈到了一个有趣的概念，即后端的微服务概念被应用到前端，以避免 DAZN 的 web 应用程序中出现一个整体。每个微前端都是一个单页面应用程序(SPA ),根据 Max 的说法，这被证明是一个让开发人员快速启动和运行的好方法，因为 HTML、CSS 和 JavaScript 的知识足以开始构建。

[![Microfrontends architecture diagram](../Images/c82da10c49942087859b2bb615683086.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--HqV1Z2Na--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/0kwr45ktys4vcxrk5w8i.jpg)

这个概念似乎还允许他们的团队适当地分担工作负载，避免过多的通信开销，因为他们必须同意的唯一公共接口是引导层。这反过来导致了自主性，并使开发人员能够尝试新事物和创新。马克斯说，这样做，他们可能会更快乐，从而留下来。我完全同意！

尽管之前听说过微服务，但 microfrontends 曾经是(现在仍然是！)对我来说是一个全新的概念。你想参加的会议正是这种类型的谈话，以激发新的想法，并跟上最新的趋势。

# [](#food)食物

作为一个从早上 6:30 就醒了的夜猫子，我立刻去买了面前的咖啡来储存急需的咖啡因。就早餐的食物而言，有很多糕点小吃可供选择，但我选择了更丰盛的燕麦和酸奶罐，这为我一天的第一次谈话提供了充足的燃料。

午餐也组织得很好，因为食物被分配到多张桌子上，以避免进一步增加已经很长的队伍。素食者、杂食者和任何有特殊饮食要求的人都可以选择咖喱饭。

唯一的缺点可能是晚上的食品车，因为当我排到队伍前面的时候，他们很快就用完了食物；更令人沮丧的是，我已经白排队 30 多分钟了。

请记住，食品车的大小是有限的，因此他们可以储存多少食物，希望会议可以通过在未来安排更多的食品车来解决这个问题。

技巧 3:在最后一次喝咖啡休息时去食品车，而不是等到最后，去打败排队的饥饿人群！

# [](#the-after-party)后党

从这一点开始，这是会议的可选部分。反正我晚上也没打算做什么，目的是让我们在一天学习新东西(或工作，即促进工作和娱乐的平衡)后玩得开心，所以我决定留下来看看这是怎么回事。

派对以电子和电子音乐的混合开始，伴随着小吃和当地的 React 主题金发啤酒:

[![React Blond beer](../Images/73c70ce6f9096eb7abacf2d1b56e8e06.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--hXWuoug6--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/7bf2afrc9aaa2nzcfowz.jpg)

这使得每个人都更加放松，小吃可能足以让我们这些没有从快餐车上得到任何东西的人开心！

派对结束后的节目亮点应该是“智能手机乐队——社会分类实验”游戏。由于场地有限，只有大约 100 人能参加。

这款游戏名副其实，是一项社会实验。参与者必须使用手机登录一个专门的网站，并按照屏幕上的指示进行操作。这些说明显然旨在鼓励我们相互交流，从与你的邻居交谈，到凝视和评价你邻居耳朵的吸引力！

当参与者彼此不说话时，他们必须填写一些问卷，最终将数据输入游戏的机器学习算法。鉴于只问了一组简短的问题，最后的结果，比如按顺序确定参与者的身高，却出奇的准确！

几个小组站成了吃豆人规则的形状，这鼓励像我这样的单独参与者走上前去开始交谈，如果他们不是，大多数人通常会接受聊天。所以，如果你要参加下一次聚会，没必要害怕自己来。

# [](#tour-day)巡日

对我来说，这就是 React Amsterdam 的独特之处，因为我不记得读过许多其他会议(当我在研究要参加的会议时)，包括帮助你探索你所在的城市并与其他与会者社交的扩展计划。会议投入的这些额外努力有助于培养一个伟大的社区。

# [](#personal-regrets)个人感慨

参加 React Amsterdam 是一次非常棒的经历！这次会议无疑让我充满活力，并受到新想法的启发——我迫不及待地想在我当前和未来的项目中尝试它们！

话虽如此，总有一些事情我希望我已经做了，包括:

*   记住，更重要的是，多鼓起勇气和新认识的人自拍。对于一个似乎回避镜头的人来说，这对我来说是一个棘手的问题，但绝对是一个需要努力的问题！像这样回顾事件时的记忆会是值得的！
*   在演讲者演讲后与他们交谈！他们中的大多数人都很友好地与我联系和聊天，所以这将是一个很好的机会来跟进他们的谈话中可能与我当前项目相关的问题。
*   这有点超出我的控制——因为我自己真的买不起票——但是参加一个会前研讨会！也许这是多样性奖学金将来可以帮助的事情。
