# 为什么人们喜欢 GraphQL

> 原文：<https://dev.to/smizell/why-people-like-graphql-221c>

人们喜欢 GraphQL，但我不相信这主要是因为技术原因。我认为这是更人性化的东西。人们选择工具是因为它们解决了一些潜在的挫折，而 GraphQL 的当前实现解决了人们在使用 API 时感到的一些更大的挫折。

# 网络的开始

蒂姆·伯纳斯·李为“是什么让你想到了 WWW？”这个问题提供了一个答案：

> 嗯，在那个年代，不同的电脑上有不同的信息，但你必须登录到不同的电脑才能获取，这让我感到沮丧。此外，有时你不得不在每台电脑上学习不同的程序。所以弄清楚事情是如何运作的真的很难。当人们在喝咖啡时，去问他们往往更容易。

有趣的是——尽管并不令人惊讶——网络诞生于挫折之中。Berners-Lee 厌倦了编写与如此多不同系统交互的软件。“难道我们不能转换每一个信息系统，使它看起来像是每个人都可以阅读的一些虚构的信息系统的一部分吗？”他问。他说这就是后来的万维网。

# GitHub 对 GraphQL 的理由

已经有很多关于选择它的技术原因的文章，但是我相信那些相对于人们在使用 API 时的挫败感来说是次要的。再看[GitHub 为什么迁移到 GraphQL](https://githubengineering.com/the-github-graphql-api/) 。

> [我们]从集成商那里听说我们的 REST API 也不是很灵活。有时需要两到三次单独的调用来组合一个资源的完整视图。似乎我们的回应同时发送了太多的数据，并且没有包括消费者需要的数据。

这里描述的技术问题是所谓的“过度蚀刻”或“蚀刻不足”但是如果你仔细阅读这句话，你几乎可以听到他们的客户说，“我知道我想要什么。为什么我要做所有这些工作才能得到它？”或者更好的是，“你能不能转换你所有的 API 资源，让它们看起来像是某个虚构的信息系统的一部分，每个人都可以阅读？”

如今，每个 web API 都有不同的感觉。它们都有自己的 SDK、自己的客户端、自己生成的工具，以及自己在通过 HTTP 公开数据方面的细微差别。即使在他们内部，重复请求一系列资源来实现共同的目标也会让人感觉不舒服。GraphQL 试图解决这个问题，GitHub 也看到了这一点。

# GraphQL 统一了一个支离破碎的世界

GraphQL 让人们思考数据以及数据之间的关系。它使开发人员不必拼凑资源列表来了解如何获得相关数据。它允许消费者忘记 HTTP 方法的细微差别。人们要求他们想要的，这就是他们得到的。这伴随着失去一些 HTTP 久经考验的超能力的代价，但是对于大多数人来说，这是值得的。

# 下一代 web APIs

我希望 web APIs 的下一步将是遵循这些步骤并解决其中的一些挫折。我觉得那一步会比技术上更优越，更务实，更以人为中心。
