# 调试网站问题的 5 大实践

> 原文：<https://dev.to/lambdatest/the-top-5-practices-for-debugging-website-issues-4761>

编码既有趣又简单。但是，并不总是这样。

好吧，实际上什么时候开始不好玩了？当你偶然遇到一个叫做“错误”的东西时，就不再那么有趣了。

突然之间，世界开始看起来像一个黑暗无望的隧道，你的注意力从代码转移到了悄悄潜入你的代码中的“bug”。程序的大小、应用程序和特性与开发人员自己解释错误的能力成反比。

然而，编码的世界并没有那么灰暗。如果你精通这门语言和一些调试工具，调试代码并不困难。遵循一些提示和技巧，你可以轻松地过河，而不会被“虫子”咬到

在这里，我们给你带来了调试网站的五大方法:

### 语法可能是罪魁祸首

与逻辑错误相比，语法错误在任何代码中都大量存在。缺少分号和括号会对代码造成超乎想象的损害。尽管有经验的开发人员倾向于更多地关注驱动代码的逻辑，但是语法也需要并且值得关注。

在深入研究代码的技术细节之前，解决所有的语法错误是一个很好的实践。

*   所有的分号和括号都必须到位。
*   所有的 HTML 元素都应该关闭。
*   打字错误应该消除。

这种看似无意义的错误会改变代码的整体含义。

### 黑盒一些脚本

由于其异步的本质，许多开发人员发现调试 JavaScript 非常具有挑战性。然而，随着库和框架的引入，这个任务已经简化了很多。[库和框架](https://www.lambdatest.com/blog/top-21-javascript-and-css-libraries/)是应该根据软件的需求使用的代码片段。开发人员已经对它们进行了很好的测试，并使它们没有 bug。在调试你的代码时，你不需要重新测试它们！

因此，将这些脚本黑盒化是一个很好的做法，这样调试器就不会浪费大部分时间来确认已知的事实，即它们没有 bug。该原则甚至可以扩展到您自己的脚本。如果您已经单独测试了一个相对较大的脚本，并且确信它没有错误，那么在调试程序的其余部分时，您不需要再次测试它。黑盒子，让你的生活更轻松。

### 检查跨浏览器问题

大多数与 HTML 和 CSS 相关的显示问题都是跨浏览器的问题。你的代码可能写得非常好，但是因为浏览器兼容性问题而给你带来困难。在这种情况下，您必须解决跨浏览器方面的问题，而不是在代码中查找错误。

减少跨浏览器问题的一个专业建议是首先确保它们不存在。如果你正在利用一个体面的 CSS 重置，这是很好的。使用这种方法可以消除浏览器之间的不一致性，使它们的行为更加一致。

另一个推荐的做法是只使用 HTML 和 CSS 的那些属性和特性，它们被多种浏览器所支持。你可以使用像 LambdaTest 这样的[跨浏览器测试](https://www.lambdatest.com/)工具来测试你的网站在 HTML 和 CSS 中的[跨浏览器问题。](https://www.lambdatest.com/blog/finding-cross-browser-compatibility-issues-in-html-and-css/)

### 缩小——是还是不是？

在不改变源代码功能的情况下，去掉所有不必要的字符，如空白字符、换行符、注释和块限制符，这就是缩小。

这些通常被添加到代码中以增强其可读性，但是在执行代码时没有任何意义。然而，调试精简的代码可能是一件痛苦的事情。大多数时候你不需要处理精简的代码，但是少数时候你必须处理，这会给你带来一些严重的麻烦。漂亮的打印是 Chrome 的一个选项，可以让你更容易地调试缩小的代码。毫无疑问，它不会像原始代码那样有帮助，但至少它会帮助您弄清楚实际发生了什么，从而简化调试。

### 稍微优化一下开发环境

在开始调试过程之前对开发环境进行一点优化也可以使它变得更容易。例如，想想 Symfony。它在系统的缓存中创建大尺寸的 PHP 文件，包含您的软件对每个请求所需的 PHP 类的集合。然而，这种行为会使调试器感到困惑，因为同一个类可能位于两个不同的位置，即原始的类文件和 Symfony 创建的大文件。建议您事先禁用引导文件和类缓存，以避免调试器混淆，并有效地完成其工作。

因此，这些是调试代码和网站的最佳推荐实践。除此之外，还有一些通用的提示和技巧，任何开发人员都必须遵循。例如，在盲目开始编辑代码之前，您应该总是考虑重现一个 bug。此外，您应该考虑编写一个重现 bug 的测试用例。

不要想当然
你应该知道你在用代码做什么。徒劳无功只会浪费你的时间。接下来，您应该了解堆栈跟踪。不是所有的 bug 都有，但是对于有 bug 的，你必须彻底了解它们。除此之外，你应该知道所有的错误代码。

另外，你要明白(更重要的是承认！)你无法修复所有 bug 的事实。

如果你已经在一段代码上花了一两个小时试图定位错误，你应该停下来，打电话给朋友或同事，接受一些友好的建议。你会惊讶于其他人能如此轻易地发现你一直忽略的错误。

最后，你必须庆祝所有的调试成功——不管它们是大是小。这将提高你的士气，并帮助你更容易地找出剩余的错误。

调试愉快！
