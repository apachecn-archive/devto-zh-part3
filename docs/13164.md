# 编程很难

> 原文:[https://dev.to/somedood/programming-is-hard-2p87](https://dev.to/somedood/programming-is-hard-2p87)

# [](#what-programming-is-not-about)编程不是为了什么

编程不是敲键盘和尽可能快地打字。这并不是宗教式地记忆键盘快捷键，最终导致鼠标过时。这并不是说要学习每一种编程语言，如果一开始就有可能的话。一个好的程序员不是由他们的电脑的品牌、价格、性能和操作系统来定义的，也不是由他们对代码编辑器和[ide](https://en.wikipedia.org/wiki/Integrated_development_environment)——[VS 代码](https://code.visualstudio.com/)、 [Atom](https://atom.io/) 、 [IntelliJ IDEA](https://www.jetbrains.com/idea/) 、 [Vim](https://www.vim.org/) 、 [Notepad++](https://notepad-plus-plus.org/) 或其他的偏好来定义的。由于许多好莱坞电影，与流行的观念相反，编程绝对不等同于黑客。

此外，它不仅仅是记住编程语言的语法和内置功能。逻辑、条件、`if`语句和算法——尤其是排序算法——并没有描绘出编程的全貌。数学、递归、计算机科学和设计模式也没有公平对待它。尽管它们是编程的重要组成部分，但它们只是拼图的一部分。

# [](#designing-and-planning)设计和策划

在编写一行代码之前，项目的设计和架构都经过了周密的计划，以确保~~或至少增加~~顺利开发周期的可能性。这就是软件设计发挥作用的地方。工具链、管道、公共和内部[API](https://en.wikipedia.org/wiki/Application_programming_interface)的抽象层、模块化、对象关系和数据库结构都是在这个开发阶段计划好的。

# [](#we-are-living-breathing-debuggers)我们是活生生的、会呼吸的调试者

编程的艺术要求我们跳出框框思考，用最务实、最有效、最可行的解决方案来解决问题。这可能就是为什么我们最有可能成为家中的“it 人员”或“客户支持人员”的原因。实际上，我们的工作就是修复损坏的东西。似乎“编程”是“解决问题”的一种美化的说法。

换句话说，我们是活生生的调试器，在我们的计算机上和计算机外呼吸，正因为如此，我们知道如何读和写文档是很重要的。适当的文档——以详细文档的实际页面的形式出现，或者简单到对代码库进行有价值的注释——是程序员最重要的生命线之一。没有它，我们会迷失在黑暗中，无法履行我们作为调试器的职责。几乎没有进展，因为我们的大部分时间都花在实验和研究一个框架或遗留代码库是如何工作的。总的来说，这会给开发者带来非常糟糕的体验。

# [](#consider-all-possible-scenarios)考虑所有可能的情况

调试已经够难了。更糟糕的是，代码的执行通常不是线性的。由于带有`if`语句的程序逻辑，大型项目需要多个可能执行路径的“分支”。我们必须考虑到*每一个*可能出现的情况和错误，尤其是当它涉及到用户输入的时候。跟踪*每一个*可能的执行路径所需的认知负荷使得编程更加困难。

# [](#user-experience)用户体验

走出开发的世界，我们填补了一个普通用户的鞋子。除了提供功能、添加新功能、修复错误和记录我们的代码库，我们还关注普通用户如何与我们的应用程序或软件交互。我们考虑多种因素，这些因素会带来出色的用户体验，例如(但不限于)可访问性、可用性、用户友好性和可发现性、UI 设计、颜色主题、[功能动画](https://uxplanet.org/functional-animation-in-ux-design-what-makes-a-good-transition-d6e7b4344e5e)以及性能。

# [](#performance-and-optimization)性能和优化

说到这里，性能本身就是编程的一个重要方面。我们，尤其是那些有计算机科学背景的人，努力使用和编写最节省时间和空间的算法。为了最大限度地利用我们的可用内存、CPU 和 GPU，我们沉迷于深不可测的微秒时间尺度。

在 web 开发的背景下，网络优化是一个需要掌握的重要概念。我们千方百计地缩小和压缩我们的 HTML、CSS 和 JavaScript，只是为了最小化服务器响应的负载。图像和其他各种资源也被压缩和[延迟加载](https://developers.google.com/web/fundamentals/performance/lazy-loading-guidance/images-and-video/)，以最小化用户在一个页面变得有意义和可用之前必须下载的数据量。

然而，有些时候我们变得过于痴迷于性能。**过早优化**当我们不必要地专注于优化代码库的某些部分，而不是专注于需要为实际进度和生产力做些什么时，就会成为一个问题。在这种情况下，我们必须有智慧来判断代码库*的哪些部分真的*需要优化。

# [](#security)安全

除了我们软件的 UI 和逻辑，作为程序员，我们还要对用户的安全负责。在我们这个数据被极度渴望并被大量货币化的时代，确保用户个人信息的安全比以往任何时候都更加重要。我们采取额外措施保护私人数据，因为我们的用户信任 T2 的软件。如果我们不坚持这种责任，我们肯定不是真正的程序员*，甚至远远不是。*

当接近安全时，我们永远不会太安全。根据一般经验，我们*“从不相信用户输入”*。竭尽全力净化数据和用户输入甚至可以被认为是“最佳实践”。如果我们不小心使用，不仅会使我们的软件和基础设施面临巨大的风险，还会危及敏感的用户数据，而这些数据正是我们作为程序员承诺要保护的。

然而，安全性并不仅仅局限于用户数据和输入。病毒、蠕虫、特洛伊木马、广告软件、键盘记录器、勒索软件和其他形式的计算机恶意软件继续传播并困扰着全世界数百万台计算机和设备。即使经过几十年的硬件和软件技术改进，也不存在无懈可击的系统。安全只是一门不断被磨练的手艺，但永远不会完美，因为总会有一些好奇的人调查和寻找每一种可能的方法来“入侵”系统。

出于这个原因，不管使用案例和用户基础如何，我们在设计软件时都会将安全性作为首要任务之一，如果不是*首要任务的话。我们这样做是为了保护我们的用户免受上述威胁，这些威胁可能会造成不便，例如数据丢失、文件损坏和系统崩溃等等。*

# 团队合作让梦想成真

即使不一定与编程相关，团队合作在软件开发中也起着极其重要的作用。由于任何大型项目的复杂性和不断变化的部分，仅仅一个人不可能以常规迭代的快速步伐或者在客户或任何监督实体的严格期限和时间限制下开发出高质量的软件。

这就是为什么我们有不同的团队，他们专门研究编程的许多方面中的一个。一个人永远也不会具备有效地、有凝聚力地将每个方面粘合在一起所需的所有技能和知识。一个团队可能负责 UI 设计和可访问性，而另一个团队可能负责软件本身的功能。如果将各种专业团队的所有能力结合起来，最终的软件将具有在财务和实际限制内可能具有的最佳功能、用户体验、性能和安全性。

对于时间管理和满足截止日期，工作流组织和自动化是最重要的。我们花时间来适当地配置我们的构建工具和管道，因为这样做将在未来为我们节省大量时间。总的来说，投资回报使久而久之增加了。

# [](#working-well-with-others)与他人合作愉快

为了阐述团队合作的理念，我们与同行建立了健康的关系，因为最终，项目的成功在很大程度上取决于背后团队的成功。我们努力营造一个支持性的工作环境，让有经验的老员工体贴地指导新员工。

因为我们是团队开发软件，所以我们必须小心别人阅读我们的代码。为了保持开发周期的长期可持续性，可读性和可维护性被认为与项目的逻辑和功能一样重要。我们始终如一地编写好的、可读的代码，同时提供信息丰富的提交消息和文档，因为这些肯定会帮助我们和其他人更好地理解我们的代码。

说到其他人阅读我们的代码，代码审查是学习更多编程最佳实践的好机会。这也是我们熟悉代码库及其底层设计和架构的另一种方式。虽然建设性的批评令人不愉快，并且在接受端很难处理，但是为了提高我们作为程序员的水平，把它作为合理的建议是很重要的。

# [](#programming-is-hard)编程很难

编程包含了功能之外的许多方面，如用户体验、性能、安全性和团队合作。仅仅严格关注一个方面而忽略其他方面是不够的。对于具有显著规模和重要性的项目，这不是键入几行代码那么简单。它需要大量仔细的计划、设计、考虑和团队合作才能成功。事实上，编程时花在思考上的时间比打字多，尤其是在长时间的调试过程中。

最后，编程实际上是关于持续不断的学习。适应性和不断学习是在这个行业生存的关键。如果我们不尽力保持学习，我们就不能指望保持相关性。在这样一个瞬息万变、技术日新月异的行业，我们必须跟上它的快速发展步伐，以免落在后面。

在这篇文章的最后，我想感谢全世界所有开发人员的辛勤工作。为了写这篇文章，我不得不反思一个开发团队的日常工作流程。我不得不研究编程和软件开发的许多方面，这些方面通常不被人注意。从那以后，我对安装在我电脑上的所有软件有了更大的欣赏。为此，我鼓励每个人今天感谢一个程序员，不管经验如何。没有他们我们会怎样？

永远不要认为他们的努力是理所当然的。
