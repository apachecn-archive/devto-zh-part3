# 像建筑师一样测试

> 原文:[https://dev.to/damnjan/test-like-an-architect-25bo](https://dev.to/damnjan/test-like-an-architect-25bo)

大多数开发人员将测试策略理解为开发过程结束时的事情，通常由开发人员自己或 QA 手动完成。因为不是每个人都看到了这里的问题，所以让我们和建筑师进行一次平行的旅行，观察他们是如何建造一座建筑的。

### [](#build-a-building-like-a-software-developer)像软件开发者一样建造一栋大楼

你会相信建筑师建造了一整栋建筑，然后在建筑工人完成他们的工作后站在一边观察稳定性吗？或者，建造建筑，然后让它空几个月，里面有试验居民。在此期间，架构师可以观察所有因素，如稳定性、基本可用性、材料质量等。几个月后，她/他将自己的房子介绍给实际的住户，然后让他们发现所有的故障、问题、放错地方的门、不通电等等。更不用说他们能找到的更严重的问题了，比如太薄的墙，承受不了书架重量的地板，等等。

### [](#integration)整合

好吧，这些都不会在架构中发生，但是在软件开发中却经常发生。我看到很多次软件被开发，并且在开发周期结束后第一次被测试。建筑师不会这么做。他们有公式和规则，让他们在开始建造之前，把任何建筑放在试验的路上。建筑由许多块、混凝土块等组成。所有这些和平放在一起，相互作用，改变整个建筑的行为。建筑师在现实中把两个元素放在一起之前，就确切地知道它们之间的力和动量。这可以与软件开发中的集成测试相比较。你有一堆组成部分，它们应该一起形成一个结构，每个组成部分可能有不同的属性，但重要的是一个组成部分的输出是另一个组成部分的输入，它们应该是和谐的

### [](#unit)单位

建筑是由单一元素构成的。每一种元素都有自己的特性。建筑师预先知道他们可以使用什么材料，一个元素的预计重量是多少，建筑内部每个元素的容量和强度是多少。这与软件开发中的单元测试非常相似。单元测试确保结构的每个部分都能够处理它们被设计的任务。它们是架构中简单的静态公式，确保每个元素的行为始终保持不变。如果您使一个元素变弱，那么您的集成测试应该向您显示一个问题，但是单元测试是向您确切地显示哪一个元素是弱的。

### [](#functional)功能性

建筑师不需要一个真正的人进入建筑，并开始使用它来发现她/他把电源插槽放在不可能到达的地方，门不能去任何地方，窗户不能打开。他们对所有东西都有一套标准，包括电力、门的位置、灯等等。如果标准没有帮助，他们可以使用常识。在将软件投入实际使用之前，软件开发人员可以进行一系列功能测试，这些测试与实际数据库一起工作，模拟用户的旅程并预测用户的常用路径。

#### [](#personal-note)个人笔记

> 我每天都在遭受痛苦，因为我参与的项目中没有测试。大部分代码都是匆忙完成的，并且第一次在产品上测试。此外，测试的泄露花费了公司大量的金钱和时间，因为他们必须支付人工 QA。我仍然相信我们可以从学究式的建筑师那里学到一些东西，不是复制粘贴，而是从中获得一些纪律。
