## [](#d%C3%A9finition-des-jobs)作业的定义

循环作业是一系列指令。
对于开源项目，我们最多可以同时运行 4 个作业。
实际上，三个脚本中的每一个脚本都有一个专用的作业，这样我们就可以同时启动 3 个作业，每个脚本一个。这样，每个脚本都将在不影响其他脚本的孤立环境中启动。

以下示例显示了作业定义的外观:

```
build:
  docker:
    - image: circleci/node:11.10.1
  steps:
    - checkout
    - run: yarn install
    - run: yarn build 
```

*   `build`:作业的名称
*   `docker`:我们工作将在其中运行的环境，这里停靠
    *   `image`:dock 要使用的图像，这里是 Node.js
*   `steps`我们的指示清单
    *   `checkout`用于定位到 circleci 创建的项目文件夹
    *   `run: yarn install`安装依赖项
    *   `run: yarn build`构建用于生产的静态应用程序

注意:shell 命令前必须有一个“`run:`”

对每个脚本重复上述步骤，得到以下三个作业的配置:

```
version: 2.1
jobs:
  build:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn build

  lint:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn lint

  test:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn test:unit

workflows:
  version: 2
  integration:
    jobs:
      - build
      - lint
      - test 
```

*   `version`配置文件版本，当前为 2.1
*   `jobs`我们不同工作的清单
*   `workflows`不同作业的执行情景
    *   `version`工作流程版本，目前为 2
    *   `integration`:工作流的名称
    *   `jobs`要执行的工作清单。在这里，我们的三个作业将并行运行

## 部署

现在集成工作正常，当分支“`master`”上出现新提交时，自动部署我们的代码将是一个很好的选择。无论是通过拉式请求还是推式(力🙄）

我在 Firebase 做了一个小公寓但谁知道呢也许有些人会感兴趣🤷‍♂️.

要通过 CI 将静态文件部署到 Firebase 托管服务，需要:

*   使用`yarn global add firebase-tools`在本地安装 firbase 工具。
*   发射一个`firebase login:ci`得到一个令牌

此令牌将用于授权 CI 部署到 Firebase。

要添加标记，请转至设置:

[![circle-ci-env-variables-2](../Images/666780953e755d39e1a08e2690464d14.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--YwiF0-j9--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/76v95hce6smpwnb2219s.png)

并添加令牌作为环境变量，在此处将其命名为`FIREBASE_TOKEN`:

[![circle-ci-env-variables](../Images/e39ddf60634a181e5d79ad4499a2ae33.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--oX6JkwCk--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/tu1krp4pn7bnrlae2yqr.png)

然后可以在我们的配置文件中重复使用环境变量:

```
version: 2
jobs:
  ## les autres jobs...

  deploy:
    docker:
      - image: circleci/node:11.10.1
    steps:
      - checkout
      - run: yarn install
      - run: yarn build
      - run: yarn firebase deploy --token "$FIREBASE_TOKEN" --only hosting

workflows:
  version: 2
  integration:
    jobs:
      - build
      - lint
      - test
    - deploy:
          requires:
            - build
            - lint
            - test
          filters:
            branches:
              only: master 
```

在新工作:`deploy`中，使用了刚才创建的变量`$FIREBASE_TOKEN`。
接下来只要将这项工作加入我们的工作流程，为其添加选项即可:

*   `requires`:运行此作业前必须完成的作业
*   `filters`用于指定哪些分支、标记-我...。此工作必须执行。就我们而言，只限于`master`分支

[![circle-ci-v1](../Images/7f2c08422ebaf611879249ec714dcc8c.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--Fedkd6BI--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/51jtjfpnwys57lyv3f7e.png)

## 摘要

作业是一系列指令。
工作流定义了各种作业的运行方式。我们的作业只能在某些分支上按顺序并行运行-我...。
您可以透过 UI 加入包含机密资讯的环境变数，并在组态中使用这些变数之前加上`$`。
