# 结对编程，三人编程，六人编程？！

> 原文：<https://dev.to/schmowser/pair-programming-trio-programming-sextuplet-programming-9k2>

除非你是一只蜘蛛，否则你很难一个人对你的代码多加关注。(无论如何，我有时会被代码中的*纠缠住*😉)所以偶尔，我能听到一声*“哇，老兄。这是什么？”*从我旁边的桌子上。紧接着又是一句*“我觉得你的方法根本就不叫”*。我把转椅滑向困惑的同事。

和💥！你在进行结对编程。

*2*

一起看五颜六色的算法并不是你必须要计划的事情。(尽管所有的敏捷团队可能至少每个月都会记下一次，作为一个必须的行动点。)就这么发生了。这是你工作的本质。天哪，如果不是为了别人，你不会用 Git。还记得当你在一个你是唯一贡献者的仓库上拉请求的时候吗？哦，那些瞬间融合的时代！

所以，准备好享用双人晚餐吧。实际上，这是一场盛宴，一场体验和学习的盛宴。正如在我之前的无数开发人员所指出的那样:通过解释你对承诺的使用或者你能做的其他魔法，你会学到更多。不仅仅是作为一个初学者。解释是双倍的学习——这是真正的魔法！

我的一些让这种经历更有价值的技巧是确保双方有一个共同的工作基础。在纸上画一个草图来可视化软件架构？洞见——好的，请！重新表述那些对你来说是解释过的和新的东西？你猜对了！

即使是最小的事情也会让一些最有经验的程序员感到惊讶。在 IntelliJ Idea 中，在不接触鼠标的情况下从一个方法切换到它的测试会引发一个*“等等，你是怎么做到的？”*并推广能为所有人节省时间的快捷方式！

尽管它是社会性的，但我们经常会忘记时间和空间。有时，我们正在激烈地讨论和生成代码，却没有注意到我们给团队其他成员造成的干扰。尤其是当你和你的评审同事相处融洽，开玩笑，同时对每个编译错误感到惊讶的时候。所以，要知道其他人的工作流程可能会被打乱。(这就是为什么我支持为世界上的每一个开发者提供免费的降噪耳机！🎉)

所有这些喧嚣会吸引其他掠食者。😉

*3*

然后，第三个程序员加入了战斗！我的代码被两个专业人士攻击了！(对此我很感激，因为我还有很多东西要学。)作为一个大三学生，安静一会儿，听两个开发者辩论，其实挺有意思的。你开始了解行话，如何谈论代码，如何以一种有意义的方式分析和构建你的思想。

三个学长在一起就变成另一回事了。这些辩论可能会很激烈，但也很有启发性和鼓舞人心。这可能就像一个三居室的情况。相处起来，大家都是赢家。但不幸的是，这个星座倾向于 2 比 1 的多数，这可能会扭转局势。

因此，在三人组中，我倾向于扮演倾听的角色。当解释发生时，只有一个人应该这样做，并由第三个人在最后加上潜在的附加评论。否则，它会变得令人困惑。也许，是时候进行白板会议了。但是你知道:三个人在白板上画画会导致伸长脖子...

*6*

我知道我的代码可能看起来像一场车祸。但是我不知道它会像真的一样吸引这么多的游客。与此同时，其他五个开发人员站在我身后，看着我的注释做奇怪的事情。对于泡沫之外的人来说，我必须看起来像是成功地侵入了联邦调查局的数据库。(而我没有。)

这种情况不会出现太多次。但是当这个问题非常重要并且足够困难的时候，它可能会吸引大量的观众。这个质量可以作为一个整体工作，并打破坚果。我注意到这些聚会不会持续太久。当我们围坐在一张桌子前讨论一个特定的 bug 时，不仅一些同事会失去兴趣，还会让我们所有人变得没有效率。

因此，上一次我和其他一些优秀的同事一起回顾我的发现时，我已经预先用屏幕记录了执行步骤，这样我就不会偶然发现不可预见的错误并浪费任何人的时间。或者用类似 GUI 的 Postman 设置替换冗长的 curl 命令。让你的工作能被几个人理解比花时间和精力把它传达给一个人更棘手。

然而，有足够多的人关心你的问题，这是一个令人欣慰的想法。愿意和你组队。在这些时刻，我感受到了我们都渴望的团队能量和能力！

*1*

一天结束后，每个人都回到自己的屏幕上。你可以休息一会儿，或者回家，或者开始另一段属于你自己的心流。与大多数会议不同，在结对编程会议之后，我满怀信心地回到了我的办公桌前。虽然没有产生很多代码行，但很有成效。

这也是所有二人组、三人组和六人组编码一个有趣的副作用。珍惜你的私人时间！
