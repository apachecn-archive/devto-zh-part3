# 如何衡量一个开发者的生产力？

> 原文：<https://dev.to/rlxdprogrammer/how-to-measure-the-productivity-of-a-developer-17g8>

如何比较开发者的生产力？这可能吗？生产率更高的开发人员也是更好的开发人员吗？如何提高你的生产力？这些都是本文的主题。

# 为什么衡量生产率有意义？

让我们从头开始。为什么有人会测量开发人员的生产力？我认为这一切都来自于管理层。管理层需要有对策。管理层需要知道开发人员在决定工资和晋升方面有多好，多有成效。当然，管理层的目标是提高生产率，但是如果没有明确的衡量方法，他们如何证明他们在这个领域是成功的呢？生产率的定义之一是单位时间的结果。但是结果可以量化吗？销售团队需要量化结果，以便为任务和项目定价。他们可以用琐碎的方式来做吗？

我认为所有这些问题的答案都是。不可能量化生产率和结果。如果你告诉程序员 A 比程序员 B 更有效率，那只是你的个人观点，并不客观。让我们看看为什么我会这样想。

# 测量方法

我读到过或亲身经历过几种衡量生产率的方法。我试着把它们列出来，一个一个地说明为什么它们不是真正正确的。

**代码行数/天**

我觉得这是最容易证明完全扯淡的一个。我可以在 100 行代码中实现相同的特性，或者通过使用环境的内置功能在 10 行代码中实现。从质量的角度来看，重用已经工作和测试过的特性比重新发明轮子要好得多，但是要花更多的时间来决定使用哪些功能。因此，如果我选择 10 行解决方案，我很确定我的 LoC/day 会比选择第一个解决方案少。

**每天提交的数量**

提交可以很小，也可以很大。提交可能很简单，也可能很困难。有时需要几天才能找到 bug 的根本原因并修复它，我只需要在长时间的调查后修改一行代码并一次提交即可。其他时候，我实现小而琐碎的新特性，每次提交一个特性，我一天可以实现 4-5 个小特性。我在修复 bug 方面效率低下吗？我想并非如此...

**每天实现的功能**

这也是连接到上一点。有时实现一个新功能不需要调查，可以很快完成，有时要花很长时间才能弄清楚如何在当前环境中实现一个新功能。程序的用户看不出区别，有时最简单的特性对用户来说是最有价值的。

**预计小时数/已用小时数**

这也是一个流行的策略，有人(开发人员自己，技术领导或团队一起)对任务进行时间估计，然后这个估计的时间作为目标来处理。因此，如果你比预计的时间更快地完成了任务，那么你是非常有效率的，如果花的时间更长，那么你就是没有效率的。这里出了什么问题？谁在做评估？如果开发人员自己比他只需要高估一切，他将是世界上最有生产力的程序员。如果是技术领导或者团队一起做的话，情况会好一点，但是在很多情况下，特别是如果你对代码库没有深入的了解，做评估一点都不容易。我经常会遇到一些问题，这些问题根本没有被计算在内，需要很长时间才能解决。我也曾多次认为一项任务很难完成，但我在一个小时内就找到了一个简单的解决方案。在这种情况下，这又一次与生产率无关，而是与运气有关。

**赚到的钱/天**

这种方法多为自由职业者使用。他们只是用挣得的钱/天来衡量他们的生产力。如果你把钱算作结果，那么它是正确的。但是当你需要重新安装你的工作环境或者学习新的东西的时候呢？你没有赚到任何东西，但是你整天都在努力工作。那时候你是一个没有生产力的开发者吗？不，我不这么认为。或者，如果您为一个开源项目或一个高收入客户开发相同的特性，会有什么不同呢？这对你的工作效率有影响吗？不，不是的。

# 生产力与质量之间的联系

另外很重要的一点是生产力和质量之间的联系。我认为没有一个是可以量子化的，这是很普遍的。你不能证明你的工作质量比同事的好。简单有太多的视角。

但是我在这里想说的是，产生质量差的代码会更快:不要考虑测试、代码可维护性、文档等等。毫无疑问，你在那一项任务上会更快(更有成效)。但是后来，当它需要被扩展，或者一个 bug 出现时，你将会在弄清楚你所写的东西上失去所有的生产力。你需要在质量和生产率之间找到一个折衷点。

# 团队的生产力与个人的生产力

大多数程序员都在团队中工作，但是在这种情况下衡量个人的生产力有意义吗？

让我举个例子:在我在家工作的最后几天，我完成了我在办公室能完成的大部分任务。但是两天后，我的同事们开始给我打电话，因为他们遇到了技术问题，无法接听。因此，直到我个人发挥出巨大的效力，我的队友们才被阻止，因为我不在那里帮助他们。由于这个问题，团队的总体生产力很差。如果我支持队友，那么我的个人生产力会更差，而团队生产力会更高。这就是为什么我认为在个人层面上衡量生产力真的会适得其反。它激励开发人员支持他们的同事，这使得整个开发变得更慢。如果需要衡量生产率，请在团队层面进行！

# 生产力的影响者有哪些？

生产力取决于成千上万的东西。我不能给出完整的列表，所以我只是提到最重要的几个:开发人员需要对任务有正确的技术知识，对代码有良好的体验，良好的工作环境(良好的硬件，平静的环境)，应该避免不必要的会议和交流，需要良好的精神状态，也许最重要的一点是动机。这个话题太复杂了，我会写一篇单独的博文。

# 总结

正如你所看到的，生产力是一个非常困难，但也很重要的话题。我建议避免量化生产力，如果必须量化，最好在团队层面而不是个人层面进行。
