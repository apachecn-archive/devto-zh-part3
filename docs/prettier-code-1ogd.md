# 漂亮的代码

> 原文：<https://dev.to/maxlmator/prettier-code-1ogd>

如果你关心代码格式，你可能想看看更漂亮的。它极大地改变了我对编码风格的看法。

所以我过去常常花很多时间摆弄代码风格。从争论空间和标签，到比较 Symfony 的编码标准和谷歌的风格指南。随着 JavaScript 开始成为大多数新项目的首选语言，我选定了 [Airbnb JS 风格指南](https://github.com/airbnb/javascript)，通过匹配的 linter 规则，这个话题已经解决了相当长一段时间。

但是半年前，我们在工作中决定在一个新项目中使用更漂亮的。这从根本上改变了我对代码样式指南的看法:我不再在乎了。

pretty 做了什么:pretty 解析代码并以它自己的格式打印出来，而不是在代码编写时查看代码样式并对其应用规则。所以经典风格指南给每个开发者的余地不再是一个值得思考的话题。

像许多 linters 一样，它会在保存时自动重新格式化文件。起初，这感觉像是对我工作的严重干扰。毕竟，我——至少是假装——在我写的代码风格上投入了一些努力和自豪。但是几天后，我几乎完全停止了对代码格式的思考。几个月后，我又回到了另一端:我写代码，如果代码不能自动转换成现在熟悉的漂亮风格，我总是感到非常困惑。

所以，如果你碰巧开始了一个新的项目，就让漂亮一点的人尝试几天吧。
