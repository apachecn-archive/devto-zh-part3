# 故意做出错误的选择

> 原文:[https://dev.to/kspeakman/wrong-choices-on-purpose-5e0p](https://dev.to/kspeakman/wrong-choices-on-purpose-5e0p)

一天晚上，我正沿着一条偏僻的小路回家。令我惊讶的是，有一辆大卡车正行驶在同一条乡间小路上。我立刻被激怒了。“难道他不知道他会破坏道路吗？它不是为负重而造的。道路已经很糟糕了。”我想阻止他并责备他。为了自己的安全，我很快打消了这个念头。然后我想到把他的路线报告给卡车运输公司甚至交通部。但后来我有了一个想法，让我决定放弃整件事:如果他确切地知道自己在做什么呢？尽管表面如此，但如果这实际上是一个经过深思熟虑的选择呢？

## [](#back-to-dev)回到开发

我之所以会有这种想法，是因为我不时会遇到来自我帖子的评论者的同样的反驳。大多数时候评论似乎是假设我所做的选择是由于缺乏基本知识或经验，他们决定以正确的方式教育我。然而，我从事开发工作已经将近 20 年了，并且一直是一个热心的学习者。我当然不是什么都懂，继续通过改正接受学习。但是最基本的...是的，我收到了。

我经常有一个很好的理由得到纠正。有时候我会故意做出“错误”的选择。

### [](#example)举例

*从 F#*

下面是我在[上发表的理解折叠操作](https://dev.to/kspeakman/understanding-the-fold-operation-24p1)的一个例子:

```
// F#
let runChallenge someList =
    someList |> List.fold update init 
```

这是我在自己的代码中编写函数的方式。但是，函数式编程 101 和 DRY 说我应该把`runChallenge`函数写成这样。

```
// F#
let runChallenge =
    List.fold update init 
```

这不是我第一次收到评论说我应该使用后一种风格。它更短，避免了声明`someList`只是为了立即将它传递给另一个函数。那我为什么不用呢？嗯，经验。我们有一个完全用函数式语言编写的应用程序，已经投入生产好几年了。在这个过程中，我尝试了多种表达问题的方式。其中之一是第二个例子中的无点风格。

每当我一个月后回到无点代码时，我注意到我总是必须跟踪实现。仅仅定义函数名`runChallenge`并不能告诉我任何关于它的信息。那么无点定义通常意味着实现在内部使用了一点函数组合。所有这些使我们一眼看不出一个函数使用哪种类型，因此也看不出发生了什么样的行为。

尝试向新开发人员解释无要点代码也很“有趣”。这让我隐约想起解释 Perl 代码，它隐式地使用了 [`$_`变量](https://www.quora.com/What-are-some-criticisms-of-the-programming-language-Perl/answer/William-Havelin)。这是困惑和疑问的来源。然而，当参数是显式的，我必须输入额外的字符，但我没有得到一个关于它的问题。

由于这些经历，我一般不使用无点符号，即使这是理论上的“正确”选择。这在实践中根本不是最好的开发体验(至少在 F#中是这样)。而且不止我一个人这么想。(仅供参考，说话者是 F#语言的创造者。)

总有例外，对吧？免点的一个例子是我有意制作一个简洁的 DSL，比如 URL 路由。

## [](#ignorance-vs-intention)无知 vs 意图

幸运的是，人们听不到/读不到我内心的想法(也许谷歌和 FB 正在努力)。但不幸的是，他们也无法知道我是否出于无知或深思熟虑的意图做出了特定的选择。我想大多数人都不想阅读以一系列假设或警告开头的文章，所以我不会在每篇文章中都花时间来解释。因此，当我收到类似的纠正性评论时，我并不感到惊讶，我会花时间解释这一点。

亲爱的读者，我认为这篇文章的目的是鼓励你考虑文章的作者和背景。也许他们*已经*错过了一些明显的东西。(我当然有时会。)不确定的话一定要提问。但也可能给他们一些好处。如果某个刺激与文章的主题无关，也许就不值得一提了。我也在这里自言自语。

## [](#epilogue)后记

至于卡车司机。我跟在他后面，观察他的驾驶行为。从他驾驶卡车拐弯的样子来看，很明显他是在轻载货物。然后我想到了他的替代路线。主要的一个是一个繁忙的 4 路站。他可能不得不等到一辆车停在足够远的地方，让他来一个大转弯。走那条路会耽误每个人在车站的时间。所以最终我决定，也许司机知道他们在做什么，我给了他们好处。
