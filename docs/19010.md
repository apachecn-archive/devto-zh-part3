为了制作更好的软件，你需要理解人们是如何工作的，这也是我推荐的书籍倾向于关注的。

那么你为什么在这里等待下面给出的列表:

1.  **代码完成 2**
![](../Images/31555fbcf535f1525561b5abebf39b00.png)

史蒂夫·麦康奈尔(Steve McConnell)的《代码完整 2》(Code Complete 2)是软件开发人员烹饪的乐趣。阅读它意味着你喜欢你的工作，你对你所做的事情很认真，你想不断进步。在《代码全集》中，Steve 指出普通程序员每年阅读的技术书籍不到一本。阅读这本书的行为本身就已经让你从 90%的开发伙伴中脱颖而出。从好的方面来说。

**T3】链接:** [代码完成 2](https://blog.codinghorror.com/content/images/2014/Jul/code-complete-2.jpg)

4.  **神话中的人月**

[![](../Images/cb1d63b801103c4bcc7c100aaf792e90.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--ayYIJb-m--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/t20qnx5dhhzl0yczrksg.jpg)T3】

这本 25 年前的书大胆地阐明了一个观点:计算机可能会改变，但人不会。

阅读这本经典著作肯定会比研读最新的千页技术巨著更好地利用你的时间。

9.  **不要让我思考**

[![](../Images/9afd557fdafcca515893f7ffc03489d6.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--S754DmUT--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/5ug0g5epehjxip5vu2cz.jpg)T3】

我读过的关于可用性的最好的书。标题是“网站可用性”,但是不要被它虚假的特性所迷惑。史蒂夫克鲁格涵盖了本书中每一个重要的可用性概念，并且涵盖得很好。这几乎是有趣的。如果你选择只读一本关于可用性的书，选择这本。它充满了大量的信息，并且以简洁、易懂的格式呈现。它适合任何受众:技术人员、非技术人员、用户、开发人员、管理人员，你能想到的都可以。

13.  **快速发展**

[![](../Images/d14dbab1eee11ba03c5fedf85b54f615.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--jqCes9BJ--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/pr2baw33j5y4mk5t4k83.jpg)

这本书的全名是《快速开发:驯服疯狂的软件开发时间表》,这不仅冗长而且有点荒谬，这也是一个不幸的用词不当。

快速开发不是快速开发。是关于*失败的现实*。绝大多数软件开发项目都会失败:它们会超出计划，产生不合标准的结果，或者有时甚至根本没有完成。这不是争论。这是一个统计事实。令人不快的事实是，你的团队必须非常优秀才能避免失败，更不用说成功了。虽然这听起来令人沮丧——好吧，的确令人沮丧——但你还是会想读这本书。

为什么？因为成功的一半不是重复你或其他人犯过的同样的错误。本书给我们的启示是，犯错误是好的——只要它们都是新的，都是歌唱和舞蹈中的错误。如果你在犯[同样的经典老错误](http://blog.codinghorror.com/escaping-from-gilligans-island/)，你甚至还没开始就已经失败了。你可能不知道你现在犯这些错误的可能性有多大。

我们的领域是少数几个变化是唯一不变的领域之一，所以拥抱这种变化并尝试不同的“快速”开发技术是很自然的。但是反过来就不正确了。我们不能假设自 1970 年以来发生了如此大的变化，以至于所有旧的软件开发课程与我们热门的新技术相比都已经过时和不相关了。还是老一套:电脑变了；人们没有。在开始之前，至少对什么有用，什么没用有所了解——用麦康奈尔的话说，“在油漆之前，阅读油漆罐上的说明。”当然，这听起来是显而易见的，直到你读了这本书，并意识到这在我们的领域实际上很少发生。

根据这本书，严格来说，是四分之一。但我认为不止如此。

***链接*** : [飞速发展](https://mycomputersciencebooks.files.wordpress.com/2017/08/steve-mcconnell-rapid-development-taming-wild-software-schedules.pdf)

21.  **Peopleware**

[![](../Images/1963f078c2db7ec657eae68dc9a7c1fa.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--e60sN0QR--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/y73c53oxve151st02fvm.jpg)

如果你曾经看到过全明星运动队的表现由于糟糕的教练而受到影响，你会喜欢这本书。当没有一个人能相互交流，或在任何事情上达成一致时，你有多少“编码超级明星”也没用。没有一个开发人员，不管他多么有才华，在不断受到干扰的情况下能够有效地工作。开发人员本身并不以他们的人际交往技能而闻名，但讽刺的是:你的项目的成功可能就取决于此。如果你有任何正当的愿望，想成为一名实践中的“团队领导”，而不仅仅是名义上的，你需要拿起这本书。

虽然 Peopleware 充满了伟大的、完全有效的观点，但它也意味着员工对工作场所的一定程度的控制，这在大多数公司都是纯粹的幻想。但至少你会知道什么时候你的工作环境或你的团队是真正的问题——更重要的是，知道该怎么办。

25.  日常用品的设计

[![](../Images/43c6ea2200b91f1301daaecd89d75f94.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--tMGQJUdY--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/l2teqdqdbm9qtrtm3pa1.jpg)

开发软件可能会非常令人沮丧，因为太多的事情可能会出错。我们做的很多事情都是防御性的:试图在事情变糟之前预测到它。这让人精神疲惫，最终会以一些消极的方式表现出来。我有时会向非技术人员描述这种情况，就像用一千个运动部件组装一只手表，所有这些部件都可能在最轻微的刺激下随机出现故障。美好时光！

诚然，设计软件很难，但设计一扇门也很难。设计的细微差别延伸到你接触的每一个物体，无论是某个热门的新 SQL 引擎，还是一只不起眼的鞋子。这本书会让你对“细节中的魔鬼”有一个新的认识如果设计一扇门并不像我们想象的那样简单，也许是时候让我们自己休息一下了，因为我们不能完美地设计软件。

***链接*** : [日常事物的设计](http://www.nixdell.com/classes/HCI-and-Design-Spring-2017/The-Design-of-Everyday-Things-Revised-and-Expanded-Edition.pdf)
