# 充分利用编码课程和教程

> 原文:[https://dev . to/joncalhoun/get-the-most-out-of-coding-courses-tutorials-3 ooe](https://dev.to/joncalhoun/getting-the-most-out-of-coding-courses-tutorials-3ooe)

我最初通过电子邮件将它的一个变体发送到我的[邮件列表](https://www.calhoun.io/#subscribe)，我在这里分享它，以防它能帮助其他人。

作为一名围棋老师，我倾向于听到各种各样关于学习编码的不同方法的反馈。例如，曾经有人告诉我，视频课程弊大于利，因为它们会导致开发人员无法独立思考，而是依赖他人来构建任何东西。另一方面，我也有学生使用我的视频课程，然后继续获得围棋开发人员的工作。

那么是什么原因呢？

一些开发人员如何使用课程和教程并变得依赖它们，而另一些开发人员如何成为成功的开发人员？

我认为这两种结果之间的关键区别是每个开发人员如何使用他们可支配的资源，在这篇文章中，我们将探索一些方法，以真正最大限度地利用现有的大量材料来学习编程、围棋和其他任何东西。

我将对其中的每一项进行更详细的介绍，但这里是我想要涵盖的要点的大致轮廓:

*   不要只是照搬教程里的代码；学以致用。
*   学习资源应该被多次使用。
*   扩展教程以获得更多信息。

## [](#dont-just-copy-the-code-apply-it)不要只是复制代码；应用它

我想讨论的第一点是，实践是学习任何新技能的必需品，简单地复制教程中的代码不是实践。至少，这不是充分的实践。

太多的开发人员拿起一本书或一门课程，期望阅读它并在完成后“知道如何编程”。事实是，这很少发生。为什么？因为你不能指望仅仅通过阅读就知道如何去做几乎任何事情。你需要练习！

旁注:这也是为什么像“24 小时自学围棋”这样的标题具有滑稽的欺骗性。你最多可以在这段时间内熟悉这门语言，但是要真正学习任何一门编程语言，你都需要用这门语言写一些代码。

虽然跟着教程一起编码是一个很好的起点，但是简单地按照别人告诉你的去编码是很愚蠢的。你不用去想你在做什么或者真的证明你理解了材料。因此，仅仅按照教程进行编码不可能产生想要的结果。

相反，我强烈推荐的是(a)在使用资源之前尝试自己编写项目代码，(b)将您的资源用作大纲或指南，但不直接依赖它，或者(c)在学习后将您学到的知识应用到其他项目/问题中。

这在现实世界中是什么样子的？我们来看几个例子。

首先，让我们看看一个想要使用 [Gophercises](https://gophercises.com/) 来获得一些编写真正的 Go 代码的练习的开发者。他们怎么能这样做？

我的建议是观看介绍每个练习的视频，然后停止播放视频，尝试自己编写每个练习。这将是艰难的；在许多情况下，一小时内可能完成的代码可能要花上几天。但是没关系。这里的目标不是速度，而是强迫自己阅读 Go 文档，并尝试拼凑如何使用各种必要的库来解决手头的问题。

你可能会在途中遇到困难，这时你可以通过观看视频来寻找灵感。这就是为什么会有这些视频——向你展示另一个开发人员可能会如何处理这个问题——但是我确实认为首先给自己一个解决每个问题的机会是很重要的。

或者，或者如果你卡住了，需要看视频，另一种方法是先看视频，了解你在做什么，然后给自己一些时间——也许几个小时或一天——然后回来，试着不看视频，进行同样的练习。不要担心得到和视频一模一样的结果，只要试着自己解决，因为你已经在视频中看到了。

对于这两种方法，我更大的观点是，我们不仅仅是让某个人牵着我们的手，告诉我们到底要编码什么。当然，观看这些视频有助于理解解决问题的方法和学习新技术，但是我们不能指望仅仅通过观看来提高我们自己的开发技能。我们需要编写一些代码。

这种代码优先的方法适用于像 Gophercises 这样以练习为导向的工具，但是当我们想从一门课程或教程中学习一套我们没有经验并且没有组织成练习的新技能时会发生什么呢？

对于我们的第二个例子，假设您实际上并不知道如何构建一个 Web 应用程序，让我们看看几种方法来充分利用我的 [Web Development with Go](https://www.usegolang.com/) 课程。

附注:我在这里给出的例子是针对这个例子的，但是我们在这里看到的技术普遍适用于任何教程或课程。

与我们的第一个例子不同，不先学习如何开始编写 web 应用程序实际上是不可行的。我们可能不知道如何正确地创建一个认证系统，这真的不是我们应该即兴创作的。那我们该怎么办？

在这种情况下，我鼓励开发人员观看视频(或阅读书籍)，然后将它们用作大纲。例如，在我的 Web 开发课程中，我们创建了一个照片库应用程序，您第一次浏览课程视频时，构建相同的应用程序可能是有意义的，但这不是我要停止的地方。然后，我会第二遍浏览课程材料(或者，如果你愿意，在你完成每一章的时候，第二遍浏览每一章),并尝试建立一些类似的东西，但不完全相同。也许是 Twitter 的克隆，或者 Pinterest 的克隆，甚至是迷你 Reddit。

这里的目标是挑选一些与原始项目足够接近的东西，我们可以用它作为一个好的指南，但要让它足够不同，我们必须思考我们在每个视频中学到的东西，并将其应用到我们独特的应用程序中。通过这样做，我们强迫自己真正理解材料，因为如果我们不这样做，我们就无法将我们所学的应用到我们的具体项目中。

旁注:如果你是一个从另一种语言过来的有经验的开发人员，你通常可以在第一次浏览材料时这样做。

最后，将会有一些教程，在那里你不能真正地把它们作为一个大纲，你也不能真正地编码所有的问题，如果你不跟随并且首先学习材料的话。一个很好的例子是算法，如果不理解冒泡排序是什么，你就不能写冒泡排序，但是如果不直接复制代码，很难将教程或截屏作为大纲。

在这种情况下，我强烈建议学习这些材料，然后将其应用于其他项目或实践问题。回到我们的算法例子，真正确保你知道算法如何工作的最好方法是找到许多使用该算法的问题，并尝试解决它们。

旁注:像 Topcoder 和 HackerRank 这样的网站可以很好地找到实践问题，但是不要太沉迷于那些你没有学到实践技能的网站。

我真的不能再强调这一点了，即使你是那种仅仅通过观看视频或阅读教程就能“明白”的人。当我在大学学习计算机科学时，我发现我的大多数同龄人可以耐着性子听完一堂计算机科学课，然后如果我跟他们谈论我们学到的新算法，他们可以解释得很好，但如果你真的让他们编写代码，他们会很难理解。

几乎所有的编码概念，你都需要在真正理解之前，在你自己的项目中付诸实践。所以不要光看视频或者看书。实际上，用你所学的去建造一些东西或者解决一个练习题。

## [](#use-resources-more-than-once)多次使用资源

我将看到的另一个常见错误是“一劳永逸”的错误。我的意思是，开发人员将阅读一个教程或经历一次课程，然后就完成了。这是一个坏主意，原因有很多。

第一个是，你经常没有你需要的所有知识来真正地把整个图片放在一起。我的意思是，我们在编程(以及生活中的大多数领域)中学到的东西有这些奇怪的循环依赖，我们需要理解 A 才能理解 B，但我们也需要理解 B 才能理解 A，那么我们如何开始呢？

答案是我们从一点点开始，然后让开发人员稍后再回到他们不理解的部分。

例如，当一个开发者第一次学习围棋时，我们可能会告诉他们，“不要担心这个`package main`部分——我们以后会知道它做什么。现在只要理解我们的`main`函数将要运行，无论我们放入什么代码，只要我们启动程序，它就会运行。”

然后，一旦程序员理解了他们的程序，我们就可以教他们关于包的知识，最后再回来解释这个特殊的`main`包是做什么用的。

当我们进入更复杂的主题时，如测试和 web 开发，这变得更加困难，因为有太多的活动部分。事实是，当你第一次读到它的时候，你不可能完全理解它。所以只要浏览一下材料，从中获取你所能得到的，但是一旦你有了一个坚实的基础，你可以计划以后再重温它。

## [](#expand-upon-what-is-taught)展开所教的东西

我看到的课程成功者和不成功者之间的另一个巨大差异是，成功者几乎总是试图扩展所教授的内容。

一个简单的例子是一个简单的教程，教你如何用几个页面构建一个 web 服务器，并添加新的页面，但这可以远远超出这个范围。

例如，我的 Web 开发课程是特意设计的，以便您可以非常容易地用几乎任何您想要的实现替换 DB 实现，我鼓励学生们这样做。许多学生已经利用了这一点，并使用不同的数据库库(例如另一个 sql 库 [sqlx](https://github.com/jmoiron/sqlx) )重写了应用程序的这一部分，而其他人则用 MongoDB 完全替换了我们的 Postgres 数据库。

作为一名开发人员，期望一个教程涵盖日常工作中需要做的所有事情是不现实的。更成功的开发人员不会为此感到沮丧，而是将他们所学的东西拿出来，尝试超越极限，然后与他人讨论什么可行，什么不可行。

这在最初的几次可能是有挑战性的，但是随着练习，你开始积极地思考如何根据你的需要来改编教程，随着时间和经验的积累，它会变得容易得多。
