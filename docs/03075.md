# 单元测试的重要性

> 原文:[https://dev.to/pcloudy/importance-of-unit-testing-haj](https://dev.to/pcloudy/importance-of-unit-testing-haj)

我们都知道测试您的应用程序是非常重要的，因为从长远来看，它可以确保安全性、客户满意度并节省资金。有时它也能拯救生命。这里有一个例子来说明我的观点；1994 年 4 月 26 日，中国航空公司的一架飞机因软件故障坠毁，夺走了 264 条生命。在软件测试中，单元测试是第一级，在这里大多数问题都可以被纠正，这样可以节省时间。让我们来理解什么是单元测试。

什么是单元测试？
在单元测试中，软件应用的单个组件在开发阶段被测试。单元测试通常由开发人员而不是测试人员来完成。测试一段代码形式的函数，以验证其准确性。驱动程序、单元测试框架、模拟对象和存根用于执行单元测试。

**如何执行单元测试**
单元测试通常是自动化的，但有时也可以手动完成。在指导性文件的帮助下，可以采用手动方式。单元测试可以在所有[类型的移动应用](http://www.pcloudy.com/types-of-mobile-apps/)上执行。

在自动化单元测试中，开发人员在应用程序中编写代码来测试功能或过程。一旦应用程序被部署，代码就可以被删除。该功能可以被隔离以严格测试应用程序，它揭示了被测试代码和其他单元之间的依赖关系。那么可以消除依赖性。大多数开发人员使用单元测试自动化框架来记录失败的测试用例。

如何改进单元测试
在执行单元测试时，有一些要点需要记住。使用一致的命名约定，一次测试一个代码。如果代码中有任何变化，确保模块有相应的单元测试用例。在进入下一阶段之前，必须修复所有的错误。为了避免错误，最好在提交代码时进行测试。更多地关注影响系统行为的测试。在线 Android 模拟器在某些情况下可以用来执行单元测试。

**单元测试的优缺点**

有了单元测试，开发的速度会更快。如果您执行开发人员测试而不是单元测试，那么您需要设置断点，启动 GUI 并提供输入。但是如果你做单元测试，你写代码，写测试，然后运行测试。您不需要提供输入或启动 GUI。最终，你有了一个更可靠的代码。与系统或验收测试相比，在单元测试中找到并修复 bug 花费的时间更少。

可重用代码减少了工作量并节省了时间，因为在单元测试中代码需要模块化。

更可靠的代码，因为错误在初始阶段就被修复了。

易于调试，因为如果测试失败，只需要调试最新的更改。然而，如果你在一个更高的层次上测试，那么你将不得不扫描在几周或几个月内发生的变化。

由于单元测试的模块化本质，我们可以测试项目的一部分，而不必等待其他部分完成。

单元测试的唯一缺点是它不可能检查所有的执行路径，并且它不能捕捉广泛的系统错误或集成错误。

结论
很明显，单元测试不能也不应该被避免。相反，开发人员应该选择测试驱动开发，他们可以编写测试，然后在测试的基础上编写代码。您只需要获得合适的工具，这将进一步减少您的测试工作。
