# 与雨果一起写博客

> 原文:[https://dev.to/detunized/blog-with-hugo-1i08](https://dev.to/detunized/blog-with-hugo-1i08)

我决定恢复我的个人博客，不再仅仅依靠 [dev.to](https://dev.to/detunized) 来满足我的博客需求。尽管使用 dev.to 来写技术博客真的很愉快，但它的某些方面还可以做得更好。我不太喜欢在浏览器中输入我所有的帖子，而宁愿使用一个合适的文本编辑器。我也不喜欢总是在预览和降价视图之间切换。我尤其不喜欢我的内容没有版本控制。

到目前为止，我只是离线编辑，每 5 分钟粘贴到浏览器中，看看它是什么样子。完成后，我提交并粘贴最终版本，然后发布。当我发现一个错别字时，我会修复、提交并再次发布。我想有一个稍微不同的工作流程。

我曾经有一个 Jekyll 博客，总体来说很不错，但我对它有些不满。我想尝试别的东西。我确信在我不注意的时候，许多令人兴奋的新工具出现了。我看了之后...我被震惊了，被选择淹没了。雨果、赫克索、盖茨比、Next 等等。甚至有一个专门的[网站](https://www.staticgen.com/)来跟踪他们的 GitHub stars、forks 和其他东西。哇！

这不是它。我还得找一个主题。每个项目都有一两个主题商店。在浏览了其中一些并阅读了一堆关于什么更好、更容易、更强大或诸如此类的内容后，我决定选择雨果和我挑选的一些主题。

Hugo 的酷之处在于它是一站式商店。它处理一切，从降价到为我的网站编译缩小的资源。不需要额外的插件。安装它非常容易。建立博客甚至更容易:

```
$ hugo new site blog 
```

搞定了。

添加一个主题并不难:

```
$ hub clone Track3/hermit themes/hermit 
```

然后我把它部署到了网络生活中。这也很容易。告诉 Netlify 用 GitHub 回购，用 Hugo 发布就行了。现在，当我推 GitHub 时，Netlify 拿起它，构建它，不到一分钟它就上线了。太神奇了！

能全是阳光和彩虹吗？当然不是。从那以后事情就开始发生了。首先，我试着给自己弄点 CSS。我不确定我们人类是如何发展到这种技术的。我想，这里错了一个弯，那里又错了一个弯，现在我们被困在这个烂摊子里了。你有没有试过把一个`div`居中？

[![People who know how to center a div](../Images/adf85006a39c24f114af621dba16c307.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--CjKpdR5I--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://i.imgur.com/sEOwTQo.jpg)

幸运的是，我不需要这样做。但是我想解决一些小问题。我在这上面花了太多时间，还创建了对原始主题的 pull 请求，并多次尝试修复 git repo。专业提示:对每个拉请求使用一个新的分支。

另一个主要的问题是 Netlify 没有接收到我在主题 git 子模块中的修改。我又花了太多时间追踪这个。我甚至给支持人员发了电子邮件，他们反应很快，很有帮助，尽管他们并没有真正帮助解决实际问题。这并不明显。经过几个小时的折腾，我得出了一个解决方案:**将生成的 CSS 放入 repo，并在每次更改后提交**。

显然，Netlify [目前不支持 Hugo pipelines](https://github.com/netlify/build-image/issues/182) 。这意味着生成的文件夹`resources`必须包含在 repo 中作为解决方法。因为我选择的主题在回购中有这个，Hugo 只是把它复制到`public`文件夹中，跳过了服务器上所有的 SCSS 处理。我真傻，试图修复 CSS。

如果某个可怜的人正在读这篇文章，并且有同样的问题:把你的顶层`resources`文件夹提交给回购。

Netlify 的 Criss 写信给我，说我可以[加入测试版](https://github.com/netlify/build-image/issues/254)，在服务器上构建我的 SCSS。一会儿我会试一试。现在我想享受一下工作的乐趣。

我的新博客的预览在这里。它仍然有许多问题，还没有一个合适的家。在公开之前，我还想把它和我休眠的[照片博客](https://detunized.net)合并。不管已经做了多少，总有两倍的工作要做。
