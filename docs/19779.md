# 激情项目很有趣——除了在生产中

> 原文：<https://dev.to/openforge/passion-projects-are-fun---except-in-production-1lmj>

就在 Ionic 团队推出 [Stencil.js](https://stenciljs.com/) 的时候，我们公司完成了从 Webjunto 到 [OpenForge](http://openforge.io) 的品牌转型。因为我们都非常热爱所有的科技事物；我们说“OMG -玩模板的机会！让我们使用 Stencil 作为 web 组件预编译器来构建我们的网站，并删除 Wordpress B.S .它将*所以*快得多！”。

从一个企业主的角度来看，这是一个从 Wordpress 过渡到我们亲手创建的东西的绝佳机会。我们可以[开源](https://github.com/openforge/OPENFORGE.IO)并讨论它。它让我们可以在培训团队新技术的同时将它应用到有用的东西上。

但是-我们毫无准备。这时疼痛袭来。

# 我们的假设

我们严重低估了 Wordpress 真正内置了多少我们从未想过的东西。在博客、页面、用户管理和 SEO 功能之间——这是一个巨大的功能集，消除了团队内部设计、开发和业务部门之间的大量*流程*需求。

我们还假设 Wordpress 中的任何东西都可以很容易地构建，值得我们团队称赞的是，我们**能够**构建这些特性；然而，我们缺少的是确定**哪些**需要跨部门评审的流程。

比如说；有多少商务人士会想到 manifest.json 文件？(在一个完整的设计和开发团队中)有多少人有资格识别内容和 url 路由之间的 SEO 不匹配？这些东西都是 Wordpress 和 Yoast 自动提供和识别的，我们甚至没有意识到。

好处是它创造了更紧密地整合我们公司的 3 个主要部门(设计、开发、业务)的需求；然而，不利的一面是，我们必须为此创建一个流程，而我们完全没有准备。

# 我的建议

值得吗？我*认为*是因为对团队的培训、PWA 专业知识和经验教训；然而，从“多么值得”的角度来看，这场胜利显然不如我预期的那样成功。

我会把我的建议分解成一个列表，列出哪些是值得的，哪些是我们错过的。我的希望是，如果你想为你的公司尝试同样的事情，你会对可能出现的问题有所了解。

### 值得

*   体验和培训(随着我们转向单一来源 PWA)
*   速度(和 WP 相比感觉像跑路的)
*   该控件(完全可定制，没有任何插件开发)

### 比预料的/意想不到的更难

*   SEO 管理。Yoast 的替代品是什么？
    *   Robots.txt
    *   清单文件
    *   URL 路由策略(谁审查这个？)
    *   旧的 404 页，即(我们只是重定向)
    *   规范链接
    *   将文本转换为图像(我们总是忘记-需要更新流程)
*   多填充物
*   Lighthouse 优化(本地与部署)
*   平滑滚动
*   易接近

还有一些项目没有进入我们的[变更日志](https://github.com/openforge/OPENFORGE.IO/blob/master/CHANGELOG.md)，当我想到它们时，我会更新帖子。

希望这对人们有所帮助！
