# 对于刚接触 web 开发的人，你会如何描述可访问性？

*   我们应该把 a11y 看作是我们网站的一个特色
*   一个网站越容易访问，对每个人来说就越有用
*   它不仅仅影响残疾人(15%的人口)
*   如果你的网站是可访问的，你会得到更多的客户
*   “这是一个很好的商业实践，是正确的做法，但也为每个人创造了最佳体验”

# [](#what-are-the-various-ways-people-with-disabilities-use-the-web)残障人士使用网络的各种方式有哪些？

*   视障人士使用的屏幕阅读器
*   放大镜用于放大文本、视频或图像，供视力不佳者使用
*   字幕和音频转录用于听力受损者
*   一种开关轴设备，用于有运动障碍的人通过开关的翻转来移动通过接口
*   眼球追踪也可以用来在屏幕上移动光标
*   根据人们的需要，有许多住宿设施

# [](#what-advice-would-you-give-to-people-who-may-be-more-experienced-developers)你会给那些可能是更有经验的开发人员什么建议？

*   公司将网络作为他们品牌的延伸，他们希望独一无二，与众不同
*   但是人们不想学习使用网络的新方法，因为网站是如此独特
*   自上而下的导航(如选项卡)可能会产生意想不到的结果
*   不同网站的不同工作流程为需要重新学习工作流程的用户设置了障碍
*   用户有肌肉记忆，例如，他们可能按下开关特定次数去某个地方，但当网站试图过于不同，他们可能(意外地)打破预期
*   提示:遵循现有的惯例！
*   意识到不是每个人都使用鼠标或触摸屏
*   人们可能通过辅助软件使用命令，这可能很难与您的网站一起工作
*   有许多可访问性标准，WCAG 是最突出的——熟悉这些标准并遵循指导方针，以确保你与已建立的模式保持一致，比如键盘导航
*   提示:实际上使用辅助软件测试你的网站，以了解瓶颈！
*   例如，chromebooks 配备了屏幕阅读器和其他工具，如放大，颜色过滤，android 音量按钮模拟开关轴

# 你会对那些用心良苦却过度描述 ARIA 属性的开发人员说些什么？

*   保持简单，依靠上下文，例如视频上的“缩小”是基于上下文的自我解释
*   不要假设用户不能理解一个简单的概念，比如视频的最小化，并且用冗长的描述来过度补偿
*   使用简单的语言，不要使用复杂的词语
*   也不要仅仅依靠 tab 键来进行键盘导航——“tab 不是键盘上唯一的键”
*   ARIA 提供了许多快捷键。tab 只是一个。

# [](#how-can-semantic-markup-help-assistive-software-to-understand-the-contents-of-the-page)语义标记如何帮助辅助软件理解页面内容？

*   例如`<input type="email">`传达语义并提供有用的建议(同样适用于日期类型、列表框等)
*   使用这些提示，除非你有一个非常好的理由不这样做
*   也许很难定制/风格化一个内置的表单控件，但是它们带有有用的语义，应该被保留下来

# 随着时间的推移，残疾人使用网络的方式发生了怎样的变化？

*   残障人士感觉被网络应用解放了
*   例如，在物理世界中，有像楼梯这样的物理限制，更不用说有人可能对残疾人有偏见
*   网站没有物理限制或偏见，它们对所有用户一视同仁
*   残疾人可以通过计算机完成传统上不可能完成的任务，如秘书工作，创造新的就业机会
*   当你和其他人一样使用相同的网站和网络应用程序时，你会有被包容的感觉，好像你受到了同样的对待
*   另一方面，web 开发者有责任解决所有的挑战
*   在现实世界中，真实的人可以看到有人有困难并给予相应的帮助，但网站和电脑是不灵活的

# 有哪些标准化努力，例如由网络无障碍倡议(WAI)所做的？

*   ARIA 指南非常广泛，定义了可以添加到 HTML 中的特定属性，使元素能够继承语义(例如，按钮或复选框功能)
*   复选框直观地传达状态(选中或未选中)，但这也必须通过数据属性(如`aria-checked`)非直观地传达，以便屏幕阅读器理解它
*   web 组件标准使得可访问的小部件能够被打包并在 web 应用中使用，增加了可用性
*   辅助功能对象模型(AOM)使 web 应用程序能够公开 a11y 信息并执行一些只有本机应用程序才可用的 a11y 操作，例如，用户手势/命令可以传递给 web 应用程序本身，后者可以采取适当的操作
*   AOM 定义了一种为图形用户界面(如 canvas)提供语义等价的替代方案的方法

# [](#what-are-the-tools-available-to-developers-to-understand-how-accessible-a-web-page-is)开发者有哪些工具可以用来了解网页的可访问性？

*   Chrome 内置了几个 a11y 工具
*   DevTools 在“元素”面板中有一个“辅助功能”选项卡
*   a11y 选项卡包括 A11y 树，它基于 DOM 树和布局树，它向辅助软件传达关于页面的信息
*   审计面板包括 Lighthouse，它有 11y 个审计，并推荐需要改进的地方
*   Lighthouse 不能捕获所有的错误，手工测试总是需要的，但是它仍然提供了有用的结果
*   记住:100 分并不意味着满分
*   HTTP Archive 从数百万个页面中收集关于 Lighthouse 得分的统计数据:A11y 得分中值为 62%，42%的页面正确使用了`alt` attr，12%的页面正确标记了表单元素
*   数据显示有很大的改进空间，我们需要加倍努力，并考虑使用自动化解决方案
*   注意 WCAG:内容是可理解的(描述图像、视频标题、简化的替换文本、禁用快速动画等)，网站是健壮的(HTML 是有效的，使用正确的属性，表单域有标签等)
*   标准可能是模糊的或过于技术性的，所以想要了解 a11y 的开发者可以找到一些视频，看看人们如何使用辅助技术设身处地地为他人着想

# 我们如何推动网络向前发展，而不把残疾人抛在后面？(例如 AR/VR)

*   难以等待渐进过程，如触摸屏可能无法使用
*   人们应该有一份职业或一个学位，挑战极限
*   发挥 a11y 的创造力，提供新的解决方案
