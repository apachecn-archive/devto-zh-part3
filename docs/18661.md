# 如何避免合并冲突

> 原文:[https://dev.to/samuyi/how-to-avoid-merge-conflicts-3j8d](https://dev.to/samuyi/how-to-avoid-merge-conflicts-3j8d)

当使用 git 与其他开发人员在一个项目上合作时，合并冲突是每个程序员的噩梦。只要有不同的开发人员在一个项目上工作，它们就一定会发生。它们需要时间来修复，非常烦人。解决合并冲突可能会占用非常活跃的团队宝贵的工时。如果经常发生，这通常表明项目成员之间缺乏沟通和协调。在这种情况下，沟通是很好的，但不时会出现合并冲突，这是本文的主题。我们想要的是尽可能减少合并冲突的所有实例。下面我列出了避免合并冲突的步骤。

## [](#use-a-diff-tool)使用差异工具

用 diff 工具比较分支总是一个好主意，这有助于在合并之前发现潜在的问题点。要使用 diff 工具，您需要配置 git 来使用 diff 工具，如下所示:

```
$ git config –global diff.tool <diff-tool> 
```

区分两个分支的命令是:

```
$ git difftool  branch1 branch2 
```

在运行上述命令之前，需要设置一个比较工具。Diff 工具还可以用来轻松解决合并冲突中的冲突。在合并之前区分两个分支对避免合并冲突大有帮助。

## [](#use-git-fetch)使用 git 获取

与在原点上进行 git 拉取相反，进行 git 取取可以在合并冲突的形式上减轻您的负担。不幸的是，许多 git 图形工具只执行 git pull。git pull 由一个 git fetch 和一个 git merge 组成，这意味着在它们被合并到您的本地分支之前，您不能检查更改。进行 git 提取使您有机会在本地分支和远程分支之间进行 git diff，从而发现流程中的潜在冲突。

```
$ git fetch 
```

就我个人而言，我使用一个名为 k-diff3 的 diff 工具来比较分支，它给出了分支之间变化的一个很好的图形显示，如果你不喜欢的话，还有很多 diff 工具。在检查了各种变更并与开发人员沟通了不清楚的变更之后，您可以继续将变更合并到您的本地分支中。至少使用这种方法，大多数合并冲突甚至可以在实际合并之前得到解决。

## [](#use-git-rerere)使用 git rerere

如果您是喜欢每天或者更频繁地进行合并的开发人员。然后你会遇到一种情况，你会一次又一次地解决同样的冲突。那么 git rerere 就是你需要的解决方案。这意味着重复使用记录的分辨率。基本上，它记录已经解决的合并冲突，并在合并冲突再次发生时再次重用它。这意味着我们不会花时间去解决反复出现的合并冲突。要使用 git rerere，需要通过
启用

```
$ git config rerere.enabled true 
```

一旦它被启用，那么如果有一个先前记录的相同场景的合并冲突，rerere 会自动为您解决它。

## [](#conclusion)结论

如果合并冲突不断出现，那么这通常是项目团队内部沟通和协调较差的标志。在这种情况下，除非团队内部的协调和沟通得到改善，否则上述解决方案都无法完全发挥作用。
