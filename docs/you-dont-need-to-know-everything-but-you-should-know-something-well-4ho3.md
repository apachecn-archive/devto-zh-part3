# 你不需要知道所有的事情(但是你应该知道一些事情)

> 原文：<https://dev.to/rhymes/you-dont-need-to-know-everything-but-you-should-know-something-well-4ho3>

丹·阿布拉莫夫最近发表了几篇文章，让我深思。我把这两个帖子视为珍贵的礼物。让我告诉你为什么。

在第一篇[截至 2018 年我不知道的事情](https://overreacted.io/things-i-dont-know-as-of-2018/)中，他列出了他所知甚少或根本不知道的技术和话题。我不打算在这里列出它们，因为这是题外话。

许多开发人员都在与无所不知的程序员这一神话般的表现作斗争，这些人吸入了所有的酷技术(现在、过去和未来)。这个神话可能源于自尊的波动、同辈的压力、不切实际的期望、作为一项运动的时髦措辞、软件的喧闹的“忍者”、夸大的招聘广告等等。沿着这条路走下去，要么导致精疲力竭，要么导致社交生活的逐渐消失。

所以，作为一股新鲜空气，我要热情地建议你去读读[那篇第一篇文章](https://overreacted.io/things-i-dont-know-as-of-2018/)。几个引语:

> 我们可以承认我们的知识差距，可能觉得自己是骗子，也可能不觉得自己是骗子，但我们仍然拥有非常有价值的专业知识，这些知识需要多年的努力才能获得

这让我想起了许多我乐于追随的开发者的态度。成为某方面的知识分子或专家，并且毫不犹豫地承认知识的尽头在哪里，这才是真正的品质。作为一个普通人。

> 我知道自己的知识差距(至少是其中的一部分)。如果以后我变得好奇或者某个项目需要它们，我可以填写它们。这并没有贬低我的知识和经验。我能做好的事情很多。例如，在我需要的时候学习技术。

这是主旨。好奇心驱动的开发(或“必要性驱动的开发”)是许多成功的开发人员所拥有的特征之一。学会如何学习更重要。

在第二篇文章中，[UI 工程的元素](https://overreacted.io/the-elements-of-ui-engineering/)，阿布拉莫夫谈到了他所熟知的东西，但有一点扭曲，使这篇文章成为真正的礼物。

如果第一个列表是关于工具和堆栈的，这一个是关于方法学、模式和 UI 开发中的重要事情。

阿布拉莫夫强调了一个方面(这里我们可以第二次喘息):

> 我最大的学习突破并不是关于某项特定的技术。相反，当我努力解决一个特定的 UI 问题时，我学到了最多。有时候，我后来会发现对我有帮助的库或模式。在其他情况下，我会提出自己的解决方案(好的和坏的都有)。
> 
> 正是这种对问题的理解、对解决方案的实验和不同策略的应用的结合，给了我一生中最有收获的学习经历。这篇文章只关注这些问题。

在关于 dev.to 的讨论中经常出现的另一件事是基础知识和问题解决相对于这个或那个工具的价值。

如题:不需要什么都懂，但是要学好一些东西。

我有一种感觉，这两个帖子会在 2019 年一而再、再而三地冒出来，作为各种技能水平开发者的阅读建议。
