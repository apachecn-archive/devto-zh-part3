# 有效测试

> 原文：<https://dev.to/jessekphillips/effective-testing--1lk8>

这是一个我有许多矛盾的轶事的领域(不是我在这里报道的)。这包括一系列相关的想法，但没有令人满意的结论。

我们已经知道，测试软件永远不会确认软件没有错误，有无限的可能性，在某些时候我们必须停止。

也有许多不同的方法来促进测试。例如机器验证、人工探索和人为失误。我们还想涵盖软件、性能、可用性等更多方面。

还有更接近代码的测试的巨大推动力(是左移吗？).导致单元测试，CI/CD 系统将代码和序列组合在一起。我们甚至让开发人员和质量保证人员更紧密地合作(或者完全消除质量保证)。

我提出这些是因为这是关于平衡的。当你意识到人们有不同的技能(广度和深度)时，事情也变得更加复杂，你需要识别、雇佣和培训来平衡所有这些领域。期望每个人都一样是不好的。人与人之间的差异对测试来说是一种资产。要是我知道那看起来像什么就好了。

我不认为我能满足每个方面来有效地测试应用程序。我认为这可能是公司的一个问题领域，他们期望一个人或一个团队能够满足有效覆盖产品的所有需求。

我发现我更加依赖不同的项目角色来捕捉问题，从开发到项目管理。但是如果项目经理发现了一个问题，这往往会被视为测试失败(注意，这并不等同于指责，而是在批准之前添加新步骤的位置)。我还在想怎么把这个变成官方测试甚至是预期。
