# 鲍勃叔叔的保龄球形-对 TDD 和 OOD 的关注

> 原文：<https://dev.to/alexmcconnell/uncle-bobs-bowling-kata---concerns-about-tdd-and-ood-4fle>

最近，我看了几次鲍勃大叔的保龄球得分形([http://butunclebob.com/ArticleS.UncleBob.TheBowlingGameKata](http://butunclebob.com/ArticleS.UncleBob.TheBowlingGameKata))，然后...我对他的解决方案极其失望。我对他的解决方案有两个大问题，希望得到一些反馈。

问题 1 (TDD):第三次和第四次测试都是垃圾。

根据我的理解和实践，当你进行 TDD 时，你要用尽可能简单的代码通过测试，如果通过测试的最简单的代码是一些非常愚蠢的代码，那么你需要写一个更好的测试或者写更多的测试。他的第三个测试是测试备件(他对罢工做了同样的事情)，他通过制作一个游戏来测试备件的得分，其中前两帧是[5，5][3，0]，其余帧都是零。他继续拿着这个，继续写得分备件的代码，但是...解决这个问题最简单的代码是“如果 roll == 3，那么 score += 3。”现在，这是一些愚蠢的代码，你不想真的把它放进去，但这并不意味着你可以忽略它，继续写你想要的代码。这意味着你要停下来写一个更好的测试，无论这意味着用多个备用组件做一个更复杂的游戏还是你可能有的其他东西。我错过了什么吗？

问题 2 (OOD):他怎么从来不上框架课？

当我查看 Game 类时，我看到它的所有 5 个私有方法都以完全相同的方式传递，这是对每隔一个 roll(一帧的开始)的一个引用，对我来说，frame 类需要被拆分是非常直接的。只需要给这一帧掷骰子，并告诉它下一帧，然后它可以返回一个分数，游戏类可以对这个分数求和。

因此...我错过了什么让这些事情变得有意义？
