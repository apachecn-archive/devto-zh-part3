# 我的面试经历

> 原文:[https://dev.to/simoroshka/my-interview-experiences-2c32](https://dev.to/simoroshka/my-interview-experiences-2c32)

每个面试过程都是不同的，通常很难想象一个人能期待什么，尤其是如果你是一个初学者或者通常不经常参加面试。我决定分享我的技术面试经验。

除了第一次面试，所有的面试都发生在芬兰。

#### [](#a-talk-over-a-cup-of-coffee)边喝咖啡边聊。

我说了几件我在大学做的事情，找了一份没有合同的兼职。最短最简单的面试，最糟糕的工作，让我离开这个行业好几年了。

#### [](#whiteboard-interview)白板面试。

这是一份大学暑期实习的工作。有人问我一些简单的算法，我必须为未来的项目(用 javascript 构建动画)提供一些数据结构。

#### [](#paper-programming)纸编程。

这几乎就像白板面试，但更混乱。我被要求写一个函数来计算斐波那契数，这就是这一部分。

#### [](#home-tasks)回家任务。

我即将毕业，没有什么经验，迫切需要一份工作，所以我高兴地抓住每一个机会来展示我的一些技能。我通过这种方式获得了我的第一份全职开发合同。

最好的事情是当你的工作得到好的反馈。最糟糕的是当反馈仅仅是“你太初级了”或者“你的代码质量低于我们的标准”。甚至是简单的“对不起，但是我们决定不继续这个过程了”。我知道写反馈可能需要一些时间，但是家庭作业也是如此(还有更多)。

#### [](#technical-discussions)技术讨论。

通常，所有的技术面试都有这样或那样的方式，但有时只是这样，没有别的。没有任务和解决问题的问题。

这种类型，如果你是大三，可能会有点压力。我被来自不同领域的各种问题轰炸，以探索我的知识。回答“不知道”太多次，很容易对自己失去信心。当你实际上很了解一些事情，并且至少听说过许多其他事情时，那就更好了。

一个是最近的，有点奇怪。大部分时间我都在听他们的系统，并就此提出问题。没有人试图专门测试我的知识。

#### [](#code-challenges-pair-programming)代码挑战/结对编程。

在我看来，这是最具挑战性的，但也是最有趣的。我参加过两次这种技术面试，但都是在同一家公司。我不得不在一个共享的环境中写代码，在这个环境中有一些任务和测试需要通过，同时大声思考并在必要时提问。提示:最好带上自己的笔记本电脑，以免因为不熟悉的布局/操作系统而产生额外的压力。

第一个失败了，因为我对现代 javascript 没有什么经验，也因为我对这种格式非常紧张和不舒服。如果你满脑子都是对自己的怀疑，而且不知道花括号藏在哪里，那么大声思考是很困难的。

第二个要好得多，即使给我的任务有点难。我知道会发生什么，也不觉得我必须在解决问题的每一步都做到完美。控制台日志记录是可以的。

#### [](#showing-your-own-code)显示自己的代码。

在最近的一篇帖子中，我就此征求了一些建议。出于某种原因，我期望仔细检查并检查项目的大部分(至少是结构和技术栈)，但最终只显示了一个组件和一个测试套件，以及后端的一部分。这很好，因为我可以展示我最新的代码，它比应用程序的其他部分都要好。:D

#### [](#doing-a-code-review)做代码评审。

这是迄今为止最有趣也是我最喜欢的一部！我被留下来独自处理一些非常糟糕的 javascript 代码 10 分钟。之后我就把我觉得有毛病的都说了。

我认为这比“有人监督的按需编码”测试潜在候选人的方式要好得多。此外，由于感知到的角色转换，压力也减少了:现在你是评价别人的人。
