# 为什么我们从项目开始就让前端开发人员参与进来

> 原文：<https://dev.to/browserlondon/why-we-involve-front-end-developers-from-project-start-8op>

当我作为前端开发人员开始我的职业生涯时，我花了大部分时间从设计模型制作 HTML & CSS 布局。我是一个线性过程中的最后一步，由设计师、项目经理和客户做出决定。除了编写代码，我的输入仅限于确认是否有可能构建某些功能。

我怀疑我不是唯一有这种经历的人。虽然项目肯定可以以这种方式交付，但它对成果的质量施加了一个相当低的上限。在最好的情况下，最终的结果只是忠实地再现了网页设计师的想法，在最坏的情况下，这可能意味着严重的问题直到为时已晚才被发现。

在 [Browser](https://www.browserlondon.com/) 我们有非常不同的理念。在新项目的第一天，我们组建了一个[核心产品团队](https://dev.to/browserlondon/why-small-teams-are-mighty-16oj)，该团队在整个产品构建过程中紧密合作。

[![Research and front-end developer](img/1fa665eda652bdf64cc378dd23ff658c.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--_kVLpCVK--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://assets.browserlondon.com/app/uploads/2018/12/JAMS-2-1024x683.jpg)

作为前端开发者，这真的是解放了。这意味着我能够表达自己对某样东西的外观和感觉的看法，并对为什么需要这种产品有更深的理解。当你见过用户，经历过公司的痛点，了解市场对你所做的东西的需求时，做点东西会更令人愉快。

所以在个人层面上，这被证明是非常宝贵的。但我们这样做也是因为我们真诚地相信这种方法能创造出更好的产品。原因如下。

## 更多视角=更好的产品

以我的经验来看，一个更多样的观点集合会产生一个更好的、考虑更周全的产品。作为一名前端开发人员，我习惯于考虑我们正在做的事情的“如果”和边缘情况，如果没有前端开发人员在早期参与这个过程，这些事情就会被忽略。思考这些问题通常意味着以更健壮、可伸缩的方式解决问题。

## 问题早就凸显出来了

如果开发人员只有在设计被签署后才有机会提出问题，那么最终结果很可能是妥协。

相比之下，让每个人都参与到项目的所有阶段，包括[发现阶段](https://www.browserlondon.com/blog/2017/03/01/building-an-app-what-to-expect-from-the-discovery-phase/)，意味着问题可以被强调出来，并在早期提出解决方案。这导致了更好的终端[用户体验](https://www.browserlondon.com/blog/2016/06/01/user-experience-design-getting-basics-right/)，也产生了其他连锁效应，比如…

## 鼓励知识共享

我最近遇到了一个情况，我对我们团队的一个设计想法提出了一个潜在的问题。我们想出了一个解决方案，但最终还是组织了一个每周一次的设计和开发知识分享小时，帮助我们所有人从新的角度看待事物。

## 客户赞赏

对于任何不在技术行业工作的人来说，开发可能看起来是一个非常陌生的领域，如果开发人员隐藏在客户之外，那么它会给人一种神秘的印象，即它只是发生在后台的某个地方。它在客户和制造产品的人之间制造了一个障碍，如果一些东西在技术上不可行，那么客户就很难理解为什么。

因为我们让整个产品团队参与到客户互动中，所以我们能够更好地解释任何技术挑战，回答问题，甚至让客户以新的方式思考。

这个清单还可以继续下去，我一直在发现新的好处。作为前端开发人员，我们今天的角色[不仅仅是编写代码](https://css-tricks.com/the-great-divide/)，最好的公司认识到这一点，并为我们提供平台，以最大限度地利用我们的技能。
