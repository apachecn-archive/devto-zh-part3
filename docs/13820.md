# 不要“因为谷歌这样做”就去做

> 原文：<https://dev.to/joshhadik/dont-do-it-because-google-does-it-58oh>

这是我不久前写的一篇评论，你可以在这里看到完整的背景，但对话的前提是，像谷歌和脸书这样的公司在招聘程序员时，与初创公司或小公司相比，有着完全不同的要求。

> [...]像 FB、Google 和 Amazon 这样的公司需要测试程序员的数据结构，因为某些结构在某些情况下效果更好，而且当他们的软件工程师知道使用哪些工具来优化一个过程以在 5μs 而不是 10μs 内运行时...这么说吧，对于一家运行数万或数十万台服务器的公司来说，每节省一微秒就意味着公司需要运行的服务器越少。服务器越少=成本越高。
> 
> 但对于大多数初创公司或中型公司来说，开发人员知道如何启动和运行更重要，而他们为此编写的实际代码则不那么重要。
> 
> 问题是，许多较小的科技公司仰望像谷歌这样的公司，并试图复制他们的面试过程，因为嘿，这是谷歌。这些公司从来没有考虑过，一个拥有数十亿用户的应用程序与拥有 500 个用户的应用程序相比，可能会有稍微不同的需求。

再读一遍最后一句话:

> 这些公司从来没有考虑过，一个拥有数十亿用户的应用程序与拥有 500 个用户的应用程序相比，可能会有稍微不同的需求。

我认为这在今天的环境中非常重要。许多 MVP 团队、初创公司或中型公司试图模仿谷歌或脸书这样的公司，这在表面上是有道理的...

`I want my app to be the ${topTenApp} of ${industry}, so why wouldn't I use the same tools and processes as them?`

区别就在这里。

谷歌使用这种工具或流程来解决他们成长过程中面临的一个非常具体的技术问题，相比之下，山寨公司使用这种工具实际上没有解决任何问题，但实际上*制造了更多的问题*。

Kubernetes 是一个扩展服务的强大工具，**但是如果你没有真正的用户来扩展服务的话，你就不需要使用它。**

使用它来开发 MVP 只是增加了不必要的复杂性，减慢了开发时间，并且使得雇佣合适的开发人员更加困难。只要坚持 Heroku 的爱好 dyno，并在你真正需要的时候改变它。

同样的规则也适用于许多工具和模式。微服务。机器学习。反应自然。CDNs。詹金斯。GraphQL。区块链。弹性搜索。这个清单还在继续。

我并不是说你不应该在你的初始版本中使用任何这些东西。如果你的应用程序的整个前提是为房地产建立一个基于智能合同的账本，那么*当然*你需要使用区块链，但如果你正在为马术运动员建立一个社交媒体网站，你可能不*需要*在开始时使用机器学习来建议内容。

开始时，您应该出于以下两个原因之一使用工具:

1)对你的 MVP 绝对必要。它提高了生产率。

**就这样。**

其他一切都可以改变，修复或添加。

不要做一个完美主义者，不要解决你没有的问题。

只要做一些足够好的东西，让用户真的想使用它，然后在你需要的时候担心可伸缩性。

如果你反过来这么做，你可能会浪费几个月的时间来构建一个*完美*无用的应用程序。
