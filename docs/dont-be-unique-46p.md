# 不要独一无二

> 原文：<https://dev.to/nukesz/dont-be-unique-46p>

今天是有趣的一天。作为我工作的一部分，我审查申请人的测试，并决定他或她是否应该进行下一步。通常我会看到类似的解决方案或错误，但今天不同。我收到了一个测试，在我的 IDE 中打开，开始为它挠头。这是一段 java 代码，但是有奇怪的变量(字符串 my_string，可能之前花了几个小时使用 python)，混合了空格和制表符。到处都是不必要的评论。我像往常一样记笔记，然后继续前进。测试本身有两个部分。第一个有点简单明了，它旨在考察候选人对正常编程任务和挑战的了解。看看代码将如何构造和组织，是否使用了某种编程模式，等等。所以我对第一部分不满意，甚至没有真正的工作，但我对我们的英雄有了一个整体的了解，我继续回顾第二部分。但是当多萝西进入奥兹国的时候，我突然变成了她。我感到惊讶和震惊的同时。

代码几乎是完美的，发生了什么？我运行了所有的单元测试，我甚至添加了一些新的单元测试，甚至尝试提出一些最奇怪的用例，但是当然，它仍然有效。我不能争论第二部分，我只是在第一部分和第二部分的课程之间切换。几分钟后，我开始质疑一切。有没有可能这个人不知何故匆匆看完了第一部分，在第二部分花了更多的时间？他是否只是在最后进行了重构，却没有时间清理作业的开头？或者他只是在编码的时候变得更好了？似乎很荒谬，我不相信。当我思考并打开项目中的文件时，感觉不止一个人在工作。会不会是这个人只是让一个朋友帮他做第二部分，然后他就把这些加在一起了？是的，就是它了！但是我能证明吗？这个人应该被转移到过程的下一个阶段，并尝试亲自谈论代码和编程，看看他是否写了这两部分？

也许我可以，但觉得只是浪费时间。我仍然不确定该做什么，仍然不知道发生了什么，我有了另一个想法。如果没有其他人，而他只是从某个地方“复制”过来的呢？所以我在看代码，看看我是否能找到它的任何特殊部分，我发现了一个错别字。他添加了一条评论，但是打错了一个单词，所以我想我可以搜索一下。在谷歌上的搜索没有返回任何东西，所以我去了 github，开始搜索神奇的拼写错误的单词，然后我又一次变成了多萝西。我找到了他第二部分代码的精确副本。然后一切都说得通了。他找不到第一部分，所以他不得不在没有太多帮助的情况下做一些事情并实现，他不得不添加单元测试，但代码质量还是很差。当然，如果他多花一点时间来重构第二部分，也许他会欺骗我，但我想你现在应该猜到了，他没有成为我在公司的同事。

当然，我的错误是在 github 上给了一个作业，但是我们倾向于告诉人们把结果上传到那里，我真的不喜欢打开拉链。但是在那里找到它也不容易(我也试过)。我想改变测试，但后来我意识到这一次也很棒。或许我们可以抓住并区分那些只是想复制粘贴解决方案的人和那些也在思考解决方案的人。

在这之后，我开始思考不同的代码风格，管理项目，我意识到了一些事情。我们倾向于有不同的风格，我们写代码就像我们做的那样，也许我更喜欢短的变量名，小的类和围绕我的代码的一些注释，但是如果我有一个真正喜欢打字和做相反事情的同事呢？长变量名，详细注释？谁对谁错？我们不仅仅是一群分别编写代码，然后把它们放在一起的人。我们应该是一个团队，一起合作。不应该有“我的”或“你的”代码。所以每次你打开一个类或方法，你可以“猜测”是谁写的代码，那么很可能你很不幸地失败了。

这个故事的寓意是为你的项目创造一种风格。最简单的方法就是遵循该语言的常规编码惯例并采用它。但是当然，它不会自动解决你的所有问题，在一种风格的情况下没有银弹(使用静态代码分析工具和代码检查可以帮助你)。但是试着跟随它在项目中的样子，看看它看起来和感觉如何。只要你能在代码中看到你的足迹，那么你还没有完成，代码已经准备好成为项目的一部分。当然这对于别人的代码也是一样的。在代码评审过程中不断向你的队友提及代码风格，或者重构现有的代码以匹配公共的编码风格，如果它已经是项目的一部分，但是不遵循该风格。

采用共同的风格将有助于您和其他人了解项目的“规则”，并指导您编写高质量的代码。
