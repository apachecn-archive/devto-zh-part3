# 为什么我在前端框架大战中选择 Vue

> 原文：<https://dev.to/hellkat_/why-i-chose-vue-in-the-front-end-framework-war-3ppb>

[![Vue.js](img/b486c8d0a5c663fcd622eef45c6fdb0a.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--AmwQ5TXH--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/g2j5emyvclf3ym3nj3mf.png)

几年来，Framework Infinity 之战围绕着三个选择，每个选择都有自己的优势和劣势。在脸书的支持下，似乎是前端世界中一大部分思想领袖所拥抱的闪亮宠儿。 [Vue](https://vuejs.org/) 是一家斗志昂扬的新贵，其使用量正在上升，但在 2019 年仍然缺乏 React 所拥有的社区和文档。最后， [AngularJS](https://angularjs.org/) ，框架游戏的老手，许多企业和项目仍然需要它的开发者。(在这场皇家之战中还有其他竞争者，但为了简洁起见，我将坚持这三个。)

现在，这不是那些**我的框架更好，让我告诉你为什么**帖子。有很多这样的例子，现存的开发者文献中肯定充斥着支持和反对的观点。这是一个关于一个公司决定采用的故事，一个开发者对该决定的反应，以及如果你也这样做，你可能会发现的好处和缺点。

Vue.js 是一个麻省理工学院许可的项目，正如其 Github 页面将证明的那样，由赞助商资助。在撰写本文时，最新的版本是 2.5.22，维护该项目的社区非常活跃。

2017 年，我工作的公司( [Twisted Rope](https://twisted-rope.com) )决定为少数中小型项目采用 Vue/Laravel 堆栈。有很多理由说明为什么这是我们的最佳选择；我不会在这里深究这些细节。然而，作为团队中的一名开发人员，我有点失望。当时，我和其他许多人一样，听到了高高在上的关于 React 的号角声。它注定是最伟大的。对 React 开发人员的需求将如何在未来五分钟内暴涨。如果我不学习它，我会被落下。在我的框架中——我相信你能理解的 FOMO，我想“为什么不做出反应？”。(特别是因为我已经花了相当多的空闲时间来学习它，以便在不可避免的 React 项目失败时做好准备。)

然而我喝了一杯咖啡，挽起袖子，说了一句“咩”。准备疯狂地潜入 Vue。这就是我的第一点:

你可以逐步采用 Vue，也可以逐步学习它。

对于像我这样的开发人员来说，这绝对是一个福音，他们在任何给定的一周内都要在截止日期前完成十个不同的项目，并且需要在自己的时间里对新技术进行调查。核心库实际上就是视图层，仅此而已。您可以使用任何 JavaScript 项目，安装 Vue，并在几行代码中运行一个实例，该实例可以为您的项目处理一位逻辑。你只是想要一点点 Vue，把一些文本输入镜像到另一个元素？Vue 会很乐意这么做的。

另一方面，Vue 也会很乐意处理你的整个项目。很容易。除了核心库之外，您还可以在闲暇时学习和安装其他支持库。 [Vue 路由器](https://github.com/vuejs/vue-router)，顾名思义，为你的单页应用提供路由。 [Vuex](https://github.com/vuejs/vuex) 将处理你的项目状态(类似于 Redux)。在你需要它们之前，你甚至不必*关心它们。随着项目增长的可伸缩性？太棒了。*

**2) Vue 可以活在你当前的 JS 代码中，也可以使用 Vue 组件。**

项目的核心 Vue 实例只是一个带有选项对象的构造函数。这是什么意思？这意味着您可以不引人注目地将 Vue 实例弹出到当前源代码中。或者，实例可以存在于自己的`main.js`文件中。无论哪种方式，这意味着如果你的项目使用 JavaScript，那么你可以使用 Vue。

然而，对我来说，Vue 的组件实现是我向它的神致敬的原因。通过对主 Vue 实例进行小的修改，您可以使您的项目基于组件。这些是扩展名为`.vue`的文件，每个文件包含一个用于 HTML 的`template`标签、一个用于逻辑的`script`标签和一个用于 CSS 的`style`标签。每个组件都是应用程序的一个模块化部分，一切都以一致的方式组织起来。没有 JSX。没有 CSS-in-JS。没有恶魔般的担忧。您的样式可以是当前组件的`scoped`，或者是应用程序的全局样式。可以用`sass`代替。父组件通过 **props** 与子组件通信，子组件通过 **events** 与父组件对话。组件提供了一种简单、易于推理的方法来处理您的项目。(另外，如果您只使用组件，您可以使用 Vue 的运行时版本，它避开了编译器，减少了依赖项的大小。)

太冷了，还不如给你倒杯玛格丽塔酒

vue-cli 是一个神奇的东西。全局安装它，只需一个命令，您就可以创建一个定制项目，完成您现在需要的一切(并且您可以随时添加您以后可能需要的内容)。有多种模板可供选择，但我最常用的是一个简单的 *vue + webpack* 设置。它为您提供了一个已经配置了 Webpack 的 SPA 设置，以及一个开箱即用的热重装开发服务器。你要做的就是开始添加你的代码。

不仅如此，新的项目流程还会问你几个问题:要不要加 Vue 路由器？需要 Vuex 店吗？您想用 Mocha 添加单元测试吗？输入`yes`可以让你在一瞬间设置好这些库。我不能过分强调它有多容易使用，或者它能节省多少时间。你在创建自己的 Vue 教程吗？您可以针对您的项目制作一个自定义模板，并分发给用户，让他们立即学习。那么，那些玛格丽塔酒怎么样？你说还要龙舌兰酒？

**4)文档是不可思议的**

基本文档和[API 文档](https://vuejs.org/v2/api/)非常好，你几乎不需要任何其他资源。我还没有遇到过很多这样的库，在那里我没有在问题出现的第一个迹象时就搜索 StackOverflow，或者在 [devdocs.io](https://devdocs.io/) 中寻找方法，因为 repo 中的 README 还不够。Vue 避免了你的困难，提供了主题的粗略浏览，但在你需要的时候提供了深入的探讨。

也就是说，有几次我遇到了文档没有解决的难以调试的问题或边缘情况。这就是广受欢迎的 React 的独特优势:论坛成倍增长，你可能在其中一个论坛找到你的问题的答案。Vue 还没有达到这一步——虽然 SO 和其他网站上肯定有主题，但有时信息并不像你想要的那样新(Vue 2 有突破性的变化，使 Vue 1 的文档变得不同，足以忽略不计)，或者对你的问题没有一个好的答案。在你绝望地跳下屋顶之前，是的，你可能找不到可以复制粘贴的示例代码，但是稍加挖掘，你通常会找到解决方案。如果你能抽出一点额外的时间，我认为这几乎更好:它迫使你理解你的解决方案，而不是现在粘贴，以后思考。

它真的很快，而且和其他人玩得很好

Vue 2.5 在速度方面有了很大的改进，包括对虚拟 DOM 和反应引擎的升级和简化。如果你是那种喜欢阅读技术细节的人，不要相信我的话- [阅读变更日志](https://github.com/vuejs/vue/releases)自己去看。

已经有一个后端，并考虑添加一个 JS 框架吗？Vue 会掩护你的。默认情况下，Laravel 可以包含一个 Vue 前端(这也是我们选择这个堆栈的原因之一)。但是自从我们采用了它，我也在 Django、Node 和 Serverless 上使用了 Vue，没有什么大问题。

**结论**

孩子们，这就是了。正如你可能会告诉，我认为 Vue 很酷。虽然这确实是一开始强加给我的，但从 Vue-noob 到有经验的 Vuester 的旅程总是让我更喜欢我的编码，而不是更少。不，我没有在任何地方花同样的时间在基于反应或角度的项目上。因此，虽然我可能听起来有点偏见，但老实说，我看不出 Vue 为什么不能与同重量级的其他顶级框架正面竞争。过去的论点(Vue 更慢，Vue 没有那么受欢迎，Vue 可能明年就不存在了)不再适用。如果你已经牢牢掌握了 HTML/CSS/JS，那么你就有了入门所需的工具，没有 JSX 需要应对(尽管 Vue 组件可以用 JSX 渲染，如果你喜欢的话)。

现在，如果我能让 [@wesbos](https://twitter.com/wesbos/) 做一集 Vue...；)

干杯 fam，继续学习！
