# [](#making-fun-of-methodology)取笑方法论

> 我们最优先考虑的是通过早期和持续交付有价值的软件来满足客户。

在大多数情况下，最优先考虑的是用最少的工作从客户那里获得尽可能多的钱，这要感谢一个大嘴巴的销售经理、顺从的项目经理，以及知道如何长期进行重构的编码人员。

> 欢迎不断变化的需求，即使是在开发的后期。敏捷流程利用变化为客户带来竞争优势。

当需求改变时，几乎所有的软件工程师都会抱怨和大叫，因为这个世界应该足够静态以适应他的脑力劳动，但又不能静态到让他离开并停止从他的创造中获得报酬。

> 频繁地交付工作软件，从几个星期到几个月，优先选择较短的时间尺度。

冲刺一周很好，你同意吗，开发伙伴？*不受控制地眨眼*

> 业务人员和开发人员必须在整个项目中每天一起工作。

在提出请求后不到 3 天就能得到 BA 的答复是一件非常有趣和愉快的事情，因为大多数时候他们都在忙着让你的生活变得更加艰难。另一方面，开发人员并不真的想和那些人道主义程度的农民相处，他们不接受另一个 Javascript 框架的伟大之处，而且一点也不时髦，一个企业根本不关心开发人员，并且现在就需要工作功能(他们还没有决定实际的工作，但是你必须做点什么)。

> 围绕有积极性的个人建立项目。给他们需要的环境和支持，相信他们能完成工作。

有上进心的人是很棒的人，你公司的雇佣率接近于零。大多数人工作是为了吃饱穿暖，而不是给别人的生意带来价值。不幸的是，现实世界打破了对未来的美好愿景，当专注的、自我控制的、有动力的专业人士的方法论被应用到拖拉的、无节制的、气馁的业余爱好者身上。

> 向开发团队传达信息的最有效的方法是面对面的交谈。

外包现实:您的经理是客户唯一的输入/输出数据源，这导致了不幸的数据损坏。你的队友除了编码和喝咖啡什么都不愿意做，所以面对面的谈话变成了对敌方间谍的审问，而不是友好的护送。

> 工作软件是进步的主要衡量标准。

你的经理的翻译:不要在意美国国债规模的遗留债务和技术债务，直到系统开始像 90 岁的咖啡狂人的心脏一样破碎。当它出现时，赶紧去修理它，但不要太久，你不会因为修理东西而得到报酬。

> 敏捷过程促进可持续发展。发起人、开发人员和用户应该能够无限期地保持恒定的步调。

这就是开发的速度——你破坏并腐烂项目，直到每一个新功能的实现过程都感觉像印第安纳琼斯电影《盗墓》,然后客户给你安排了一次重构和修复技术债务的冲刺。经过几个小时的修复，客户感到困惑，该项目仍然不是优雅的理想。

> 对技术优势和良好设计的持续关注增强了敏捷性。

编写代码和打破东西，生产将信号所有的错误，离开“技术卓越”在大学，我们在现实世界中的发展，在那里只有血，铁和比特...什么？我们的测试套件运行数小时，有些组件长达 3000 行？来吧，我们带来的价值，没有必要担心。

> 简单——最大化未完成工作量的艺术——至关重要。

客户最知道该做什么，是的，我们有很多这样的情况，需求在 sprint 中期或之后发生了变化，即使每个人都在努力发布没有人需要的特性。这很简单——你是被雇来写代码的，不是来争论的。

> 最好的架构、需求和设计来自自组织团队。

他的左手不知道右手做什么，所以他们自组织成天鹅、梭子鱼和螃蟹。

> 团队定期反思如何变得更有效，然后相应地调整自己的行为。

当你的经理认为效率取决于你在办公室呆的时间时，你很难理解。

# [](#on-the-serious-note)在严肃的音符上

现代计算机行业有稳定的新来者，每年都在增长，但是，他们中的一部分是专业的，为该领域所向往，受过适当的教育(通过自己或在大学),即使工资较低，没有炒作，他们也会留在该领域，有绝大多数人在这里赚钱，做报酬最高的工作。这些人有不同的意图，但两种类型都受到工作场所的欢迎，因为需求仍然大于供应，而且从第一眼看，很难将两者区分开来(甚至不参与关于从一种状态转移到另一种状态的辩论，因为或筋疲力尽，在编码中找到乐趣，等等)。

人们的不同意图、关注点和工作质量会直接影响雇主实现目标所需的流程和待遇。敏捷的创建是为了更好的代码、更好的质量和价值，从而为客户的业务注入活力。这意味着在这个领域工作的人是自我管理的、有动力的专业人士，他们知道自己在做什么。有些人就是这样，所以敏捷适合他们。另一方面，其他人...他们还没有那么独立，所以在某种程度上，方法论对他们来说是失败的。In 可以无处不在，在业务层面上，在管理或开发层面上。事实是，有些人无法离开福特工厂附近的流程工作，在那里他们只负责自己的那部分流程，不需要做任何其他事情。

# [](#fastfood-pipeline)快餐管道

让我们打个烹饪的比方。麦当劳(和几乎任何其他快餐连锁店)的主要流程特征之一是稳定和相对简单的食谱。他们试图按照任何有智慧的人都能理解和复制的方式来准备食物，并达到预期的效果。这样的菜肴永远不会非常美味，但它是可以食用的，即使没有什么特别之处，顾客也能感觉到这是一顿像样的饭菜。他们生产半制成品，编写食谱，并以某种方式形成流程，使流水线能够产生稳定的结果，不管厨房里的人有什么烹饪技能。

# [](#perfect-cuisine)完美美食

在幽灵的另一边，你有一个精英级的米其林星级餐厅，有一个敬业的专业厨师团队，他们非常清楚如何用任何给他们的东西烹饪一顿合适的饭，除了可能是黄瓜牛奶。他们知道如何团队合作和单独工作，他们尝试了厨房里的每一件设备并知道何时使用，他们学会了随机应变并与食谱保持一致。如果他们不那么喜欢一起工作，他们每个人都会开一家自己的餐馆。他们可以自由选择食物，可以做任何他们想做的事情，只有一个小小的限制——客户应该总是对他们吃的东西感到满意。

# [](#so-what)那又怎样？

大多数公司都介于这两种类型之间。问题是，同样的方法并不适合所有人。你不能给一个想要被约束的人自由，让他对需要做什么有适当的措辞，做好步骤并把它传递给链条中的下一个员工，同样你也不能限制并把一个米其林厨师放在机械流程中，期望他在这样的环境中工作得更好。在这个行业中，有时你会遇到高度受限和详细的流程，这些流程只将开发人员作为序列中的代码生成螺栓，以及“做你所能做的”流程，这些流程需要链中每个人的机智、开放的思维、动力和专注来完成工作。奇怪的是，这两种过程配置都被称为敏捷，尽管它们唯一的共同点是围绕交付过程的仪式，但没有任何价值。也许我们应该用新的术语来解释宣言是如何在野外实施的？
