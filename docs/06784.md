## [](#execute-execute-execute)**执行，执行，执行**

#### [](#invest-in-iterate-speed)*投资迭代速度*

1.  为什么要投资？
    1.  CI/CD–自动化测试和交付
    2.  评估指标的工具——提高质量、流程和速度
    3.  小块交付速度更快
    4.  较大功能的配置标志
    5.  选择性部署
2.  动作快学得快
    1.  迭代得越快，你就能越快地了解什么可行，什么不可行
    2.  消除瓶颈
3.  投资省时工具
    1.  如果你需要做一件事超过两次，就写一个工具
    2.  编译时间的减少
    3.  增量变更采用
    4.  用 20%的时间进行优化
    5.  切换到高效的工具，即使切换需要时间——以后复合
    6.  从小处着手，构建它，展示和改进
4.  缩短您的调试和验证循环
    1.  最小工作流程–从小处着手
5.  掌握您的编程环境
    1.  微调简单的动作，即使在这里或那里节省一秒钟
    2.  熟练使用文本编辑器
    3.  学习至少一门高效的高级编程语言
    4.  熟悉命令
    5.  更喜欢键盘而不是鼠标
    6.  自动化您的手动工作流程
    7.  在交互式解释器上测试想法
    8.  使只为变更运行单元测试用例变得快速和容易
6.  不要忽视非工程瓶颈
    1.  亲自解决瓶颈
    2.  沟通对人相关的瓶颈很重要
    3.  优先构建原型从决策者那里获得反馈
    4.  在你的发布清单上开始吧
7.  对您的迭代循环采取整体观点

> "*过早优化是万恶之源*"

#### [](#measure-what-you-want-to-improve)*衡量你想提高什么*

在决定使用哪些指标时，选择那些:

1.  最大化影响
2.  是可操作的
3.  反应灵敏而又强健

记录正在发生的一切——有助于诊断问题

1.  监控您的软件
2.  确定如何使用它
3.  衡量一切事物
4.  分析学
5.  将所有测量值投影到仪表板上

内化有用的数字——了解这些有用的数字可以让你:

1.  通过一些粗略的计算
2.  快速评估设计的性能属性，而无需实际构建它
3.  帮助您发现数据测量中的异常
4.  澄清需要改进的领域和范围

对数据完整性持怀疑态度

1.  用数据来支持你的论点是强有力的
2.  正确的衡量标准可以穿透哲学偏见和产品争论
3.  快速解决讨论
4.  不幸的是，错误的衡量标准会造成同样的结果——带来灾难性的后果。我们必须小心使用数据。
5.  增强数据完整性信心的策略:
    1.  大量记录数据，以防以后有用。
    2.  构建工具以更快地迭代数据准确性。实时分析很有用
    3.  编写端到端集成测试来验证您的整个分析渠道
    4.  更快地检查收集的数据
    5.  通过以多种方式计算同一指标来交叉验证数据准确性
    6.  当一个数字看起来不合适的时候，尽早挖掘它

#### 尽早经常验证你的想法

1.  尽快优化反馈
2.  了解客户真正想要什么，然后重复反馈
3.  做正确的事情
4.  建立 MVP 并发布测试版以获得反馈
5.  10%的精力花在小型信息原型上
6.  假装完全实现一个想法来验证它是否可行是非常有效的
7.  通过 A/B 测试验证产品变更
    1.  使用可用的工具，而不是自己写
    2.  产生可理解和可操作的知识
    3.  通过小型验证降低大型实施的风险
8.  小心单人团队
    1.  没有反馈和验证
    2.  代码审查
    3.  单调的工作
    4.  对人的依赖性
    5.  成就庆典不会在那里举行
    6.  反馈渠道策略
    7.  开放并接受反馈
    8.  尽早并经常提交代码
    9.  向彻底的批评家请求代码审查
    10.  请向你的队友征求意见
    11.  首先设计新系统的界面或 API
    12.  在投入精力写代码之前，先发出一份设计文档
    13.  如果可能的话，组织正在进行的项目，以便与你的队友有一些共享的环境
    14.  在投入太多时间之前，征求对有争议功能的认同
9.  为你的决策建立反馈回路

#### [](#improve-your-project-estimation-skills)提高你的项目估算技能

使用准确的评估来推动项目

1.  将项目分解成粒度任务
2.  根据任务需要多长时间来估算，而不是一个人完成任务需要多长时间
3.  承担这项任务的人必须给出一个估价
4.  避免随意猜测
5.  使用多种方法进行评估——从精细到完整的项目，以及以前工作的历史数据
6.  避免虚构的人月/人日——九个女人不可能在一个月内生下一个婴儿
7.  使用时间限制来避免任务范围的扩大
8.  允许其他人质疑评估

为未知做预算——建立缓冲时间，为可能发生的灾难留有余地

定义具体的项目目标和可衡量的里程碑

1.  必须拥有和最好拥有的区别
2.  关键利益相关方之间的明确性和一致性

尽早降低风险——首先处理有风险的领域，建立 e2e 脚手架，尽早进行系统测试。

以近乎谨慎的态度对待重写项目——增量重写对于阶段是有效的。

不要在马拉松中途冲刺——通过加班避免精疲力尽，现实一点，保持消息灵通。
