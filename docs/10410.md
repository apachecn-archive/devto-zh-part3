# 你想要一个 ACME 公司的设计系统吗？

> 原文:[https://dev . to/albinotonnina/do-we-want-a-design-system-at-acme-Inc-4ak 9](https://dev.to/albinotonnina/do-we-want-a-design-system-at-acme-inc-4ak9)

## 如果是的话，我会这样做

一般来说，设计系统可以粗略地定义为*一个为团队服务的系统，更有效地开发和发布功能，以形成跨所有应用程序的更具凝聚力的客户之旅。通过一致性提高生态系统质量。*

产品开发是我们的核心业务。在我们的世界里，我们的软件永远处于进化状态。

功能不断增加，而我们的积压不断得到纠正和更新。资源不是无限的，工程师和产品设计师总是忙于遵循路线图。

当我们几乎已经被要构建的功能埋没时，我们如何认为我们将能够构建甚至另一个产品，例如设计系统？我们没有足够的资源或时间。我们有许多**特色**等待建造。我们现在不能停止机器来构建和实现一个设计系统。

#### [](#cohesiveness-is-a-feature)凝聚力是一种特性

我们产品设计部门的最终目标是在所有产品中实现统一的用户体验。这对我们的业务很重要。是市场想要的，是客户想要的，是公司每个员工想要的。保证。

设计系统本质上就是为了提供跨产品的一致的用户体验。它通过整合视觉语言和其他职责来做到这一点。也许你的公司已经有了一个由设计团队构建的视觉语言，尽管缺乏一个连接它和开发者的系统会导致开发过程中的问题。此外，分散的团队和远程办公室可能会给流程增加更多障碍。设计系统承诺内聚性是主要的结果。

设计师肯定会更开心，更投入。设计师讨厌不一致。

#### 我们将更快地推出新功能

设计系统产生 UI 组件库和 UX 模式。它是设计者和开发者通过系统的共享语言交流的结果。

产品设计和工程可以从这一结果中受益匪浅。通过将小的解决方案让给共同的努力，产品团队将逐渐加速到一个新的水平。

*经前综合症肯定会这样。*

#### [](#we-will-produce-better-code)我们会产生更好的代码

工程师将更少担心与产品 UI 方面相关的问题，更多地关注业务逻辑。无论如何，工程师们普遍讨厌这部分工作。

花在修改布局上的时间更少也有可能使视觉质量保证的过程更加流畅和精简:拥有一个自主测试和控制的 UI 库自动保证了 UI 的完整性。

组件库也有助于我们的软件更加模块化和可伸缩。

开发者会喜欢这个的。

#### [](#will-produce-less-documentation)会产生较少的文档

共享的语言，共享的实践，它们都有助于使每个特性所需的文档变得精简和智能。

谁喜欢阅读文档？

#### [](#everyone-will-participate)每个人都会参与

这里有一个关于设计系统的很棒的事情，它们非常适合协作。**召集所有人参与**。从部门到团队，再到个人。

每个工程师和设计师在日常工作中都会很自然地接触到 UI 库，因此对它的熟悉可以鼓励知识共享和参与。这在我们的世界里是好事，考虑到高流动率和基本上**永远没有足够的时间**来做入职培训。

#### 这将提高我们公司的声誉

未来我们会越来越多地听到设计系统，因为公司正在朝着开发以用户为中心的产品和精益流程的目标改进。

这将有助于**吸引有才华的专业人士**，他们自然倾向于跟上最新的前沿技术。

我们也可以通过文章和演示分享构建设计系统的经验来吸引他们。

### [](#it-sounds-interesting-now-estimate)听起来很有趣，现在估计

> 设计系统是一个活的、有资金支持的产品，有一个路线图&backlog，服务于一个更大的生态系统，以实现统一的用户体验。 [内森·柯蒂斯](https://twitter.com/nathanacurtis)，[八个形状](http://eightshapes.com/nathan-curtis.html)

#### 我们需要一个多学科的团队

[![A common configuration of a team dedicated to the Design System](../Images/e77ba976da2bbd254d4015ea28f55468.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--nJr82jyU--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/kwsuu5sfopxguzbvgh6i.png)

一个产品需要一个团队。所有成员都致力于一半或更少的能力，否则仍然致力于其他产品。团队成员可以根据需要轮换/变异。

团队不会单独工作。他们的工作是在需要时让团队参与产品开发。**将系统团队视为建设者和研究者**。

#### 我们需要一个路线图

路线图揭示了系统的方向。

它使设计系统团队能够传达:

*   **承诺**交付价值，证明其明确定义的资金是合理的
*   **透明的优先级**，在为利益不同的产品和平台提供服务时尤其重要
*   客户依赖的可靠**节奏**

路线图突出了每个版本中最重要的元素，描绘了系统的发展方向，而没有淹没细节。将这个简单的故事组织成一个基于里程碑的柱状路线图格式可以创造奇迹。

设计系统团队的首要任务之一是定义一个粗略的、可修改/扩展的、按优先顺序排列的任务列表。

在执行这一重要任务的过程中，团队的一项责任是与其他团队和个人合作，为我们的产品开发做出贡献。

#### [](#we-need-a-workflow-to-design-and-build-components)我们需要一个工作流来设计和构建组件

该系统的团队将通过简单明确的步骤——发现、**设计**、**构建**、**文档**和**发布**,来制作大多数特性——视觉风格、UI 组件、文档工具、UX 模式等等

已定义的工作流有助于团队相互理解范围、审阅者、工具以及每个步骤中“完成”的定义。有了工作流，协作变得紧密，交付变得更快，团队可以预见地交付特性。

[![Example workflow.](../Images/4ea201d0a68d896918b9eb20245b9de1.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--8_Pq-atZ--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/vtx1uniq0c2ev6jnurz8.png)

质量保证清单也是必不可少的。视觉质量、功能、兼容性是清单的一部分。

#### 我们需要一个发布策略

可能是四分之一，也可能是冲刺。

系统团队将交付一个可以在可预测的时间内采用的初始库。最初的工作是发布一个`1.0`，它提供了一个强大的视觉基础和 12 到 16 个 UI 组件。

这些组件将是设计师用来构建体验的工具。开发人员将做同样的事情，让这些部分准备好在他们的代码库中实现。

**在 1.0 之后，我们需要一个采用策略**

该系统的模块化特性让团队逐渐采用它。例如，我们可以确定哪个产品可以成为我们的 beta 测试者。或者开始在每个产品上逐步实现系统。

**我们需要等 1.0 才开始采用系统？**

随着快速的发布周期或持续的集成，使用花费在特性上的时间来逐步构建库听起来是正确的，基本上是在我们进行的时候。

我发现这种方法有一个问题。与基于特性的发布不同，我认为设计系统有责任成为一种已经建立了语法和词汇的语言，即使在早期的采用阶段。字母(组件)必须以无限有意义的方式(接口)组合，这样我们才能用它们构建有意义的句子(产品)。语言(系统)已经创建好，可以使用了。

### [](#conclusion)结论

越来越多的公司证明拥有/维护设计系统是一个好的选择。我认为设计系统将会是贵公司重要且无价的产品。它本身不会产生功能，但会为其他产品服务。是的，在理论上，它在开始时会有成本，但没有有形的回报。实际上，我从第一天就看到了好处:**设计和工程在同一个房间里**生产产品，这将使他们能够说一种共同的语言。
