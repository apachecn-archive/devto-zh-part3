# 年度首次会议:Git Merge 2019

> 原文:[https://dev . to/mindsers/first-conference-of-the-year-git-merge-2019-3202](https://dev.to/mindsers/first-conference-of-the-year-git-merge-2019-3202)

2019 年 1 月 31 日-2 月 1 日，我在布鲁塞尔参加了 **Git Merge 2019** 大会。如果你不知道那是什么，这是一个完全致力于 Git 及其工作方法的会议。门票销售的所有收益都捐给了[软件自由保护协会](https://sfconservancy.org/)。

因为你似乎喜欢我的#devlife 和#devtravel 帖子，所以我决定简单回顾一下我的比利时之旅。

*我为**欢迎来到丛林**写了一篇[所有会谈](https://www.welcometothejungle.co/articles/git-merge-2019-recap)(会议日，2 月 1 日)的详细帖子。去看看更深入的技术回顾吧，虽然没有这篇文章那么个人化。*

这篇文章也是为那些不是科技行业的人写的，他们想知道当我们聚集在一起开会时我们在谈论什么。例如，我想到了我的家人和朋友，他们对我来说根本就不是开发者。这些事情我们不一定要在社交媒体上谈论，因为这对我们来说太常见了，但我要带你一起去兜风。

# [](#the-journey)**旅程**

我乘火车去参加这个会议。这相当便宜又快捷。我坐的是里昂到布鲁塞尔的高铁。

如果回家是轻而易举的事，那么去布鲁塞尔就是一场真正的噩梦。我的火车晚点了(1 点 30 分)，当我在火车上的时候，他们决定不再去布鲁塞尔了。所以我不得不在法国北部停留，去赶另一趟同样晚点的火车...等等。

我很高兴我们是两个人，而且我不负责拿到我们租的 Airbnb 的钥匙。我也很高兴我决定提前一天坐火车，否则我会错过整个第一天的会议。我建议这么做，即使这意味着多付一个酒店/airbnb 晚上的钱，最终也是值得的。

## [](#the-talks)**会谈**

正如我之前所说，我已经写了一整篇关于这次会议的文章。你可以在**欢迎来到丛林**的网站上找到[这篇文章](https://www.welcometothejungle.co/articles/git-merge-2019-recap)。我链接的帖子有 2 月 1 日的每次演讲的摘要。

当然，我有几个最爱。以下是一个简短的列表，排名不分先后:

*   自由软件的未来来自 Deb Nicholson
*   基于 Git 的教育如何培养更有弹性的开发人员
*   *耐心的艺术:为什么你应该不厌其烦地教设计师 Git*
*   Johan Abildskov 的文章

我在总结帖子中没有明确的是，还有一天的活动是专门用于研讨会的。这些基本上都是带实际练习的讲座。

尽管我知道这些研讨会主要是针对初级/初学者的，但我还是签了名。我没有学到任何技术上的东西，但是我发现了一些使用这个工具的新方法和一些向别人解释 Git 的方法。我也看到了其他开发人员如何使用 Git，他们有什么技巧。

每个研讨会持续一小时。我们一直忙到下午 5:30，非常像一个学校日。

### [](#visualizing-git)**观想饭桶**

Katrina Uychaco (来自 Github)全力以赴开始了第一次研讨会。它非常容易访问，是为不了解 Git 的人设计的。她讲述了 Git 的基本命令，这是一个很好的基础，可以让大家在接下来的时间里保持一致。

[![](../Images/21d6a7b74a0a0e91fd96de035067fc29.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--eHtUvKB---/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://blog.nathanaelcherrier.com/content/images/2019/01/IMG_1720.jpg)

为了形象地描述她的话，她使用了[一个模仿 Git 工作方式的工具](https://git-school.github.io/visualizing-git/)。这个工具根据你输入的命令实时制作一个图形。

这是她工作室的伟大之处。因为就命令和可见的结果而言，我什么也没学到，但多亏了她的可视化工具，她展示并教授了通常不可见的东西。

### [](#git-legit)**饭桶合法**

[![](../Images/754401b2da60d8379b339ea142ae7350.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--wQzEgYUB--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/tac26hfaakw7xfqpejq2.jpg)

在这个研讨会上， [Pauline Vos](https://twitter.com/vanamerongen) 想要表明我们可能遇到的 Git 问题通常是因为我们没有很好地使用它。

她选择了一些开发人员在日常工作中遇到的问题。对于他们每个人，她都解释了为什么会遇到这个问题，以及如何通过了解更多和更好地使用 Git 来解决这个问题。

这个研讨会与下面的类似，但他基于 Pauline 作为开发人员的经历触及了这个主题。

### [](#ten-git-problems-amp-how-to-fix-them)**十个饭桶问题&如何解决它们**

Briana Swift 是 Github 的一名培训师。她处理所有类型的概要文件，但是经常有初学者使用 Git。

她得出结论，人们遇到的许多问题都是相似的。这是她工作室的主题。她列出了她的学生遇到的 10 个问题，并展示了如何解决这些问题。

这个工作坊与波林的类似，但它关注的是学生们最常问的问题。

### [](#git-workflows)**Git 工作流程**

我渴望参加这个研讨会！如您所知，Git 的工作流是我非常关心的一个话题。我总是对 Git 的可能性印象深刻，如果他使用适当的工作流，可以将开发人员的生产率提高 10%。

对于他展示的每个工作流程， [Johan Abildskov](https://twitter.com/RandomSort) 询问观众有无利弊。然后，他会列出自己的优缺点，并根据三个标准对工作流程进行评级:

*   工作流程简单吗？
*   它需要很多不同的工具吗？
*   用的时候有没有把脚放进嘴里的感觉？

这些是相当个人和主观的标准。如果你想知道他的选择:两个第一分别是 Github 流和 Gitflow。他更喜欢 Github 流，因为它要轻得多。

作为 devops，他解释说 Github 流比 Gitflow 更适合 CI/CD。然而，Gitflow 更适合使用 V cycle 或逐版本部署。

## [](#the-people-i-met)**我遇到的人**

会议不仅仅是在交谈、吃饭和享受会议的过程中学习东西。最后一点很重要:与其他开发者会面。

由于 Git Merge 是一个国际会议，我很高兴见到了来自几个国家的开发者，否则我可能永远也见不到他们。

例如，演讲者中有来自以色列的 Ben Greenberg，来自英国的 CB Bailey，来自 T4 的 Johan Abildskov，来自印度的 Aniket Subhash Kadam，来自荷兰的 Pauline Vos，来自德国的 Briana Swift，他们中的大多数人来自美国。

我还和两位谷歌开发人员共进午餐并聊天，他们也是 Git *核心*贡献者:非常鼓舞人心。我和 Gitlab 的工程师喝了杯啤酒。我还和 SourceGraph 的创始人和主要开发者聊了聊。

我真的很感谢这次会议，我希望你喜欢这个总结。
