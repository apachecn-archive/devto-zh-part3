# 如何给老板反馈？

> 原文:[https://dev . to/rlxdprogrammer/how-to-feedback-to-your-boss-4l3b](https://dev.to/rlxdprogrammer/how-to-give-feedback-to-your-boss-4l3b)

大多数开发人员会和他们的老板进行某种反馈讨论。在某些情况下，它是定期的(每月、每半年等)。)，其他情况下只有有目的才做。

很多程序员觉得这些会议基本上只是从老板那里得到下一次的命令。但不应该是那样的。这应该是一个双向反馈会议。

当然，你的老板对团队有一些长期目标，对你个人也有一些计划。这些会议通常是关于这些目标的:给你一些支持团队长期目标的个人目标，并在出现问题时给你反馈。这取决于你的老板，但有时这是你会议的全部内容，仅此而已。幸运的是，优秀的老板知道:没有优秀而热情的员工，团队的目标就不会实现。这就是为什么好的老板也会问员工他们的感受和反馈。但是对这些问题给出一个正确的答案并不容易。一些开发者只是说一切都很好，即使这完全不是真的。其他人开始抱怨一切。这些都不是很有效的策略。在第一种情况下，你的老板永远不会知道你的问题，所以他们很可能不会改变。在第二种情况下，在抱怨了很多之后，你的老板不会关心你的问题。

但是怎么做才正确呢？

首先，让我们为会议做好准备。想想自上次会议以来的时间段，收集所有你成功完成的事情:实现了里程碑、良好的客户反馈、实现了与老板一起定义的目标的一小部分、改变了上次负面反馈。想想那些不顺利的事情。如果你错过了什么，试着找出原因(你正忙于其他任务等等)。).除此之外，总是试着想想在最后一段时间里，什么对你是积极的，什么对你是消极的。

在会议上，总是让你的老板开始讨论，永远不要打断他/她的谈话，即使你强烈反对某事。总是等到直接问你。在会议的第一部分，总是尝试展示你的成果:总是从成功的故事开始，然后提及一些你失败的成就。总是试图告诉别人你尽力了，但不幸的是这还不够。比如:“不幸的是，我们交付晚了，我们在里程碑上努力工作，但是出现了一个棘手的技术问题，需要花很大的努力才能解决。最后我们找到了一个很好的解决方案，但不幸的是为时已晚，希望下次会更快。”。

在会议的下一阶段，试着把你的反馈告诉你的老板:总是提及一些你喜欢的积极的事情(好的团队、有趣的任务、灵活性等等。)之后你可以改变你的负面反馈。总是试着用积极的态度告诉他们，试着提出你的相关想法。例如:“不幸的是，我们需要经常加班，这是我不喜欢的。当然我能理解有时候需要。我认为，如果我们可以用一个新的开发人员来扩展我们的团队，情况会好得多。如果一个问题反复出现，你可以更严格一点，你可以说“这个问题已经存在几个月了，我已经告诉你很多次了，它让我失去了动力”。但重要的是:永远不要给你的老板惊喜，如果你是第一次提到一个问题，不要太咄咄逼人。

很有可能你的老板也给了你一些反馈，大多数情况下也有一些消极的部分。重要的是:即使你强烈反对，也要回答与负面反馈相关的问题。“你这一点到底是什么意思？”“我怎样才能做得更好？”等等。最后，告诉他们“好的，我会试着改变它”。那么你的老板应该对这种反应感到高兴，事实上在那之后他/她就说不出任何不利于你的话了。

在会议结束时，你很可能会谈论你的下一个目标。你的老板很可能有一些想法，或许你也有一些想法。让你的老板先陈述自己的想法。之后你可以这样回答:“好的，很清楚，我喜欢这个主意，但是我认为这样会更有效”。在这里你可以提出你自己的想法。注意:总要从老板的建议中保留一些部分，但总要带上自己的想法和一些好的理由。如果你在这一部分做得很好，那么你的老板会认为你同意他的想法，但事实上这主要是你的想法，这与他一开始的想法完全不同。

还有一件更重要的事情:永远试着记录这些会议中的每一件事，它会对以后有所帮助。

下一个要点:如果你的老板没有和你组织这样的会议，但是你有反馈，你可以随时要求你的老板进行反馈讨论。你的老板会认为你很有上进心，这也是你要求反馈的原因。但是在他反馈之后，你也可以自由地提出你的反馈。

最后，永远不要忘记:你的工作场所当然对你很重要，但你对你的工作场所也很重要。如果你不喜欢某样东西，总有办法去改变它。而每一个改变的第一步，都是给一个明确的反馈！
