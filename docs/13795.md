# 删除工作数据库(不！不要！不要！)

> 原文：<https://dev.to/saral/dropping-my-working-model-no-no-no-269>

今天应该是我使用 rails 的第一个项目的最后一天。我已经实现了博客的文章页面、用户认证和降价功能。我有一件事，我觉得我需要做的，添加一个用户编辑功能。为了这个。出于某种原因，我选择了使用`devise gem`。我认为这将是一个很好的实践与此宝石。到目前为止，我一直在和`bcrypt`一起工作。由于 devise 附带了所有与用户相关的功能，我想尝试一下。

我在我的 git repo 上创建了一个单独的分支，这样我就可以处理这个特性了。至少我认为我已经确定了这个过程。在不同的分支上工作会使我的工作代码安全。我知道无论我在哪个分支工作，数据库都是共享的。然而，我决定删除我的用户数据库，并重新开始设计。这是我犯的一个非常非常大的错误。

我开始使用 devise 构建整个用户特性。我认为这是一个直截了当的方法。但是，灾难降临了，我开始收到各种各样的错误。那时，我意识到，我应该为我的用户创建一个单独的数据库，我可以给它取别的名字。然而，我无法想出一个数据库名称，这意味着我放弃了我的工作模型。我现在想回去使用 bcrypt，至少我的应用程序可以工作。然后我可以按计划把它部署到 heroku。

因此，我没有添加新的功能，而是在修复我制作的 howler。不过，最终我还是修复了它，并让这个应用程序在 heroku 上运行。

我将使用 devise 再次实现该应用程序，但这一次，我可以创建一个新的模型，而不用接触我的工作数据库。
