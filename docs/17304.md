# 从创业发展中学到的经验

> 原文：<https://dev.to/koolamusic/lessons-learned-from-developing-for-a-startup-4503>

尽管我们喜欢迭代地开发产品和服务，但事实是组织会考虑严格的期限和预算。我们产品负责人需要考虑如何管理这个问题。

浏览器定期更新，进行频繁的，(大部分)小的改变。这对用户造成的干扰很小。你很少会下载一个大的更新，然后发现一切都完全不同了。这就是我们希望在未来几年内开发软件的方式，尤其是嵌入式系统。

但是，假设你有一个有限的预算，因此，一个固定的发射日期-就像上图中的船。人们的期望是，产品会准备好，即使撞到冰山也不会沉。

如果你的最佳反馈来自真实的使用，你可以计划一个测试阶段，就像我们在开发图书馆自助服务软件时所做的那样。这是有用的，但是在一个地区的不同机构内的 20 多个图书馆中安装 kioskss 和在 4 个小型图书馆中有控制地部署 kiosk 之间有很大的不同，在这些机构中，员工对新产品的投资较少，并且需要理解才能利用该产品的优点。

我们在发布期间发现了用户问题，我本想更快地解决这些问题，但是我们遇到了大的发布日期诅咒:没有更多的项目资金。

因此，得到的教训很简单:

如果你有一个正式的发布日期，确保在这个日期之后有一些项目预算可以做进一步的改变。

注意:重要的是这笔钱是为项目预留的。如果你的产品不完美，那么预算不足没有什么价值(除了一些暂时的良好公关，这些公关会在你职业生涯的后期反噬你)——而且永远不会完美。
