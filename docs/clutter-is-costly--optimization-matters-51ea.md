这是我的博客[mokacoding.com](https://www.mokacoding.com/blog/clutter-and-optimization/)的一篇交叉文章。

在他的书[中，数字极简主义](https://geni.us/7HNOJAz)卡尔·纽波特定义了一种新的*技术哲学*，旨在通过与人们使用的应用和设备建立更好的关系来帮助人们过上更好的生活。数字极简主义的两个原则是杂乱是昂贵的，优化是重要的。这些想法也转化为软件开发，应用它们可以给我们的日常工作带来显著的改进。

## 杂乱是昂贵的

卡尔·纽波特认为，盲目地注册每一项新服务和使用每一个新应用程序，会让你的数字生活变得杂乱无章。你使用的大量技术冲淡了少数对你真正有价值的技术所能提供的好处。混乱在软件开发中也是昂贵的，你应该尽可能地避免它。

努力保持你的函数简短，你的类型和对象只专注于做一件事。考虑阅读和理解代码所需的精神负担。越混乱，代码就越难解析，这使得处理起来越困难，越慢。对于你写的所有东西，考虑它的*本地推理*成本，理想情况下，当你看着代码，就在你面前，你不应该像[苹果工程师所说的那样“不得不考虑你的代码的其余部分如何与那个函数交互”](https://developer.apple.com/videos/play/wwdc2016/419/)

每个第三方依赖都伴随着保持更新的隐藏成本，以及引入 bug 的可能性。不要误解我，库和框架通常是构建好软件的基础。毕竟，你应该专注于解决你的领域问题，而不是重新发明轮子。我只是想让你意识到，当你把完成某些事情的责任推卸给你无法控制的代码时，你正在做出的权衡。还记得 [`left-pad`事件](https://blog.npmjs.org/post/141577284765/kik-left-pad-and-npm)吗，当一个模块从 npm 中移除时，破坏了 JavaScript 生态系统中的大多数框架。

关注依赖性不应该局限于第三方代码。总是试图控制单个对象所依赖的领域概念的数量。杂乱会降低软件的灵活性，比如你想在一个专用的模块中提取你的部分逻辑，这样你就可以在隔离状态下工作，并且有更快的反馈周期。如果你的代码没有被很好地隔离，你将有大量的工作要做，经常会带来与你的原始代码无关的领域概念。

与产品和代码中的混乱作斗争。试图做太多事情，拥有所有功能，取悦所有用户的应用程序，最终变得臃肿而没有吸引力。不要害怕做出取舍，只要是有意的。每当着手构建一个新特性时，要确保它对用户的价值是明确的。质疑功能并不是一个懒惰的开发者的标志，而是一个有思想的、致力于产品成功的开发者的标志。询问变革背后的假设是否得到了验证。建议先在办公室里的人身上用一个模型或纸原型来试一试。人们会为解决实际问题的产品付费，关注这一点。

无情地删除不用的代码。养成自己打扫卫生的习惯。未使用的代码和过时的注释是代码库中最混乱的部分，它们使得阅读、构建和使用代码更加困难。如果管理层不接受在发布功能后花时间清理的想法，向他们解释这是卫生问题。就像你不会在做完饭后把厨房弄脏，或者在客人到来之前把房子弄得乱七八糟，所以代码库应该总是保持整洁，以免腐烂。

## 优化很重要

数字极简主义者优化他们与技术的互动，以充分利用它们，经常超越默认的使用模式。例如，他们可能会订阅一些策划好的时事通讯，将他们关心的文章存储在口袋或 Instapaper 中，以获得一个注重可读性的无广告版本，并在周末喝咖啡时阅读这些文章，而不是使用脸书来了解最新消息。

就像数字极简主义者和他们的服务一样，我们软件开发人员不应该仅仅满足于完成工作，而应该总是寻求充分利用我们的工具。

优化你使用 IDE 或文本编辑器的方式将会消除你的想法和它们在屏幕上的具体化之间的摩擦，这会节省你的时间和精力。查看菜单以发现新功能，并学习键盘快捷键- [如果您使用 Xcode](http://xcodetips.com/) ，请查看。大多数工具支持插件，浏览它们的目录，你会发现各种有用的东西来改善你的工作流程。

ide 和文本编辑器不是软件开发人员的唯一工具。熟悉你所使用的编程语言的标准库，以及基于它们的框架文档。永远不要假设你做事的方式是最好的。你不知道什么是你不知道的，那就花时间深入挖掘吧。增加您对任何给定语言或框架的知识的一个好方法是阅读您还没有使用过的每个类或方法的文档。无可否认，这种做法在开始时很耗时，但是随着时间的推移，你会逐渐熟悉它，并且需要减少这样做的次数。一路上你会学到很多东西。有了开源库，你还可以看看他们是如何实现的，从而学习技巧和最佳实践。

目标是掌握版本控制。学习使用 Git 将使你在增量阶段编写软件变得更加容易，而不必担心错过正在进行的工作。因为我们在代码上进行协作，所以通过小的提交和拉取请求可以使协作变得更容易，这将优化您获得工作反馈并最终合并反馈的速度。

调整您的编译和测试阶段，使它们尽可能快。您希望测试和编译器提供的反馈及时。盯着进度条对任何人的自控力都是一个挑战，让他们专注于手头的任务，而不是去社交媒体。所以不要让你自己更难做到最好，不要让你的构建和测试变得缓慢。

* * *

从数字极简主义剧本中吸取经验。减少混乱以避免付出精神上的代价。优化您使用工具和工作流程的方式，以便快速获得反馈，更快地完成工作。少即是多，目标是更好，而不是更大。
