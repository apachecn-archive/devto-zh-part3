# 代码质量被高估了

> 原文:[https://dev.to/ross/code-quality-is-overrated-3bnn](https://dev.to/ross/code-quality-is-overrated-3bnn)

如果你像我一样，你会非常担心你写的代码是否“足够好”。

在过去的一年里，我一直在开发一款个性化的指南软件 [Gush](https://gush.com) ，这是一款关于在纽约要做的事情的指南，目前正处于私人测试阶段。我经常问这样的问题:“这是适当的抽象吗？”，“这有足够的文档吗？”，“这个好重用吗？”，“从长期来看，这是可维护的吗？”，或者“有没有更聪明的方式来写这个？”。毫无疑问，回答这些问题是一项有价值的工作。

但是 Gush 的未来是不确定的。如果我们不能获得用户的关注，代码质量并不重要。我不得不学会比我希望的更早地停止改进我的代码，因为用户并不关心代码有多好。对于像我这样的情况，我认为“代码质量”需要一个稍微不同的定义。

代码质量的第一个衡量标准应该是“它能工作吗？”。下一个最重要的质量衡量标准是“它是否为我们的用户提供了优秀的体验？”。在你证明你的产品有吸引力之前，所有其他的质量指标都是不可或缺的。

在代码质量上花费时间是一种特权——必须通过构建人们喜爱的产品来获得。

我并不提倡草率的代码。我关心代码质量，并尽我所能避免积累技术债务。但是对于早期阶段的项目，你必须找到一个平衡，而这个平衡并不总是美好的。
