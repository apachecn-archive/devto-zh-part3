# 我们安全事件管理流程的幕后

> 原文:[https://dev . to/atlassian/我们的安全事件管理流程的幕后-3pb6](https://dev.to/atlassian/behind-the-scenes-of-our-security-incident-management-process-3pb6)

在安全团队中，我们不像其他 Atlassian 团队那样管理任何 Atlassian 产品。我们的主要产品是信任，这是一项永无止境的工作。

对我来说，安全感更多的是一种心态；一个不断努力，不断改进，并寻求创新的方法。

有时候，安全团队更像是一个阻挡者，而不是推动者，用繁琐的关卡检查阻碍产品。但是我的团队努力协助安全软件的创建和管理，确保团队能够协作，以尽可能安全的方式完成工作。当伟大的产品以安全可靠的方式运送时，我们知道我们已经完成了我们的工作。为此，有三项原则指导我们的工作，并为我们的行动计划和应对安全事件提供信息:

## [](#our-guiding-principles)我们的指导原则

### [](#1-be-open-and-available)1。保持开放和开放

没有人会从在暗处工作或不共享信息的安全团队中受益。我们知道，如果我们想参与安全软件的开发，我们必须平易近人，可以为每个人服务:工程人员、支持人员、合作伙伴和客户。我们鼓励任何有问题的人 [<u>向我们</u>](https://getsupport.atlassian.com/secure/Dashboard.jspa) 报告，无论问题的严重性或他们的角色如何。

**[![](../Images/53c6e97cb98647cf0c7fd9fe65202d04.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--ehKblFcn--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://atlassianblog.wpengine.com/wp-content/uploads/2019/02/simple-workflow%402x-e1550024154650.png)T4】**

### [](#2-be-consistent-in-word-and-action)2。言行一致

如果我们对待工作的方式不一致，那么对人们开放就没有多大意义。为了维护和执行政策和程序，我们都需要保持一致，并以可预测的方式行动。我们记录我们的工作方式，然后通过我们的内部协作工具与我们更广泛的客户和合作伙伴社区共享这些信息。我们还在我们的公共 [<u>信任网站</u>](https://www.atlassian.com/trust) 上发布安全统计、数据、政策和程序。

**[![](../Images/4d2de8d229649e69936c3bf6ed589212.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--Vn4Vg-FF--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://atlassianblog.wpengine.com/wp-content/uploads/2019/02/simplify%402x-e1550024220891.png)T4】**

### [](#3-always-seek-better-ways)3。总是寻求更好的方法

如果我们总是错的话，一致性对任何人都没有好处。因此，我们一直在努力改进我们的监控、我们的工具、我们的程序，并领先于潜在的风险。我们有一个团队，其主要目的是研究野外的漏洞，然后想出加强我们的系统抵御它们的方法，这样我们就不会因为一直处于反应模式而失去优势。

## [](#meet-our-team)认识我们的团队

安全性是我们所有产品和流程的一部分，在公司和社区范围内共享。Atlassian 有三个主要小组积极致力于预防和应对安全事故:

**安全工程。**这些人审查代码并研究我们产品的安全性，确保他们及时发现漏洞并解决它们，最好是在产品发布之前！我们希望我们所有的产品从一开始就是安全的。通常，产品安全工程团队的成员会参与对安全事件的响应，以便他们在补救漏洞时拥有所需的上下文。

**安全情报。**该团队积极寻找整个网络上正在发生的可疑事情，或*可能会发生的事情*。他们回应来自客户和合作伙伴的报告，以及与他们合作的内部团队的报告。这是一个积极保护我们的产品和系统免受漏洞攻击并直接响应事件的团队。

**政策和信任。**这个团队负责传达我们的安全规则和政策，并将它们发布到我上面提到的 [<u>信任网站</u>](https://www.atlassian.com/trust) 。他们发布的信息旨在对更广泛的安全和开发社区有用，而不仅仅是 Atlassian 的客户。再一次，回到我们的三个指导原则，我们希望让每个人都可以获得信息，并打破通常围绕安全讨论的障碍。

## [](#how-we-build-our-stack)我们如何构建堆栈

人们经常问我们在我们自己的堆栈中使用了什么，特别是当它涉及到安全性时。我们混合使用我们自己的产品，并集成了一些其他同类最佳的工具。能够在整个组织内共享信息，以及与合作伙伴和客户共享信息，是我们的首要任务，因此我们自然要确保我们的堆栈为此目的而协调一致。

这里列出了我们使用的产品以及使用方法:

* *全面的检测和分析:** 我们使用 [Splunk](https://www.splunk.com/) 来查询我们的系统，并根据我们的安全情报团队编写的策略使用启发式分析和异常检测。谈到风险，我们希望涵盖所有的基础，所以我们根据历史和理论事件制定政策。

**敏捷沟通循环:**如果我们发现需要快速解决的问题，我们会立即处理，然后记录发生的事情。但是对于更复杂的事情，我们首先在[吉拉](https://www.atlassian.com/software/jira)记录事件，然后也在 Slack 中记录，以便进行更广泛的交流。

**关联对话:**有时，警报通过[吉拉服务台](https://www.atlassian.com/software/jira/service-desk)从客户或合作伙伴处传来，然后传到 [Opsgenie](https://www.opsgenie.com/) 以提醒相关人员。这些警报也会发送到吉拉，然后发送到 Slack，因此透明度很高，每个人都可以看到。

* *知识获取和转移:** 我们的许多行动手册都存储在 [Confluence](https://www.atlassian.com/software/confluence) 中，如果我们需要使用其中任何一本作为回应指南，我们会在吉拉入场券中引用它们。从那里，如果对话也发生在电子邮件中，这些信息也会被记录在吉拉。如果事件在吉拉得到更新，我们将在 Slack 中看到。我们创造了一种非常顺畅的方式，让人们可以通过对他们来说最有意义的工具提供输入，我们不必担心人们被留在沟通的筒仓中。每个人都可以接触到任何东西。

## [](#how-we-respond)我们如何应对

我们遵循行业最佳实践，以预定义的方式来响应事件，安全情报团队花费大量时间来详细说明这些流程，并培训人们如何遵循这些流程。我们这样做是为了避免反应太快而做出错误的修复，而是真正花时间调查事件，并确保我们就补救方法达成一致。

1.  * *检测分析。** 正如我之前所说，这是安全情报团队花费大量时间关注的事情。我们编写查询，寻找某些易受攻击的服务，并衡量任何问题的严重性，以便我们可以确定我们的响应。
2.  **调查。**一旦发现问题，并且我们对问题的性质有所了解，我们将进行调查以确定问题的严重性和紧迫性。为此，我们使用 VERIS 社区的安全分类系统。这有助于我们确保有效利用我们的资源，而不是对事件反应过度或反应不足。
3.  * *遏制并根除。** 事实上，在某件事被确定为威胁之前，我们不称之为事件。但是，一旦我们做出决定，并对其进行分类，我们就会调用安全情报团队花费大量时间创建的计划响应。我们找出谁易受攻击，构建补丁，与产品团队合作准备补丁，并确保一切就绪。
4.  * *沟通。** 一旦我们准备好要部署的补丁，我们就会与营销部门(有时是法律部门)合作，以确保我们与客户的沟通是及时和清晰的，并且每个人都知道如何安装代码补丁。这项工作的大部分是在 Confluence 中完成的，我们可以在这里审查草案，进行评论和编辑，并确保公告坚如磐石。
5.  * *事件后审查。** 在我们确信问题已经解决，每个人都有了他们需要的东西之后，我们会进行事后审查，称为 PIR，并在吉拉对此进行跟踪。这通常是我们分配给自己的任务的集合，以处理我们需要采取的任何行动，比如对我们的响应过程进行任何调整，或者任何需要一些新培训的人，我们为这些任务分配最后期限。我们在事件发生的第一周内这样做，那时我们头脑中一切都很清晰。部署修补程序后，这是确保我们的产品和系统安全的最佳方式之一。

我们已经在网站的“信任”部分发布了更多关于我们团队如何应对事件的详细信息。

正如您所看到的，Atlassian 的每个人都非常重视安全性，我们投入了大量精力来确保我们有专门的团队来构建安全的软件和维护我们的系统，以尽可能确保它们的安全性。我们认为，专注于三个指导价值——开放和可用、一致和总是寻求更好的方法——是与我们的社区建立信任的一个很好的方法。
