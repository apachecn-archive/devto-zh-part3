# 如何教成年人编码

> 原文:[https://dev . to/Nielsen Jared/how-to-teach-adults-to-code-4k8h](https://dev.to/nielsenjared/how-to-teach-adults-to-code-4k8h)

*本帖最初发表于[jarednielsen.com](http://jarednielsen.com/teach-adults-code/)。这里的观点是我自己的，不一定代表我的雇主的观点。*

我的第一份全职工作是在犹他州的沙漠里给被裁决的年轻人讲授野外生存。虽然这些技术很原始，但我学到了一项贯穿我整个职业生涯的基本技能:将复杂的过程分解成简单步骤的能力。要成为一个终身学习者，我相信解决问题是任何人都能获得的最重要的技能。用 bow and drill 点火与处理异步 HTTP 请求没有太大区别。我现在在一所著名的大学训练营教成年人编程。在从 web 开发和数据可视化的六个团队毕业后，我发现并实施了许多方法，通过培养解决问题的心态来确保学生的成功。我在这里特别强调如何教成年人编程，但是基本概念可以很容易地应用于任何知识领域。

## [](#programming-is-problemsolving)编程就是解题

> “孩子们以求知欲强的学习者开始他们的生活。他们必须学会学习有困难……”

以上引文摘自头脑风暴的开篇段落之一，由[西蒙·派珀特](http://www.papert.org/)于 1980 年出版。头脑风暴的大部分内容探索了 [LOGO](https://en.wikipedia.org/wiki/Logo_(programming_language)) 的历史和哲学，这是 Papert 和他的同事在麻省理工学院开发的向孩子们介绍计算机科学的编程环境。LOGO 的激励因素之一是创建了一个沙箱，在这个沙箱中，儿童可以使用计算机作为中介和界面，通过反复试验和即时视觉反馈来发展解决问题和批判性思维的技能。

在 *Mindstorms* 中，Papert 定义并描述了[数学恐惧症](https://en.wikipedia.org/wiki/Mathematical_anxiety)的现象，在这种现象中，学生对数学产生厌恶，通常是由于强调(正确)结果而不是过程的糟糕教学。这种社会文化现象将孩子分为两类:“聪明人”和“哑巴”。孩子在某个领域的第一次不成功或不舒服的学习经历会让他们给自己贴上“T4 x T5”的标签。“孩子们会根据自己的局限性来定义自己，这种定义会在他们的一生中得到巩固和加强。”(Papert)

在 *[心态](https://en.wikipedia.org/wiki/Carol_Dweck#Mindset_work)*[卡罗尔·德韦克](https://profiles.stanford.edu/carol-dweck)勾勒出两种不同的学习者心态，*固定*和*成长*。心态固定的人认为智力是静态的，与生俱来的，不变的品质，而心态增长的人认为智力是一种可以随着时间发展和改变的品质。我们可以很容易地看到这些二元思维模式是如何映射到上面 Papert 描述的现象的。下表摘自*心态*的总结章节，快速总结了两者的区别。需要注意的是，这些类别并不是真正的二进制。一个人可能在生活的一个领域有一个固定的思维模式，但在另一个领域有所发展，或者可能根据情境在两者之间摇摆不定。

| 固定思维模式 | 成长心态 |
| --- | --- |
| 认为智力是静态的... | 相信智力是可以开发的... |
| ...这导致了想要看起来聪明的愿望，因此倾向于... | ...这导致了学习的欲望，因此倾向于... |
| …避免挑战 | …迎接挑战 |
| …变得有防御性或轻易放弃 | …面对挫折坚持下去 |
| …认为努力是徒劳的 | …将努力视为通往成功的道路 |
| …忽略有用的负面反馈 | …从批评中学习 |
| …对他人的成功感到威胁 | …从他人的成功中寻找教训和灵感 |

我们不仅仅是教成年人编码。我们在教授解决问题的方法。这种区别尤其重要，因为作为初学者，很容易变得沮丧和放弃。从根本上说，编程语言是逻辑和语法的结合。我们需要强调编程的逻辑方面，而不是语法，教会学生只见树木不见森林。当发现问题的解决方案仅仅是一个丢失的逗号时，我的最刻板、固定思维的学生严厉斥责自己，这是一个即使是专业人士也会犯的常见错误。非常重要的一点是，我们不允许学生纠结于语法，并保持对所有部分如何组合在一起的全局观点。

我们如何教会学生首先成为问题解决者，其次才是程序员？我喜欢从共同点开始:语言。

## 我的语言的极限意味着他们世界的极限

> “这不是一个‘这可以沟通吗？’。相反，这是一个‘我们如何提高沟通能力’的问题？"
> 
> 西蒙·派珀特

对大多数学生来说，最大的挑战之一是从自然语言过渡到正式语言，或者学习像计算机一样思考。为什么不从日常生活中学习程序性思维？给出指示或遵循食谱是程序性思维的练习，“但是在日常生活中，程序是活的和使用的，它们不一定被反映出来。”(Papert)

这可能也是教成年人编程的导师面临的最大挑战。当我们从自然语言过渡到正式语言时，精确的语言变得非常必要。在编程中没有含糊不清的余地，也没有错误的余地。教师的责任是找到并使用能以最清晰的方式解释概念的语言。由此，学生将学会模仿机械思维，并在被要求回答技术问题时掌握描述性的形式。

一个人如何学习这个过程？向一个五岁的孩子解释这个概念。如果不能，那就精炼你的语言，再试一次。这是由[理查德·费曼](https://en.wikipedia.org/wiki/Richard_Feynman)开发的一个思维实验(因为不是每个人手头都有一个五岁的孩子)。用清晰的、描述性的语言，复杂的概念，甚至理论物理，都可以解释给任何人听。

> “成为优秀学习者的一个重要部分是学习如何拓展我们能用语言表达的领域。”
> 
> 西蒙·派珀特

比我们教授的语法更重要的是我们用来描述它的词语。在 *[思考，快与慢](https://en.wikipedia.org/wiki/Thinking,_Fast_and_Slow)* ，[丹尼尔·卡内曼](https://en.wikipedia.org/wiki/Daniel_Kahneman)中描述了一个由约翰·巴格在 NYU 进行的实验，在该实验中，学生们被要求从五个单词中组合出四个单词的句子。实验组的学生拿到了与老年人相关的单词集，如“佛罗里达、健忘、秃头、灰色或皱纹”。完成单词集后，学生们被送到隔壁房间进行下一项任务。真正的考验是从第一个房间走到第二个房间。研究人员发现，实验组的学生，即那些研究与老年人相关的单词组的学生，比控制组的同龄人走得慢得多。

这是*灌注*。接触特定的词汇会对我们的行为产生显著的影响。这和编程有什么关系？我用“有挑战性的”、“要求高的”和“有趣的”，而不是“困难的”、“艰难的”或“无聊的”。编程就是解决问题。问题从来都不难:它们具有挑战性；有些人比其他人更喜欢。正是通过接受挑战，我们才能学习和掌握。当一个学生对我说，“这很难”时，我会探究为什么。答案最常见的是“我不擅长这个”。这就成了关于目标和成长的对话的起点。

## [](#teach-solving-not-solutions)教人解，不教人解

> “没有人指望第一次就能成功。一个人不能用“对-你得到好成绩”和“错-你得到坏成绩”这样的标准来判断。相反，人们会问这样一个问题:“我该如何解决？”为了解决这个问题，我们首先要理解在它自己的术语中发生了什么。"
> 
> 西蒙·派珀特

在*头脑风暴*中，Papert 引入了*微观世界*的概念。在编程中，微观世界是一个自包含的环境，学习者可以在其中建模和测试假设或直觉。每个活动，每个家庭作业，每个项目都是一个微观世界。微观世界的边界同时由语言和学习者使用语言的方式来定义。当学习一门编程语言时，我们会有关于它应该如何工作的直觉，但我们可能会看到它不工作的证据(bug！).然后，我们被迫面对和评估我们直觉的局限性。

Papert 认为计算机以两种方式帮助学习者面对他们的直觉:

*   计算机允许或迫使(学习者)将直觉预期具体化。当直觉被转化为程序时，它变得更加突出，也更容易被思考。
*   计算思想可以作为重塑直觉知识的材料。

编程长期成功的关键在于这种双重的反思和重塑过程，或者说，学习如何思考。最终，每个学生需要的是更好地了解自己，但对每个人来说，获得 *meta* 并不容易，尤其是那些思维固定的人。为什么他们认为这种方法可行或不可行？为什么他们认为它有或没有？这种情况对他们的假设和直觉揭示了什么？他们的偏见？

> “关键的技巧是问你自己为什么会犯一个特定的错误，而不是仅仅纠正错误然后继续前进。”
> 
> -V. Spraul

错误是“学习过程中固有的一部分”，不是应该避免的。通过调试，我们最了解自己。我们的目标是教成年人如何调试自己，并在这个过程中编写一些代码。我们如何做到这一点？

在经典的 *[如何解决](https://en.wikipedia.org/wiki/How_to_Solve_It)* ，中，乔治·波利亚概述了一种在数学背景下促进问题解决的方法论。这些试探法有两个目的:

*   帮助学生解决手头的问题。
*   培养学生自己解决未来问题的能力。

Polya 的启发法可以提炼为四个纵坐标类别，每个类别都有具体的问题要问学生:

1.  理解问题
    *   有什么问题？目标是什么？让学生尽可能描述问题或错误。控制台或终端在告诉他们什么？如果没有反馈，为什么会这样，我们能做些什么？
2.  设计一个计划
    *   我们以前见过这个问题或者类似的问题吗？如果我们之前没见过这个问题，可以用我们熟悉的术语重述一下吗？我们能把问题分解成更小的、熟悉的、容易理解的问题吗？
3.  执行计划
4.  检查解决方案
    *   解决方法是什么？请学生解释一下。可以修改吗？如果是这样的话，有哪些可以改进的地方？我们可以用这个解决方案来解决其他问题吗？

开始时，这些步骤可能非常耗时，但在课程结束时，学生们已经习惯了这种方法，他们只会在最棘手的问题上寻求帮助。

在 *[像程序员一样思考](http://www.vantonspraul.com/TLAP)* ，[诉安东·斯普劳](http://vantonspraul.com/)同样地概述了同样的过程:

1.  永远要有计划
2.  重述问题
3.  把问题分开
4.  从你知道的开始
5.  减少问题
6.  寻找类似物
7.  实验
8.  不要沮丧

最后一点非常重要。学会如何冷静自信地调试应用程序不是一朝一夕的事。初学者，尤其是那些有固定思维模式的人，当面对应用程序中断错误和时间紧迫时，往往会抓狂，而不是阅读日志并仔细思考，迷失在 StackOverflow 的黑洞中，并抛出他们可以找到的所有复制/粘贴的解决方案，使事情变得更糟，更难纠正。

我们如何教成年人平静而自信地编码？

> 试图解决问题时，你必须观察和模仿别人解决问题时的做法，最后，你通过实践学会解决问题
> 
> 乔治·波利亚

尽可能多地对活动使用现场代码解决方案。询问学生的尝试，即使不正确或未完成，也要在全班面前调试。如果你的学生给了你可行的解决方案，重构并修改它们，证明找到解决方案的方法不止一种。波利亚写道:

> 一个好的老师应该理解并让他的学生记住这样一个观点:没有什么问题是完全没有的。总是有事情要做；通过充分的研究和渗透，我们可以改进任何解决方案，而且在任何情况下，我们都可以提高对解决方案的理解。”

我们如何保持在解决问题上的投资？我们如何不仅让学生参与进来，也让我们自己参与进来？

## [](#put-the-fun-in-functions)把乐趣放在功能上

编程很有趣。如果你不同意，你可能进入了错误的领域(或者你可能有错误的思维模式)。这就像玩一个游戏，但在你玩任何游戏之前，你需要学习规则。从[维基百科对游戏](https://en.wikipedia.org/wiki/Play_(activity))的定义中，我们读到:

> “[游戏]按照固定的规则和有序的方式在自己的时间和空间的适当界限内进行。”

听起来很像编程，不是吗？再加上 Papert 的以下引用，我们开始看到一个将乐趣融入函数的公式:

> “这个过程让人想起修修补补:学习包括建立一套人们可以处理和操作的材料和工具。也许最重要的是，这是一个利用现有资源的过程

我们如何创造一个强调过程而不是产品的学习环境，或者，如果你愿意的话，是玩耍？

*   将新的概念与学生已经知道的联系起来
*   要求学生接受新概念，并将其变成自己的概念

设计活动时，学生必须知道规则，并配备找到解决方案的工具。解决方案不应该是一个尚未引入的概念，而应该是(如果不总是)一个以前学习过的概念的集合。开放式的任务将鼓励我们努力投入，因为当我们着手解决自己创造的问题时，我们学到的东西最多。

如果你不开心，他们也不开心。

## [](#how-to-teach-adults-to-code-including-yourself)如何教成年人编码(包括自己)

> "人们认识到所有想法中最强有力的想法是强有力的想法的想法."
> 
> 西蒙·派珀特

教成年人编程的好处是双重的。首先，全球:我们生活在一个矛盾的时代。每天，我们都会看到关于“成年人”抵制变化，但却完全接受促进他们生活变化的技术的新闻。人类历史的每个时代都有悖论的特征吗？如果选择是“ *[编程还是被编程](https://rushkoff.com/books/program-or-be-programmed/)* ，答案显然是前者，不是因为某种经济必然性，而是因为一个人通过编程获得的首要技能是解决问题。解决问题的心态不仅能让一个人具备批判性地思考特定问题的能力，还能让他批判性地思考自己，将变化视为机遇而非阻碍。在这方面，我们作为教师可以成为社会变革的推动者。第二，本土:我们作为导师也是终身学习者。如果你想成为一个更好的程序员，教成年人编程。除了巩固你对基本概念的理解之外，它还将教会你递归地应用反思/重塑启发法，这不仅会成倍地提高你解决问题的技能，而且，也许更重要的是，让你具备清晰有效地交流任何概念的能力。
