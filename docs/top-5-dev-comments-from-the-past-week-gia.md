[![phlash profile image](img/a2df406b96aafece7e3457e240cb8d91.png) ](/phlash) [ Phil Ashby ](/phlash) • [<time datetime="2019-03-22T22:01:44Z" class="date-short-year"> Mar 22 '19 </time> • Edited on <time datetime="2019-03-25T07:56:34Z" class="hidden m:inline-block date-no-year">Mar 25</time>](https://dev.to/phlash/comment/9ib0) 

我来咬:)

我拥有所有这些头衔，这对我的工作几乎没有影响，但对我的印象却大不相同。

与供应商或客户的第一次接触:设定对权威的期望是有用的，因此我是一名架构师——他们感觉受到重视，我们可以更快地一起工作(或达成一致的分离)。

对于将与我一起工作的团队成员，我介绍自己是对糟糕的决策负责的傻瓜，减轻他们的压力，暗示好的决策是由他人做出的，并展示在工作环境中生存所需的幽默:)随着时间的推移，他们可以自己决定我的实际技能！

被称为“编码员”会冒犯我吗？不，恰恰相反:这意味着有人认为我实际上可以编写代码，并且会理解我在看什么(可能是他们的努力，可能是我最近的 PoC 乱七八糟)，是一个极客，而不是一个在某个象牙塔里的怪异的不食人间烟火的高尔夫球场决策“建筑师”。

就我所做的而言:架构是团队中的一个角色，与其他角色同等重要，有不同的关注点和结果(大的目标，与涉众的沟通，等等。)这是我的 T 型专长，其他人享受他们的专长，我们一起把事情做完。如果我们做得对，那么指导就会发生，其他人会从我这里学到一些系统和商业规模模式，而我也会从他们那里学到一些东西。

[![chrismcleod profile image](img/6a0eb12853fdc48f36b6cd500d20310d.png) ](/chrismcleod) [ Chris McLeod ](/chrismcleod) • [<time datetime="2019-03-25T00:19:25Z" class="date-short-year"> Mar 25 '19 </time>](https://dev.to/chrismcleod/comment/9j7f) 

架构师**选择**最适合工作的工具、框架、范例。画蓝图的人已经选好了房子的形状和材料。

工程师**使用软件/算法原理解决**现实世界的问题，并与利益相关者协调。例如:工头使用一份蓝图，但决定事情的完成顺序，组装房屋组件的最佳实践，指导实施等。

开发者**实现**解决方案；他们通过修复错误、重构等方式开发软件。普通工人在建造房屋的时候钉钉子，在改造房屋的时候拆墙。

所有这些都不是头衔，而是一个人在任何给定时间都可以工作的运作模式。每个人对每项重要任务的熟练程度都不一样，所以在给头衔命名的时候，可能更喜欢一个特定的头衔。

[![gabrielecimato profile image](img/6718b38ffd9f92a079994cab3ca10b10.png) ](/gabrielecimato) [ Gabriele Cimato ](/gabrielecimato) • [<time datetime="2019-03-21T05:17:42Z" class="date-short-year"> Mar 21 '19 </time>](https://dev.to/gabrielecimato/comment/9h5p) 

老实说，当我不久前开始时，我也在同一条船上。我不再担心寻找一个我能为之做出贡献的开源项目，因为它感觉很有力量。相反，我继续学习和练习。

现在，在我使用了库、节点包、框架之后，我为我正在从事的任何项目(无论是个人的还是工作需要的)制作 PRs。有时是文档上的一个打字错误，有时是在文档上添加更多的例子，最终是一个新功能或一个 bugfix。正如其他人提到的，如果你真的想深入了解的话，有些项目对初学者来说是友好的。

但是我发现更“自然”的是做一个项目...变得沮丧！然后，我处理这种挫败感，希望防止其他人陷入同样的陷阱。我通常用公关的方式来称呼它！

所以继续努力，试着建立一些东西。如果你用的是图书馆，而文件令人困惑，提交一份 PR！你发现一个错别字，提交一个公关！你发现了一个 bug，尽你最大的努力去修复它，如果你做不到，就开一期！
