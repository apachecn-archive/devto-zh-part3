# 停下来，让我们在标记该功能之前谈谈

> 原文：<https://dev.to/jessekphillips/stop-lets-talk-before-you-flag-that-feature-1872>

特征标志正在巡视，我想介入。

当你阅读链接的文章时，它有一个坏的部分，不能使用功能切换不是一个缺点。你还会看到它对 QA 很有帮助，因为你可以将 QA 分解成特性测试人员。

让我告诉你特性标志的实际成本。

*   您正在向代码中添加非生产逻辑
*   QA 现在有更多的软件版本需要测试

我认为第一点很清楚，更多的代码意味着更多的漏洞，修改现有的代码来提供切换，这本来不需要修改，否则会增加变化，变化等同于增加风险。

QA 不需要更多的变量来控制。有一个[算法](https://dlang.org/phobos/std_algorithm_setops.html#.Cartesian%20Product)可以帮助你计算所有新的测试场景。这是最精彩的部分，我们不打算全部测试。

功能标志可以工作，而且它们可以很好地工作。但是在它有机会不造成巨大混乱之前，还有其他的软件开发实践要遵循。

被吹捧的 QA 好处之一是一个损坏的环境需要几天才能修复。明确提交/合并需要尽快恢复。

功能标志会增加风险，尤其是如果按照本文所宣扬的方式使用的话。另一方面，A/B 测试也有同样的问题，但是你被告知需要额外的生产监控。
