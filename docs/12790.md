在我的工作中，作为网络安全部门的主要专家和代表，我经常与客户沟通，以便彻底讨论所有请求和所有问题。

因此，在这种电话和会议的过程中，一些客户没有完全意识到渗透测试这种服务的意义。出于某种原因，许多人将这种类型的服务与漏洞评估甚至简单的扫描混为一谈，在实施与渗透测试相关的项目的方法类型中也存在一些混淆。

有些人无法决定他们需要什么- [渗透测试](https://www.scanforsecurity.com/articles/what-is-penetration-testing.html)或[漏洞评估](https://www.scanforsecurity.com/articles/vulnerability-assessment.html)，有些人考虑到项目的具体情况，不能完全正确地确定所需的方法。

真实案例——一个注册过程有限的 web 应用程序，而客户希望以黑盒模式进行检查，尽管根据客户的预期，我立即意识到这是一个灰盒，以及从用户的个人帐户执行检查的可能性。当然，综合这些信息，我们得出了一个共同点。

是的，我知道许多客户没有自己的信息安全部门，他们根本不可能正确制定这类项目的要求。当然，我们的主要任务是帮助和引导他们走上正确的道路，这一点我做得非常成功。

但问题的本质是什么？也许这个主题没有被充分涵盖？或者太多复杂的术语和不完全可理解的方法/过程？

让我们在这里尝试一点来处理这些问题。

### 安全测试的主要类型

*   渗透测试
*   脆弱性评估
*   安全审计

### 渗透测试的类型

1.  黑盒(Black box)-我们在几乎没有信息的情况下工作，我们要么有网站链接，要么只有公司名称。目标之一当然是危及应用程序/网络/任何数据(包括员工)。
2.  **灰盒** -在这种情况下，我们将使用某种接近最小的数据集，大约在黑盒级别，但我们可能已经有了用于输入应用程序的数据，例如。
3.  **白盒** -这通常适用于直接从客户的基础设施和基础设施内部工作的情况，当我们有网络拓扑时，我们知道服务器的 IP 地址和使用的应用程序类型。此外，这种方法部分适用于与源代码分析相关的项目。

### 其他类型的安全测试

**漏洞评估**是一个更加自动化的过程，其中有一个额外的步骤来验证获得的结果并消除误报。主要任务是找到尽可能多的漏洞。

安全审计(Security audit)——我们手中握有所有的牌:我们有访问权、管理员级别的特权，或者，我们有配置文件的副本。主要任务是研究使用的解决方案、架构、使用的软件及其版本、限制系统访问和系统间通信的方法。

在这篇文章中，我能够容纳远非所有的方法和手段，但总的来说，画面发展成这样。我希望这份材料至少能够帮助那些希望对其系统进行 pentest 或 audit 的人正确选择所提供的服务类型和使用的方法。
