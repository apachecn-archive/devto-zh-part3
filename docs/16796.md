# 乐高 WordPress 开发的真实成本

> 原文:[https://dev . to/no _ fear _ Inc/the-true-cost-of-Lego-WordPress-development-4 mni](https://dev.to/no_fear_inc/the-true-cost-of-lego-wordpress-development-4mni)

当我十多年前转到 WordPress 的时候，我的目标是最大化 RAD(快速应用开发)。

但不是每个人首先假设的原因。

## [](#coding-can-last-forever)编码可以永久持续

我的工程背景是企业级开发。我花了几年时间学习 Java (JavaServer Faces，Google Web Toolkit ),并对. NET 有所涉猎。

召集一个 20 多人的团队在一个项目上工作了 5 年，这与在两三个月内启动一个强大的应用程序有很大的不同。

在整个过渡期，我在 web 和移动 Java 开发、联合国儿童基金会项目的 Python (Django)、智能电视出现之前的机顶盒编程，以及使用 CodeIgniter、CakePHP 和定制框架的 PHP 开发之间切换。

我甚至试图出售自己定制的 Java CMS，但这毫无意义。我无法将“上市时间”与 PHP 的能力相匹配，也无法将服务器成本追溯到过去。

## [](#wordpress-can-be-a-rad-framework)WordPress 可以是一个 RAD 框架

这也是为什么 WordPress 目前为将近 33%的网站提供支持的原因之一。

PHP 驱动的 CMS 承诺向后兼容，提供无限的设计机会，简化的编辑体验，以及强大的模块化基础设施，是企业的主要卖点。

从我记事起，WordPress 的市场份额每年都保持 2%的增长。

这个网络巨头的座右铭是“*使出版民主化*”。平心而论，它确实提供了一个快速简单的安装，被每个虚拟主机提供商采用，并为学生、业余爱好者、博客和摄影师或初学者提供了一个练习发布的机会，以及基本的高级用户活动。

## [](#the-dreadful-diy-mentality)可怕的 DIY 心态

WordPress 最出名的是它无穷无尽的免费(而且非常便宜)插件。

只要有足够的时间和毅力，浏览初学者教程的足够带宽，以及一堆插件，初学者就可以设置:

*   电子商务网站
*   小型宠物项目网络
*   一个论坛
*   下一代社交网络
*   投资组合网站
*   《福布斯》或《企业家》的翻版

方便的是，这引发了新一波没有技术背景的服务提供商提供 WordPress 开发服务。数百万自学成才的超级用户都有这样的头衔:

*   WordPress 开发者
*   WordPress 专家
*   WordPress 专家
*   WordPress 大师
*   WordPress 忍者
*   摇滚之星

> 注:[大写字母 P](https://codex.wordpress.org/Function_Reference/capital_P_dangit) 很少出现，但为了准确起见在此声明。

这导致了多年的“竞相降价”和价格低至每份 50 美元的“专业”服务。

一旦我们搞清楚了这一点，让我们来讨论是什么让 WordPress 成为一个拥有 DIY 超级用户的危险社区的强大平台。

## DIY 应用的危害

几年前，我花时间写了一篇关于定制构建和 DIY 站点的比较，这一点至今仍然有效。

我的团队建立了所谓的“3S”框架来处理每一个新的线索，这代表着**稳定性**、**速度**和**安全性**。

或者是我们在着手一个由 DIY 构建者或经验较少的代理机构构建的大规模应用程序时最常遇到的三个问题。

这主要是由于大量免费的“瑞士军刀”插件捆绑在一个 web 应用程序中，以及一个强大的，badass premium WordPress 主题——通常是一个多用途的主题，提供 150 个模板，1300 个主题选项，以及无数的滑块和页面生成器。

我明白了，这让编辑体验...更简单。也就是说，一旦你阅读了一本 120 页的手册，其中披露了 Visual Composer 中的滑块选项和所有组件，或者是另一个替代的重型构建器。

然而，一旦你开始扩展，事情看起来就不一样了。更重要的是，处理专有的第三方解决方案变得极其复杂(尽管是开源的)。

将质量与成本或执行时间进行比较是一场持久战。

## [](#sample-study-updating-an-article-layout)样本研究:更新文章布局

背景:一家月浏览量达到 2000 万的新出版商打电话给你，要求对他们现有的帖子布局进行一系列修改。

1.  您要求他们访问他们的 git 存储库(不存在)。
2.  然后你想浏览他们的临时服务器。太令人惊讶了，这里也没有。
3.  SSH 访问？幸运的是，我们提供 SFTP，甚至 FTP 服务。

一旦你深吸一口气，意识到没有可用的 changelog、任何文档或合理的部署应急计划(回滚)的方法，你就提供了一个合适的托管解决方案(从 Digital Ocean 到 Elastic Beanstalk，再到支持应用程序核心需求的托管 WordPress 供应商)。

对工作的评估被拒绝，更不用说“迁移的风险”了。让我们假设他们至少有一个 SSL 证书。

您设置了 DeployBot(或任何其他通过 git repo 中的 webhooks 连接 SFTP 的桥),并跳转到第二步。

## [](#handling-maintenance-in-wordpress)在 WordPress 中处理维护

正如所料，现有的设置很混乱。除了多用途的高级主题，你会发现 54 个激活的插件:两个滑块(天知道为什么)，一个页面生成器，一个带有大量复选框的广告管理解决方案，一个随机激活的“相关帖子”，以及几十个其他插件。

一个明智的方法需要彻底的代码审查和一些概要分析。然而，这是要花钱的，在过去的几年里，客户并不习惯于支付这笔钱。

您的客户渴望为他们在特定类别上的文章设置“无限滚动”。一旦你仔细检查了 DevTools 控制台，发现了由构建器和滑块插件导致的 JavaScript 错误的无尽列表，你就有了修复这些错误的行动计划。

> “不用担心，搞定就行，谢谢。”

选择一个合适的库来实现无限滚动是可行的，您可以做到这一点。或者甚至使用 Ajax Load More，这是一个带有一组扩展的插件，与 WordPress 一起工作还可以(带有一组缓存和功能限制)。

## [](#how-to-perform-the-update)如何执行更新？

我知道你现在不是一个厌恶风险的人。所以让我们玩 YOLO 吧！

你提出了一个无头 WordPress 构建的想法，以防万一。这将使布局开发变得非常容易，并减少对核心平台的依赖。当然，这是行不通的，所以你只能靠自己了。

当你在处理旧版本时，你突然想起 WordPress 在其最新的 WordPress 5.0 版本中推出了自己的页面生成器 Gutenberg。由于您正在运行一个过时的版本，您试图升级您的本地副本，所有的地狱打破松散，即使安装经典编辑器保持其核心行为。

无论如何，您足够坚持将加载帖子的核心逻辑分离到一个单独的模板中，将其附加到一个类别映射列表或一组复选框以便于控制。

你在本地帖子上运行了几个测试，看起来不错。由于缺乏一个搭建的环境，客户唠叨着要你尽快上线，你照做了，并在之后“祈祷和希望”。

## [](#exceptions-warnings-and-errors)异常、警告和错误

虽然最初的测试看起来很好，但一小时后您会收到一封紧急电子邮件。

有些帖子根本不管用！

事实证明，这些 JavaScript 错误毕竟发挥了作用。动态广告管理解决方案全面嵌入随机破碎的视频，完全消灭了核心帖子布局，使页面的 HTML 及其对应的 JavaScript 因无限滚动逻辑而崩溃。你回复(由于你自己的临时魔法)并且再看一眼。

快进几个无休无止的通宵，你已经用你旁边满满一桶汗水和泪水修补了整个事情，要求一个适当计划的代码重构、清理、部分重建、托管迁移等等。

> 这贵得离谱——从那以后，网站一直运行得很好，而且变化可以瞬间发生！

## [](#moral-of-the-story)故事的寓意

WordPress 本身还不错。

但是传达价值和“技术债务”的危险，除了他自己的个人品牌和信誉之外，还需要一种*格兰特卡尔多内*式的销售技巧来度过难关。

伟大的工程师对实现这一点不感兴趣(而且通常缺乏技能)。

在此之前:

*   缺少分期应用将使迭代开发几乎不可能。
*   不存在的 SSH 访问也是如此，甚至无法在服务器上设置 git repo 来进行自动部署。
*   主题显然是混乱的。应用微小的更改需要完整的重构，并将核心逻辑分离成[模板部分](https://developer.wordpress.org/reference/functions/get_template_part/)或它们相应的包含。
*   安装无数的“瑞士军刀”插件对性能、稳定性和安全性没有任何好处。每一次更新都有平台部分崩溃的风险。
*   通过管理面板控制大量的逻辑导致数百万种可能的排列，使得 QA 几乎不可能，更不用说稳定性因素了。

与十几家服务于数千万甚至上亿页面浏览量的出版商合作后，在 WordPress 上建立一个健壮的大规模平台是可能的。

很多东西都要从零开始建造。轻量级的定制主题，考虑了“关注点分离”和“松散耦合”。

遵循 UNIX 工具开发模型的简单且经过深思熟虑的插件。

一个连贯的编辑体验。

考虑到质量保证和管理，有足够的时间设置新布局或构建功能。

这是一个让包括产品经理、我们的团队和所有相关第三方供应商在内的所有各方都参与进来进行仔细评估的过程。

其他的一切都有帮助，但是我们应该从一些事情开始。

但是，向一个在未来几年数百次崩溃后一直在修补平台的中型企业销售这种产品是一项挑战。

让我们期待更好的未来。
