# 开发人员招聘的误区

> 原文：<https://dev.to/denisviklov/myths-with-developers-hiring-5827>

我想我们都知道“团队就是一切”这句名言我也同意，尤其是早期阶段的团队。关于统计，90%的 IT 创业公司第一年都活不下来。从字面上看，他们会在产品开发上失败。所以，让我们假设以下几点是某种阻止你加入梦之队的障碍，梦之队能够将想法转化为可行的产品。

我雇佣开发人员已经有七年了，作为商业开发人员已经工作了十四年，所以下面的一切都是关于雇佣过程中最常见的误解。

## 简历中的著名公司

首先让我们意识到大公司下面我们指的是什么？比如谷歌和迪士尼，名气大，做 IT 领域的。此外，一切都来源于上下文。例如，如果你正在为 3D 艺术家创建一个渲染农场，而你有一个熟悉迪士尼相同流程的人的提议，我认为这将比谷歌的任何员工都好，因为 3D 图形不是谷歌的东西。此外，不要忘记公司是如何运作的。在一个大公司里，每个人都有一个最小的责任范围，所有的工作流程看起来就像一个传送带。例如，这是一件普通的事情，然后一个好的开发者可以坐在支持和修复热 bugs streamline，多年来从不编写新的功能。老实说，从我的经验来看，在大公司工作比在创业公司更轻松，我认识一些长期坐在大公司的人，他们的生产力非常平均，直到管理层同意为止。在我看来，简历中的公司证明一个候选人肯定有一些技能，可以在一个大的团队中工作，但你只有在实际工作中才会知道这些技能的水平。

## 编程挑战

围绕像 hackerrank、codefight 等编码挑战的大肆宣传。因此，作为一名工业开发人员的日常工作和用秒表解决奇怪的编程难题之间存在很大的误解和巨大的差异——从字面上看，与实际工作相关的编码挑战就像占星术和天文学一样。在我看来，如果创业公司在招聘过程中使用这一点，只会说明管理层在团队建设中的低能力和正式方法。还有，谁想在一个连基本流程都不关心的公司工作。编码挑战对每个人来说，暖脑并从竞争中获得乐趣是一种令人愉快的爱好，但对于雇佣真正的程序员来说，这是一种软弱的做法。

## 小测验

你见过好的编码小测验吗？我看过很多，但都是低质量的，目的不明确的。任何测验的黄金问题都是“如果(' a' * log(3))^'a'/'5')会显示什么”。不幸的是，没有“自杀”这个选项。从字面上看，我甚至不愿意和那些把字母分成数字，然后把它们提升到“狗”这个词的能力的人呆在同一个地方。真的，谁知道对数整个字母表后他们会做什么。不要忘记什么样的编程语言在不断发展，有不同的版本，它们对同一事物的行为可能会有很大的不同。你可以打开 Python 的标准库，尝试全部读取。是的，这是一门包含电池的语言，只是试着阅读完整的标准文档，我甚至没有提到记住所有方法的签名或其他什么，但这种方法在测验中广泛使用。
大概，有史以来最没用的东西不要用。

## GitHub 账号

这是一个普遍的误解，即每个程序员都必须在 GitHub 上有一个帐户。我的第一个问题总是:“为什么不是 GitLab、Bitbucket 或者自己的 git 服务器？”。Git 是一个分布式 VCS，它不需要任何中央服务器等等。如果我们谈论参与开源项目，这是一把双刃剑。一方面，流行的开源项目是一个大生意，有高薪的专门团队，没有人在那里等你。是的，可以发送一个 pull 请求来修复文档中的错误类型。然而，我能从事实中得到什么信息呢？字面意思，没什么。询问当前项目的代码样本是一个很好的做法。当然，如果你在做一个商业产品，你需要 NDA，但是展示一些不违反你的协议的样品总是可能的。这是一个几乎无用的要求，简单来说，如果我们只雇佣有“无国界医生”志愿者经验的外科医生，这也是一样的这看起来是个愚蠢的观点，但是许多雇主仍然试图强迫它。

## Stackoverflow account
在乞讨时，应该注意到 Stackoverflow 大大降低了可接受问题的门槛，最终使其成为一个强烈关注问题而不是答案的资源。我的意思是，对于更常见、更新手的问题，你可能得到的最高分。从字面上看，你可以在官方文件的前二十行中找到这些问题的答案。我认为有经验的程序员会多次面临这样的情况，当他们对棘手而深刻的问题没有回应时。StackOverflow 账户对我来说什么都不是。
