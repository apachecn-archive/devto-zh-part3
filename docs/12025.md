# 如何走得快

> 原文：<https://dev.to/quii/how-to-go-fast-2195>

这篇文章是写给任何从事软件工作的人的，他们希望生活简单，压力小，但是仍然让他们的雇主开心。您将看到如何通过做出简单的选择并坚持被充分理解的、被证实的原则来发布软件，而不会让每个人都筋疲力尽。

企业武断地给了你的团队一年的时间来制作网站，他们有很多宏伟的想法。不要试图花大量的时间预先计划，想象大的架构和所有这些雄心壮志的积压。这是通向失败、过早优化和失信的道路。

你要做的唯一承诺是每两周展示一次你所做的事情，并接受业务部门对实际问题的指导，而不是做出决策。

我们知道走得快很重要，但我们正在制造的系统的质量和我们团队的心理健康也很重要。我们会走得很快，但我们会一直以可持续的方式前进。

我们不希望因为大爆炸式的发布而感到紧张，不希望不得不进行令人恐惧的大规模重构，也不希望为那些我们可能从未真正做过的工作纠结于不相关的评估。

## 迭代

从 hello world 开始，全面部署并持续交付。

我们总是希望团队有一个小而专注的愿景。开发小东西很容易，也很有可能成功。

考虑交付大量的特性会导致团队过于复杂，并使决策变得困难。

随着许多功能悬而未决，有扩大团队的诱惑，这也减慢了事情的速度。[见:人月神话](https://en.wikipedia.org/wiki/The_Mythical_Man-Month)

相反，我们致力于下一个最重要的事情，基于业务专业知识、研究、开发者知识和来自用户的反馈。

我们将继续不断地交付小功能，无限地迭代。

## 规则和原则

### 1 个小型同地团队

小团队比大团队交付得更快，因为沟通的渠道很少，一次完成的工作量也很小。大型团队做很多事情会产生混乱和复杂性。

让多个小团队从事明确分开的事情*可以*起作用，但是需要良好的协调。一旦你有了一个成熟的产品，在架构上定义那些团队就容易多了。我认为把一个*的新*项目和许多未知因素分开是错误的，因为架构在这一点上需要非常灵活，因此团队*将不能独立工作，他们将被耦合*。

根据我的经验，能够很容易地与人交谈是很重要的。你与同事进行的非正式聊天对于打破分享想法和建立关系的障碍非常有价值。对我来说，这就是为什么我更喜欢在同一地点的团队中工作，因为这样可以最大限度地减少沟通中的摩擦。

### 配对编程的大部分时间

结对有助于你写出更好的软件，并有助于技能、想法和领域知识的交叉传播。这将有助于你的团队成长，并改善关系，这通常会让每个人都更开心！

不要从朝九晚五开始，因为这很快就会变得很累。人需要时间和空间去思考。

### 无分支，无拉取请求

你有一个小团队。你*应该*一直互相交谈，这样你就不需要拉请求来检查代码是否正常。你最好把它寄出去并得到反馈。大多数时候，你应该相信两个人会把事情做好。

拉请求对于开源项目非常有用，因为在开源项目中，你并不完全信任每个贡献者。在一个小团队中，你不需要它，你需要通过减少编写软件中的仪式来利用非常紧密的反馈循环。

这些规则的一个重要主题是消除反馈循环的障碍，这意味着我们可以轻松地更改软件。有人发布了不理想的代码**并不是世界末日，只是修复它并继续前进——不要引入复杂的过程来减慢一切**

### 拥有最少的部署管道

维护管道和各种非生命环境会变得很麻烦。至少在项目开始时，你真正需要的是:

1.  运行测试
2.  部署到现场
3.  进行烟雾测试

如果你发现自己试图证明一个非现场环境，问问自己

*   有什么是我不能在本地或生产中测试的？为什么？
*   只能在特定环境中测试一个东西的成本和后果是什么？
*   这给生产发货造成了什么样的延迟？

大多数时候，这些问题你可能没有好的答案。

### 持续交付绿色构建。

这给了你:

*   快速反馈
*   强迫你自动化测试
*   强迫你写真正可发布的代码
*   迫使你进行良好的监控
*   使发布成为无压力、无事件的

关于这一点已经写了很多。

手动运送一周或两周的工作既费时又有风险；不断发布小东西很容易，风险也小。

### 写测试

它们使你能够自信地快速迭代。没有它们就不能连续交货，如果你不得不一直手动检查东西，你也不能走得很快。

### 独石，不微服

编写分布式系统是一件痛苦的事情，我宁愿避免这种痛苦，直到我真正需要分发我的系统。换句话说就是 YAGNI。

重构进程内方法调用远比更改 API 调用容易。部署一个应用程序比协调多个系统部署要容易得多。

我过去曾做过微服务项目，但我的上一个项目是一个整体项目，我们进展得非常快，因为我们剔除了大量需要应对的问题。

为了做微服务，你必须考虑如何将你的系统分解成小组件...

### 不要预先设计

只要发布一些代码，如果你做一些有用的东西，并不断重构和迭代，设计将会从现实中产生，而不是开发者在白板上争论。

### 连续重构

糟糕的代码会让你变慢。糟糕的代码往往会变得更糟。所以持续重构，不要有重构冲刺之类的傻事。不要请求许可，如果它很糟糕，就修理它。

monoliths 一半的问题来自于编写糟糕代码的人。如果你保持代码的质量*真实*基于*真实用法*的抽象将会出现，然后如果你需要打破你的铁板一块，将会有明显的方法去做。

### 避免写 SPA，拥抱渐进增强

spa 带来了大部分项目不需要的复杂性。你可能不会创建 Gmail，所以不要假装你需要 T1 来写一个 SPA。你可以在你的简历上得到 10%的反馈。

相反，我更喜欢使用渐进式增强技术，通过编写语义 HTML，用 CSS 和少量 JavaScript 进行修饰，以在必要时稍微增强体验。更好的是，如果可以的话，不要编写任何客户端 JavaScript。

### 不要沉迷于像素完美的设计

如果你有一个设计师过分看重他们的设计在每个浏览器上看起来都像像素一样完美，那么编写 HTML 和 CSS 会很有挑战性并且非常耗时。

我们是为*用户*而不是设计师写代码，大多数时候他们不会在意火狐浏览器和 Chrome 浏览器相比是否有所不同。

通过挑战对用户来说足够好的东西来节省时间、精力和压力。

**简单的设计实施起来简单，精细的设计实施起来复杂**。如果你的设计师让你很难实现网页，提醒他们我们正在努力提高速度。

### 先发货 80%，以后再打磨

与上述相关的是，在设计和 UX 方面真正完善某个特性完全是浪费时间，如果结果证明这是一个不需要的特性。

我见过一个团队花了数周时间，从设计和 JavaScript 方面对网站上的一个搜索功能进行润色，结果几个月后就被抛弃了，因为根本的想法是错误的。

*发送足够的信息以获得反馈并证明想法*，一旦你完成了，你就可以花时间润色它。

### 编写软件的人是软件的支持者

这种责任必须与授权相结合，以使系统可维护。

如果创建系统的人因为系统停机而面临凌晨 3 点被叫的威胁，他们更有可能发布更健壮的代码。

### 使用成熟的、经过验证的工具

如果你离开了，没有人能再构建或理解你的代码，那么即使你用*闪亮的东西*走得很快也没有关系。

在你的团队中就你喜欢用什么来写作达成共识，这对于手头的任务是合适的。不要学习一门每个人都不熟悉的新编程语言——记住我们正在努力加快速度。

### 构建软件一定很简单

运行和测试项目所需的设置应该是最少的。仅仅因为几个月没碰或者某个关键人物走了就无法建项目，这是完全不能接受的。如果一个人花了一个多小时来设置项目，那你就失败了。

有了 Docker 这样的技术，配置一个项目变得很简单，因此它拥有所有的第三方依赖项，比如本地可用的数据库。

它不应该依赖于计算机的特定设置(除了安装一些关键的东西，如 JVM、Go 或其他)。不得依赖第三方系统在本地运行和测试。

为了帮助实施这一点，配置您正在使用的任何 CI 工具，以便每天早上自动构建软件；即使没有人承诺过。

### 如果构建为红色，则停止

测试必须总是通过。如果你有不可靠的测试，停止并修正它们。过去，古怪的测试花费了我很多时间。

### 与用户坐在一起，看他们工作

业务认为用户需要的东西和用户*实际*需要的东西往往大相径庭。我非常幸运地亲眼目睹了这一过程，通过实际观察用户的工作情况，我们节省了大量工作，并且我们能够确定他们实际需要的非常便宜和简单的解决方案，而不是企业提议的极其昂贵的解决方案。

### 用户故事是对话的开始

如果一个用户故事明确规定了要写什么样的软件，那么是谁做的决定？为什么？他们怎么知道自己是正确的呢？

这种由“技术领导”或其他什么人来决定如何编写软件的方法，创造了失望的开发机器人，他们没有所有权感，故事被“封锁”，因为他们没有得到一些过度控制的领导者的祝福。

用户故事应该描述一个用户问题，当这个问题被发现时，团队会想出一个方法来解决它，然后他们会执行这个计划，就这么简单。这样成功的几率更高，因为更多的人在思考问题，让人们有参与感。

每张票据都有某种成功的衡量标准，当收集到足够的数据/研究时，应该对其进行验证。

### 明白了*我们都是软件开发人员*

在我想象的公司里，如果你认为自己是“后端”开发人员，结果因为“前端”开发人员生病而被屏蔽，那你就认为自己被解雇了。同样，如果 UX 的设计师不在，而你不想猜测某个特定的交互应该如何进行。

你应该对软件开发的各个方面有所了解。这并不是说你必须成为所有方面的专家，但是因为某个人不在而被屏蔽是不可接受的。

你应该感到有足够的力量去尝试任何任务。如果它不完美也没关系，因为最好交付 70%好的东西，然后当专家回来时，你可以在软件上**迭代**以使它变得更好。

当我前面提到结对时，我并不仅仅指“后端开发人员”或其他什么。每个人都应该参与进来。如果一个 UX 设计师在面试用户，他们有责任带其他人来获得这方面的经验并了解用户。这种知识分享对于让个人感受到对项目和团队的投入是至关重要的。

### 有 10/20%的时间

给每个人分配时间，让他们做自己想做的事情，并严格遵守。

我们从事创造性职业，我们需要一些空间来呼吸、学习和尝试不同的想法。

这对团队来说是一个很大的好处，可以帮助避免精疲力竭，并且可以帮助带来新的想法，这些想法可以给你的系统带来很大的改进。

### 不要计划超过 6 周的事情

对未来有一个愿景是有趣的，但是要保持一个模糊和有抱负的愿景。记住，专注于一两个特性的团队比担心 10 个想象中的需求的团队更有效率。

### 没有吉拉，没有估计，没有计时，没有消耗图表

所有这些都是对时间的巨大浪费，导致了摩擦，在我的职业生涯中，我还没有见过一张燃烧表能提供任何真正有用的信息。

记住我们已经承诺经常展示真实的工作软件***，**而不是想象中的图表**，这些最多是有根据的猜测。手指在空中的猜测是好的，为没有希望在未来 3 个月内被实际观看的门票增加故事点是没有价值的。***

> [流程和工具上的个人和交互](https://agilemanifesto.org/)

无论如何，要有一面简单的物理墙，上面有“准备开发”、“开发中”和“完成中”的柱子，但比这多得多的东西只会变得愚蠢。

## 包装完毕

快速前进的方法不是想象完美的产品、架构，设计一个待办事项，然后执行它。

你快速前进的方式是通过用低努力的、已被证实的方法快速迭代，并且在你通过迭代学习的过程中互相交流。

优化你的过程，这样你的团队就可以无畏而快速地改变软件。通过这种方式，你将能够在不断的反馈循环的指引下，打造出一款优秀的产品。

没有一个真正的方法可以创造出能运送伟大产品的伟大团队，但以我目前的经验来看，这些原则似乎站得住脚。你的方式可能会有所不同，这很好，但问问你自己，你的团队是否有可能做到最好。回顾对于帮助你的团队适应新环境总是很重要的。
