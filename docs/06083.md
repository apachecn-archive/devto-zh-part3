# 通过贡献学习

> 原文:[https://dev.to/agoldis/learn-by-contributing-27g6](https://dev.to/agoldis/learn-by-contributing-27g6)

## [](#contributing-to-open-source-projects-helps-you-to-improve-your-professional-skills-and-discover-new-opportunities)为开源项目做贡献有助于你提高专业技能，发现新机会。

成为一名专业的软件开发人员需要不断的提高和学习。

不管你是新兵训练营的菜鸟、大学毕业生还是软件开发老手，探索不同的软件开发方法对你自己的职业发展(和好奇心)都是有用的。

跟上节奏相当困难。你已经“水平地”成长了——意识到新技术并从中获得经验，用主流流行工具提高你的专业技能；你也可以通过解决更复杂、更抽象的问题来“垂直”进化。

有经验的开发人员受到团队在工作场所采用的技术堆栈的限制。了解不同的方法会给他们带来优势，并对团队/公司中常见的解决方案和备选方案展示不同的视角。

对于处于职业道路起点的开发者来说，这更具挑战性。工具和技术的多样性是压倒性的！如果没有类似问题的经验，很难知道在特定的任务中应该关注什么，什么工具是最好的。

课程和教程，即使是高级的，通常也不能提供所需的信心水平。总有一面“超越教程”的墙。你偶然发现了一个超出教程范围的问题，并开始在别处寻找解决方案。

# [](#it-is-very-good-for-you)这对你很好

为开源做贡献对你有好处。事实上，这对每个人都有好处，原因如下。

### [](#you-help-others)你帮助别人

无论你的贡献类型是什么——文档、代码修改或者仅仅是打开一个问题——你都改进了我们所有人使用的工具，并使其他开发人员的生活变得更容易。

### [](#excellent-projects-quality)优质工程

开源软件不再处于劣势。对于内部代码库来说，代码的质量通常是一流的。通过从事如此高质量的项目，你会看到其他人是如何做好软件的。

### [](#glimpse-into-professional-software-development)专业软件开发一瞥

你可以看到并了解微软、谷歌或脸书(以及许多其他以优秀工程闻名的公司)是如何开发软件的——这些公司使用开源许可证来共享内部编写的代码。

### [](#collaborate-with-experts)与专家合作

识别、实现和维护一个成功的软件项目需要相当多的经验和专业知识。

这些工具背后的人对他们的创作充满热情，热爱他们所做的事情——他们是真正的专家。与他们一起工作时，你得到的反馈是有价值的。

### 你在证明自己的专业技能

在 Github(或任何你选择的平台)上拥有令人印象深刻的投资组合是你未来雇主的可靠指标。而且很多猎头用 Github 找人才。

当您在第三方开源项目中遇到实际问题时，开始会容易得多。一个对你很重要的问题。

幸运的是，Javascript 开发非常依赖于公开可用的软件(通过 npm 包),而且解决方案并不完美。

每当你看到一个问题——无论是配置、文档还是实际问题，这都是一个改进、贡献和学习/提升你的专业水平的机会！

# [](#the-contribution-checklist)贡献清单

### [](#create-an%C2%A0issue)创建议题

这是你能做的最少、耗时最少、最重要的贡献。

人们可以说这不是真正的贡献，但我确实认为它非常重要。事实上，Github 确实把它算作一种贡献:

<figure>

[![The contribution 4 types as considered by Github](../Images/4872b561ad3a4e501ebf297638e5d8b8.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--a91FgPxP--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://cdn-images-1.medium.com/max/800/1%2AVaWdY8grXqENUij___ZBVw.png)

<figcaption>The contribution 4 types as considered by Github</figcaption>

</figure>

让我们从一个显而易见的观察开始——没有产生问题，维护人员永远不会知道他们的软件有问题，他们也无法改进它。

但是，你是最大的受益者！

你有没有发现自己试图解决一个问题，但没有成功，在尝试了几次后，打电话给同事寻求帮助？当你开始描述问题的时候，我发现了！

你突然明白了！(通常是错别字吧？)

很难有效而精确地描述一个问题✍🏻。今天的大多数项目都使用某种新问题的模板。按照模板，你强迫自己从不同的角度看问题。

通过用你自己的话描述问题，你可以突然意识到解决方案是什么，然后…应用它！

或者，用你自己的话描述这个问题可以激发新的想法/假设。如果是这样的话-🙌🏻干得好-去告诉你的同事吧。

如果没有，你只是制造了一个有着合理解释的问题。这将有助于维护人员。您帮助了其他开发人员——现在他们能够确认问题不在他们自己的代码中。人们会发现你的问题👍🏻

通过第一次参与，您为维护人员提供了一个指导您的机会，可能会提出一个解决方案，并激励您打开一个带有修复的 pull 请求(稍后将详细介绍)。

看，你已经在帮助很多人了！

额外收获:在 Github 上创建一个问题时，你会得到一个已经存在的问题的自动建议列表，这有助于发现解决方案并防止重复！👍🏻

### [](#provide-a-minimal%C2%A0example)提供一个极小的例子

对于维护人员来说，处理您报告的问题的一个非常具体的、最小的表示会容易得多。

虽然比较费时间，但是对你也是有好处的。通过提取问题，你得到:

*   确认这实际上是工具/库本身的问题，而不是您自己的代码的问题
*   更好地理解问题是什么(这可以在以后有效地描述问题)
*   通过处理简化的上下文来解决问题的机会

通过去除不必要的复杂性，同时构建一个最小的示例，您很有可能更好地理解问题，找到解决方案，甚至意识到这根本不是问题😼

以下是一些可用于创建可重复示例的平台列表:

*   Github 上的新知识库
*   [Codesandbox](https://codesandbox.io/)
*   [小提琴](https://jsfiddle.net)
*   [密码笔](https://codepen.io)

### [](#find-a-workaround)寻找变通办法

在这个阶段，问题得到了确认和验证，或者可能没有得到完全的验证，但是…你想要向前迈进，摆脱它！

大多数情况下，你没有时间去钻研第三方库的源代码，但是你可以找到一个解决方法！

这可能是一个丑陋的猴子补丁，配置变化或一些创造性的方法来克服这个问题。与世界分享是你的义务！

创建新问题或添加评论。你会得到大量的❤️，👍🏻和🎉！

### [](#suggest-a-solution-in%C2%A0comments)在评论中建议解决方案

好消息——你刚刚找到了一个解决方案，或者对什么是好的解决方案很有信心！

它在您的代码库中工作得很好，但是您没有时间用测试和/或文档创建一个拉请求。

分享你的发现仍然是有价值的——其他人会从你离开的地方捡起它。

顺便说一句，通常问题甚至不在代码中——它可能是一个配置问题或者缺乏清晰的文档，将项目用户引向错误的方向。重点是分享这一点点知识——它是有价值的，即使它看起来不像是重大贡献。

### [](#implement-a-solution-in-your-own-repository)在您自己的存储库中实施解决方案

经常发生的情况是项目维护人员不太响应。大多数情况下，他们只是忙于自己的生活，或者在工作场所太忙。一个好的临时解决方案可能是分叉项目，并在您自己的存储库中推进变更。

您的对等用户可以将您的临时存储库用作依赖项。虽然不理想，但它可以帮助你的团队前进。✅

同时，提交你的修复作为一个公关。最终，作者会检查你的修正并发布一个新的版本。创建 PR 是非常重要的，否则，您将面临无法更新的依赖关系的风险。

有时，作者会建议加入核心贡献者名单，这很棒——你有机会承担责任，维护一个拥有活跃用户的项目。

### [](#create-a-pull%C2%A0request)创建一个拉式请求

拉请求是开源开发的基础。

每个人都可以做出贡献，参与更大的项目，留下技能和专业知识的数字痕迹🌟

最重要的是，这是个人学习和提高的最大、最有效的机会💪🏻

通过提交拉取请求，您可以获得几层有价值的经验——让我们列出它们:

*   你看到复杂的问题是如何解决的，发现新的架构方法，组织项目和代码的不同方法

*   您会发现新的编码风格和技术

*   您会发现测试框架、CI 管道、linters 的新的或不同的用途，并丰富您使用其他补充工具的经验和知识

*   你正在创造一个从有经验的专业人士那里获得有价值的反馈的机会

*   你正在学习你正在使用的工具的内部实现——这将帮助你更好地使用它

*   你正在创造一个真实、透明的记录，记录你的成就和专业精神

*   你和其他开发者联系，这创造了合作、雇佣的机会，这很有趣

> 这就是你如何学习并成为一名更好的专业人士——你做更多不同的事情。

你看到并尝试不同的方法，学习好的和不好的部分，在不同的情况下应用你所看到的，并获得关于你的决定的反馈。

通过不断重复和定期用更复杂的主题挑战自己，你会获得专业的自信和经验🥋

# [](#discovering-opportunities-to-contribute)发现做出贡献的机会

正如我提到的，当你有一个必须解决的实际问题时，开始做出贡献会更容易。但是，如果你不使用 OSS 工具，但仍然想寻找贡献机会，你该怎么办？

### [](#online)在线

Github [建议](https://help.github.com/en/articles/helping-new-contributors-find-your-project-with-labels)项目负责人给更容易开始的问题分配标签。通过`label: good first issue`进行简单的[搜索](https://github.com/search?q=label%3A+good+first+issue)，将会发现超过 20 万个问题可供选择😄

你使用`label: help wanted` [搜索查询](https://github.com/search?q=label%3A+help+wanted)来发现不一定对初学者友好的更复杂的问题。

有大量的资源根据项目的技术、难度和受欢迎程度进行某种聚合。

下面列出了几个这样的项目来帮助你开始:

*   面向初学者的令人敬畏的列表 -一个按技术堆栈分类的项目管理列表
*   [http://github-help-wanted.com/](http://github-help-wanted.com/)——《寻人启事》发布聚合器
*   [https://www.firsttimersonly.com/](https://www.firsttimersonly.com/)-开源新手的有用阅读清单
*   [https://giistr.com/](https://giistr.com/)-轻松搜索、过滤和帮助您标记的存储库的问题
*   [https://www.codetriage.com/](https://www.codetriage.com/)-选择你最喜欢的回购，每天在你的收件箱里收到不同的公开问题
*   [http://issuehub.io/](http://issuehub.io/)-按技术和难度标签发布聚合器
*   [http://www.pullrequestroulette.com/](http://www.pullrequestroulette.com/)——提交审查的拉请求(或 PRs)列表，属于开源项目

还有很多很多…

### [](#offline)离线

很有可能你所在地区的更多人对开源贡献机会感兴趣。尝试找到一个 meetup 事件。

我想强调的一个特别的例子是[善良小队](https://hackernoon.com/goodness-squad-a8704d594a7a)。几年前，我亲自参加了其中的一次活动。这是非常有效和有益的。
