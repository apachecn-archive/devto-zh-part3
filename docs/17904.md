## CSS 很硬

## 链接标签

块渲染，只运送你需要的 CSS，即关键的 CSS

## 边界元法([块元素修改器](http://getbem.com/)

`.block__element—modifier`

赞成的意见

*   易于设置
*   框架/库不可知
*   修正了一些特殊性问题
*   刚性结构

骗局

*   命名很难
*   需要记住规则并严格遵守

## 预处理器

赞成的意见

*   嵌套
*   变量
*   SSR 只是工作

骗局

*   过度嵌套
*   更多依赖关系
*   很难创作库→要修改，您需要运行预处理器
*   未针对基于组件的工作流进行优化
*   没有基于运行时值的动态样式

## CSS 模块

用类编写普通的 CSS，并在你的组件中使用它们。最终这些将被随机的类名所取代。

通过 [PostCss](https://postcss.org/) 使用它(Css 的巴别塔)

赞成的意见

*   可以使用所有新的 CSS 功能
*   自动修复 css
*   SSR 只是工作
*   更加基于组件和可维护

骗局

*   不可读的类名
*   更多依赖关系
*   需要与预处理器配对，以提供可变支持
*   全局类名可能会被滥用

## CSS-in-JS ( [样式组件](https://www.styled-components.com/)

只反应成分。没有现成的 SSR 支持，这可以通过使用 Razzle 来缓解(它将提取所有的样式，并给你插入到 HTML 中的样式标签)

赞成的意见

*   动态样式(基于组件的状态和属性)
*   使用`ThemeProvider`组件进行主题化非常简单，它会将主题作为`prop`传递给子组件。
*   仍然可以使用层叠，但是重点是你的组件的样式只适用于那个组件。
*   没有更多的特殊性问题。
*   自动修复一切
*   不需要担心命名的事情

骗局

*   需要改变你对 CSS 的看法
*   团队需要学习的另一件事是
*   您正在使用的库可能只支持基于对象的 CSS
*   SSR 很难自己实现

没有正确/错误的答案。选择解决方案时，背景/用例很重要。例如:对于原型，你可能想要使用[超光速粒子](https://tachyons.io/)，而如果你需要可伸缩性，你可能会选择不同的东西或者在它上面做些什么来避免重复的类名。
