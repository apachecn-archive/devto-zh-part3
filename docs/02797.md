## [](#compiled-vs-interpreted)编译 vs 解释

在写应用程序之前，你的第一项工作是选择一种语言。有许多语言可供选择，但几乎所有的语言都有一个共同点；你的电脑不能理解它们。

我说“几乎所有”的原因是因为有一种语言计算机可以阅读；机器代码。

> 机器代码是一种计算机编程语言，由计算机可以直接响应的二进制或十六进制指令组成。

虽然开发人员可以用机器码编程，但大多数人不这样做，而是选择高级语言。

[![Translation and Types](../Images/282356e26934818a85cd9f15a19f50dc.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--TBeO0J1I--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://www.codetips.co.uk/content/images/2019/05/image-4.png)

低级语言和高级语言之间的区别可以自成一文，但基本的总结是:

*   低级语言是为一种类型的架构(Windows、Linux 等)编写的，不能在另一种架构上工作。它更难阅读和理解，但具有更高的性能水平。
*   高级语言基本上是相反的。你可以写一次，它就可以在所有的，或者至少是大多数的架构上运行。对于人们来说，阅读和理解要容易得多，但却没有那么有表现力。

哪些语言是高级的，哪些是低级的？每种语言都有一个谱系。

那些对于人类理解和编写来说不太复杂，但是对程序员能做什么提供较少灵活性的，被认为是高级的。处于这个范围顶端的语言可能是这样的(**注意:**这种高水平的语言还没有被创造出来):

```
Create an application that accepts two numbers,
subtract the second from the first,
return the result. 
```

那些对人类来说理解和编写起来更复杂，但是在程序员能做什么方面提供更多灵活性的，被认为是低级的。这个光谱底部的语言是机器码，看起来像这样:

```
0x 60 00 00 80
0x A4 00 00 00
0x 60 01 00 84
0x A4 01 01 00
0x 60 02 00 00
0x 60 03 00 04
0x 60 04 00 00
0x 60 05 00 01
0x 08 00 00 02
0x 20 00 00 03
0x 20 04 04 05
0x 11 20 04 01 
```

不知何故，我们写的代码必须被翻译成计算机能理解的语言(机器码)。这可以通过编译或解释来完成。

### [](#compiled)编制完成

如果源代码在运行前被翻译，那么这种语言就被认为是编译过的。编译器将有一个程序员源代码的完整输入，和一个目标语言的输出(通常是机器码)。

有时候有一种中间语言，叫做汇编，如果这种语言不能直接编译成机器码。

### [](#interpreted)释义

如果源代码在执行过程中被即时翻译，则该语言被认为是被解释的。当一段源代码需要执行时，解释器会将这一段源代码翻译成机器代码。

最基本的是，如果你需要运行相同的源代码两次，它将需要被转换成机器码两次；这是对资源的浪费。然而，大多数解释器都进行了优化以减少这种资源浪费。

### [](#why-does-all-this-matter)这一切为什么重要？

如果您选择使用一种使用解释器的语言，那么从编写/编辑源代码到运行应用程序的时间就会减少。不必等待源代码被编译可以节省大量时间(取决于项目的大小、编译器和机器的性能)。

您也不需要生成多个输出文件，以在每个不同的架构上运行。您将能够在所有架构上以同样的方式运行您的源代码，并让解释器处理其余的工作。

如果你选择使用一种使用编译器的语言，它通常会更有性能。它不需要额外的即时翻译源代码的步骤，因为所有的源代码都已经被翻译了。

最后一点:仅仅因为一种语言使用解释器或编译器，并不意味着它不能使用另一种。通常，会有一个标准或公认的方法可以使用，但是有人可以为任何语言编写解释器和/或编译器。

## [](#static-vs-dynamic)静态 vs 动态

每种编程语言都有[数据类型](https://codetips.co.uk/beginner/what-is-a-data-type/)。这样做的一个原因是为了让计算机知道可以对它们执行什么操作。

假设我们正在编写一个接受两个输入的[函数](https://codetips.co.uk/beginner/what-is-a-function/);从另一个中减去一个。

```
subtract(input1, input2) {
    input1 - input2    
} 
```

注意，我说的是输入，而不是数字。从技术上讲，我们可以将任何东西传递给这个函数

```
subtract(5,3) // 2
subtract(19,40) // -21
subtract(9,3) // 6

subtract("hello","world") // wait, what?!! 
```

什么是`"hello" - "world"`？不知道吗？为什么不呢？因为都是字符串，没有任何意义。没错。

计算机也不能理解这一点，所以我们需要一种方法来检查我们是否犯了这些错误。

这可以使用静态类型或动态类型来完成。

静态类型语言将在运行前检查这些错误，如果发现任何错误，它将报告问题并通常阻止应用程序运行，直到问题被修复。

动态类型语言会在执行过程中动态检查这些错误。您的应用程序可能运行时从未出现过问题，或者如果出现意外错误，它可能会在几天后突然崩溃。

让我们看一个例子。下面是我们应用程序的开始；我们已经声明了两个[函数](https://codetips.co.uk/beginner/what-is-a-function/)，一个具有我们之前讨论过的相同减法误差，另一个具有完全有效的减法。

```
subtractionWithError() {
    "hello" - "world"
}

validSubtraction() {
    9 - 4   
}

validSubtraction() 
```

注意只有`validSubtraction`功能是如何执行的。

如果您使用的是静态类型语言，在应用程序运行之前会显示一个错误。

如果您使用的是动态类型语言，那么永远不会抛出错误，因为带有错误(`subtractionWithError`)的代码永远不会被执行，并且您的应用程序会继续运行，而不会意识到有错误。

如果我们的代码调用了`validSubtraction`函数，等了 4 个小时，然后调用了`subtractionWithError`函数会怎么样？

```
subtractionWithError() {
    "hello" - "world"
}

validSubtraction() {
    9 - 4   
}

validSubtraction()

wait(4 hours) // wait for 4 hours before continuing

subtractionWithError() 
```

如果您使用的是静态类型语言，行为将完全相同。在应用程序运行之前会显示一个错误，因为无论执行与否，它都知道有一个错误。

如果您使用的是动态类型语言，那么您的应用程序在前 4 个小时会运行良好，但是一旦执行了`subtractionWithError`函数，它就会显示一个错误并可能崩溃(取决于问题是什么)。

### [](#why-does-all-this-matter)这一切为什么重要？

静态类型的语言在运行时会有更好的性能，因为对类型的了解和对一切都已检查过的信心，使得优化成为可能，而这在动态类型的语言中是不可能的。

动态类型语言不需要在运行前检查所有东西，所以从编写/编辑源代码到运行应用程序会更快。就像解释语言和编译语言一样，这可以节省你大量的时间。

总结一下:

*   编译型和静态类型语言——从编写源代码到运行它的开销更大，但在运行时速度更快。不同架构的不同可执行文件。
*   解释型和动态类型语言——从源代码运行应用程序要快得多，但在运行时会慢一些。对所有架构运行相同的代码。

## [](#stronglytyped-vs-weaklytyped)强类型 vs 弱类型

强类型计算机语言是一种程序员预先声明什么是数据类型的语言。

```
var x string = "Hello world"

func example(input1 int, input2 int) {
    input1 - input2
} 
```

弱类型语言是一种数据类型由赋予它的内容决定的语言。

```
var x = "Hello World"

func example(input1, input2) {
    input1 - input2
} 
```

严格的类型将帮助您避免错误，就像我们之前看到的在编译或运行阶段之前尝试减去字符串(`"hello" - "world"`)时出现的错误。

弱类型允许你在你能做的事情上有更多的灵活性。最严格类型的语言不允许你把一个整数加到一个浮点数上，而弱类型语言允许你把一个字符串和一个整数相加。

```
6 + "5" // 65 
```

只是在使用弱类型语言时要小心，如果你自己不处理类型检查，你可能会得到意想不到的结果(例如，你认为你是在把整数加到整数上，但实际上你是在把整数加到字符串上。)
