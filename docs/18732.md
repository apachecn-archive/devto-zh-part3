# 进行第一次 SCRUM 培训后的回顾。

> 原文:[https://dev . to/dev Monte/retrospect-after-conducting-first-time-scrum-training-3706](https://dev.to/devmonte/retrospective-after-conducting-first-time-scrum-training-3706)

几周前，在 Java 训练营期间，我有机会进行了一次为期 3 天的关于敏捷方法的培训，重点是 SCRUM 框架。这是我第一次当讲师，所以我之前感到有点压力，尽管我认为我已经准备好了。当我开始时，一切都如我所愿(大部分:)，从我的角度来看，这三天都很好。然而，我注意到一些我应该改进或改变的事情，还有一些进展顺利的事情，值得再做一次。因为一些开发人员得到教新人的机会是正常的，他们可能会遇到同样的问题，所以我决定分享一些我在正式培训期间教别人的想法。

1.  在比赛开始前 20-30 分钟。
    尤指第一天。你将有时间适应你的班级，并有机会解决一些意想不到的障碍。找到最好的位置站着说话。设置好所有设备，深呼吸几次:)

2.  **80%的实践，20%的理论！**
    我尽可能遵循帕累托原则。几乎所有人都更喜欢动手训练，而不是理论。如果演讲太多，你可能会让参与者感到厌烦。相反，至少增加一些实践练习。尤其是如果这是你第一次和陌生人见面。想一想可以两人一组甚至是一组进行的活动。人们会理解的。

3.  训练前先练习。
    实践练习是好的——也就是说，你必须向团队解释这些练习，但这并不总是容易的。最好是和你的朋友或同事一起做一个测试练习。真的很有帮助。人们会问一些与练习规则相关的问题，这些问题对你来说可能很明显，但对他们来说却不明显，最好是从你认识的人那里听到这些问题，而不是在真正的训练中听到。即使你认为这个任务很简单，我建议你至少大声说出解释。它会帮助你意识到你是否错过了什么。

4.  不要看你的幻灯片！看看你的观众！
    如果你想被认为是专业人士，了解你的幻灯片，只有当你想指出某个特定的东西时才看。

5.  **给出简短有价值的答案。**
    我喜欢说话，也喜欢解释事情，但当你时间有限，想尽可能多地分享知识时，最好给出简短的答案。

6.  保持乐观，保持微笑！
    这是显而易见的。这对我们来说更好更健康，但是在压力大的情况下，我们可以忘记这一点。在演示过程中添加一些关于微笑的隐藏提示，即带上你最喜欢的杯子，或者在幻灯片的笔记中添加“微笑”信息。

7.  **总会有人的！**
    不听话的人。与此无关的人。会失望或不满意的人。很难让每个人都满意，所以当你听到一些对你或你的班级的抱怨时，不要气馁。倾听建设性的批评，忽略反对者！

仅此而已。你对自己第一次演讲或主持某种训练营有什么想法吗？
