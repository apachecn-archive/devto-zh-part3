# 增加价值，而不是复杂性:改变你对“云”的看法

> 原文：<https://dev.to/clicktravelengorg/add-value-not-complexity-change-the-way-you-think-about-the-cloud-8b6>

[![Cloud](img/0d21a55537727c45b0ece90b81ba15e9.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--c-jSn_2B--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/gvrlielz8upyy3fug5t3.png)

我不喜欢用“云”这个词，我并不孤单，但我们在这里；它已经成为任何与“不在你的硬件上的东西”相关的事物的标准术语。无论是手机上的个人存储还是多区域冗余数据库集群的部署，它们都被归入同一个总括问题“为什么不将其迁移到云中？”

然后是对“云”的过分轻视。当技术部门试图说服他们的业务迁移时，最大的挑战之一是很难描述好处。团队花了很长时间策划一场推介来概述好处，结果却被一个简单的回答否决了:“但这不就是其他人的数据中心里的相同计算机吗？”

## 它通常是如何售出的

所以技术团队一直在讨论，他们真的想把系统转移到云上。为了说服决策者，他们已经去搞清楚将会涉及到什么以及要花多少钱。几周后(如果一切顺利)，他们带着数字和迁移计划回来了。对话通常是这样的:

*   决策者:“那么这要花多少钱？”
*   *团队:“将花费大约 X，000 美元来配置架构以复制我们现在拥有的东西”*
*   决策者:“好吧，但是你去年说服我买的这些昂贵的东西又是怎么回事呢？这些东西能让我们的系统在未来的几年里经得起考验？”
*   团队:“我们不再需要它了:它会在云上！”
*   *决策者:“…………”*
*   *团队: "......嗯，我们可能会卖掉它来补偿一些东西，但可能不会很多，但搬家确实是正确的做法"*
*   决策者:“好吧，让我们先忽略成本。这要多久才能实施？”
*   团队:“嗯，建立一个复制当前系统的架构需要大约 6 个月的时间。完成后，我们将关闭当前系统，迁移所有数据，然后再次启动系统"
*   *决策者:“关掉……”*
*   团队:“总有一天我们会从一个系统切换到另一个系统。这可能会导致一些停机时间，但会保持在最低限度。”
*   *决策者:“…..好吧，但那时我们会比现在过得更好，更有弹性？”*
*   *团队:“是的……..实际上，这只是复制了我们现有的当前系统，但使用了专用资源来保持服务器运行等。*如果有故障
*   决策者:“我们不是已经有一个团队了吗？”
*   团队:“我们不再需要它了:它会在云上！”
*   *决策者: ".....好的，谢谢你的时间，我会再联系你的。”*

我在这里说得有点苛刻——人们会指出，有许多方法可以实现无缝迁移，而不需要停机等。-但事实是，这些对话通常是这样进行的。

决策者往往不理解这些好处，那些试图施加影响的人也不知道如何量化这些好处。

当重新开始时，在云上开始是非常有意义的。很简单，无论是公有还是私有的云提供商都比我们有一个优势:购买力。他们购买了足够多的硬件，能够以我们可以直接购买的一小部分价格提供给我们。我们将能够在我们需要的确切时间内准确地购买我们需要的东西——按分钟计算——而不是在我们自己的硬件上投资多年。事实上，我们很可能能够提供比我们的预算通常允许的规格高得多的硬件。

然而，更为成熟的企业往往拥有价值数千英镑的设备，这些设备是多年积累下来的。它是由他们的团队维护的，代表着一项巨大的投资，他们为什么要为了迁移到云而放弃这一切呢？

## 因为他们所拥有的还不够好！

没有人有足够的能力应对突发事件，但我们都希望自己的事业成为下一个萧条或停滞。我们需要不断考虑如何增加产能，这样当我们赢得大客户或成为下一个一夜成名的人时，我们就不必等待新套件的交付。随着增长而来的是对弹性的需求——如果所有硬件都并排放置在一个位置等待恢复供电，那么配置大量硬件是没有意义的——因此我们跨多个站点进行部署，并投资购买高可用性故障转移设备。这就是它一直以来的工作方式；我们根据现有的最佳数据进行赌博，希望在需要的时候得到我们需要的东西。但现在已经不是这样了。

刚起步时，我们可以以很低的成本进行小规模部署，并真正试用您需要的东西，而无需任何长期承诺。随着我们的增长，在云上是有意义的，因为我们可以通过在几分钟内(而不是几周内)配置或取消配置硬件来快速响应增长。当我们很大的时候，在云上是有意义的，因为你能够在多个地点甚至多个国家灵活地部署这些服务。

没有“云”,我们也能做到以上所有的事情，但归根结底就是:你为什么想要这么做？

对你的企业来说，什么是重要的？是保护您的硬件投资和传统流程，还是为您的用户提供最优质的服务？你的重点不应该是招聘员工和专业知识来管理基础设施；它应该是招聘员工和专家来解决客户的问题和推进产品。现实是云提供商比你更擅长管理这些东西！那么，你究竟为什么要浪费时间和精力去复制别人做得更好、更便宜的东西呢？接受你的约束，把你的努力集中在让你变得伟大的事情上。

如果你到了这一步，并且开始接受使用云的想法，那就太好了！但是说实话，你可能还在用错误的方式思考这个问题。虽然到目前为止我提到的所有好处应该都很清楚，但这仅仅是在您迁移到云之后所能实现的皮毛。

## 不仅仅是“别人的电脑”这么简单！

以上所有都集中在非常明显的好处上:成本、规模和维护。但是你可能已经注意到，我花了很大的篇幅来使用短语“在云上”而不是“在云中”,这是因为这两个概念之间有很大的区别。

一旦你通过了最初的步骤，那么一个全新的世界将会打开。随着工程师完全专注于解决实际的业务问题，而不是基础设施问题，你可以利用虚拟服务器以外的大量服务。

这两个概念之间的差别很微妙，但这里有几个例子:

*   在云上——你提供一个虚拟实例并部署你的应用。It 通过互联网处理请求，当您需要扩展时，您可以提供更多相同的实例，以便分散负载和提高性能等。
*   在云中——您只需提取您的应用程序逻辑，并将其部署到您的云提供商帐户内的一个功能即服务(FaaS)提供商。应用程序的扩展、执行和弹性由您的云提供商管理，您只需为客户实际提出的请求付费。
*   在云上——你的应用程序架构良好，系统不同部分之间的消息通过传递事件来处理，以保持系统的解耦。您可以部署更多的虚拟实例作为代理和队列来支持这个分离的系统。
*   在云中——为了管理您的解耦系统，您只需利用云提供商的消息传递和队列系统。能够处理数百万条消息，您的事件现在由本机解决方案处理，该解决方案旨在跨多个可用性区域扩展您的使用。

最终结果是一样的，但解决方案才是关键。在讨论成本和可伸缩性时，这又回到了同一点:任何人都可以将他们的系统迁移到云中。

任何人都可以获得成本收益:但这不是真正的价值。真正的价值不是成本；这甚至不是弹性。而是一旦你到了那里能做些什么。几乎每天都有新的服务提供给云提供商；从新数据库到机器学习，再到卫星通信，所有这些都是为了让你的生活更轻松。

停止关注与之相关的成本；接受现实吧。不要试图拿一个云提供商和你自己能做的事情做比较；接受他们能做得更好的事实。停止考虑你的服务目前的形式；接受它需要进化以满足现代需求。

集中精力做你能做的最好的产品，并利用一切能让它变得更容易的东西。没有人曾经使用过一个产品，因为他们喜欢数据库的配置方式或它所在的特定数据中心。

## *增加价值:而不是复杂性。*
