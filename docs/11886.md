# 编程现实与期望

> 原文:[https://dev . to/prot ICM/programming-reality-vs-expectations-5a1f](https://dev.to/proticm/programming-reality-vs-expectations-5a1f)

你好，我的开发伙伴。欢迎阅读另一篇文章。今天我们将谈论编程和它实际上的方式。你会注意到现实与预期不同。

如果你是一个刚刚开始职业生涯的新开发人员，你可能会认为你的工作将是整天都在编码，编写超级酷的代码，这将在野外产生影响。

事情不是这样的。为此做好准备。

编码不仅仅是写代码，而是解决客户的问题。我的意思是，如果任何代码不能解决某个问题，那么它又有什么用呢？它可能很好，可能很酷，但是如果它不能提供真正的价值，它对任何人都没有用。

# [](#starting-out)首发出场

一开始，当你开始工作时，特别是如果你是一个新的开发人员，编码任务，至少是严肃的任务，不会马上出现在你的盘子里。你的雇主或导师可能需要一些额外的知识验证。除此之外，他们可能还想看看你能多快多好地融入新环境。一种常见的做法是让你参与一些内部项目，他们可以监督你的工作以及你如何接受他们的做事方式。

每个公司都有一个或几个高级人员负责项目架构，这取决于公司的规模。多年来，他们已经形成了领导团队的习惯和方式，你必须适应。根据你的适应程度，任务难度会有所不同。

最后，他们将评估你解决某个问题的速度，你偏离项目架构有多远(有时按照规则来做并不容易，尤其是如果创建架构的人没有时间快速浏览它)，以及你是否明白你需要做什么？最后一点由前两点不言自明。

重要的是要强调，编程通常是关于**什么**而不是关于**如何**的。

# [](#working-for-the-client)为客户工作

### [](#requirements-and-specs)要求和规格

现在你可能会想:*“好吧，当我开始为实际的客户工作时，编码就开始了。”*

上面的陈述只是部分正确，因为这是真正的最后期限开始的地方。截止日期是需要完成的工作中最重要的事情。超过最后期限对你没有好处。我们可以将实施过程分为以下几类:

1.  阅读说明书
2.  思考
3.  想得更多
4.  也许更多
5.  编码

读完说明书后，会花很多时间思考如何解决所呈现的问题。换句话说，你必须明白实际上需要做什么。客户要求的价值是什么？一旦你成功地做到了这一点，你就可以开始编码了。

截止日期在这一点上被缩短了，您必须快速编写代码并使其尽快可访问。很可能，你不会满意你写的方式(只有有经验的开发人员才会满意，因为他们花更少的时间寻找解决方案，花更多的时间编写正确的代码)，但客户通常只看重最终结果，你对此无能为力。这将我们引向代码审查部分，这是极其重要的。

如果代码审查不是由某个项目高级人员完成的，这真的很糟糕，会出现很多问题，尤其是如果团队有很多成员。代码风格因开发人员而异，如果没有控制，每个实现都会不同。不仅如此，因为不知道架构中有什么可用，不知道另一个开发人员做了什么，您可能会重复自己的工作，重写现有的代码。

例如，另一个团队成员可能创建了一个扩展，在字符串突破一定长度后，在字符串末尾添加三个点。由于不知道它的存在，您将重写代码，而不是使用扩展。

### [](#bugs)虫子

为此做好准备。会有很多 bug。这意味着你将花费大量的时间去解释别人的代码，而不是创建一个新的代码。这可能非常令人沮丧。我不想打断你，但是维护阶段比创建阶段要长。这意味着代码被发布到产品中，用户正在使用它，而你所做的只是修复他们发现的问题。

您将修复与您从未见过且不熟悉的代码相关的错误。通常，有更多经验的开发人员看到部分代码可以改进，可能会想要更改它。由于实现和重新测试现有功能需要额外的成本和时间，这一愿望通常不会实现。很难向客户解释重构的过程，因为他们看到的只是最终的价值。他们不会从技术的角度来看待产品，通常很难理解“为什么我们要改变已经完成并支付的东西，以便再次获得相同的最终价值？”

作为一名有经验的开发人员，您会为此而挣扎，有时会被迫接受低质量的代码。

### [](#communication)沟通

[![communication](../Images/47c879e4320affc06398c0cfb15395c7.png)T2】](https://res.cloudinary.com/practicaldev/image/fetch/s--LKJl-VRC--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_auto%2Cw_880/https://www.toolshero.com/wp-content/uploads/2017/09/communication-cycle-shannon-weaver-toolshero-696x348.jpg)

啊，每日聚会的快乐。这是你要做的另一件事，而不是写代码。没有什么比得上上午 9 点的会议。你早上起来精神饱满，渴望写一些代码，但是如果遵循敏捷方法，你要做的第一件事是参加一个 scrum 会议。

在这个会议上，团队分享他们前一天完成的工作，关于项目需求的想法和想法...等等。它的一个优点是应该很短，不超过 15 分钟。这仅适用于日常会议。持续时间可以根据冲刺的长度和会议的频率而变化。你拥有它的次数越多，它就会越短。

你要做的另一件事是写/回复大量的电子邮件。通常，会议之间的交流是通过电子邮件或一些任务管理系统如 Jirra 或 Trello 进行的，但很多时候你会亲自与你的同事交谈，分享想法和想法。

所有的变更和对话都应该在某个地方记录下来。通常，每个特性或 bug 在管理板上都有自己的标签。这意味着每次您有问题或创建与项目功能相关的变更时，都需要更新票证。这可能很耗时。

记录部分非常重要。为什么？首先，你这么做是为了你自己。一段时间后忘记一些事情并不罕见。

想象一下，你正在做一个长期的项目，并且在没有正确记录变更的情况下实现了一个问题。一年半后，你会拼命抓头，以至于流血。如果在这种情况下被发现，要做好重新进行规范分析的准备，考虑到你可以在第一时间找到它。

其次，你是在为“下一个”人做。下一个人是在你因为某种原因不能继续工作的时候继续你的工作的人。

当一个特性被赋予我继续工作的时候，我曾无数次遇到这种情况，通常，我做的第一件事就是检查票据文档。惊讶，这是一个沙漠，什么都没有，死胡同...这张票只有原始的规格，没有别的了。没有评论，没有关于变化的更新...等等。

看到这个之后，我从椅子上站起来，喝了杯咖啡，有点不情愿地开始阅读说明书。之后，我开始代码分析和检查，找出“作者想说的”是什么。

# [](#wrapping-up)包装完毕

我不想打击你，我非常喜欢做一名软件开发人员，你也应该这样。我喜欢创建软件，喜欢看到人们实际使用它，因为它为他们提供了一定的价值。项目成功完成后的感受是不可替代的。客户的满意是项目结束后你能收到的最好的礼物。

你应该承认，作为软件开发人员或工程师，你的工作是解决某个问题，而不仅仅是整天写代码。这将使您的开发人员生活变得更加轻松。
