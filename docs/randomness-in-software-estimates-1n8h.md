# 软件评估的随机性

> 原文：<https://dev.to/pbeekums/randomness-in-software-estimates-1n8h>

构建软件时，评估可能很有用。抱有期望会让你立即感到安慰，但也有实际的好处。估计可以正确地确定工作的优先顺序。我们不仅致力于重要的事情，我们还致力于为我们投入的时间带来巨大价值的事情。

不幸的是，估计往往不准确。估计值越大，误差越大。一个 5 分钟的任务很少会膨胀到 5 天，但是一个 5 个月的项目很容易失去控制。我曾经在一次会议上讨论了一个涉及 3 个开发团队的 4 个月项目，以及我们如何得到精确到一天的估计。我指出在这个房间里有超过 100 年的软件开发经验。房间里有人见过这么大的项目有精确到当天的预算吗？没人回应。

一些人认为，如果开发人员足够努力或者足够好地发展他们的评估技能，他们可以获得准确的评估。然而，在一个大型项目中，误差是不可避免的。在项目评估中会有一定的随机性。

为了理解这种随机性，让我们看一下掷骰子。你可能认为单次掷骰子是随机的，无法预测。那不完全正确。任何人都可以完美地预测一对骰子的下一次滚动。你所需要的是:

*   模具的重量
*   掷骰子所用的力量
*   骰子的起始角度和高度
*   空气阻力的大小
*   与表面碰撞的角度
*   表面上的摩擦力

获取所有这些数据现实吗？也许有足够多的传感器和一个掷骰子的机器人。值得尝试去做吗？没有。然而，由于收集这些数据需要不合理的工作量，单次掷骰子似乎是随机的。

不管我们喜欢与否，完美地评估软件项目也需要不合理的工作量。对于需要单个开发人员花费几个小时或几天的任务，我们可能会得到准确的估计，但是对于有许多开发人员的几个月项目的估计会有如此多的因素，这将增加看似随机的不准确性。

这很大一部分是因为在构建软件的过程中不仅仅有技术因素，但大多数开发团队只考虑了技术因素。软件还不是由软件构建的。它是由人建造的。人们有各种各样的事情让他们慢下来。

最明显的就是疲劳。疲劳增加了开发人员引入 bug 的可能性，并增加了修复这些 bug 所需的时间。我认识一整个开发团队，他们花了一个长周末来修复一个 bug，结果却成了大小写敏感的问题。这些开发者并不是糟糕的开发者。他们刚刚经历了持续数月的死亡行军。

疲劳也会影响创造力，而创造力在软件设计中是必不可少的。我记得我绞尽脑汁了一个月才得到一个版本。最后一部电影花了我两天 14 小时。后来我设法睡了一会儿，但第二天早上回来时发现里面有各种各样的虫子。在用新的眼光看了我写的代码后，我意识到它有内在的缺陷。我选择了一种愚蠢的方法，这种方法需要大量的代码，而且永远不会真正奏效。我最终删除了我所有的工作，并在 2 小时内重写了一个工作版本。睡眠不足使 2 小时的任务耗费了 28 小时，并且生产出了劣质产品。

虽然开发人员的疲惫可以通过良好的管理策略来控制，但是身体疲劳并不是疲劳的唯一形式。我们在思考和感受人类。我们可能睡眠充足，但情绪上仍然感到疲惫。糟糕的心理健康很容易影响防止错误所需的警觉性和优秀软件设计所需的创造力。

虽然经理可以确保开发人员不会被迫在办公室呆上不合理的时间，但他们无法控制开发人员生活中的所有其他因素。他们的个人关系有问题吗？他们关心自己的人生方向吗？他们有需要克服的瘾吗？不管我们喜不喜欢，不管开发者自己想不想阻止，所有这些问题都会渗透到办公室生活中，影响工作效率。

团队成员之间也会发生冲突。一个项目中的人数越多，少数人相处不好的可能性就越大。不喜欢对方的人不太会妥协。他们不太可能确保对方知情。他们也不太可能进行富有成效的辩论。

软件设计是非常主观的，每个开发人员都有自己喜欢的方法。大多数运作良好的团队可以根据每种方法的优点进行合理的辩论。不喜欢对方的人会用固执来解决争论。会议时间会更长，解决方案会更差，这两者都会增加项目的时间。

大多数经理不会意识到这些冲突。告诉你的老板你不喜欢某人并不太专业。但是仅仅因为它不专业并不意味着人类不会有那种感觉。它只会被隐藏起来，并以一种不可预知的方式在工作中表现出来，这将增加项目评估的不准确性。

有一种观点认为，您可以通过合计估计值，然后应用乘数(如+50%)或将估计值乘以 3，来考虑所有这些隐藏的因素。单个事件是不可预测的，但是所有事件的集合可以做出准确的估计。我对这一点的主要反驳是，在金融危机期间，人们对抵押贷款支持证券提出了同样的论点。它在那里不太管用，在评估软件开发时也不太管用。

尽管所有的随机性都会影响项目所花费的时间，但是创建评估仍然是一项值得做的工作。我们只需要意识到估计只是估计，而不是精确的。项目越大，估计值就越可能是错误的。问题不是我们如何使这些估计更准确，而是当事情变得明显不准确时，我们如何处理。
