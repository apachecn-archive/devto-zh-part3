# 如何管理自由网络开发客户的期望

> 原文：<https://dev.to/raddevon/how-to-manage-freelance-web-development-clients-expectations-kln>

为你的潜在客户和客户设定明确的期望会让他们感到轻松。😌不知道从订婚中期待什么是焦虑的来源。😱人们想要制定计划，他们需要围绕你要为他们做什么来制定计划。

需要多长时间？费用是多少？他们应该期待什么结果？不知道这些事情，你的客户就不能有效地计划接下来的事情。这让他们感到失落和失控。直到你做了你的工作，他们才能前进，他们甚至不确定那个*是什么*。

这里有一些你应该设定期望值的关键转折点，以及一些让你的客户对明确的期望值放心的策略。

## 初次接触

如果你遇到对你的服务感兴趣的人，在你第一次接触他们的时候，对你下一步要做什么以及什么时候做设定一个明确的期望。当你开始这段关系时，这会让你的潜在客户感到安慰。当你真正兑现承诺时，你就开始建立一个可靠的记录。这会建立信任。

这里有一个例子。你在一次活动中遇到一个潜在客户。通过与他们交谈，你了解到他们使用两种不同的 CRM(客户关系管理)应用程序，因为每一种都有他们离不开的功能。不幸的是，这两个应用程序不能互相交流。每个客户必须输入两次。除了浪费时间之外，这也为人为错误敞开了大门，从而导致公司失去客户。

你认为你可以为他们解决这个问题，但你需要对这两个应用程序提供的 API 进行一些研究。要设定适当的期望值，你可以说，“我可能有办法帮你解决这个问题。让我做一点调查，我下周末给你答复。”这给了你的潜在客户一个清晰的预期，知道他们什么时候会收到回复。

第一次会面是给你的潜在客户一个大概的价格，或者从他们那里得到一个大概的预算的好时机。这里没有任何约束，你要确保潜在客户明白这一点。对于前面的例子，您可能希望从了解问题给他们带来的成本开始。

在解决方案更明确的其他情况下，您可以这样说，“我的服务从 x 美元开始，这在您的预算之内吗？”如果你想从等式的前景开始，你可以问这样的问题，“你能大概估计一下这个项目的预算吗？”如果他们拒绝，询问“是接近 10000 美元、10000 美元还是 100000 美元？”可能会让他们更开放一点。

如果你无法在客户的预算范围内为他们服务，那么一开始就说明这一点，可以节省每个人的时间并赢得他们的尊重。即使客户现在请不起你，他们也会对以后雇佣你的预算有所期望。

## 项目前

在项目开始前与你的潜在客户交谈，确保你们双方都同意项目的条款。对你和你的客户的期望保持一致。一旦你通过谈话解决了所有这些细节，确保你们双方**签署一份合同，正式确定所有这些期望**。

随着你的发展，事情无疑会发生变化。项目需求可能会改变。截止日期可能会错过。你正在使用的技术可能会改变。你的合同是项目前进的基线。

## 启动项目时

就在新客户支付了你的定金之后，他们感到非常脆弱。他们处境艰难:他们给了你一些价值(以金钱的形式)，而你什么也没给他们(或者相对来说很少)。这是一个很好的机会**让他们知道你下一步要做什么，他们何时会收到你的最新消息，以及你需要他们做什么**。

最后一点非常重要，要记住。作为 web 开发人员，我们倾向于只考虑对我们的期望，但是我们的客户也有责任确保我们的项目成功。没有他们的帮助，我们无法完成工作。

## 进行项目时

主要的期望应该在合同中列出，但是你应该在项目进行过程中重申它们。并不是每个期望都会在合同中列出，那些较小的期望需要随着你的进展而设定。“我会在这个星期五给你发电子邮件，”或者“我应该在明天结束第一阶段”，这些都是让客户知道他们的钱花得值的好方法。

## 当事情发生变化时

在某些时候，你会错过最后期限。让客户知道这将会发生的最佳时间不是截止日期的那一天，而是当你知道你将错过截止日期时的**。如果你在[审查你的客户](https://raddevon.com/articles/how-to-get-fewer-freelance-clients)方面做得很好，他们应该理解并感激你让他们了解情况。**

即使客户不高兴，给他们一些时间为错过的最后期限做准备也比在最后一秒突然告诉他们要好。下次他们需要网站开发者的帮助时，他们会记得你是如何处理这种情况的。

就像期望一样，变化不会单向流动。客户通常希望在中途对项目进行更改。当这种情况发生时，一句简单的“是的，我能做到”是不够的，除非变化足够小，不会对交易的任何条款产生实质性影响。在大多数情况下，**你需要围绕成本和时间表设定新的期望**。

## 当事情按计划进行时

当事情似乎进展顺利时，你会忍不住继续滚动。花时间交流你的进展会占用执行时间。但问题是:如果你认为一切都很顺利，但你的产品并不符合客户的愿景，该怎么办？你想在工作一周后发现这一点，而不是在项目完成时。

达到目标的方法是定期交流和展示你的进步。你可以通过电子邮件和截图来做到这一点，但我更喜欢展示现场，或者在不可行的情况下，分享一段屏幕记录。我用 [ScreenFlow](https://www.telestream.net/screenflow/overview.htm) 记录了我目前为止的工作演示。然后，我通过 Google Drive 与我的客户分享。

## 沟通！

管理期望的关键是沟通！多交流总比少交流好。这并不意味着你想分享每一件小事。考虑你的客户关心什么。你的客户不在乎你正在使用的 API 版本被弃用……*除非*这种弃用会导致你错过最后期限。

如果你选择通情达理并尊重你的客户，他们会很高兴听到你的消息，即使事情出了差错。他们将理解他们的选择不是在遇到问题的开发人员和没有遇到问题的开发人员之间，而是在直到最后一刻才隐藏问题的开发人员和尽早公开交流问题的开发人员之间。
