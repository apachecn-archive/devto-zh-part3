# “左移”测试如何帮助你的产品质量？

> 原文:[https://dev . to/lambdatest/how-shift-left-testing-can-help-your-product-quality-4 nm0](https://dev.to/lambdatest/how-shift-left-testing-can-help-your-product-quality-4nm0)

随着软件行业的发展，新的趋势和运营模式也在发展，每个“软件模式”都旨在为“软件开发”的每个阶段带来更高的效率。提高效率是当务之急，因为项目执行和交付中的任何延迟都可能会延迟产品的运输或发布。

最广泛使用的软件开发模型之一是“瀑布模型”，在这个模型中，软件开发生命周期中的所有活动，即计划/需求收集->软件设计->编码(开发)->产品测试，都是按顺序执行的。**'[* *瀑布模型](https://www.lambdatest.com/blog/agile-vs-waterfall-methodology/?utm_source=devto&utm_medium=organic&utm_campaign=jul19_rd&utm_term=rd&utm_content=blog)** '**的主要缺点是测试活动不是在每个阶段都进行；因此，bug(主要/次要)只有在“产品开发”完成后才被发现。

如果 bug 的严重性较低，开发人员可以修复问题并提交变更进行验证。如果严重性“非常高”,并且修复可能会有副作用，情况将会发生巨大变化。在这种情况下，向客户发布产品也可能会有潜在的延迟。在这样的模型中，测试阶段在 SDLC 的最右边，如果您的产品没有 bug，这种方法是理想的，但是任何产品都不会出现这种情况。

## 左移测试:授权给“测试团队”

随着不同软件模型的发展，例如敏捷模型、增量模型、螺旋模型等。；人们意识到了“软件测试”的重要性。保持“测试为一次性”活动涉及到大量的风险，因为它对项目的最后期限和成本有很多影响。这种认识产生了“转移提升”的概念，即将测试阶段转移到左边，并通过让测试团队参与对项目执行至关重要的活动来增强他们的能力。在早期的测试方法中(称为左移)，测试阶段在开发周期的末尾处于“最右边”，而在左移方法中，测试涉及到开发的每个阶段。

在传统的软件开发方法中，测试团队在竖井中工作，因为他们的主要工作是通过识别 bug 并向开发团队报告来提高产品质量。在产品规划和开发的重要阶段，测试人员很少或根本没有参与。有了“左移”的概念，测试团队更多地参与到产品开发的每个阶段。通过这种方法，测试团队可以与其他利益相关者合作，如开发团队、产品规划团队、产品开发团队、营销团队等。从而在开发的早期灌输“测试的心态”。由于测试团队有机会与来自不同团队的成员进行互动，这种增强的理解将有助于他们编写有效的测试用例，从而有助于提高产品质量。

因此,“左移”的概念不仅有助于在产品开发的早期阶段发现错误；但是它也帮助测试团队与其他利益相关者合作，提高领域能力，并提出更现实的测试用例。

## [](#major-benefits-of-shiftleft-testing)左移测试的主要好处

当您将左移测试作为 SDLC 的一部分时，您可能会意识到许多好处。在我看来，下面提到的是最有益的。

**降低开发成本&测试**

左移概念的创始人拉里·史密斯曾经提到“抓到小虫子便宜”。在典型的 SDLC 中，测试是在产品开发周期的末尾进行的。如前所述，修复 bug 所涉及的成本和影响会随着发现时间的推移而成倍增加。按照拉里·史密斯的说法，错误必须在变得不可发现之前及早发现。

一旦你“左移”，测试就被认为是产品开发每个阶段不可或缺的一部分。因此，每个构建都要进行一次测试，以便在早期发现并修复错误。一旦代码变得越来越庞大，修复简单的问题也会花费更多的时间&可能会导致一些副作用。左移测试策略可以降低开发、测试和修复的总成本，因为“错误总是在年轻时被发现”！

**早期缺陷检测确保更好的“代码”&“产品”质量**

左移方法确保项目的不同涉众之间有及时的沟通。开发人员可以合作开发单元测试和系统测试。自动化是左移测试的一个重要部分，有了自动化脚本，测试团队可以一天进行几次测试。他们的产品反馈以“错误”的形式再次反馈到产品开发漏斗，这有助于提高代码质量(以及通过测试用例&测试套件的开发来提高代码覆盖率)。

这导致生产阶段遇到的问题数量减少，并且生产阶段的“关键错误”可能会给团队带来“大破坏”，例如，*测试团队遇到由于一些数据库问题，他们的 web 产品上的注册页面无法工作，或者移动电话中呼叫场景的功耗远远高于相同价格范围内竞争对手的*产品。这意味着通过严格的代码质量检查，整体的代码质量将会提高，从而确保向客户交付一个“更稳定的最终产品”。

## [](#still-not-sure-about-shiftleft-testing-here-is-how-it-can-help-your-product-quality)还不确定左移测试吗？以下是它如何帮助你的产品质量。

“右移”和“左移”测试方法的根本区别在于测试团队参与了软件开发的“每个关键阶段”。从开发环境中的单元测试到 s [标记环境](https://www.lambdatest.com/blog/13-reasons-why-staging-environment-is-failing-for-your-organization/?utm_source=devto&utm_medium=organic&utm_campaign=jul19_rd&utm_term=rd&utm_content=blog)中的迁移，然后将最终代码推入生产环境。然而，实现这种方法说起来容易做起来难，因为测试团队需要走出他们的“舒适区”,并且需要每个利益相关者的心态转变，以便他们可以[与测试团队合作，成功执行项目。然而，你一定想知道左移测试是否真的能帮助你交付比以往任何时候都更健壮的产品，更好的团队协作&效率。

下面是一些主要的亮点，通过这些亮点，左移测试可以帮助你的产品质量做得更好。

1.  在“左移”测试方法中，测试团队参与**重要的项目讨论**，这使他们更加了解项目需求。在这个过程中，测试团队将收集大量关于项目计划&执行的知识。这将激励团队在日常活动中做得更好，因为团队每天都会遇到重要的学习。

2.  除了产品/项目规划团队和开发团队之外，没有多少成员意识到“业务目标”,因为他们从未接触过这些信息。通过“左移”测试，软件测试人员可以在确保他们的工作有助于**满足团队/组织的业务目标方面发挥至关重要的作用。**

3.  在今天，所有的注意力都集中在“上市时间”上，交付和执行时间变得越来越短。这就是 Agile & [DevOps](https://www.lambdatest.com/blog/automating-testing-backbone-devops/) 需要更多自动化来加速测试过程的地方。如果自动化脚本必须由测试团队开发，它可能会减慢“自动化”过程。使用左移测试，**测试团队将与开发团队**紧密合作，提出自动化测试用例，这些用例可以用来发现产品中的关键问题。通过遵循这样的方法，将会有改进的“跨团队的协同作用”，这是实现行为驱动开发(BDD) &测试驱动开发(TDD)的一个非常重要的方面。

4.  有一种普遍的误解，认为开发人员只需要开发代码。为了提高代码质量，开发人员应该树立“开发&测试”的思想，这样开发人员就可以自己发现并修复 bug。通过实现左移方法，开发人员可以通过同时关注开发和测试变得更加负责。这将有助于增强开发人员的软技能，这是变得更加熟练所必需的。

5.  参与自动化脚本实现的测试团队成员在技术上对多种测试框架和编程语言都很熟悉。然而，来自测试团队的成员从来都不是评审委员会的成员。一旦实现了 Shift-Left，测试人员就可以成为定期站立会议、代码评审和其他重要活动的一部分，这些活动是产品开发的一部分。这些知识在未来的项目中会很有用，因为测试人员可以想出更多的[测试策略](https://www.lambdatest.com/blog/why-you-need-to-understand-test-management-strategy-to-become-pro/)，测试计划&测试用例，因为他们参与了‘开发的每个阶段’。

当正确实施时，左移测试可以提高领域技能(开发人员、测试人员等)。)并改善团队内部的沟通，这对于任何项目的成功都是至关重要的。

## [](#which-shiftleft-testing-model-is-best-suited-for-you)哪种左移测试模式最适合你？

左移测试可以通过 4 种不同的方式进行:

1.  **左移传统测试:**传统的左移测试方法更侧重于单元级测试和集成测试。这是通过使用 API 测试和测试工具来实现的。它没有把更多的重点放在验收测试&系统级测试上。

2.  **左移增量测试:**增量左移测试广泛应用于具有非常高复杂性的项目中。为了降低复杂性，项目任务&可交付成果被分解成更小的部分。这种方法使测试任务变得有点简单，因为测试也是在那些较小的部分上执行的。这些部分是建立在彼此的基础上的，每一部分都在增加；软件也交付给客户。在每次交付之后，开发&测试被逐渐转移到左边。涉及硬件的项目被认为极其复杂，因为有许多依赖因素，即硬件、软件、中间件、应用层等。增量左移测试广泛应用于这类复杂项目中。

3.  **在 Agile/DevOps 中左移:**顾名思义，这种左移测试会在许多 sprints 中执行。它主要用于开发测试，而不是操作测试。敏捷/DevOps 左移测试慢慢流行起来，组织正在根据项目需求&进度使用这种左移测试方法。

4.  **基于模型的左移:**左移测试的整个概念是在为时已晚之前识别问题/错误。然而，在先前类型的左移方法中(到目前为止已经讨论过了)，测试将在开发周期的早期阶段开始。因此，在需求收集/设计阶段发现的问题，一旦软件准备好了，就会暴露出来。主要问题是将近 45~65%的缺陷是在需求收集阶段、设计阶段引入的。在基于模型的左移测试中，测试可以最早开始，以便在软件开发周期开始之前很久就报告错误。

总而言之，左移测试更多的是关于每个阶段的连续测试。转移到左移测试有很多好处，但是是否采用这种方法取决于项目类型和项目进度，这取决于产品开发团队。
