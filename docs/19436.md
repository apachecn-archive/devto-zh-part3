# IT 运营从脚本到开发运维再到开发运维即服务的演变

在本帖中，我们概述了基础设施管理从大型机计算时代，然后是企业计算到今天的云计算的演变。通过引用这一历史趋势中的细节，我们提出了一个案例，即服务于应用程序的最佳方式是通过基于开发人员需求自动生成配置的抽象，让开发人员直接访问基础架构。它必须制造(或雇佣)一个为用户服务的机器人。DevOps 应该关注机器人不能服务的特殊用例。如果开发人员经常需要操作员；机器人不工作了。

> 为应用服务的最佳方式是让开发人员直接访问基础设施。制造(或雇佣)一个机器人。如果开发人员经常需要它；这个机器人不工作。

从表面上看，it 需求似乎是基础设施技术发展的催化剂，但由于开发人员的生产力需求，根本的转变已经被接受。它服务于开发者生产力的意愿。不管安全理由是什么，如果他们在基础设施周围建造一个围墙花园，开发商将会离开。

## [](#enterprise-computing-it-operations-with-scripting-amp-cli)**企业计算:IT 运营与脚本& CLI**

90 年代末和 2000 年初，大型机计算迎来了辉煌。个人电脑很普遍，公司在内部部署软件。他们为了“生产力”而离开了大型机。

微软和 Sun 在使开发者能够使用。NET 和 Java。应用程序变得越来越花哨。在基础架构方面，VMware 凭借虚拟化颠覆了行业，Cisco 凭借网络，EMC 凭借存储技术。活动目录加强了微软对视窗系统的垄断。随着时间的推移，这些基础设施技术变得如此复杂，以至于成为一种利基技能。现在，IT 部门中有了服务器管理员、网络管理员、存储管理员、系统管理员和数据库管理员。从表面上看，人们并没有忘记基础设施的目的是更快地运送软件，这反过来意味着开发人员需要更有生产力。理想情况下，开发人员应该能够通过一个自动化底层细节的抽象层直接访问基础设施。但是它反而加强了控制。他们确实发展到使用越来越多的脚本工具，如 bash、power shell。他们通过厨师和木偶实现了更高的灵活性。然后出现了软件交付的 SAAS 模式，它再也跟不上了。开发人员将内部存储留给了云。“云计算”时代就这样到来了。同样的原因又是“生产力”。。

> IT 运营消亡的一个核心原因是，没有一家基础设施公司专注于让开发人员能够访问基础设施。

## [](#cloud-computing-devops-engineering-with-infrastructureascode)**云计算:基础设施即代码的开发运维工程**

云消除了物理基础架构的所有方面。以指数级的速度交付基础架构即服务，实施的复杂性大大降低。虽然 AWS 始于 IAAS，但事实上微软始于 PAAS，尽管这被证明是超前的，但他们调整了方向，专注于 IAAS。然而，愿景是明确的。

完全消除了服务器、网络和存储管理员各自为政的现象。通过代码提供基础设施是事实上的模型，并且比它的前身快得多。

> 虽然核心突破是由云本身的基础架构实现的，但运营工具的改进并没有带来突破性进展。开发人员仍然无法直接访问基础设施。

虽然核心中断是由云本身的基础架构实现的，但运营工具的改进并没有造成中断。地形，云的形成，厨师和木偶在这种情况下稍微好一点。另一方面，IAAS 云协调器专注于混合云，通过将所有云标准化到 IAAS，从而将数百个本机云服务(如 DynamoDB、SQS、SNS、Kinesis 等)排除在外。从根本上说，所有这些基础设施工具都不是供开发人员使用的。新环境需要几天而不是几周；连续交付是一种规范，但仅限于静态基础设施，其中只有应用程序代码(构建)发生变化。即使是最高效的公司，运营支出与资本支出的比率仍约为 1:1，例如，如果他们在 AWS 上花费 100 万美元，他们将拥有 6 到 10 名 Devops 工程师。它仍然是一个大的成本中心。

## [](#devopsasaservice-with-bots)**使用机器人开发软件即服务**

我们认为，IT 方法的根本改变不会来自那些以运营商为核心受众的公司。它将来自亲身经历过痛苦的云供应商或开发人员。

世界正越来越多地由机器人服务。他们驾驶我们的汽车，管理我们的商场和停车场。为 DevOps 开发机器人的想法有那么牵强吗？一个有趣的事实是，为了建立一个 Devops 机器人，我们甚至没有人工智能，DevOps 是一个用例，其中有一个有限的输入集和一个有限的输出集。计算机科学告诉我们，这可以通过规则引擎来实现。事实上，我的经验告诉我，大多数优秀的 DevOps 团队都在尝试构建他们自己的 bot。他们的需求是独一无二的吗？

> 开发机器人的想法有那么牵强吗？…..要建造一个 DevOps 机器人，我们甚至不需要人工智能..对于有限的输入集和有限的输出集，计算机科学告诉我们这可以通过规则引擎来实现。

将会有一些 DevOps 机器人来控制用例，一个由云供应商设置，另一个由云不可知的供应商设置，其版本将适用于每种云。基础设施即代码的人力 IT 将缩小到专门的用例，并发挥监督功能。
