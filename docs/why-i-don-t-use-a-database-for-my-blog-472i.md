# 为什么我的博客不用数据库

对于大多数开发人员来说，数据库是存储数据的首选方式，这是有原因的:它非常适合保存数据。那么，为什么我决定不在博客中使用数据库，而是选择 JSON/YAML/Markdown 文件呢？简单！可移植性、版本控制和性能。哦，学习新东西很有趣...

## 便携性

当我重新设计我的旧网站时，它仍然使用数据库。我已经两年没有做网站了。我从 Github 中取出代码，并试图在我的本地机器上启动它。我让它工作了，但是很明显，我没有任何数据让它显示。我没有在本地安装 MySQL，也没有找到一个好的理由来安装数据库引擎，下载数据库，并导入它，只是为了 5-6 个以前的工作记录和大约 30 个我无论如何都要替换的内容块。因此，我决定对之前的工作使用 Markdown，并完全去掉数据库。

这意味着无论我在哪里打开我的博客的本地版本，我都可以毫无障碍地获得我的所有内容。不需要外部系统，只需要一个包含一些内容文件的 Laravel 应用程序。这意味着我有一个一致的开发和生产环境，我可以在大约 2 分钟内在另一个地方建立一个相同的博客，而无需任何配置。

用文件代替 CMS 和数据库一起工作，允许我使用任何我想要的文件类型。我选择对我的内容使用 Markdown 文件。只需要关心标题、文本和其他基本内容类型的重要性就非常自由了。当我和其他 CMS 一起工作时，我总是觉得自己被 HTML 束缚住了。如果我想添加另一个段落，我必须使用一个很棒的编辑器来生成它，或者手动编写 HTML 元素。这变得非常乏味，慢慢地阻止了我创建内容。这很可悲，因为我喜欢创作内容，但我不得不经历的创作方式却让我失去了快乐。能够使用 markdown 并完全放开它让我重新获得了创作内容的乐趣。

## 版本控制

我的所有内容都保存在文件中，这意味着您可以将这些文件保存在某种版本控制中。这大概是这个项目我最喜欢的“特色”之一。我可以准确地看到我什么时候修改了我的帖子，就像你在 WordPress 中一样，但是没有任何数据库。对于 Git GUI，我有很多种选择，或者只是命令行，如果那是我当时想要的。我可以在任何支持 Git 的系统上编辑我的任何帖子，并且在我需要的时候可以在另一个系统上使用它。对你来说，这可能听起来像一个愚蠢的噱头，但我在任何时候都在 3 个设备上写文章。

## 表现

从数据库获取数据一直是我所有项目中最大的瓶颈。这可能是由于草率的查询设计造成的，但通常与您的系统正在为某些数据请求外部服务这一事实有关。即使数据库在同一台机器上，在获取和接收数据之间也会有一点延迟。当您拥有远程数据库时，您会立即注意到性能下降，因为数据是通过互联网连接获取的。对我来说，变数太多了，尤其是一个简单的博客。应用程序只需要读取数据并显示给用户，为此增加外部依赖似乎是不必要的复杂。

将所有内容与应用程序放在同一个存储设备上，使得读取数据近乎即时。它让你以你认为最容易使用的方式来写内容。我选择用 JSON 和 YAML 来编写一些配置，我可以这样做，因为我对自己决定保存内容的方式有绝对的控制权。你可以把它变得简单或复杂。这样，你就可以以自己喜欢的方式快速添加或更改内容。

## 学习新东西很有趣

如果我想做同样的事情，我会用数据库。但是那样我就会错过很多学习的机会。因为通过不允许自己使用数据库来限制自己，我学会了解析 YAML 文件和处理以其他文件类型保存的数据，并以我认为合适的方式使用它。我觉得我对自己的内容有绝对的控制权，无论我在哪个设备上工作，这非常自由，使创建内容成为一种真正的乐趣。
