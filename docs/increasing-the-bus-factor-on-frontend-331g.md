# 增加前端的总线系数

> 原文：<https://dev.to/steffenpedersen/increasing-the-bus-factor-on-frontend-331g>

ℹ️:这是我去年在媒体上写的一篇文章。

现在，当我写这篇文章时，我是我的团队中唯一的前端开发人员。这意味着我非正式地负责所有的前端，也意味着它没有被很好地测试。这听起来可能适合居住，但从长远来看，如果一个团队成员在一个特定的领域工作，这可能是危险的。

自从另一个前端开发人员找到新工作后，就一直这样。顺便说一下，他也是一名高级开发人员。现在所有的前端问题都有我的名字，这真的不是管理团队工作的最佳方式。如果我也找到了一份新工作呢？那么球队的处境就非常糟糕了。

通常，这被称为总线因子。如果你的团队中有多少人被车撞了，他们会让你的项目陷入困境。我们的前端和风格指南的总线因子大约是 1.25。这显然是一个奇怪的数字。当然不可能有 0.25 人。我的意思是，我最了解前端和风格指南，但也有一些团队成员研究过这个项目。顺便说一下，风格指南是一个定义所有前端的项目，也是 CSS 和 JS 输出的地方。

关于我们的前端没有太多的帮助或讨论，这让我很沮丧。考虑到这一点，我选择给我们的团队写一封电子邮件。我写了它，所以我肯定会得到适当的制定。

> 嗨伙计们，
> 
> 我想我们应该谈谈风格指南。
> 
> 自从 X 说再见，我就一直一个人在做前端开发和风格指南。如果我被一辆公共汽车撞了，团队就会丢失这个特定领域的知识。我们可以很容易地继续这样下去，但我不认为这对我们的团队、我在团队中的角色或我的进一步发展有好处。
> 
> 我认为一个解决方案是结对编程和更深入的测试。如果风格指南是主项目的一部分，这可能也会有所帮助。那么它就不会是一个异化的项目。这也许是另一个讨论。
> 
> 这并不像风格指南是一个完全不同的世界。如果你知道 HTML、SCSS、ES2015 以及架构 ITCSS 和命名约定 BEM——那么你就知道最基本的。
> 
> 知识共享很重要，尤其是当知识依赖于一个人的时候。
> 
> 考虑一下，我们很快会谈谈的。
> 
> 斯特芬😀

我们在 Sprint 回顾会上讨论了这个话题。这里我们定义了一些行动点，这是我们可以在接下来的 sprint 中改进的地方。在这种情况下，更多的是**结对编程**和风格指南的**展示**。其他团队成员也会做更彻底的**代码审查**。我也试图在前端开发上做更多的文档。我现在正在准备一个关于前端和风格指南的**工作坊**。

我很想知道你是否有过类似的经历。你是否也独自参与过某个项目的某个部分？
