# 网页无障碍设计

> 原文：<https://dev.to/vuild/web-accessibility-design-1db5>

注意:这些信息直接来自 W3，为了简单起见进行了重写。请访问 [W3](https://www.w3.org/WAI/tips/designing/) 了解更多信息。

尽你所能让你的网站为所有人服务。

网页可访问性是让尽可能多的人使用网站的艺术和科学。它包括设计和添加功能，以帮助盲人，部分视力和许多其他类型的残疾。可访问性应该在最早的阶段考虑，否则有一天你可能连自己的东西都用不上了。也有助于搜索引擎优化。

## 1。正文&背景之间的对比

文本颜色应该与背景颜色形成充分的对比，包括图像、背景渐变、按钮等。WCAG 要求的文本的最小对比度(亮度对比度)是 4.5:1。徽标、非活动用户界面、装饰或复杂视觉效果的一部分中的文本没有对比度要求。

### 提示

*   粗体小于 14 磅，非粗体小于 18 磅文本在纯色(黑色或白色)背景上显示时，对比度为 4.5:1。
*   粗体文本(或图像中的文本)至少为 14 磅，非粗体文本至少为 18 磅，对比度为 3:1。
*   图案化或复杂的背景、图形、图表、空文本字段等。所有这些都需要额外的对比度考虑。(例如:文本内容后面的阴影保持正确的对比度。)
*   文本/背景的红/黑组合也需要额外的对比度考虑。
*   淡色背景上的黑色文字(而不是白色)使得对比度不那么极端&对于一些用户来说更容易阅读。
*   如果由于某种原因无法满足对比度要求，应提供一个显著的链接或控件(具有足够的对比度),允许用户控制对比度。
*   避免使用包含文本的图像，使用 Unicode 文本和样式表。
*   理想情况下，文本和背景颜色应该保持“未指定”,这样它们可以完全由用户或他们的辅助技术来决定。
*   组件应该使用光谱的中间(绿色、黄色、橙色)作为浅色，外部边缘(靛蓝、红色)作为深色。
*   尽可能简化图标(线条画是理想的)&它们也应该符合文本的对比度规定。

### 它是如何帮助

最佳的对比度可以让所有用户更容易阅读网站上的文本内容，尤其是那些视力不好或难以分辨颜色的用户。

## 2。不依赖颜色传达信息

不是所有的用户都能感知所有的颜色(尤其是红色和绿色)，所以为通常使用颜色传达的信息提供替代文本标识是非常重要的。这不仅对视力受损的用户有益，对各种辅助技术也很有用。

### 提示

*   彩色表单控件标签也应该提供文本提示(例如:表单上的必填字段)。
*   使用一种颜色(及其替代品)来传达信息时要保持一致。
*   提供可选的(多余的)视觉提示(下划线、粗体、斜体、字体大小等)。)当文字颜色单独传达信息时。
*   当在图像或图形中使用颜色来关联信息时，应该使用图案来进行额外的区分。(例如:条形图使用图案和颜色来区分不同的条形值。)
*   当页面上的元素是交互式的时，应该使用具有明显视觉突出的悬停状态(不仅仅依赖于颜色)。

### 它是如何帮助

使看不到或无法区分颜色的用户能够访问与颜色相关的信息并与之交互。

提供单色或纯文本显示和触觉(盲文)界面，更好地访问与颜色相关的信息/与之交互。

## 3。交互元素应该易于识别

确保所有的交互元素都是高度键盘可访问的&当它们处于焦点时(轮廓，突出显示等)是明显可见的。).此外，为交互元素(如链接或按钮)使用独特的(一致的)风格和名称使用户易于识别和预测。

### 提示

*   在悬停状态或接收键盘焦点时突出显示链接和控件(可以在 CSS 中完成)。
*   所有的交互组件都应该有可选的文本解释，以及当它们被激活时会发生什么的信息。
*   大多数操作系统/辅助技术都提供了本机焦点指示器，用户可以操纵该指示器以获得最佳可视性。在一个平台上设计时(WordPress，Joomla 等。)理想地，聚焦指示器设置应该保持默认，否则它们会干扰用户的原始设置。
*   如果操作正确，还可以在内容中创建默认对焦指示器设置(更高的对比度、3px 黄色轮廓等。)以获得更高的可见度。

### 它是如何帮助

使通过键盘(无鼠标)操作页面的用户能够直观地确定页面上的哪个组件将随时受到键盘交互的影响。

为注意力、短期记忆或执行过程受限的用户提供清晰的交互焦点。

为依赖替代文本的用户创造更可预测的体验，也使他们能够在单独的页面上搜索一致的标签。

## 4。逻辑一致的导航

导航应该是合乎逻辑的&使用户能够理解他们在页面中或站点上的位置。页面应该提供清晰(有用)的标题，以及一致的布局、导航、命名、样式和定位等。

### 提示

*   提供网站导航的替代方法(例如:面包屑、网站搜索、网站地图)。
*   重复的多个组件应该总是以相同的相对顺序出现，即使插入了新的组件。(例如:菜单应该总是在相同的位置和顺序。)
*   在可能的情况下，整个页面布局、定位、分层和对齐应该在网站范围内保持一致。
*   使用页面模板维护页面和导航的一致性。
*   创建一个链接到站点上所有部分/页面的站点地图提供了站点的概览、系统化的导航&帮助用户理解/预测内容是如何组织的。
*   当添加或删除页面时，必须维护网站地图&如果它们没有链接到网站的所有部分，呈现的组织结构与网站的组织结构不匹配或包含断开/无效的链接，则作为“网站地图”无效。
*   为长而复杂的文档(尤其是如果它们跨越多页的话)创建一个(链接的)目录提供了一个文档概览&使用户能够跳到特定的部分。
*   添加搜索功能使用户能够在不需要网站导航的情况下使用特定的关键字来定位内容(在大型网站上尤其有用)。一个提供基本的纠正技术(拼写检查、词干分析等)。)增加可访问性。
*   对于小网站来说，在每个页面上提供一个链接列表是可以接受的(例如:页脚链接)。如果列表变得比页面的实际内容长，应该实现一个替代方法。
*   较小的网站也可以提供从主页到所有网站页面的链接(相反，所有页面都链接回主页)。
*   使用 link 元素进行导航为 HTML 页面的位置提供了元数据，并有助于定位一组页面中的特定内容。
*   目录和概念图也应该包括关于展示模式的信息。

### 它是如何帮助

页面之间组件的一致性使视力差/无视力、认知限制或智力残疾的用户能够准确预测站点元素的位置和功能。

多种导航类型使患有各种不同障碍的用户能够以最适合其特定需求的方式找到信息。

## 5。表单元素&关联标签

所有的表单域都应该提供描述性的标签&标签应该明显地与它们所标识的域相关联。对于从左向右阅读的语言，标签位于字段的左侧或上方；对于从右向左阅读的语言，标签位于字段的右侧或上方。

### 提示

*   包括对引起上下文转换的表单控件交互/变化的效果的清楚描述。(例如:网站顶部的语言转换单选按钮框应该有文字说明。)
*   表单应该遵循线性设计，相似的项目分组，并提供预期的数据输入格式的例子。
*   描述必要交互的文本说明应该在一个(小)表单或一组字段之前提供。
*   一般来说，标签应该紧接在输入字段之前(为了一致的对齐和易用性/位置)。
*   然而，复选框/单选按钮的宽度是一致的(它们的标签并不总是一致的)，所以它们的标签应该位于字段之后(这是一个通用的‘默认’)。
*   必填字段应标有符号或文字，表明其必须填写的状态。当符号或文本以编程方式与其字段相关联时，用户在第一次使用它之前会被告知其含义。
*   任何需要一组特定(定义的)输入值的字段都应附有文本说明，包括允许值/相似值建议的列表。
*   对于接受具有许多常见变量(日期、时间、电话号码等)的数据格式的字段。)提供偏好选项使用户能够输入对他们来说最容易的格式。
*   “label”元素可用于使用“for”属性将标签与表单控件相关联(表单控件“id”属性值必须=“for”属性值)。
*   可以使用带有图例(作为第一个元素)的“fieldset”元素定义相关表单控件组，图例提供标签或说明。嵌套字段集会造成混乱，应该避免。
*   每当较长表单中的一组控件需要额外的标题来进一步描述时，可以使用“字段集”和“图例”元素。(例如:一组较长形式的地址控制。)
*   默认情况下，大多数浏览器显示“字段集”元素，在分组控件周围有边框。虽然这种视觉提示对许多用户来说很有用，但是可以在 CSS 中修改它&使它更有吸引力。
*   还可以使用可访问的富互联网应用(WAI-ARIA)技术以各种方式添加描述性标签。
*   在极少数情况下，标签无法使用或可能会引起混淆，可以使用“title”属性来标记表单控件，以便辅助技术仍然可以读取它们。<small>*这种技术可以降低其他类型残障用户的可访问性。</small>
*   在某些情况下，用相邻的按钮标记输入字段是可以接受的，特别是如果这样可以减少文本冗余的话。用于标记单个输入字段的按钮通常跟在字段后面(例如:搜索框)。<small>*这种技术可以降低其他类型残障用户的可访问性。</small>

### 它是如何帮助

当与输入元素相关联时，当字段处于焦点时，屏幕阅读器会朗读标签元素。

靠近所描述的字段的可见字段标签使依赖屏幕放大镜的用户更容易理解内容。

预期数据格式的示例&明确标识的必填字段帮助有认知、运动技能、学习或语言障碍的用户准确输入信息。

写得好的标签对那些断章取义的用户来说是有意义的。)

## 6。反馈很容易识别&明白吗

需要用户操作的反馈(如确认、提醒或通知)应该突出显示&确切的操作说明应该易于识别。

### 提示

*   错误信息应该用简短的语言书写&与页面上的其他文本有明显的区别。
*   当表单成功提交时，总是在显著的位置向用户提供成功的反馈。
*   在可能的情况下，通过在输入错误出现时直观地强调它们来实时突出显示输入错误(使用客户端脚本)。这将触发一个带有错误文本描述的警告对话框(对话框关闭应该会将键盘焦点返回到同一字段)。
*   当用户输入不符合要求/允许的格式或值的信息时，提供文本反馈。
*   提供文本反馈，确定未完成的必填字段和说明/示例。
*   当用户输入的信息不正确时，提供一个纠正建议列表(拼写、格式等)。)可以有用。建议(或其链接)应放在需要更正的字段附近。
*   如果可能的话，表单应该接受所有通用格式的输入数据。
*   如果提交的表单有错误，它应该重新显示错误的轮廓，在页面标题中的通知&跳转(链接)到/在错误之间的能力。
*   WAI-ARIA 技术也可用于指示错误字段。

### 它是如何帮助

显而易见的通知使有认知、语言、学习障碍的用户&有视觉障碍的人在错误发生时能够意识到。

清晰的信息和纠正错误的示例允许有各种障碍的用户以最小的努力更新不正确的信息。

帮助在认知、语言和学习方面有残疾的用户理解由视觉线索表示的信息。

## 7。分组相关内容使用标题&间距

使用邻近和空白来确保内容之间的关系对读者来说是显而易见的。提供描述性的页面/章节标题，以便于浏览和理解内容。

### 提示

*   为页面和章节使用独特的标题。
*   将长的文本块分解成更容易理解的段落和带有相关标题的部分。
*   描述性标题概述了内容及其组织方式。这使得用户能够识别与页面上其他部分相关的内容部分&与页面本身相关。
*   在标题的早期添加重要信息可以通过辅助技术更容易地浏览内容。
*   应该对标题进行标记，以便可以使用(适当嵌套的)标题标签(h1、h2、h3 等)以编程方式识别它们。)来传达逻辑层次。
*   内容的每一部分都应该以标题标签开始。这不仅表明了主要内容部分的开始，而且还划分了导航部分&使用户能够快速地按部分导航。

### 它是如何帮助

使用描述性标题有助于阅读困难、视觉障碍或短期记忆有限的用户预测每一部分内容的有用性。

使行动不便的人能够通过有限的击键次数找到他们想要的内容。

写得好的标题对那些断章取义的用户来说是有意义的。)&在页面导航中为视障人士提供。

逻辑标题标签是用户快速浏览页面、预测组织和访问最近更新的内容的简单方法。

## 8。响应式或移动设计

网站信息在不同的设备/浏览器/窗口大小上显示不同。基本站点元素的定位应该充分利用空间&文本/行的高度应该最大化任何屏幕尺寸的易读性。

### 提示

*   内容丰富的网站可以提供简化内容的移动版本，以适应较小的屏幕/有限的空间。
*   响应式设计可以使用 CSS 简化移动内容交付。
*   按钮、链接和其他触摸控件应该足够大，以便在移动屏幕上轻松访问(即使对于那些行动不便或视力低下的人)。
*   短网址更容易阅读、输入和在较小的设备上使用。
*   在移动纵向布局中，表单标签应该始终显示在各自字段的上方。
*   强制特定的移动显示方向会使行动不便的用户难以使用&如果可能的话应该避免。
*   在响应式设计中，页面上组件布局的一致性尤其重要。
*   在页面滚动之前，重要的页面元素应该(一致地)显示在移动屏幕上，使行动受限的用户能够在不与页面交互的情况下访问和预测内容。
*   当多个元素执行相同的动作时，它们应该被组合用于移动设计(例如:文本链接和图标/图像链接组合在同一个可操作的元素中)。这不仅增加了触摸目标的尺寸，还减少了聚焦目标的冗余。
*   触发变化的任何元素(链接、按钮等)。)应该以不需要单独视觉检测(可通过辅助技术检测)的方式清楚地区分。
*   冗余编码的原理确保元素被指示为可由一个以上的特性特征(形状、颜色、风格等)操作。)
*   在移动界面中使用触摸屏和设备操作手势时，应提供说明(在显著位置，滚动前),解释手势控制以及可能的替代方案。用户应该能够在需要时重新访问这些说明。

### 它是如何帮助

让你的网站对移动设备有用对你的所有用户来说都是至关重要的(他们中的大多数可能会通过手机访问——你正在你的手机上阅读这篇文章吗？).

帮助行动不便或能见度低的用户在较小的设备上理解、预测和导航。

## 9。为非文本内容提供替代内容

任何媒体或非文本内容也应该提供一个用于相同目的的短文本。这包括图像、图形、视频、图形、图表等。对于长/复杂的内容，书面或音频抄本对许多类型的用户是有用的。

### 提示

*   所有图像都应该包含使用 alt 属性的可选文本。该文本应该简要地传达图像的含义，以及在图像中找到的任何文本。
*   为了更好的可访问性，可以省略图像标题标签，删除潜在的冗余/令人困惑的额外文本(注意:当图像处于悬停状态时，标题标签会显示文本)。
*   为没有文本的图标或按钮提供文本说明标签。
*   在显示表格、图形和图表之前放置标题或描述。
*   可以使用可访问的富互联网应用(WAI-ARIA)技术以各种方式添加描述性标签。
*   一起使用的非文本内容组应该为整个组提供一个文本选择&将它与一个分组项目相关联。组中的剩余项目应该被标记，以便辅助技术将忽略它们，限制冗余。(例如:已满/未满星级评定系统。)
*   当非文本和文本内容包含同一资源的相邻链接时，辅助技术的用户可能会感到困惑。将文本和非文本内容链接合并到一个元素中，并保留图像 alt 标签为空(这对于屏幕阅读器来说是无效的，消除了重复的文本)。
*   使用 alt 属性来标记 applet，并在 applet 元素的主体中提供一个替代文本(两者都是必需的，因为用户代理对 alt 属性和 applet 主体文本的支持各不相同)。
*   为视频的音频和视频描述文本的可视链接创建一个一致的位置。
*   辅助技术的用户不能呈现使用客观元素的内容。回退文本内容可以添加到元素的主体中&只有当用户代理不能(或者被指示不能)呈现目标元素时才可用。
*   对于使用屏幕阅读器的视障人士来说，ASCII 艺术可能会令人困惑&应该提供一种文本替代。在可能的情况下，跳过 ASCII 艺术的链接对于各种用户来说是理想的。
*   表情符号包括 ASCII 字符，这些字符也会使屏幕阅读器感到困惑&应该避免。当它们被使用时，它们应该提供一个文本选择(插件在一些博客和论坛软件上可以实现这个目的)。
*   包括 leetspeak 在内的文本内容也应该为屏幕阅读器提供一个纯文本选项。

### 它是如何帮助

难以感知视觉内容的用户可以使用辅助技术通过音频、盲文或简化的视觉“查看”内容。

除了对听力受损的人有明显的好处之外，文本描述对那些在视觉上理解图形、图表、照片等复杂信息有困难的用户也很有帮助。

文本替代也可以出现在网站搜索中，否则不会返回非文本结果。

## 10。限制自动内容

为所有自动启动的内容提供明显的控件&当页面加载时，播放时间超过 3 秒。这包括影响内容的移动、闪烁、滚动、自动更新、音频、视频和动画等信息。

### 提示

*   虽然在检测到辅助技术时可以关闭自动音频，但理想情况下，音频应该只在用户请求时播放。
*   在页面/章节阅读顺序顶部的明显位置提供(符合 WCAG 标准的)自动内容控件，并带有清晰的文本标签。
*   如果你的网站是自动内容为主，提供给用户一个明显的控制来禁用音频网站。
*   移动、闪烁或滚动信息，自动开始，持续超过 5 秒，与其他内容并行呈现，应该提供明显的暂停、停止或隐藏控制。
*   理想情况下，应该向用户提供停止移动(闪烁、滚动)内容的控件，即使移动在 5 秒钟后自动停止。
*   自动开始的自动更新信息&与其他内容并行呈现，应该提供明显的控制来暂停、停止、隐藏或控制更新频率。
*   移动或滚动内容应该提供控件，使用户能够根据需要暂停活动和重新启动。这可以通过(符合 WCAG 标准的)交互式控件或(记录的)键盘快捷键来完成。
*   应该为脚本创建的任何滚动内容提供一个容易访问的暂停机制。
*   辅助技术使用户能够停止某些类型的动画内容(通常通过按“退出”键)。图形交换格式(GIF)和动画便携式网络图形(APNG)技术通常不会干扰用户代理。
*   对于不容易关闭的闪烁内容，为用户提供一个明显的链接/按钮选项，用不闪烁的文本内容替代重新加载页面。替代页面应该提供与它所替换的闪烁内容的页面相同的信息。

### 它是如何帮助

自动音频干扰了屏幕阅读器的使用，尤其是那些为用户控制系统音量的屏幕阅读器。

对于阅读困难、智力障碍和注意力缺陷障碍的用户来说，停止或隐藏大声、闪烁或移动内容的能力非常重要，否则他们可能很难与页面进行交互。
