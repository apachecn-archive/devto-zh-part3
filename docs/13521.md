# 指导四个新开发人员的回顾

> 原文:[https://dev . to/darr aghor/a-回顾-指导-四-新-开发者-4n3j](https://dev.to/darraghor/a-retrospective-on-mentoring-four-new-developers-4n3j)

去年年底，我有幸帮助一个学生团队在奥克兰为一个非盈利组织建造了一个原型。非营利组织需要一个原型来筹集更多的资金，并获得对他们计划的反馈。

我在大型组织中帮助管理团队和指导初级开发人员。这要容易得多，因为组织的支持、工具、框架、系统已经可用。对于这个项目，学生团队有一张白纸，必须决定从如何与赞助商沟通到如何托管软件的所有事情。

我想写下我所学到的东西，并记下我所犯的一些错误，以便我下次帮助新开发人员时有一个框架！它可能对管理新团队的其他领导或为客户工作的新开发人员有用。

## [](#0-years-experience-but-heaps-of-enthusiasm)0 年的经验可是成堆的热情！

总共有四名学生分成两队。学生们在他们的最后一年，已经建立了一些网络项目作为课程工作。一个人维护了一个面向公众的网站，但除此之外，他们没有与客户或顾客合作过。当我第一次见到这些学生时，我完全被他们积极的态度打动了，并决定马上提供帮助。

整个团队对软件开发充满热情和激情。他们非常关心自己的社区，所以选择从事非盈利项目。他们都有强烈的动力和职业道德——每个人都做兼职工作(有时是两份！)除了大学和这个项目。

我扮演了一种技术顾问的角色，每周大约花 4 个小时在这个项目上。我认为我的角色是帮忙

1.  赞助商得到一个工作原型
2.  学生们努力取得好成绩

我已经确保一些最佳实践到位，并指导一些技术选择。当学生们真的被某件事卡住时，我也会帮一两次忙，但这种情况很少见。他们很擅长谷歌搜索和解决问题。

## [](#setting-the-scope-and-expectations)设定范围和期望值

赞助商希望在三个月内完成尽可能多的工作，学生们对开发时间非常乐观。我帮助做的第一件事是限制范围，并为开发团队和赞助商设定现实的期望。

发起者在 Excel 中创建了一个全面的规范，在每个特性项上都有一个“必须”和“可选”标签。该产品将是一个市场，具有你在 Shopify 等成熟服务上预期的所有客户和管理功能，学生们同意在三个月的周期内完成大部分功能。

我只是不认为这对一个新开发团队来说是现实的。

我们应用了一些精益原则来找出要测试的最重要的特性，并按优先级对它们进行排序。这种有限的功能集只会让非营利组织进入下一步，而不是建立一个功能齐全的市场。

*   如果你的开发团队非常新，他们会对自己交付高质量软件特性的能力持乐观态度。就像，比正常的开发者乐观多了:)通过教他们承诺不足和超额交付来帮助他们。
*   据我所见，开发团队没有考虑到诸如研究和创建基础设施、设置工具、CI/CD 之类的事情，这些都是你在开始之前就需要的。这将花费他们在构建应用程序功能之外的大量时间。
*   下一次，我会在一开始就更具体地说明需求和质量期望。确保赞助商和团队就将要交付的内容达成一致。

## [](#communications)通讯

在成熟的组织中不会有任何沟通规范，所以你必须建立它们。项目发起人希望每周和学生们聚一聚，叙叙旧，这再好不过了。我们最初通过 Skype 和聊天组织我们的会议，但很快就变成了定期安排的谷歌会议。大约每两个星期，资助者和学生将在大学见面，回顾进展。

我们也在 slack 上做了很多交流。#tech 有一个不包括赞助商的独立的松散渠道。有些讨论需要赞助商的意见，这些意见必须转移到#general channel，但上下文要少得多，如果赞助商能够看到所有讨论，他们会对工作、进度和复杂性有更好的了解。

对于像这样的小团队来说，每个人都在#tech 频道会更好。

*   通过视频会议链接建立定期会议(只要使用谷歌日历，如果你可以的话)。Skype 不太适合组织人们。
*   建立一个聊天室(懈怠或不和什么的)
*   不要在一个短项目中把“技术”聊天分开。时间太宝贵了，所以让每个人听所有的聊天吧！

## [](#technology)技术

大多数 web 项目需要源代码控制、托管、后端技术、前端技术和数据存储。我建议学生们在决定技术之前做一些研究，这样他们就可以查看

*   鉴于他们即将毕业，什么样的技术是有需求的
*   他们已经经历过的
*   对于给定的技术，可以在线获得什么样的帮助

学生们已经做了一些工作。Net，并且非常想使用 JavaScript。所以最初的讨论是围绕一个带有 React 前端的 DotNetCore 后端。

学生们很快决定 DotNetCore 不适合他们，他们将使用 node 和 React 开发完整的 JavaScript。我不认为有很强的理由不使用 DotNetCore，只是一种偏好。两个都工作得很好，所以我不太在意那个。

学生们也想使用 MongoDb。他们已经在很多教程中看到了它的使用，但是我不认为一个无模式的数据库会和一个全新的、无纪律的团队一起工作。由于 Heroku 也提供了 Postgres 的免费实例，我建议我们使用 Postgres 和带有迁移的 ORM，这样每个人都可以恢复模式的精确副本。这在后来救了我们几次。

为了节省时间，我给学生们设置了一个工作启动程序，这是我从一堆其他启动项目中拼凑出来的。该项目有

*   GitLab 上的 CI/CD
*   部署到 Heroku
*   反应组分的例子
*   节点 rest APIs 的例子
*   基本结构
*   促进地方发展的备忘录
*   设置迁移和播种的序列
*   Postgres 会话存储

你可以在[https://gitlab.com/darragh.oriordan/starter](https://gitlab.com/darragh.oriordan/starter)看到起跑器。此后，我添加了 typescript、大量测试和 graphQL (Apollo)。

我们使用 GitLab 而不是 GitHub，因为当时 GitHub 上没有 CI/CD，也没有免费的私有库。这两个现在在 GitHub 上都有。他们以前没有使用过分支模型，所以我认为他们是 gitflow 模式。我让 master 成为一个受保护的分支，所以他们必须执行 pull 请求来添加代码。Heroku 将自动从主服务器部署。

我最初做了一些代码审查来设定一些标准，并建议他们在那之后互相审查代码。我不认为这发生了，但这不是一个大项目。如果您想要强制执行评审，那么将评审接受作为合并的一个要求。

我没有添加任何类型检查系统，也没有添加任何自动化测试的例子。所以最终产品也没有这些。由于遗漏测试和基本类型错误，有许多回归。

*   让团队选择他们自己的技术，除非它绝对行不通。
*   针对他们的经验水平和您自己以后解决问题的时间，提供合理的防护栏
*   PaaS 将比 IaaS 容易得多。这将让团队马上开始开发有用的特性。
*   下次我会在初始代码库中有更多常见软件功能的例子。单元测试、端到端测试、代码结构的例子。当有好的例子可以学习的时候，你可能会惊讶于他们能建立什么。
*   不是所有的学生都有强大的开发机器，所以要考虑到这一点。Visual studio(完整版)在学生笔记本电脑上可能会很慢，因此 vscode 更好。如果学生没有 windows professional 许可证，他们就不能运行“完整”的 windows docker，他们必须运行一个包含 docker 的虚拟机。
*   他们可能没有使用迁移、ORM、源代码控制分支，所以找出并解释
*   CI 是我推行的最有价值的东西之一。在他们的机器上运行的 CI 服务器上有很多不完整的构建。绝对有 CI！
*   我将启动分为前端和后端项目，但一个单一的解决方案。我认为这是令人困惑的，因为他们通常一起部署。这意味着有三个 package.json 文件，您必须位于正确的目录中，才能在正确的位置安装给定的包。

## [](#infrastructure)基础设施

学生们将使用他们大学的私有云来托管应用程序。他们会得到一台空服务器来安装和使用。对服务器的访问(甚至 http)只能在学校内部网络上进行。考虑到这个限制和短时间，我建议 Heroku 会更好，因为…

*   有 0 个服务器配置
*   非生产/试运行是免费的
*   dpl 有众所周知部署模式
*   赞助商可以随时随地在互联网上看到进展
*   赞助商可以随时向顾客展示产品

## [](#building-the-product)打造产品

迭代运行良好。这些团队需要在他们设定的时间内完成大量工作。主环境和舞台环境始终保持在可用状态。赞助商从第三周开始就有了一个产品，随着学生们在功能上的融合，这个产品逐渐得到了改进。赞助商花了大量的时间为学生们做反馈，这些反馈都被学生们筛选、修正或放弃了。

学生使用库进行身份验证(Auth0)和支付(Stripe)。这对他们来说非常有效。它节省了开发更多定制解决方案的数周时间。仍然有相当多的工作要做，比如写邮件、发送邮件、将 auth 连接到后端、前端和数据库。付款很顺利。

我们的团队中没有设计师，所以这个网站看起来不专业。在我看来，这是一个大问题。我应该建议赞助商做的第一件事就是和设计师合作。这也会给赞助商一个很好的纸质原型展示给潜在客户。这将省去整个项目中的假设和可能的讨论。

桌面布局之后增加了移动支持。这导致花了大量时间来修改布局。我应该建议移动优先的方法。如今，许多网站的移动使用率最低为 60-70%，所以你需要拥有它。不妨从约束条件入手。无论如何，设计师可能早就发现了这一点。所以先雇个设计师:)

学生们没有调试经验，所以我有时不得不帮忙。这可能是你可以在你的新的初级开发人员陷入困境之前经历的事情。比如如何读取控制台日志，在源文件中定位文件中的某一行，以及如何设置断点。我还教了他们从一个模块中删除所有代码，然后一段一段地添加回去，直到它再次崩溃的方法:)

*   在编码之前有一个设计，而不是建议他们复制现有的市场，让发起人选择一个市场，并让团队尽可能多地复制它。但是认真的先有一个好的设计。
*   SaaS 产品通常有非常简单的例子，让它们看起来非常容易使用。现实情况是，您很可能需要做大量的工作来进行集成。仍有配置和拷贝需要添加到 SaaS，这需要时间。
*   确保及早考虑移动布局。
*   确保赞助商已经向客户提供了书面模拟，以获得一些反馈。
*   尽早教开发人员如何调试

## [](#great-results-but)成绩斐然但……

学生们完成了开始时商定的所有较大的功能块。为了按时完成项目，做了一些小小的妥协。最终产品中也有一些错误。我还没有听到什么样的客户反馈，但这个设计可能不会很好，如果客户期待一个真正完美的体验。

然而，学生们在他们班的项目中得到了一些最好的分数，对我来说，这是一个巨大的成功。

总的来说，这里出现的主要问题和任何软件项目一样。

*   在编码之前确保解决方案解决了问题(伟大的设计和 UX 将在这里提供帮助)
*   大多数技术选择都有优点和缺点。你只能忍受一些烦人的事情。
*   最终遵循良好的实践是值得的，比如 CI/CD、自动化测试、静态类型、来自利益相关者的反馈的迭代开发。即使在一个短期项目中，它也会回报前期的努力
