# 工程经理:是时候写更好的职位描述了

> 原文:[https://dev . to/cohere/engineering-managers-its-time-to-write-better-job-descriptions-1c 26](https://dev.to/cohere/engineering-managers-its-time-to-write-better-job-descriptions-1c26)

软件工程师的离职率是所有行业中最高的，他们的平均任期为[2 到 3 年](http://blog.indeed.com/2017/04/03/silicon-valley-tech-job-migration/)。为什么？因为 *[人离开的是管理者，而不是公司](https://www.forbes.com/sites/victorlipman/2015/08/04/people-leave-managers-not-companies/#a564b7347a94)* ，正如妮可·桑切斯和达尼洛·坎波斯指出的那样，[科技行业充斥着糟糕的管理者](https://twitter.com/nmsanchez/status/935654372276453376)。我也是其中之一！

一个糟糕的管理者的部分原因是缺乏对日常管理活动的投资，以及过分强调从业者的日常活动。这些日常管理活动之一是了解正在做的工作，为什么要做，以及如何做。因为这是一个不断调整的答案，所以花时间向潜在的新员工和现有的团队成员传达这些内容是很重要的。除此之外，我们还能指望别人理解对他们的期望吗？或者在你的团队或更广泛的组织中，成长轨迹会是怎样的？沟通的一种方式是写工作描述。

依赖*职位头衔*作为工作描述的代理是非常非常诱人的。但是，当我们主要依赖职位头衔时，我们会含蓄地考虑三件事:我们寻找的技能，这些技能的水平，以及我们想要鼓励或阻止的行为。这种缩减瓦解了上下文，我们最终发布了关于“*‘高级’到底是什么意思？”*或者推特上宣称“ *devops 不同于站点可靠性工程！*”。这种与未来雇员或团队的沟通通常不会发生。

为了传达你所寻求的必要技能、水平和行为，写一份工作描述，明确写出在*你的团队或部门背景*下，对于给定的职称活动，什么技能、什么水平是必要的。

“啊”你咕哝。“我看了我的工作描述，最重要的一句话是‘分配的其他任务！’“而且你说得对。很多职位描述都很惨。它们是在超负荷的工程经理的胁迫下编写的，他们试图满足人力资源，同时克服领导工程团队实现组织目标的困难。

因此，职位描述通常是一系列与多年经验相关的技术术语。这是获得某种程度上有效的职位描述的最简单途径。此外，它还允许招聘团队撰写职位信息，并对候选人简历或简短电话进行通过/失败分析。

我们可以做得更好，我们现在和未来的员工和团队成员*应该得到更好的*。他们应该得到 ***的功能性理解，了解每天对他们的期望，以及他们可以期待什么作为增长技能的回报。***

那么我们如何写出有用的职位描述呢？那些以有意义的方式向团队成员准确传达期望和职业轨迹的？

## [](#write-better-job-descriptions)写出更好的职位描述

写职位描述时， ***显性比*** 隐性好。描述你日常使用的行为、活动或技能。。不要依赖“计算机科学基础”来传达“将数据从事件流转换成可报告的格式”

第二， ***使用面向成长的语言，而不是面向凭证的*** 。当然，你可以用“计算机科学学位或相关经验”作为简写，但你真正想要的难道不是“已经向生产 web 应用程序交付了特性和错误修复”吗？

第三， ***避免对人的价值判断*** 。“优秀的工程师会及时地执行代码审查，而不会成为一个混蛋，”虽然值得称赞，但过于强调**演员**，而对活动**重视不够。相反，“通过及时提供具体的、可操作的和富有同情心的反馈来执行代码评审。”**

第四， ***去掉无关的东西。*** 试图将作为团队中的高级开发人员意味着什么的每一个细节都写进你的工作描述中，可能会鼓励对次要活动或技能的投资，这些活动或技能很好，但不是核心。相反，留下足够的空白，让你的团队成员可以用他们构建团队的独特而美丽的方式来填充空白。如果这些独特的技能最终变得非常有价值，它们甚至可能会让你写一份新的工作描述。

### [](#describing-behaviors-and-activities)形容行为和活动

当描述你的工作描述中的活动时，把重点放在该角色的人花费大量时间的活动上。例如，如果你期望你的工程师在测试自动化上投资，你将会包括一个“编写自动化测试来验证团队权限下的特性用例”的行为

此外，包括*为什么*执行活动或鼓励行为是有用的。虽然“在业务和产品团队的指导下交付特性和 bug 修复”对于开发人员来说是一个不错的开始，甚至对于工程师的第一份或第二份工作来说也是可以接受的，但它未能鼓励开发人员理解并按照特性的动机行事。这导致了一种“我按照别人说的去做”的心态。相反，考虑“通过交付与产品和业务团队的方向一致的特性和错误修复，朝着业务目标取得可识别的进展。”

我们已经明确了额外的责任，即正在进行的工作*推动了切实的进展。*现在，一名团队成员对如何评估他们以及如何调整自己的行为以支持评估有了更好的想法。

### [](#describing-skills)形容技能

技能甚至比行为和活动更难客观描述。这是大多数职位描述和职位发布失败的地方，也是你如何获得要求 [8 年以上 swift 经验](https://twitter.com/kevinkaywho/status/898622008019124224)的职位列表，我鼓励使用*流利模型*来描述技能。

流畅性模型关注的是利用特定技能所需的脑力水平。技术组织最大的成本是*时间、*、思想和*努力。明确时间、思想和努力与你正在培养的技能之间的关系，会让你关注组织能从高技能员工身上得到什么。*

流利程度与*人*关系不大，与他们的*能力*关系更大。这有助于我们更有效地分离价值判断，即“伟大的反应开发者”相反，我们可以更具体地说，相关前端开发人员“需要积极的支持来设计和实现具有数据或事件相互依赖关系的 React 组件。”它避免了对人的评判，并设定了一个会得到支持的期望。对于更高级的团队成员，相应的“支持其他团队成员设计和实现具有复杂数据或事件相互依赖关系的组件”可能是合适的；还会向您的助理级团队成员发出信号，表明他们可以通过支持周围的人来推进自己的职业生涯。

大多数人都熟悉流畅这个概念。当我们问关于流利度的问题时，我们给了人们用故事交流的机会。故事让人们更加真实地了解这个人的实际技能。问题“在 React 中实现一个特性时，您遇到的最后一个挑战是什么？”更容易提问和回答，并为后续问题提供了更丰富的领域，以便深入研究并形成更微妙的观点(或发现可能的废话)。

## [](#now-write-a-job-description)现在写职位描述！

现在花 30 分钟写一份工作描述。在你的角色中，你每天都做些什么？你最常用的技能是什么？你的角色与或多或少的高级角色相比，流畅度的优势在哪里？
