#### [T1】简介](#intro)

如果您是一名 PHP 软件开发人员，那么您目前正在开发的所有应用程序都极有可能使用任何类型的框架。所有级别的 PHP 社区开发者都崇拜框架，因为这有很大的历史和现实原因。

#### [](#historical-reasons)历史原因

从早期的 PHP 版本开始，开发人员就声名狼藉，因为不是每个人都认为 PHP 是一种编程语言，就像几年前的 JavaScript 一样。虽然强类型语言在几十年前就存在了，但 PHP 从现在开始一直是软类型，只是在版本 7 中引入了基本类型。还有一个事实是，你可以不用一个对象就用 PHP 编写脚本。但是这为框架的介入打开了一个空间，并把它们自己作为一个工具或标准来介绍，这个工具或标准将塑造项目，赋予它们权利和秩序，引入结构和规则。最后，他们做到了。框架是很好的结构的例子，使用 PHP 在每个版本中提供的所有可用的新特性，实施一些好的实践，等等。

#### [](#practical-reasons)现实原因

该框架提供了许多已经解决的常见问题。他们为其他开发者提供了一个很好的生态系统来贡献和插入他们的组件。有很多在线资源可以学习和了解任何特定框架的最新信息。此外，每个框架社区都在努力使设置和使用变得容易。

#### [](#when-php-frameworks-sucks)当 PHP 框架烂透了

最近，我有机会在一个会议上做了一个关于为什么 PHP 框架有时很糟糕的演讲。有时候，我们在框架教程中看到的东西似乎与我们正在努力实施的一些面向对象的标准和基本的干净代码指导方针不太一致。另一方面，使用一个框架并没有什么错，如果你正确地使用它的话。

本文是本系列的第一篇“试验性”文章，用来衡量 dev.to 上的读者有多少乐于以博客系列的形式了解这个主题。在本系列的每一篇新博客中，我们将更深入地探讨我在演讲中涉及的每一个具体话题。我很高兴分享这一知识，因为我看到许多开发人员患有绑定到框架的疾病。
