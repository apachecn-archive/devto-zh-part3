# 视野有限的领导

> 原文:[https://dev . to/jessekphillips/leadership-with-limited-perspective-18gf](https://dev.to/jessekphillips/leadership-with-limited-perspective-18gf)

转变为领导角色。这是一个相当有趣的挑战。我不仅要学习一套新的技能，而且要留下来做大部分的工作，在没有自己权威的情况下指导别人。

我是专业的软件测试人员，不是开发人员。即便如此，我的测试经验来自一家公司。我处于领导的最佳位置，但缺乏被视为知道如何有效管理代码的整体经验。

我对接受挑战和学习感兴趣。

我必须采用高度定制的平台，并大规模测试每个定制的实现。我不能指望一个实现能跟上构建定制实现的组件的发展。

我们利用 npm/nuget 打包来管理注入定制。这就需要有一个解决方案将这些包组合在一起。

我已经通过版本控制的旧概念提出了分支和维护更新的使用。我还提到过，子文件简化了我们用包管理器解决的许多挑战。这导致了对陈旧的编码实践的指责，这也是每个人都转向包管理的原因。

现在，我完全认为 nuget 和 npm 增加了我们交付的复杂性，所以让我们忽略我只谈论如何处理实现回购。

架构并不依赖于代码是如何构建和打包的。这意味着你可以有一个高度解耦的代码，即使它在同一个 repo 中，我甚至认为用正确的语言在同一个文件中也可以实现。我们面临的挑战是管理变更和交付变更，我们希望用包来解决这个问题。Sem 版本当然是做出不破坏任何东西的改变的解决方案(记住我是 QA，所以这意味着我要测试那个特性)。

对我来说，git 是用来管理变更、交流变更的，我认为如果使用得当，它有助于减少差异。我在之前的帖子中提到了前两个。将完整的源代码放在一个位置可以实现深度耦合和不受控制的变更冲突。

这是我的经验不足的地方，我如何创建一个系统或过程来促进好的架构，使管理变更更容易。是的，它可以被浓缩成旧的，在版本控制之前尝试过的。但是我的研究表明，这就像 cpan 对于 Javascript 一样新。我也知道我的历史。

这些想法并不是我反对开发者，我确实得到了一些开发者的支持，并且在这个方向上正在取得进展。
