# 问:“科特林方式”有多少是正确的？

> 原文：<https://dev.to/preslavrachev/q-how-much-of-the-kotlin-way-is-the-right-way-3129>

*注意:这篇文章最初是为了引起讨论，尽管它很长，但最初的目标仍然是一样的。请分享您的反馈*

* * *

我对 JVM 的体验可以追溯到 2010 年左右。从今天的角度来看，当时的 Java 有点不同。Java 6 是当时的标准，但是我参与的大多数项目都停留在 Java 4 和 Java 5 之间。那些都是强悍的野兽。编写代码更加冗长和乏味。

那时，我们开始寻找一种方法来减轻编写代码的负担，但是仍然依赖于 JVM 的广泛采用。Groovy 和 Scala 是两种选择，它们用较少的样板文件和仪式提供了更多的语言表现力。而且，虽然 Groovy 不知何故从未被视为一种更适合构建编排和运行时应用内管道的脚本语言，但 Scala 看起来像是 JVM 大胆而性感的未来。

也就是说，直到该语言及其社区在意识形态上与 Java 渐行渐远。尽管当时大多数现有的 Java 库都是兼容的，并且可以从 Scala 运行时访问，但这些库被认为是过时的，使用起来很乏味，并且根本不是以“Scala 方式”构建的。这个口号被用来证明构建一个全新的工具、库和框架的生态系统是正确的，它是用 Scala 从头开始编写的。可以肯定地说，如今对于每一个 Java 库，你都可以找到两三个由*“Scala 之路”*编写的 Scala 等价物。我不是 Scala 专家，但我认为这导致了两个社区的完全分离，每个社区都以自己的方式重新发明了轮子。

科特林应该帮助解决所有这些问题。JetBrains 加入这个游戏的时间已经够晚了，它把赌注押在了 Kotlin 与标准 Java 代码的互操作性上。为了实现这一点，一门语言必须被视为一种伴侣，而不是一种真正的范式转变。在我与 Kotlin 一起工作并跟踪其开发的三年时间里，只有少数情况下互操作性没有达到 100%,这些或多或少都是边缘情况。似乎最终达到了中庸之道，多种语言范式可以共存，他们的社区合作，以实现一个共同的目标。

然而，事实并非如此。我已经看到了一个 Kotlin-first 开发者社区的诞生，宣扬一些东西*“kot Lin 方式”*。很快，工具、库和框架开始出现，在 Java 中有完美的工作等价物，但是用*“惯用的 kot Lin”*编写。我开始想我以前在哪里见过这个🤔。

不要误解我的意思，我 100%支持挑战现状，但要在已经行之有效的基础上更进一步，而不仅仅是抛弃它，因为它不是按照*“XYZ 方式”*建立的。
