# 撤回讣告

> 原文：<https://dev.to/codemouse92/retraction-of-an-obituary-19n>

我有一个通知...

**Javascript 已死！**

[![Graveyard](img/0ec0af52e91ac8f75aec2abeb4d0ba1c.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--7bi4rrP8--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/9ggh5hbiur0tqrcky8qx.gif)

好吧，现在，说实话。你们中有多少人瞬间想到“哦，天哪，那我应该用什么来做我的项目呢？”

请放心:Javascript 是*而不是*死亡。你可以毫无畏惧地继续做你的 JS 项目。但是那句偏离主题的话产生了影响。想象一下，如果本文的其余部分建立在这个前提之上。人们会如何回应？

在过去的一年里，我听到了一些令人震惊的死亡宣告。Perl 死了！C++已经死了！Java 死了！灯灭了！巨蟒死了！

我们在编程领域有一个巨大的问题，它不是平台死亡率。

# 定义“死”...

我们经常使用这个术语，但我不认为我们中的很多人真正理解“死亡”意味着什么。也许韦伯斯特先生能帮我们。

> 被剥夺了生命；——反对活着活着；降低到一种状态，在这种状态下，运动和生命的器官已经不可逆转地停止了它们的功能；如，一棵枯树；一个死人。

我想诺亚的意思很明显。**死了就是...死了。不生长，不生活，不可行。**什么都没发生。当你死了，你就是虫子的食物。

[![Worms](img/02d2dc1cc8d7691d713f29c9dd676b64.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--wVWVFbYk--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/l2r4sqho30ny2yeqwiu7.gif)

也许这将有助于为你提供一个死亡的技术例子...

[![Visual FoxPro](img/c8e33a7f6bc2229d62b2ebc066a42a0f.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--SBtF5cnG--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/vm0w279us12tj6k9nrtx.gif)

Visual FoxPro 已于 2007 年停产。截至 2015 年，已达到支持结束。无论你多么想，今天都不可能构建和发布一个 Visual FoxPro 项目。 **Visual FoxPro 已死。**

[编辑:我也要撤回那篇讣告！正如 Herb Wolfe 在评论中指出的，今天不仅可以用 FoxPro 开发，而且它有一个活跃的社区。我甚至找到了一个[开源克隆](http://foxincloud.com/opensource.php)。惊喜惊喜！]

现在，相比之下，让我们看看最终促使我写这篇文章的说法。“Ruby on Rails 已经死了。”据他所说，除了少数少数边缘用户，没有人再用 Ruby on Rails 构建任何有意义的东西了。让我们想想这个...

*   DEV 实际上是基于 Ruby on Rails 构建的！
*   Rails 在 2018 年 12 月 4 日刚刚发布了 5.2.2 版本
*   他们的官方 GitHub 上现在有 718 个开放的拉取请求。
*   2019 年春天在明尼阿波利斯将举行一场真正的铁路会议。
*   您当前查看的网站是基于 RAILS 构建的！(值得重复。)

Ruby on Rails 连一个“死”的标准都达不到。我可以为 Perl、C++、Java、Python、LAMP 等等提出类似的论点。

甚至像 C、FORTRAN 和 COBOL 这样的老语言实际上也不能被认为已经死亡。几乎所有的现代操作系统都支持它们，并且它们在各自的领域中发挥着强大而持久的作用。如果你怀疑我，走进一个 Linux 开发人员的房间，大喊“C 死了，amirite？”(专业提示:当你戴头盔的时候。它会保护你免受岩石的伤害...虽然不是嘲笑的笑声。)

# 死去...至少对我来说

显然，编程社区并没有以任何理智和合理的方式使用“死亡”这个术语。

当我们说“X 已死”时，我们通常是指“X 不流行”，也就是说“X 不是最耀眼的新事物。”对我来说，这让我们听起来像一群喋喋不休的中学生。

“比如，你看到辛迪穿的是什么了吗？那条裙子就像是上周的。我绝对不会邀请她参加我这个周末的派对。”

[![The cool girls](img/99f8ec8ffc7eeb2a92dd9e5d045a1cb2.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--atzsZPbF--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/6nua33gme9rhdi1rb69b.gif)

当然，问题是这种评估纯粹基于大众的主观意见。当我们鹦鹉学舌地说“Ruby on Rails 已死”时，我们可能只是引用了某个随机的程序员群体的观点，他们的注意力就像一只幼年的金毛猎犬。

[![Squirrel!](img/0d695b55df08dc7ad5a1b24c5eaeb268.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--fH0Et32I--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/6mo5pk6pvv8u0dpttfbe.gif)

当然，有时候，我们并不是真的将“X 已死”的论断建立在任何大众观点的基础上...我们是基于自己的想法。我听说很多人声称“C++已死”，仅仅是因为他们不喜欢这种语言:手动内存管理、指针、编译...它让他们得了荨麻疹。这很好，他们不必喜欢这门语言。

然而，就像 Ruby on Rails 一样，C++非常活跃，非常好——c++ 17 最近刚刚推出。它也是我最喜欢的语言之一。我将它用于我发现它非常适合的用例。

所以实际上，这些人说的是“C++已死”...给我。我讨厌它。我希望它会消失。”

[![Dead To Me](img/06059d176aefd7345bcac167f219325b.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--k7mhHAlP--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/f8lzcady6ggmhax2pbxf.gif)

就我个人而言，我讨厌 Javascript，我希望它能消失。但是即使 20 年后，如果 Javascript 不再受大众欢迎，我可能也不会写任何关于“Javascript 已死”的文章，因为它不会！

但是说实话，**技术普及了又有什么关系**？如果它满足了项目的需求，如果它使开发人员能够解决他们的问题并将可行的软件交付给他们的目标受众，如果它为他们提供了他们个人认为有成效和有回报的开发体验，**谁会在乎**其他人对此有什么看法？

# 有什么意义？

那么，这到底有什么关系呢？如果我们都清楚“死”是什么意思，那我还在说什么呢？

很简单:**文字很重要！不是每个人都知道我们中学生对“死亡”的定义，我们也不应该强迫他们接受这个含义。**“死亡”有一个精确的、有意义的定义，它正确地阻止我们利用定量上不可行的技术。****

还记得我在文章开头宣称 Javascript 已死吗？即使你*知道*这是一项可行的技术，你也可能(尽管不是绝对)有过短暂的情绪反应。沟通和人类心理之间的联系很奇怪，但是*非常*重要！就像你不会选择雇佣一个图形设计师，如果你被告知他或她已经死了，就像你不会买一只鹦鹉，如果你知道它已经死了，那么使用一项你已经被告知已经死了的技术就感觉是错误的。这变成了精神减速带。

[![Dead parrot](img/7dfc2909242ce81835e81fe3ba27e0de.png)](https://res.cloudinary.com/practicaldev/image/fetch/s--_Zo9Mv5f--/c_limit%2Cf_auto%2Cfl_progressive%2Cq_66%2Cw_880/https://thepracticaldev.s3.amazonaws.com/i/wv055h1b1cdnl0d0a6qf.gif)

换句话说，当我们基于流行度或个人观点称一项可行的技术“死亡”时，我们实际上是在*操纵*他人做出符合我们突发奇想的技术决定，不管我们是有意还是无意！即使另一个人知道我们流行的“死了”的错误含义，他们仍然会有心理障碍。

换句话说，宣布一项可行的技术“死亡”实际上是在人们为他们的项目选择最佳技术的道路上设置了一个心理障碍。即使你的意图是好的，通过这样做，你实际上认为你比他们更适合为别人的项目做技术决策！

# 不受欢迎的好处

我以前写过关于编程的[时尚，所以你可能还记得当我说...](https://dev.to/codemouse92/all-the-cool-kids-are-doing-it-dm8)

> 一项创新只有当我们开始大规模采用它而不是它本身的价值时，它才会成为一种技术时尚！

当一项技术不再受大众青睐时，它会在最适合的用例中扮演更稳定的角色。它将从因*受欢迎*而被采用，到因*适合*而被采用。正在进行的开发，以及该技术的社区，将会更好地关注该技术的优势，通常是因为忽略了该技术不太适合的用例*的优点。*

作为历史上的例子，看看 C、FORTRAN 和 COBOL。这三者在当时都非常受欢迎，人们试图用它们做任何他们能做的事情。后来，随着它们的光芒逐渐褪去，大众转向闪亮的新领域，这些语言开始了它们最适合的用途。

即使在今天，C 仍然是大多数系统开发的基础，提供了一些最可靠的“裸机”访问。FORTRAN 是科学计算的支柱，NASA、CERN 和许多其他组织信任其最关键的代码。由于其速度、可移植性和高超的数学准确性，即使在今天，COBOL 也参与了大多数金融交易。

这些语言不仅仅是由一些过时的边缘开发者和狂热爱好者编写的。它们是真实的，专业上可行的技术。

在未来的某一天，C 可能会让位于 Rust，FORTRAN 让位于 R，COBOL 让位于，老实说，现在没人知道会发生什么。循环总是在继续，没有什么是永恒的。

但是严肃地说，**语言并不经常消亡**。即使是备受嘲笑的 Ada 也活了下来，其最新的稳定版本将于 2016 年 2 月发布。开源的优势越来越明显，这进一步保证了大多数语言一旦建立，很少会消亡。

三十年后，我敢肯定那些酷孩子会嘲笑 Python、Javascript、Rust、Haskell、Go 和 Java 开发人员坚持他们“死”的语言。

但是，老实说，我们的工具受欢迎与否又有什么关系呢？只要飞利浦螺丝在我当地的五金店有售，我就会继续使用我的飞利浦螺丝起子。事实上，我保证三十年后我会修理一台全新的高端电脑，取出大约两打十字螺丝。

然而，没有人再写文章称赞菲利普斯螺丝了(至少，据我所知没有)，但这没有任何区别。他们工作。

“这是否符合我的项目需求”是选择技术时唯一重要的问题！

# 它还活着！

关于鲁比·安·赖尔死亡的报道被大大夸大了...以及几乎所有类似的“X 已死”的宣告。

我们对彼此都有责任，尤其是对跟在我们后面的新手，不要让彼此的工作变得更困难！你不必喜欢每一种技术，但是不要把你对一种语言的看法和它的可行性混淆起来，然后再去迷惑另一个开发人员，看这种语言是否值得她诚实、客观的考虑。

那种语言还没有消亡。*这种语言*非常活跃，有一群聪明的开发者用它来开发有价值的、可行的软件。

所以不要试图告诉我们这是一个前鹦鹉。我们对用它换子弹不感兴趣，不管它有多闪亮。
