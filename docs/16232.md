# 换队！

> 原文:[https://dev.to/kpath001/switching-teams-31n](https://dev.to/kpath001/switching-teams-31n)

去年九月下旬，我决定借此机会离开我目前工作的团队和项目！我已经在这个项目上工作了两年，我熟悉的同事也离开了这个团队。我看到了这个让自己重回面试常规的机会。决定留在我现在的雇主那里，尝试不同的部门/项目/软件栈。在我的后赛季，我学到了一些有价值的东西:网络，项目，团队文化。

### [](#networking)联网

我的雇主中的一个老大学团体告诉我，他的团队中有一个职位空缺，他正在他们的团队中寻找一名高级开发人员，但我应该申请这个空缺职位，看看我是否能获得晋升。看到这个机会，我决定与我的队友分享这个空缺职位，并提交自己的晋升申请。几天后，我收到了第一轮面试的邀请，其中包括一些行为和经验问题，以及对项目经理和项目技术负责人职位的介绍。在第一轮之后的几天内，我收到了项目团队经理和业务经理的第二轮邀请，在那里我们愉快地讨论了播客并练习了我的推销技巧。结果几天后，我得到了朋友推荐给我的职位，但没有得到提升，而是平级调动。这整个过程真正让人们意识到拥有一个人际网络对一个人的职业生涯有多么重要。你网络中的人越多，你需要的时候就有越多的机会。

### [](#project-stack)项目堆栈

我以前的团队使用 Java Spring 框架为遗留内容管理系统(CMS)开发了一个 Create Read Update Delete(CRUD)RESTful API 微服务，利用 Pivotal Cloud Foundry 将微服务部署到生产和测试环境。在团队中，我学会了如何高效地将应用程序部署到生产环境中，如何利用框架开发 RESTful API，以及开发/集成 pivotals gemfire 缓存服务来提高内容交付速度。没有一项技能真正转移到我当前的项目中，该项目要求我在 Amazon Web Services(AWS)Elastic MapReduce(EMR)集群上用 python 开发分析系统。但是我喜欢过渡到一个新项目的一点是，我正在为一群新客户解决一个新问题。伴随着新问题而来的是一种新的心态和新的方法，它真的让我走出了自己的舒适区，并让我从一个新的角度看待一个已经存在的问题和解决方案。

### [](#team-culture)团队文化

我转型的另一个影响是从一个大团队到一个小团队，这对我的技能有很多好处:

1.  与更多的开发人员结对编程，以更好地理解项目架构
2.  熟悉团队，了解谁是项目各部分的联系人

### [](#in-conclusion)总之

发布我的第一个专业项目是一次有趣的经历，教会了我很多。来自与他人交流的惊人机会，问题的改变增加了我解决问题的方法，增加了我在职业生涯中可以使用的工具的效用带，最后，当我从一个大团队跳槽过来时，较小的团队及其创造的文化真的让我感觉像家一样。我鼓励每个人都采取这种跳跃来转换项目，尝试一些新的东西，看看你能从不同的东西中学到什么。
