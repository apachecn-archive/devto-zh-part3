# 作为项目经理，我能做些什么来帮助我的工程团队？

> 原文:[https://dev . to/grant gumina/what-can-I-do-help-my-engineering-team-as-a-pm-5a 88](https://dev.to/grantgumina/what-can-i-do-to-help-my-engineering-team-as-a-pm-5a88)

嘿戴夫托，

我想听听你对我在大型公司和初创公司担任项目经理时观察到的以下情况的看法。我想这个网站上的大多数人都是开发人员，他们可能会帮助我理解我在工程团队表现下看到的以下行为。我很想知道我能做些什么来更好地帮助球队获胜！

在我做项目经理的 4 年里，我有幸与一些真正优秀的工程团队一起工作。我也和一些不太出色的人合作过。伟大的团队通常是伟大的，因为我们都在充分沟通和相互信任，以实现分散决策。当一个团队成员失去同步时，他们会很快提高速度，如果他们不同意已经做出的决定，也不会惊慌失措。部分原因是因为团队在个人层面上正在形成合力，但也因为有先例表明，当发现新数据时，会重新审视决策。

对于我们奋斗过的团队，我注意到以下失败模式会导致我们的失败。

首先，有一种模式，团队要么非常初级，要么生活在一个有掩护自己文化的组织中。不管怎样，人们害怕承诺日期、目标，甚至设计决策。如果一个团队成员真的对敏捷感兴趣，我会听到很多这样的话，“我们是敏捷的——我们不约会”或者“做出这样的承诺不是敏捷的”。我完全被客户痴迷的迭代开发所吸引，但是在这种情况下，像这样的评论让团队成员看起来像是在使用敏捷教条来实现他们的病态。

因为技术团队成员不想做决定，所以我最终不得不花大量时间和整个团队一起让每个人达成共识。恐怕由此产生的作品是委员会思维设计的产物。

最糟糕的是，在我们应该达成一致后，团队成员似乎突然失忆了，即使我发了一封电子邮件总结我们后来达成的一致。接下来的一天或一周，我们将处于站立或复古状态，有人会震惊地得知客户正在期待 2019 年 3 月的“功能 X”。

所以，我的问题是:作为工程师，你见过或经历过陷入我所描述的模式的团队吗？如果是的话，项目经理是如何促成这些问题的？
