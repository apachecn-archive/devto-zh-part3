# RubyMotion 与 Swift 在 iOS 开发中的对比

> 原文：<https://dev.to/ibmdeveloper/rubymotion-versus-swift-for-ios-development-lno>

当我在 IBM Watson 和 Apple Core ML 的合作中工作时，我受到启发，去探索更多以不同方式向开发者展示机器学习的方法。我的一个想法是研究如何将 RubyMotion 与 [IBM Watson](https://www.ibm.com/watson/) 一起使用。

如果你不熟悉的话， [RubyMotion](http://www.rubymotion.com/) 可以让开发者使用 [Ruby 编程语言](https://www.ruby-lang.org/en/)快速编写 iOS、Android 和 OS X 的跨平台原生应用。应用程序被静态编译，然后调用本地平台 API。

作为一名 Rubyist，当 RubyMotion 在 2012 年首次发布时，我非常兴奋。我曾想探索为 iOS 开发应用程序，但 [Objective-C](https://en.wikipedia.org/wiki/Objective-C) 对我没有吸引力，学习这种语言很有挑战性。因此，当 Swift 到来时，这从根本上改变了等式，因为现在开始为 iOS 开发要容易得多。对许多开发人员来说，Swift 更容易使用，但功能依然强大。

作为 iOS 开发的主要语言，Swift 的快速发展极大地降低了用 Ruby 编写 iOS 应用程序的需求和吸引力。在这一点上，似乎主要的好处是一次写入的方法来获得 iOS 和 Android 支持。然而，在这一点上，有更受欢迎和广泛支持的跨平台移动开发选项，如 [React Native](https://facebook.github.io/react-native/) (使用 JavaScript)。

然而，我从一开始就看到的最大问题是，要为每个操作系统使用各种 API，开发人员仍然必须阅读和理解 API(用 Objective-C、Swift 或 Java 的相应语言编写)。因此，虽然你可以用 Ruby 编写大部分代码，但你并没有远离学习和理解这些 API 的母语。因此，除了对技术的好奇之外，我从来没有确信 RubyMotion 在这一点上的实际总体节省。

此外，如果你在谷歌上搜索 RubyMotion，你不会找到很多最近的结果。因此，虽然这看起来像是一个创建教程和其他内容的机会，但我认为更重要的原因是 RubyMotion 是在只有 Objective-C 是 iOS 开发的一个选项时构建的。既然 Swift 正在成为主要的 iOS 开发语言，对替代语言的需求就大大减少了。

除非 RubyMotion 有一些重大的改变(相对来说，它最近才被新主人拥有)，否则我对它在 Ruby 开发者中获得任何流行都不乐观。

我希望我是错的，因为我喜欢用 Ruby 编程，但我会更多地关注跨平台移动开发的原生反应。
