# 调查-你什么时候停止承担责任？

> 原文：<https://dev.to/scotthannen/survey---when-do-you-stop-taking-responsibility-56d6>

让我通过描述四个场景来澄清一下，在这四个场景中，您将承担一些软件开发的任务。这可能是一个或大或小的任务。第四种情况是我感兴趣的。前三个只是为了对比。

1.  您要么决定如何实现需求，要么给出您同意的方向或约束。(如果没有给你这些具体的指示，你可能还是会做同样的事情。)**你觉得要对结果**负责，比如它是否正常工作或者是否按时交付。当然，团队觉得集体有责任。(我们可以称之为“快乐之路”)

2.  你更喜欢用一种方式工作，但是你被一个经理、团队领导或团队成员指示用不同的方式工作。你不同意这种方法，但你不是固执的万事通。你明白他们的方式是有效的，或者你不完全明白，但你相信别人明白。你可能会学到一些东西。你以一种正常的、非责备的方式对结果负责。

3.  你被指示去做一件极其错误的事情，这件事情注定会失败。你有资格拒绝或反击，你也确实这样做了。

4.  你得到你知道不好的指示，你提出你的反对意见，然后你还是照做了。这可能是一个非常糟糕的技术方法，使用复制粘贴的代码，错误的工具，错误的库，没有单元测试，或者任何类似的事情。你预期的负面后果可能包括技术故障、技术债务或重大延误。你不觉得自己应该对结果负责。如果它不起作用，耗时太长，或者用户讨厌它，你希望给你指示的人承担全部责任。不是你想指指点点，但如果真到了那一步，你会的。

* * *

第四种情况可能会引发各种反应。我对它感兴趣是出于你所谓的研究目的。

最终结果无关紧要。如果失败了，是其他人最终承担责任还是你受到指责都无关紧要。这是团队责任的对立面，这是显而易见的，也是无关紧要的。

问题来了(终于！)

你有没有经历过第四种情况，在这种情况下，你开发软件(大到一个应用程序，小到一个类或方法)的方式你认为是不正确的，因为有人要求你这样做，而你遵从了他们(而不是你)对结果负责的期望？

重点不在于结果或谁该负责，而在于当你开发软件时，你是否觉得自己有责任。

如果你只想回答是或不是，或者“是，几次”，那太好了。如果你想详细描述这个场景，那也很好。如果你不想公开分享，你可以私下联系我——我在 gmail.com 的名字。

重点不是判断。我先来。我的答案是肯定的，我经历过场景#4。例如，有人告诉我复制/粘贴/编辑代码，我知道这些代码是不可理解的、不可维护的、错误百出的，会让未来的开发人员做噩梦。我不得不开发一些我知道用户会讨厌的功能。有时候我错了。我通常会提出异议或与团队分享担忧。有时环境使这变得不切实际。如果问题依然存在，我就去找其他工作。但问题是，有时我会按照别人告诉我的去做，我觉得如果事情严重出错，我可以说，“是的，我明白，但这不是我的决定。”我不觉得自己有责任。。

我计划写更多关于这个的东西，但是我想从收集一些观点和理解开始，而不仅仅是我自己的经历。
