# 周五爆炸#61

> 原文:[https://dev.to/horia141/friday-blast-61-1pp7](https://dev.to/horia141/friday-blast-61-1pp7)

[API 网关正在经历一场身份危机(2019)](https://medium.com/solo-io/api-gateways-are-going-through-an-identity-crisis-d1d833a313d7) -我喜欢这篇文章，尤其是因为它很好地概述了各种被称为“API 网关”的东西——后端的前端、Kubernetes 访问管理器、API 基础架构即服务产品等。

[优步的大数据平台:100+Pb，分钟延迟(2018)](https://eng.uber.com/uber-big-data-platform/) -看看优步“数据系统”的演变和现状。疯狂的东西！这也提醒我们，一旦你达到一定规模，就没有现成的解决方案了。你必须使用现成的和定制的东西来制作适合你的东西！

[不要在登录表单上耍小聪明(2019)](https://dev.to/scottw/dont-get-clever-with-login-forms-3499-temp-slug-7190786) -一堆很好的登录表单设计模式(如 UX 设计)。最突出的是*被链接*。这是所有 web 应用程序的主流都应该有的。我也没有考虑到漂亮的 UX 会给支持人员带来问题——例如，如果没有登录页面的链接，他们很难告诉客户在哪里登录。

[开放位置代码(2018)](https://github.com/google/open-location-code)——谷歌地图最近弹出的[加号代码](https://plus.codes/)的[设计文档](https://github.com/google/open-location-code/blob/master/docs/olc_definition.adoc)。看着这样一个大问题如何被解决真的很有趣，他们有什么限制(使代码清晰，使代码简短，处理投影失真，处理缩放等)..)以及他们是如何解决的。另一个选择是[what 3 words](https://what3words.com/)——它有一个完全不同的方法。

[我不想学你的垃圾查询语言(2018)](https://erikbern.com/2018/08/30/i-dont-want-to-learn-your-garbage-query-language.html) - SQL 仍然是数据库交互的通用语。大多数查询语言都基于它。但是作者指出，不应该有这么多简化的变体，相反，人们应该尝试在他们试图公开的任何内容上提供 SQL。同样，这也是无 ORM/微 ORM 运动的一个很好的论据。
