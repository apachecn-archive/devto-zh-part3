# Git 是一个通信工具

> 原文：<https://dev.to/jessekphillips/git-is-a-communication-tool--2j9k>

历史上，人们对 git 允许人们改变历史的方式有很多抱怨。关于使用 rebase、merge 和/或 squash 的讨论还在继续。我想把重点放在所有这些选择上，交流。我不想提供新的建议，有人已经写了为什么你应该做这样或那样的事情。

你想和谁交流？

*   未来的自己
*   其他开发者
*   质量保证成员
*   项目管理

你想交流什么？

*   一个特性是完整的(此时尽你所能)
*   这一代码更改是修复 bug X 所必需的
*   我有这个重要的文档更新
*   我正在做一些工作，我的编辑决定改变这些文件，我不知道为什么。
*   我不喜欢这份文件的格式，以下是我的建议
*   我完成了 X、Y 或 Z，并认为让这些上游对每个人都有利是很重要的。
*   我们能否将 visual studio 项目文件升级到。网？

作为一名开发者，如果这就是你开发作品的方式，那不是很好吗？美好清晰的目标，只做一件事的良好纪律，忽略你发现的错误，因为它不是手边的任务？嗯，不，这样不好，你会忘记那个 bug，或者你会忙于记录需要修复的 bug，你会忘记你在任务中的位置，把那些改变放到代码库中，然后继续前进。这就是为什么 Git 的历史重写如此有价值。
