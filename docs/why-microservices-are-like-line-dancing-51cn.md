# 编排

排舞是一种编排好的舞蹈，舞者排成一排，同步起舞。

就像服务需要由 Kubernetes 等总体系统来编排一样。

# 学长

看来排舞很受高年级学生的欢迎。当你有一大群高年级学生在一起，试着开心地跳排舞是有意义的。很像微服务。

但是不要被骗了。大三可能觉得自己能搞定。但是需要多年的经验，才能变得足够精通和明智，才能成功执行。

# 服务边界

在排舞中，人们排成一排，叫做墙。舞者必须待在他们的墙内，否则我们会失去凝聚力。

如果发生这种情况，你会得到高耦合。在一个有许多脆弱的活动部分的舞蹈中——这是不好的。它成为被推倒的一长串多米诺骨牌中的第一个...

# 复合 UI

一个排舞其实可能有很多墙！

许多墙壁组合成一个整体，创造了一个复合材料，一起看起来和工作更漂亮，比一个巨大的同质和单片墙。

# 服务发现

当加入一个现有的排舞时，你走到一个现有的队伍的末端，成为舞蹈的一部分。这就是我们的舞者被发现的时候。

他必须迅速确定他发现谁在他周围，这样他就不会造成任何失败。

如果他真的导致了失败，有经验的舞者会有足够的弹性回落到之前的状态。

# 最终一致性

舞蹈演员负责确保他/她开始与其他舞蹈演员合拍。

如果舞者错过了任何步骤，这通常是好的，因为舞蹈将获得最终的一致性。

# 消息传递

跳舞时，舞者不得身体接触对方，但必须倾听音乐和节拍中某些公开的暗示。

使用异步消息可以确保跳舞者在需要的时候可以有效地继续跳舞，同时指导其他跳舞者。

# 服务自主

舞者是自主的，他们可以随时离开或加入舞蹈。如果他们希望升级他们的鞋子，稍后再回来——很像服务部署——没问题。

此外，舞者可以灵活地拥有自己的个人风格。就像服务如何选择自己要使用的技术一样。

# 上限定理

有些舞蹈，像职业联赛，必须保证整个舞蹈的连贯性。如果有任何失败，那么舞蹈将被扣分，并被认为是一个糟糕的执行。

另一方面，更多的非正式舞会仅仅着眼于有一群可用的舞者。如果有任何失败，我们总是可以用其他人取代一个舞者，并重新发现他们。

# 可扩展性

排舞可以变得很大。你想要多大就有多大，真的。

如果音乐系统中出现了新的音乐，每一面墙或每一排都可以缩放，以便让新的人加入，如果他们觉得他们需要一些额外的活力。

# 健康检查

在开始跳舞之前，进行健康检查是很重要的。如果任何舞者没有通过这个测试，那么他们最好确保走出下一个。

我们不希望任何老年人受伤。

# 遗留问题

排舞很难。许多老年人对承担这样一项任务时所面临的麻烦保持警惕。

下次你想组织排舞的时候要小心。

这可能会给你的长辈留下不好的遗产。尤其是健康检查失败的时候。
