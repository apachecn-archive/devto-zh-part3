# 写下来就好

> 原文:[https://dev.to/gmeluski/just-write-it-down-5n8](https://dev.to/gmeluski/just-write-it-down-5n8)

*本帖最初发表于[meluski.com](https://meluski.com/engineering/2018/12/27/just-write-it-down.html)T3】*

如果你想知道是什么让我产生了制作单页产品推介的想法，我会从痛苦开始。更具体地说，这归结为选择我想忍受哪种形式的痛苦。我是想要错过需求的灼热、强烈的痛苦，赌我们在讨论项目需求的过程中不知何故达到了所有的角度，还是想要退出我的日常工作并进行另一个准备步骤的模糊、怪异的痛苦？

我们很容易向前冲，冒险认为我们已经想通了一切，却没有记录下我们的思考过程。也许不是当前的项目有问题，也许是下一个或再下一个。当有人说“嗯，等一下，为什么我们没有做[在这里插入全新的想法]？”时，团队可能正在琢磨发布。

这看起来像是对航运的接近向项目中无关紧要的参与者发出了一个信号，他们感兴趣但很忙，部分投资。由于他们的距离，他们有不同的视角。这可能会让人感到痛苦。

但是你需要这样的视角，因为产品概念化的观点越多，产品的效果就越好。而痛苦的是在个人、精神、情感甚至身体层面上所花费的(没错，弓着背敲键盘是一种身体上的投资)。沉没成本谬误？当然，但是对已经建成的东西有真正的依恋。

必须重新配置这一特定功能如何工作的心理模型，这是一件非常痛苦的事情。它需要时间去遗忘，将新的信息整合到现有的设计中。

所以这种模糊的，奇怪的做一些新的和不同的事情的痛苦？突然间，这成了一种可以接受的交易。

这种痛苦实际上是我自私的表现，我相信我目前的工作流程是可以接受的。我把事情写在便笺簿上，这是鸡毛蒜皮的事。我编写单元测试，但这不容易传播给非开发人员。我已经和我的团队联系过了，但是那些都是口头的和短暂的——但是任何错过会议的人都错过了会议中包含的信息。

天啊，当谈到最基本、最本质、最容易理解的交流和解释形式时，我的思想、身体和灵魂都在抗拒。

为了缩小产品文档的概念和实际实现之间的差距，还有一些额外的障碍:

*   如果我要写一个产品文档，它放在哪里？
*   人们会读这个吗？(也就是说，我是在向虚空呼喊吗？如果是的话，我为什么要浪费时间？)
*   这是否有某种结构，是什么？

首先，我喜欢在本地和私下作曲。我不知道这是什么，但有一种感觉，当我把我最初的想法倾倒出来并组织它们时，我真的不想感觉有人可以从我的肩膀上看过去，无论是数字还是其他方式。所以我喜欢至少从一个本地存储的(或者个人的，基于云的)版本开始，我可以用它来精炼和将来参考。

我确实不知道如何让公众阅读这份文件。内联或附加产品文档的电子邮件可以用作线索来跟踪对产品的评论，但如果人们需要稍后再联系的话，这就太麻烦了。一个共享的谷歌文档将解决这个问题，并允许人们以更有组织的方式进行在线评论。就我个人而言，我觉得任何能够促进流通和有组织的对话的工具都会提供一个可以接受的结果。

我收到了很多东西链接和电子邮件，我快速浏览了一下，然后就忘记了。也许这是一个信噪比的问题——在任何规模的公司里都有大量的信息，有时决定什么是关键的，什么是可删除的是一个仓促的判断。但是肯定有我关心的事情，尤其是影响我自己工作的事情。当谈到让人们了解信息而又不使他们不知所措时，有很多东西需要探索，但在这种情况下，我做了手工工作，在文档中制作了一个`People`部分，并仔细考虑了谁可能真正关心这个项目。

在继续之前，我想把这个想法加入进来——编写文档本身对作者来说是信息丰富的，即使没有其他人阅读它。

对于这种性质的文档，拥有一个模板化的结构可以使下次开始时更容易。在浏览了一些保存在口袋里的文章，并看了一些我以前(失败的)尝试后，我找到了下面的标题。这个列表很宽泛，主题之间有潜在的重叠，但在我看来，在我写作的时候删除和合并章节比创建一个章节标题并重新组织更有效。希望这份列表能帮助那些在产品报道中磕磕绊绊的人:

```
Problem Definition
Wider Context
Goals
Scariest Assumptions
Unknowns / Issues
How Will This Be Tested?
Is There Effort / Approval Required From Other Teams?
How Will We Measure Success?
People 
```

对我来说，确定这是否是一种有效的组织实践还为时过早，尤其是因为我最初是在解决最近的技术重组带来的孤立问题。

时间会证明一切？目前，我很高兴收获随之而来的自我组织的好处。
