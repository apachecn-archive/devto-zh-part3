# 纯与不纯 FP

> 原文:[https://dev.to/drbearhands/pure-vs-impure-fp-2e5e](https://dev.to/drbearhands/pure-vs-impure-fp-2e5e)

当我第一次开始写函数式编程及其各种好处时，我查阅了维基百科，以确保我对“函数式”有正确的定义:声明性的(即没有副作用)，并基于数学函数的评估。“Function *style* ”反而被用来表示作为一等公民所拥有的功能。

然而,[引起了我的注意](https://dev.to/didibus/comment/8ldn),在学术界，缺乏副作用并不总是被认为是称一种语言为“功能性”的必要条件。

不管哪个定义是“正确的”，我认为最好是明确的，以避免未来的误解。

因此，区分*纯*功能编程和*不纯*功能编程变得很重要。

纯函数式编程要求没有副作用。纯语言一般都是类型化的。这是有意义的，因为当一个字符串被传递给它时，`λa.a/7`会抛出一个错误，因为字符串不能被一个数整除。这将打破库里-霍华德-兰贝克对应关系，意味着人们将失去许多有趣的属性。
纯函数式编程语言有 Haskell 和 Elm。

不纯粹的函数式编程有副作用也很好。这可以用很多语言来完成:JS，Clojure，Lisp，C，Scala 等等。

到目前为止，我所有的写作都严格地讲是关于纯粹的函数式编程，不能应用于不纯的语言，因为它们不具备库里-霍华德-兰贝克对应的必要属性。

纯度的特性是否能弥补避免副作用的努力是有争议的。我自己坚定地认为，是的，纯洁解决了如此多的问题，它是绝对值得的。另一方面，像显示输出或读取用户输入这样的事情都是效果，所以在纯粹的功能世界中，人们总是需要某种方式来表示效果。

纯洁性在实践中的优势，以及如何表现效果，都是以后文章的主题。
