# 谁是高级开发人员？

回顾过去的一篇文章，高级开发人员实际上是指:

*   能够胜任其公司使用的核心技术
*   通常有一些专门化
*   处理高层次的架构问题和设计
*   应该有几年的经验(这是你需要通过学校的磨练获得的)
*   有构建“完整”现实世界解决方案的经验
*   积极指导和训练团队中的其他开发人员

最后一点，我要说，是一个人是高级开发者的最好指标。

如果你通过以下方式证明了自己是值得信赖的:

*   展示你能解决棘手的技术问题
*   证明自己是一个优秀的沟通者
*   拥有现实世界的经验，这种经验会影响你的许多决策

然后，你的同事会视你为领导和教练。

# 成为成年人？

我的关于[成为中级开发人员的文章](https://dev.to/jamesmh/becoming-an-intermediate-developer-mh9)将中级开发人员比作进入青春期的人——但是是为了他们的职业生涯。

高级开发人员就像一个已经长大成人的人，可以照顾自己了。他们的生活没有那么飘忽不定、自发和实验性。

这是因为，在许多情况下，他们已经“在那里做了那件事”

他们已经从错误中吸取了教训。他们已经做了许多实验，可以说已经为他们的生活打下了坚实的基础。

当你成为一个成年人时，生活就不再那么美好了。但是你确实更了解这个世界，而且你明白在你的视角之外还有一个巨大的世界。

只要把这个类比放在软件开发人员的上面，你就会感觉到一个高级开发人员是什么样的😋。

高级开发人员在解决问题的方法上更加谨慎、周到、务实、实用和简单。

他们学会了什么可行，什么不可行，什么可能太抽象，不够实用，有用，有效，等等。

高年级学生对他们根据自己的经历所知道的事情充满信心。但是他们也明白有太多他们不知道的事实。

# 广度和深度

你可能听说过高级开发人员的技能应该是“T”型的。

这意味着他们对许多主题(T 的顶部水平部分)有了基本的了解。但他们在一个或多个领域拥有非常深入的知识和技能。

> 高级开发人员对什么工具和技术可以解决特定的问题有着广泛的了解。

## 广度

例如，我从未构建过生产就绪的分布式[微服务](https://www.martinfowler.com/microservices/#what)系统。但是我知道这种架构方法解决了什么问题，并且我知道开始考虑如何构建它们的不同方法。

我从未在生产就绪的应用程序中使用过事件源。但我知道它能解决什么样的问题。

## 深度

在“T”的垂直部分，应该有一些高级开发人员非常熟练和知识渊博的特定领域。很多时候，这种专长与他们的特定角色直接相关。

> 应该有一些高级开发人员非常熟练和知识渊博的特定领域。

不管怎样，一个高级开发人员是某方面的专家，并且在其他领域也在不断成长！

这可能是特定的领域，如系统架构、特定的编程语言、编程范例(OOP、函数等。)，某一套技术，安全，性能，甚至只是对某个特定行业有广泛的了解！

# 禅

作为初级开发人员，您仍在学习如何构建软件。我想到的是:

*   什么模式可用/何时以及如何应用它们
*   如何处理交给你的业务需求
*   如何评估你的工作
*   如何将各种系统连接在一起
*   等等。

高级开发人员在构建解决方案、管理复杂性、处理混乱的业务需求、应用设计模式等方面积累了经验。因为他们已经做了很多次，一遍又一遍，他们可以“用心”解决许多常见的问题

高级开发人员可以看到初级或中级开发人员面临的问题，看到该问题中涉及的模式，并且他们知道——基于他们已经习惯的模式——应该发现或实现解决方案的一般方式。

> 高级开发人员使用通用模式来发现通向合适解决方案的一般路径。

如果你想成为一名高级开发人员，这种“禅”真的只能来自于真实世界的经历，你从失败、成功、导师等等中学到了东西。

你只需要大量的练习，需要做这些事情很多次，让它们印在你的大脑里！

当然，需要多长时间取决于强度、种类等。你在工作期间所做的工作。

顺便提一下:如果你的工作没有挑战性，那么你很可能不会快速成长。

# 举个例子

以我自己的旅程为例，我的第一份工作是作为一个团队的一员，该团队正在构建一个新的软件系统，该系统以一种新的方式处理汽车制造商的财务。

我很快就不得不处理一个非常大的代码库，在前端、后端、数据库编程以及两者之间的一切方面都有所涉猎！

在成长了几年后，我被要求帮助保险公司建立一个新的白标平台，允许人们使用网络申请人寿保险。

负责建立这个项目的整个网络部分真的让我成长了。我学到了很多！包括不要做什么😋。

这个项目有非常复杂的业务逻辑(已经被开发并移植到这个项目中)。这很难理解和管理。这促使我寻找如何在软件中管理业务逻辑的模式和工具。

我花了很多时间(下班后也是！)学习像[设计模式](https://sourcemaking.com/)，架构风格和[领域驱动设计](https://en.wikipedia.org/wiki/Domain-driven_design)这样的主题。

一旦我开始在自己的代码中使用这些技术、模式和原则，其他开发人员就会来找我并开始询问它们！

因为我花时间投资学习这些模式和技术，今天我不断地帮助我的团队解决这些类型的架构和代码组织问题。

# 一些提示

你能从这一切中得到什么？下面是一些成为高级开发人员的技巧，可以从我们所学的内容中收集:

*   找出你真正想精通的技术，并深入学习
*   天天读！(这将帮助你成为一个更好的读者，并帮助你每天学习)
*   试着接受那些提供给你的真正困难但有价值的项目
*   记住，经验是随着时间而来的——没有别的办法！
*   还记得“T”形技能集的概念吗
*   规划你的技能，弄清楚哪些是你真正想钻研的，哪些是你想学习的基础知识
*   对于你想知道基本原理的技能，最重要的是理解它解决了什么问题？
*   对于你想挖掘的技能，考虑导师或高质量的培训材料(想想 [Pluralsight](https://www.pluralsight.com/) )
*   大多数领域都会从了解设计模式中受益(同样，最重要的是了解每个模式解决什么问题)
*   如果你目前的工作没有挑战性，考虑换一份更有挑战性的工作

你可能会注意到其中一个共同的主题:**学习技巧和模式主要是关于知道什么问题有解决方案。**

熟练的高级开发人员可以指出具体的问题，记住什么技术或模式可以解决这些问题，并在需要时随时学习或重新学习如何即时实现这些解决方案！
