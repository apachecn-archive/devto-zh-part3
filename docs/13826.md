# 任何项目的快速有用笔记

> 原文：<https://dev.to/alexflorea2/quick-useful-notes-for-any-project-5apa>

我发现这些笔记适用于任何项目环境，小到个人项目，大到大型企业项目。

排名不分先后:

*   **根据预定义的模板记录用户操作**，例如用户在日期对实体/实体详细信息执行了操作。在我们的示例上下文中，它可能看起来像:

    `Mark deleted_ticket id 257 at 3/29/2019`

*   尽可能使用软删除。删除一个*_ at*
    信息比从数据库中实际删除条目更有用。即使你做了常规的数据库备份，总有一天有人会删除一些重要的东西，尽管你在应用程序中有许多警告和确认弹出窗口，你也不会(通常)浪费大量空间留下条目，而是标记为已删除。ORM 通常有这个特性，并且不会在其他查询中返回删除的条目。

*   有一个快速的方法可以在现场调试。框架通常会发布一个更升级的 **var_dump** 替代品，比如 Symfony 的 **dump($var)** 。这一切都很好，但是您可以通过使它只在特定条件下工作来增强它，甚至是简单的 var_dump，比如如果 IP=x 或者有一个`debug` Cookie。您不希望在半夜接到支持人员的电话时(假设问题)所有用户都看到所有其他用户的票证，您需要修复它，在活动页面上显示随机调试字符串。

*   用分页显示大量数据？请记住，MySql(我相信一般的 Sql)没有默认的排序顺序。在一个足够大的数据集上，当从一个页面移动到另一个页面时，可以得到 ID 为 20.000 的条目。id DESC 的一个快速**订单就足以让你放心。在没有任何先前删除/替换/更新的表上*大多数时候*虽然 SELECT 会给出插入顺序的结果。**

*   **项目意外增长，需求会变然后又变回来**，哈哈。这很正常，至少从我的经验来看是这样。有一种方法来跟踪一切是必不可少的，像 Trello，Asana，Active Collab 等产品真的很有用。但是尽量准备一个 **readme.txt 文件，里面有一些关于项目设计的重要评论**，有用的评论，从开发到生产时需要注意的事情等等。当有人从 git 中提取代码时，该文件将使 onboarding 变得更加简单，但从长远来看，当您需要添加或更新功能 x 时，它应该对您自己有用。提示:将它保存在 *markdown* 中，IDE 的 PhpStorm 等将正确地呈现它，以及 Gitlab/Github 和开发中使用的许多其他工具。
