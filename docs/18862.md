# [](#a-world-of-dialects)方言的世界

多年来，人类设法分散在地球上不同的地方，形成社区，发展出自己独特的语言作为交流的手段。其中一些语言甚至演变成不同的方言，听起来可能非常不同，这就是为什么说美国英语的人可能很难理解澳大利亚口音。在软件开发的世界里，我们可以说同样的情况也适用；在类似领域中工作的不同企业可能在它们之间使用不同的语言，例如在保健领域中，一个企业可能使用“医疗从业者”来描述他们的服务提供商，而另一个企业可能简单地使用术语“医生”。没有规则管理一个术语相对于另一个术语的选择，但是组织必须在整个团队中积极地执行自己的语言——无处不在的语言，并且不允许外来术语泄露到团队中。

在大型组织中，在不同的环境中工作的团队使用同一种语言的各种方言是很正常的，例如，在一个虚构的电子商务业务中，采购团队可能使用术语“订单”来表示客户从网站的库存中购买一组产品的请求，而运输团队将使用相同的术语“订单”来表示发送给其中一个快递员的请求，该请求将货物从一个区域的仓库运送到另一个区域。只要每个团队一致同意他们自己对“顺序”的定义，这就没问题。

当单个团队中的一部分人开始使用自己的行话与他人交流时(一个突出的例子是开发人员和业务专家在会议中使用不同的术语)，就会出现混乱，需要懂两种方言的翻译来弥合团队成员之间的差距，使团队面临误译的风险，以及由于新引入的瓶颈而导致的不必要的延迟和低效率。而且，从技术的角度来看，极有可能遇到这样疯狂的代码:(假设一个保留上下文)

```
class ReservationService
{
    void CreateReservation()
    {
       Appointment appointment = _appointmentFactory.CreateAppointment(...);
       //It is a Reservation, but Randy likes to think of it more as an Appointment ¯\_(ツ)_/¯
    }
} 
```

语言中的矛盾和歧义为团队提供了一个机会来学习更多关于领域的知识并建立一个更深层次的模型。团队成员应该非常警惕这种模糊性，并尽快解决它们，我最近在为电子商务应用程序编写产品目录时遇到的一个特殊情况是，我们互换使用术语“供应商”和“品牌”来表示提供我们产品的实体。该团队了解拥有统一语言的重要性，因此我们讨论了这一困惑，在这一困惑中，有充分的理由支持使用任一术语，最后决定在产品目录的上下文中使用术语“品牌”,直到出现进一步的见解，从而保证返回到“供应商”或使用一个全新的术语。

# [](#polysemous-integration)多义融合

一切都很好，直到团队之间的整合开始隐约出现。前面我们展示了一个“订单”多义词在购买和运输中不同用法的例子。那么当这两个团队开始整合过程时，应该如何管理矛盾呢？运输团队对“订单”的定义是否比采购团队的定义更有效(反之亦然)？肯定不是，毕竟——概念上——两者都是订单。管理层是否应该试图强迫两个团队完全放弃术语“订单”而采用“采购订单”和“装运订单”来代替？这不是一个好主意，而且可能无论如何都行不通，因为从采购团队的角度来看,“订单”是一个在他们的领域中代表单一明确概念的术语，并且因为他们可能不依赖于送货快递员的概念，因此他们为什么需要通过在“订单”前面加上“采购”这个词来区分他们的“订单”呢？

如果集成团队不得不共享他们的模型的一部分，情况会变得更加复杂，这需要语言上的突破，以提供更成熟的词汇表，方便地服务于两个团队。然而，如果不是这样，那么每个团队最好在其上下文中保持统一的词汇，而负责集成过程的团队成员应该完善不同的方言以及如何在它们之间进行翻译。

# [](#beware-the-false-cognates)谨防假同源词

维基百科对假同源词的定义如下:

> 假同源词是由于相似的声音和意义而看起来是同源的词对，但具有不同的词源。

一个错误同源词的例子是西班牙语单词“Ropa ”,它很容易与英语单词“Rope”混淆，然而“Ropa”在西班牙语中的意思是“衣服”,而英语单词“Rope”的准确西班牙语对应词是“Cuerda”。

一个特别危险的情况是，当一个软件团队的不同成员使用同一个术语时，他们认为它意味着相同的意思，但在后来的阶段发现事实并非如此。例如，在拼车应用程序中，团队使用术语“旅行”来表示由他们自己的司机操作的常规拼车。因此，当他们开始构建一个新的特性来支持出租车/出租车司机操作的乘坐时，业务分析师仍然对出租车乘坐使用相同的术语“Trip ”,这导致开发团队错误地假设他们可以在新的特性中重用他们的“Trip”模型，但是，由于团队没有持续集成的过程，所以在生产后发现“Trip”模型具有微妙的假设，这使得它不适合在新的特性中重用。哎哟...

# [](#conclusion)结论

当谈到无处不在的语言时，你可以尽可能严格地保持它的完整性。坚持不懈地练习这种语言，并时刻注意任何混淆的迹象。
