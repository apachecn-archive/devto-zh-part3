# 软件开发:有些事情永远不会改变

> 原文:[https://dev . to/check git/software-development-some-things-never-change-1lk 5](https://dev.to/checkgit/software-development-some-things-never-change-1lk5)

作为一个领域，软件开发进展很快。新技术来来去去，方法变得流行然后衰落，风格和审美偏好改变，用户的期望和技术专家的交付一样快。

这种背景可能会产生误导。有时候，事情不会改变。偶尔抓住机会反思我们是如何走到今天这一步的，这很重要。这里有一些通用的软件开发实践，在基本层面上，这些年来一直保持不变:

## [](#code-reuse)代码重用

甚至追溯到 20 世纪 60 年代，像施乐这样的早期技术公司的开发人员都在推动增加抽象和模块化，这使我们能够开发我们今天拥有的大规模软件系统。这种推动最终导致了像 C 这样的高阶机器语言，然后是像 Java 这样的字节码语言，以及像 Python 这样的动态解释语言。这些语言允许开发人员在实现细节(如内存管理和数据结构)上更少的麻烦，产生更多的功能。

主要的编程范例在这里也发挥了作用。结构化编程、面向对象编程和函数式编程都可以追溯到 20 世纪 60 年代。结构化编程通过控制流(“if”语句和“while”循环)促进逻辑的重用。面向对象将数据和算法模块化并封装在易于理解、可互换的接口下。函数式编程通过确定性行为和最小化副作用来促进重用——如果您知道可以丢弃结果中任何不需要的部分，那么重用代码是安全的。

当前的趋势似乎倾向于多语言环境——根据对领域的适应性，将软件系统分成用不同语言和范例编写的模块，但是通过标准协议可以互操作。开源的兴起也是代码重用的一个很好的例子，在这种情况下，基本的基础技术可以由更大的社区来构建和共享，从而允许项目更充分地关注它们的差异化价值。

## [](#design-amp-architecture)设计&建筑

《人月神话》的作者 Fred Brooks 曾说过，他从 OS/360 项目的经历中学到的最难的一课是一致的架构愿景的重要性。最初缺少一个，团队发现代码库变得太难理解和推理。

并不是每个项目都像 OS/360 那么大，但是基本的重要性仍然适用。一般来说，在开始编码之前，最好对一个软件有一个清晰一致的愿景。它不一定是一个会产生大量文档的正式过程，但是每一个变更都应该从理解它在更大系统中的一般范围和目的开始。

就我个人而言，在开始任何实施之前，我会努力记住坚实的原则。它不必是教条式的，因为工程都是关于上下文和权衡的，但是为了理解、分析和讨论设计决策，使用一种通用语言是很重要的。这对于长期保持质量也有很大的帮助，特别是对于大型代码库和分布式团队。

## [](#source-control)源代码控制

早在 20 世纪 70 年代早期，贝尔实验室就使用 SCCS 来管理他们的软件修订历史。适当的版本控制对于管理复杂变更的集成是必不可少的——同样，对于大型代码库和分布式团队来说尤其如此。它在质量控制中也起着重要的作用，允许开发人员将第一次报告的问题的出现锁定到版本控制中的一般时间范围，然后可以对其进行检查以帮助调试。

如今，Git 已经成为默认的版本控制系统。它的分布式模型使得为定制的变更派生现有的项目变得容易，并且为贡献者提供了一个强大而方便的机制来请求将他们的变更合并回主分支以便正式发布。无论您的团队使用什么样的源代码控制系统，都要确保您有一个合理的方法来跟踪您的系统是如何、何时以及为什么随着时间而变化的。

## [](#testing-amp-quality-assurance)检测&质量保证

传统上，QA 过程是从现有的工程学科借用的，这些学科最初是着眼于物理系统来开发它们的。大型组织会产生详细的文档，概述具体的手工测试过程，这些过程由训练有素的专门测试人员团队执行。

今天，有一个明确的趋势是尽量减少人工 QA 活动，特别是随着开发人员驱动的自动化测试的兴起。人工 QA 还是有一席之地的。首先，由除作者之外的人来测试软件是很重要的。其次，让一个非开发人员坐下来讨论软件是件好事——对熟悉项目的软件开发人员来说直观的东西对最终用户来说不一定直观。

自动化测试通常是很好的实践，但是仅限于开发和维护测试实际上对最终用户的价值有所贡献的程度，并且不能更有效地手工执行(如果不那么频繁的话)。无论哪种方式，测试都是并将继续是该领域的基本实践。

## [](#talking-to-users)与用户交谈

由于各种原因，软件开发人员有忘记这一点的倾向。鉴于它的特殊重要性，这很不幸。最终，软件最终到了最终用户的手里，他们通常用它来做软件开发以外的事情。如果你不和你的用户交流，你就不知道你在试图构建什么。

在软件开发的早期，用户是在内部工作的专家，这使得交流相对容易。随着 20 世纪 80 年代微型计算机和大众市场软件的出现，大公司发展了客户关系专家团队，以使程序员了解最终用户的需求。

今天，随着敏捷流程及其衍生产品的主导地位，开发人员直接与最终用户对话是很常见的。理想情况下，应用程序有多个内置反馈机制，社交平台可供社区围绕特定技术形成。无论你怎么做，你都应该和你的用户保持联系。

## [](#and-so)如此这般...

软件是一种特殊的动物。它的极度灵活性、易于分配和接近零的边际成本使得构建和部署如此复杂的系统成为可能，以至于没有一个人能够完全理解它们。从实践的最初几年开始，这就已经是一个确定的问题——软件开发中的限制归结为对复杂性的智能管理，以及将组件抽象到个人贡献者可以理解和推理的规模。

所有这些实践——代码重用、设计和架构、源代码控制、测试和质量保证，以及定期与最终用户交谈——对于生产良好的、可工作的软件是必不可少的。它们在软件开发的初期就已经存在，如果允许我推测的话，只要人们还在编写软件，它们就会继续被实践。
