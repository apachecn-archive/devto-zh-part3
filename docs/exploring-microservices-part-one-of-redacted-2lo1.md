# 探索微服务，第一部分

> 原文：<https://dev.to/gmeluski/exploring-microservices-part-one-of-redacted-2lo1>

*本帖最初发表于[meluski.com](https://meluski.com/engineering/reading-books/2019/02/05/exploring-microservices.html)T3】*

元工作很有趣，但它是编程的空卡路里。

在我的职业生涯中，我花了一生的时间(也就是六年)专注于 Javascript，当我意识到我已经掌握的后端架构知识正在以越来越快的速度流失时，我感到震惊。在早期，出于需要，我负责整个网站的工作，随后设法在开发运营革命中睡着了。缩放？当专注于客户工作时，如此多的扩展问题可分配给模糊的后端团队，不管他们是属于第三方还是你自己的公司。

但这不是一个好工程师的行事方式。一个好的工程师理解他们所依赖的东西，但他们不会每天都做贡献。所以这可能使我成为一个有点糟糕的工程师。一个糟糕，糟糕，无知的工程师。

这种认识让我鲁莽而惊慌地潜入水中，当我的盔甲出现缺口时，我就会这样做。我想起来这是一个愚蠢的反应——冲动和贪婪。我确实喜欢学习，所以这是好的一面。尽管如此，我认为我需要对不知道的事情感到更舒服，我不能一辈子都埋在书里...我可以吗？

这让我看了几本书，一本叫做[构建进化架构](https://www.goodreads.com/book/show/35755822-building-evolutionary-architectures)，另一本叫做[构建微服务](https://www.goodreads.com/book/show/22512931-building-microservices)。好的、高水平的书籍，关注技术背后的概念，这些概念往往从我们身边呼啸而过，如果你抽出一点时间，你很可能会被无意义的技术术语淹没。

当我花时间退后一步，认真思考事情的时候，我发现自己渴望了解整个技术——不仅仅是如何让它工作，而是让它工作的理论。回想起来，搜索这些书更符合我想要成为的人，而不是在第一个变更集就崩溃的不稳定的实现中挥汗如雨、跌跌撞撞。

两本书都在某些地方引用了康威定律，老实说，这可能是让我们远离过去十年已经习以为常的巨石的驱动力。整体的命令和控制动态将永远在架构设计中占有一席之地，但就像 HTML `<table>`一样，它们正在从公认的标准演变成有意识选择的用例。正如尼尔·福特在*构建进化架构*中警告的，“确保你的架构匹配问题域。”

因此，现在我们已经获得了我们在日常生活和工作场所中看到的更大程度的灵活性和活力。作为技术专家，迎接这一挑战似乎是关键。对我自己来说，这意味着将理论转化为行动，这就是为什么我一看到实际应用的机会就拿出一个 EC2 实例。Web 服务器是面向服务的体系结构菜单中的汉堡包——坚定不移，众所周知，尽管并不总是被理解。每个人都提供一个服务器或主机包，对不对？

已经有一段时间了，我希望做一些事情，而不仅仅是打开和关闭服务器(在撰写本文时，我已经安装了 Nginx，所以期待更多的内容)。作为一个与客户共事了很长时间的人，这是一个在 AWS 领域造成一些破坏的难得机会。几年前，我与 Lambda 有过几次罕见的亲密接触，但随着公司的发展，我发现自己越来越脱离基本功能，而这些功能需要高超的服务能力。

顺便提一句，最近有一段时间，我把一个 S3 存储桶当作我笔记本电脑上的挂载驱动器，从而产生了巨额数据传输费用。这么说吧，这些天我查看 AWS 仪表盘的账单部分的频率要高得多。
