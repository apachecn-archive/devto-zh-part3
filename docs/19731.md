# 概念证明？MVP？完成后告诉我！

> 原文：<https://dev.to/pavsaund/proof-of-concept-mvp-just-tell-me-when-its-done-3k14>

[![](img/3b8a88ef3ea976148429a6c738e94951.png)](https://cdn-images-1.medium.com/max/1024/1*BsllTh66nrleARYfwQ3-PA.png)

迭代产品开发和一个习惯于通过项目透镜来处理产品开发的公司之间的会议可能是相当苛刻的。

一方面，你设定了预算和收入目标。项目交付已经为未来几个月、几个季度甚至几年制定了详细的计划，并为今年剩下的时间制定了一系列依赖关系。

另一方面，您需要以一种真正敏捷的方式跟踪交付并做出影响范围和最终产品的日常决策。

一直以来，没有人能回答你是否能达到目标。而你，作为一个利益相关者或项目所有者，只是想知道**什么时候会完成**！

#### 事情是这样的…

当进入这个数字化组织和迭代产品开发的世界时，有一些事情需要注意。

*   Agile™️并不意味着更快或更便宜。
*   你决定什么时候结束。
*   产品开发很难。

如果你愿意相信我的话，那么你现在可以停止阅读。否则，请继续…

#### 敏捷/开发/精益并不意味着更快、更便宜或更容易

数字化已经作为“必由之路”,以及某种形式的敏捷软件开发的概念，被各种组织所接受。言下之意(或直接承诺)是，这种形式的发展将更快、更便宜，或者两者兼而有之。对此没有任何保证。

相反，这些实践意味着您能够以更加迭代的方式测试您的假设。这意味着您可以创建足够的产品，让客户在下一次迭代之前使用它并给你反馈。因此，从这个意义上来说，验证你是否在正确的道路上要便宜得多，并且纠正路线的成本比这是一个持续几个月或几年的项目要低。

这种工作方式对你的时间要求更高，因为你现在是构建和发现产品的团队的一员，你需要在一个更特别的时间表上。有更多的小型跟进会议，并根据客户反馈或新的机会不断改变优先事项。像这样工作很吸引人，很有趣，但也很累人。尤其是如果你习惯于通过更大的“瀑布”项目来更传统地“订购”软件。

#### 产品开发很辛苦

构建软件产品最困难的部分是知道要构建什么。你可能知道自己想要创造什么，但这真的能满足人们的需求吗？说你需要一个应用或服务并不能立刻让它成为人们想要使用的东西。找到适合市场的唯一方法是用真实的顾客和用户来测试它。这就是迭代开发方法所提供的——在预算(金钱、时间、信誉等)用完之前测试假设的机会。

企业和内部客户为了完成工作而接受劣质产品的日子已经一去不复返了。消费者习惯于在网络和他们的设备上使用时尚、直观的应用程序。没有什么神奇的开关能让这些人在进入工作环境时降低他们的期望值。

这种对质量的期望对想要提高标准的企业来说是一个挑战。随着产品质量的提高，构建正确的东西的需求也在提高。验证你试图解决的问题是否是使用你的产品的人实际感受到的，并且想要解决的，这是至关重要的。

要跟上这一步伐，意味着要把更多的精力放在产品开发上，把它作为核心学科，而不是外包或远程管理。

#### 你决定什么时候做。

肮脏的秘密是“它”从未真正完成——难以捉摸的产品仍然是一个移动的目标。把东西拿出来只是第一步。相反，重点应该是:

*   什么时候产品好到对用户有价值？
*   你愿意花多少钱来发现你离满足这些需求有多近？
*   你怎么知道产品实际上是在满足需求和传递价值呢？

通过迭代过程，您有机会回答这些问题，并通过改变产品的方向来做出回应。但是我发现获得真实反馈的唯一方法是在真实用户面前展示完整的功能。

草图、原型、概念证明和最小可行产品都是获得反馈的可行方法——越早越好。选择你的口味，严格要求范围，并得到一个可行的产品！

#### 终于

组织对固定预算、范围和路线图的需求会阻碍你在真正迭代的产品开发周期中实现诱人的价值承诺。这种价值就是能够响应不断变化的需求和新机遇的敏捷性。

通过持续的反馈周期来接受实验，并根据它们来调整你在产品中所处的位置。基于这些反馈周期，你会知道你什么时候完成了。
