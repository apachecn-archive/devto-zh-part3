# 周五爆炸#60

[你的 AI 技能比你想象的更不值钱(2019)](https://www.kdnuggets.com/2019/01/your-ai-skills-worth-less-than-you-think.html)——有趣的看*数据科学/机器学习/AI 技能*的价值。这并不奇怪，即使对于那些对过去几年人工智能的发展略知一二的人来说，技术水平已经有了很大的发展。无论是在模型上，还是在工具上。因此，在 2010 年可能需要一个专门团队花一年时间才能完成的事情，在 2019 年可以由一名实习生在业余时间完成。因此，了解并擅长人工智能的含义正在发生变化——就像构建软件一样。过去，编写计算机程序需要博士学位和专业知识。这也影响了公司如何使用 AI。回到这篇文章的主题，如果你的公司只是关于人工智能的，你很可能处于危险的境地。最好的设置是当你正在做一些需要做的事情时，但是你可以使用人工智能作为*杠杆*来使它更快/更好/更便宜等等。尤其是当人工智能工具似乎集中在少数大公司手中，而不是像软件公司那样更均匀地分布。

[原型制作更平滑的地图(2018)](https://medium.com/google-design/google-maps-cb0326d165f5) -这篇文章主要是关于谷歌地图的内部工作，这是一个真正大规模的软件项目。我仍然认为地图的网络客户端是*魔法*。也有一些关于使用 Canvas 来更平滑地滚动和缩放地图的讨论。但这似乎是过时的矢量方法目前在地图上使用。

[超越 web 和 worker——Heroku 上现代 web 应用的演变(2019)](https://dev.to/heroku/beyond-web-and-worker-evolution-of-the-modern-web-app-on-heroku-5c27)——互联网应用的*标准*架构概述。有趣的是，*架构*已经“降低”了复杂性。今天，甚至更小的项目可能会像这样建立，而昨天的谷歌、雅虎和亚马逊有类似的东西。我仍然认为我们如何建造这些东西不是一件好事，也许有一天我会写下更多的想法。

[Nuage -使数据系统管理可扩展(2018)](https://engineering.linkedin.com/blog/2018/08/nuage--making-data-systems-management-scalable) -数据源管理实际上是我在工作中涉猎的东西，所以这是一篇关于 LinkedIn 如何解决这个问题的好文章。编目和组织一个组织的所有数据，以及它是如何生成的，由谁生成的，等等。是一个*难题*。我不相信这里讨论的基于 UI 的解决方案是正确的*，但它至少是我们可以借鉴的方法——LinkedIn 必须有大量的数据。*

 *[Kolmogorov 复杂性和我们对意义的寻求(2018)](http://nautil.us/issue/63/horizons/kolmogorov-complexity-and-our-search-for-meaning)——Kolmogorov 或算法复杂性的外行介绍。虽然很难相处，但我一直认为这是最像我们人类的“随机”或“非随机”的概念。*
