# 10x 程序员:神话还是现实？

> 原文:[https://dev.to/ashrafalam/10x-programmers-myth-or-reality-4ge 5](https://dev.to/ashrafalam/10x-programmers-myth-or-reality--4ge5)

10x 程序员是当今软件行业讨论的一个热门话题，所以我认为分享我在这方面的观点是个好主意。

老实说，我不知道 10 倍程序员的确切定义，但我假设它指的是超高产的程序员(这可能不是精确的 10 倍生产力，而是接近或超过 10 倍的显著水平)。

基于我在为初创公司和公司交付超大型软件时取得的一些成功，以及与一些顶尖开发人员的工作经验，我可以分享一些关于这个主题的见解。

## [](#the-10x-programmer-then)当时的 10x 程序员

与现代软件相比，20-30 年前开发的软件在功能、安全性、性能和可伸缩性方面要简单得多。另一方面，当时只有少数几个库或框架可用于实现任何功能。

因此，20-30 年前编写的软件包含了大量的编程工作。

在这种情况下定义一个 10 倍的程序员是相对容易的。10 倍于普通程序员实现相同功能的核心编程工作量之间的差异。

## [](#the-10x-engineer-now)现在的 10x 工程师

在进入其他细节之前，让我们先弄清楚一件事:软件开发是一个复杂的过程，编程只是现代软件的一个组成部分。

由于消费者和组织对软件的使用越来越多，特别是在互联网上，软件变得非常复杂，从头开始构建软件可能是一个疯狂的决定。老实说，除非它是一个全新的平台或操作系统，否则开发现代软件所需的核心编程没有以前那么紧张。

虽然编程知识和努力仍然非常重要，但构建优秀软件(无错误、安全、可销售)所需的努力是在工程方面。

所以在今天的背景下，一个 10x 的程序员可能不会给行业带来太大的价值，但是一个 10x 的工程师在开发过程中创造了非常广泛的影响。

我想在这里补充我的 3 美分。

### [](#1-they-make-the-best-usage-of-tooling)1。他们充分利用了工具

软件开发是一个复杂的过程，涉及到许多不同种类的工作。除了编码，它还包括从记录过程到格式化数据，从读取日志数据到发送报告，从自动测试到手动测试，从复杂的调试到手动调查问题，等等。

通过使用一套正确的工具和平台(适合客户的预算和其他限制)，可以非常高效地完成大量工作。

#### [](#editors)编辑人员

编写软件涉及一个或多个编辑器，其中一些也称为 IDE(集成开发环境)。现代的 ide，比如 Visual Studio 或 Eclipse，提供了大量的功能来提高开发人员的工作效率，但是大多数开发人员并不了解这些功能。

流行的 IDE 也有商业和免费的插件(比如 Resharper ),这使得开发人员的工作效率更高。

除了 IDE 的，其他编辑器如 NotePad++，MarkdownPad 等。在相关的上下文中也非常有用。

#### [](#utilities-and-online-services)实用程序和在线服务

实用程序和在线服务，如读取或搜索大型日志文件，http 调试器如 Fiddler，构建和部署工具等。

#### [](#own-tools)自带工具

10x 工程师还制作自己的工具集来执行重复性操作，这些操作可能无法从外部获得合适的软件。

在很多情况下，软件开发公司的业务所有者似乎对投资开发人员的工具不感兴趣，但是使用正确的工具集将会带来惊人的生产力。

对于软件开发人员来说，雇佣他们是最昂贵的，为他们花钱购买合适的工具是值得的。

作为. net 开发团队的领导者，开发了一个[ASP.NET 核心& Visual Studio starter 模板](https://github.com/EISK/eisk.webapi)。net 编码实践帮助我的团队将整体生产率提高了三倍。

### [](#2-they-dont-reinvent-the-wheel)2。他们没有重新发明轮子

在过去的三十年里，软件业已经变得非常成熟。开发者试图通过编码解决的几乎每一个问题都已经得到了解决，并作为 API(二进制或 web 服务)可用，其中一些是商业的，而另一些是免费的。

此外，开源框架还提供了很大的灵活性来理解底层 API 行为或用户所需的定制。

在开始编程之前，10x 工程师实际上要确保这个问题以前没有被完全(或部分)解决，或者，如果是这样的话，他们不能被使用。

### [](#3-they-continuously-write-beautifully-engineered-code)3。他们(不断地)写出漂亮的工程代码

编写优秀的软件需要定义一个好的架构，这个架构不仅要遵循好的设计模式和原则，还要利用现代的基础设施。生成设计良好的代码不仅能够更快地编写新功能，还能显著减少错误。

但是，美丽的工程需要持续改进。

在我的职业生涯中，我在不同的软件项目中看到了数量惊人的低质量代码，这不断增加了技术债务。一个 5 年前开始的简单项目充满了非工程化的丑陋代码，需要 20 多人来处理开发周期(他们花费 80%的时间实际修复 bug)。

我发现，这种糟糕局面背后的一个主要原因是“恐惧”。

组织，尤其是企业实体，大多害怕变化。

就工具、框架和工程而言，软件开发是一个令人难以置信的快速变化的学科，并且需要持续的变化才能到达正确的位置。

变化是令人恐惧的，但它并不像听起来那么危险，尤其是在软件行业(使用正确的工具和人员)。但如果没有按预期完成，用不了多长时间，这个项目总有一天会被完全放弃。
